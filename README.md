# backend-labs

一个以后端系统演化为主线的学习型项目，用来连接概念、场景、取舍和代码实验。

## 为什么做这个项目

这个项目的目标不是堆积知识点，而是建立一套更清晰的后端认知地图。
重点不是单独理解某个框架或技术，而是理解：
- 为什么需要它
- 什么时候该用它
- 它解决的核心问题是什么
- 面对类似方案时应该怎么选择

## 学习方式

- 从一个很小的系统开始
- 随着压力增加逐步演化系统
- 先分析问题，再引入技术
- 以文字说明为主，以代码实验为辅
- 允许边做边调整，不追求一开始就设计完美

## 各部分作用

- `项目记忆.md`：记录项目长期记忆，包括目标、结构原则、当前决定和路线
- `学习路径.md`：记录建议学习顺序
- `docs/roadmap/`：记录系统从简单到复杂的演化主线
- `docs/topics/`：记录可从主线跳转的知识点文档
- `docs/resources/`：记录少量真正需要单独展开的深入资料
- `docs/文档索引.md`：作为文档总索引
- `labs/web-server-scaling/`：主实验项目，按阶段演示系统演化
- `labs/independent-demos/`：独立的小型实验，用来观察某个局部机制

## 四类内容的职责边界

### 1. 主线文档：`docs/roadmap/`
主线文档负责回答：
- 系统现在遇到了什么问题
- 为什么这个问题会在当前阶段出现
- 应该从哪些角度分析
- 有哪些可选方案
- 为什么当前先选择这个方案
- 引入方案后会增加什么复杂度

它的重点是**串联演化过程**，不是展开解释所有技术细节。
所以主线文档更像“剧情线”或“路线图”。

### 2. 知识点文档：`docs/topics/`
知识点文档负责回答：
- 这个技术或概念是什么
- 它解决的核心问题是什么
- 为什么会在主线的某一步被引入
- 什么时候适合使用
- 有哪些替代方案
- 主要代价和限制是什么

它的重点是**解释单个知识点本身**，不是承担整条系统演化叙事。
所以知识点文档更像“主线中的锚点说明”。

### 3. 补充资料：`docs/resources/`
补充资料负责回答：
- 某个具体实现细节应该怎么理解
- 参数应该如何看待
- 常见误区和踩坑是什么
- 某份外部资料值得提炼的重点是什么

它的重点是**补充细节和经验**，不是承担主线叙事，也不是代替知识点文档。
所以补充资料更像“参考手册和经验笔记”。

### 4. 实验项目：`labs/`
实验项目负责回答：
- 这个方案跑起来是什么样
- 能观察到什么现象
- 不同实现方式会带来什么行为差异
- 当前方案在代码层面是怎么落地的

它的重点是**观察行为和验证想法**，不是替代主线文档、知识点文档或补充资料。
所以实验项目更像“实验室”。

## 怎么判断内容该放哪里

- 如果你在讲“系统为什么演化到这一步” -> 放 `docs/roadmap/`
- 如果你在讲“某个技术点本身是什么、何时用” -> 放 `docs/topics/`
- 如果你在讲“某个技术点的细节、参数、踩坑” -> 放 `docs/resources/`
- 如果你在讲“代码跑起来会发生什么” -> 放 `labs/`

## 当前建议

先从 `docs/roadmap/服务扩展演进路线.md` 开始，把第一批阶段定义清楚。
再根据主线中出现的问题，优先补充 `docs/topics/` 中的主文档和 `labs/` 中的实验项目。
