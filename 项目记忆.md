# 项目记忆

## 项目目标
构建一个后端学习项目，把零散的知识连接成一套结构化的认知体系。

## 目标使用者
面向已经有 Java 和 Node.js 实战经验，但希望系统化理解后端开发的工程师。

## 核心学习原则
只有当系统出现具体压力时，才引入对应技术。

## 当前阶段的学习策略
当前阶段优先建立后端开发的整体脉络和全景认知，不以底层细节深挖为主。
对象头、锁、AQS、源码机制等内容仍然重要，但应在主线推进过程中按问题逐步补充，而不是一开始就作为学习主线。

这样做的原因是：
- 当前最重要的是先建立知识之间的连接关系
- 先形成“问题 -> 分析 -> 方案 -> 取舍”的判断路径
- 避免项目再次演变成知识孤岛仓库

## 主线颗粒度原则
主线阶段需要控制颗粒度，但不设死数量上限。

当前共识是：
- 不把每个细小技术点都拆成独立顶层阶段
- 顶层阶段数量保持在可理解、可维护的范围内
- 如果后续主线自然演进到 10 个以上阶段，只要结构仍然清晰，就不需要为了数字强行合并
- 更重要的是避免出现 20+ 或 30+ 个过碎的顶层阶段
- 允许在每个主阶段内部再细分小步骤或小场景

## 武器库组织原则
当前更推荐“一个主题一篇主文档 + 一个独立实验项目”的组织方式。

当前共识是：
- 主线文档负责说明“为什么现在需要这类武器”
- 知识点主文档负责统一说明“有哪些武器、怎么用、适用场景和常见坑”
- 独立实验负责展示“这些武器具体怎么工作、行为有什么差异”
- 只有当某个主题深到参数级、源码级、清单级时，才额外使用 `docs/resources/`

这样设计的原因是：
- 更贴近人类学习时的阅读路径
- 更容易形成完整的武器图谱
- 更符合当前项目追求系统化认知，而不是零散资料堆积的目标

## 主要学习方式
从一个非常小的 Web Server 开始，随着压力增加逐步演化。每一步都要说明：
- 当前出现了什么问题
- 为什么会在这个阶段出现
- 有哪些可选方案
- 为什么当前选择这个方案
- 这个方案带来了什么代价

## 结构原则
项目以文字说明为主，代码演示为辅。
主线文档负责串联“问题 -> 分析 -> 方案 -> 取舍”。
知识点主文档负责解释某个主题是什么、有哪些武器、为什么需要、什么时候使用以及常见坑。
补充资料只在真正需要更深展开时使用。
实验项目负责展示关键方案的运行行为。
不是所有后端知识都需要代码演示。

## 当前目录方向
- `docs/roadmap/`：主线演化文档
- `docs/topics/`：知识点主文档
- `docs/resources/`：补充资料文档
- `docs/文档索引.md`：文档索引
- `labs/web-server-scaling/`：主 Web Server 演化实验
- `labs/independent-demos/`：独立小实验

## 当前决定
- 采用“主线演化 + 知识点主文档 + 代码实验”的结构
- 主实验以 Java 为主
- 主实验技术栈基线为：`JDK 21`、`Spring Boot 3.5.x`、`Maven 3.9.x`、`PostgreSQL 17`、`Spring Data JPA + Hibernate`、`Flyway`
- 商城系统作为主要业务语境，但不从完整商城开始实现
- 当前优先使用 Docker / Docker Compose 管理数据库、Redis、队列等外部依赖
- 前期只安装真正需要的工具，能不装在 WSL Ubuntu 中的基础设施就不直接装在 WSL 中
- 当前环境下需要特别注意 Windows 宿主机、WSL 和代理之间的网络边界
- 当前学习顺序以“先主线、后细节”为主，理论知识按主线问题逐步补充
- 独立实验按主题聚合，避免拆成大量零散 demo
- 并发主题当前采用“一篇主文档 + 一个独立实验项目”的方式组织

## 当前路线
1. 先搭好简洁的项目骨架。
2. 定义 Web Server 演化主线。
3. 从主线中拆出相关知识点文档。
4. 为真正需要深入展开的内容补充少量资源文档。
5. 只为真正需要观察行为的技术补充 demo。
6. 随着学习推进持续迭代结构。

## 工作规则
先启动并迭代，不等待完美设计。
