# 项目记忆

## 项目目标
构建一个后端学习项目，把零散的知识连接成一套结构化的认知体系。

## 目标使用者
面向已经有 Java 和 Node.js 实战经验，但希望系统化理解后端开发的工程师。

## 核心学习原则
只有当系统出现具体压力时，才引入对应技术。

## 当前阶段的学习策略
当前阶段优先建立后端开发的整体脉络和全景认知，不以底层细节深挖为主。
对象头、锁、AQS、源码机制等内容仍然重要，但应在主线推进过程中按问题逐步补充，而不是一开始就作为学习主线。

这样做的原因是：
- 当前最重要的是先建立知识之间的连接关系
- 先形成“问题 -> 分析 -> 方案 -> 取舍”的判断路径
- 避免项目再次演变成知识孤岛仓库

## 主线颗粒度原则
主线阶段需要控制颗粒度，但不设死数量上限。

当前共识是：
- 不把每个细小技术点都拆成独立顶层阶段
- 顶层阶段数量保持在可理解、可维护的范围内
- 如果后续主线自然演进到 10 个以上阶段，只要结构仍然清晰，就不需要为了数字强行合并
- 更重要的是避免出现 20+ 或 30+ 个过碎的顶层阶段
- 允许在每个主阶段内部再细分小步骤或小场景

## 当前文档组织原则
当前项目以三条线并行为主：
- `docs/roadmap/` 负责系统演进主线
- `docs/capability-map/` 负责能力地图这一中间层
- `docs/java-topics/` 负责专题知识整理

当前共识是：
- 主线文档负责说明“为什么当前阶段会遇到这个问题、为什么先选这个方案”
- 能力地图文档负责承接“核心原则”和“专题展开”之间的空档，把原理翻译成工程能力域
- 专题文档负责由你自己整理“这个专题内部有哪些知识点、彼此怎么连接、当前 stage 用到了哪一部分”
- `references/` 作为你过去知识体系的参考材料，而不是新的唯一结构源头
- 实验项目负责展示方案行为和代码落地

这样设计的原因是：
- 主线能帮助建立问题驱动的系统认知
- 能力地图能帮助建立“原理 -> 能力域 -> 专题”的过渡层
- 专题整理能帮助你真正把知识用自己的语言重新组织起来
- 三者结合，既避免只剩场景，也避免只剩零散摘抄

## 主要学习方式
从一个非常小的 Web Server 开始，随着压力增加逐步演化。每一步都要说明：
- 当前出现了什么问题
- 为什么会在这个阶段出现
- 有哪些可选方案
- 为什么当前选择这个方案
- 这个方案带来了什么代价

同时，对主线中涉及到的专题知识：
- 由你自己整理初稿
- 我负责 review、提问、找漏洞和补边界
- 最终保留你的表达，而不是完全替换成我的表达

## 结构原则
项目以文字说明为主，代码演示为辅。
主线文档负责串联“问题 -> 分析 -> 方案 -> 取舍”。
专题文档负责沉淀你对某个专题的知识结构与理解。
实验项目负责展示关键方案的运行行为。
不是所有后端知识都需要代码演示。

## 当前目录方向
- `领域知识整理方法论.md`：跨领域可复用的知识整理方法
- `docs/roadmap/`：主线演化文档
- `docs/capability-map/`：能力地图文档
- `docs/java-topics/`：专题知识文档
- `labs/web-server-scaling/`：主 Web Server 演化实验
- `labs/independent-demos/`：独立小实验
- `references/`：历史知识清单与参考资料

## 当前决定
- `docs/java-topics/` 的建设方向，优先覆盖 Advanced Java Backend 的主干能力，而不是追求 Java 全量知识点大全
- 新增专题时优先从高层抓“核心意义、解决的问题、作用边界和使用场景”，细节暂时只点到为止
- 新增 `docs/capability-map/`，作为 `后端开发核心原则` 与专题知识文档之间的中间层，用来沉淀后端工程能力地图
- 新增 `领域知识整理方法论.md`，把项目中逐渐形成的“核心原则 -> 能力地图 -> 演进线 -> 专题展开”整理方法沉淀为跨领域可复用框架
- `领域知识整理方法论.md` 当前额外明确了三个护栏：写文档前先声明对象与边界、明确区分“演进线 / 主线文档 / 实验代码”、先判断是单线 / 多线 / 双轨结构，避免再把知识演进线误写成代码实施顺序
- AI 应用相关文档已迁移到独立的 `web-app-labs` 仓库维护；`backend-labs` 继续聚焦 Java 后端主线与相关方法论沉淀
- 每次确认新的方向、结构或整理方式后，都要回看 `AGENTS.md` 和 `项目记忆.md`，确保新会话或上下文压缩后仍保持一致
- 采用“主线演化 + 专题整理 + 代码实验”的结构
- 主实验以 Java 为主
- 主实验技术栈基线为：`JDK 21`、`Spring Boot 3.5.x`、`Maven 3.9.x`、`PostgreSQL 17`、`Spring Data JPA + Hibernate`、`Flyway`
- 商城系统作为主要业务语境，但不从完整商城开始实现
- `labs/web-server-scaling/` 不再按演进线拆成多个 stage 项目，而是作为一个主题目录，后续包含两个代码项目：一个 Spring Boot 模块化单体项目，一个基于它拆出来的 Spring Cloud 微服务项目
- `docs/capability-map/后端演进线.md` 在 `web-server-scaling` 中保留为知识解释顺序，而不再作为代码实施顺序
- Spring Boot 单体项目集中承载所有不依赖服务边界的后端能力点；微服务相关问题（服务注册发现、远程调用治理、网关、分布式事务等）进入同主题下的第二个 Spring Cloud 项目
- `labs/web-server-scaling/` 新增文档分层：`PRD.md` 负责主题总蓝图，`docs/slices/` 负责 `mall-monolith` 的 `V1 ~ V4` 训练路线，`docs/designs/` 负责关键方案设计，`docs/execution/` 负责执行清单与批次记录；当前 `V1 ~ V4` 不覆盖 `mall-microservices/`
- 当前已明确：`Kafka` 不进入 `mall-monolith/` 基线；消息队列与事件驱动能力放到 `mall-microservices/`
- 当前单体项目中的附属异步流程，先用数据库任务表 + 调度 / 线程池处理，不引入通用 `outbox`
- 当前优先使用 Docker / Docker Compose 管理数据库、Redis、队列等外部依赖
- 前期只安装真正需要的工具，能不装在 WSL Ubuntu 中的基础设施就不直接装在 WSL 中
- 当前环境下需要特别注意 Windows 宿主机、WSL 和代理之间的网络边界
- 当前学习顺序以“先主线、后专题补全”为主
- 专题文档由你来整理，我负责 review

## 文档一致性复查约定
- 先确认文档对象：它是在写领域、主题目录、具体项目、子项目，还是首版范围
- 先确认“阶段”含义：这里说的是知识演进阶段、代码实施阶段，还是项目拆分阶段
- 关键决策优先只在 `项目记忆.md` 定稿，其他文档尽量做映射和引用，不重复发明新口径
- 每次改动项目结构、技术边界或阶段落位后，至少回看 `README.md`、`项目记忆.md`、相关 `docs/roadmap/`、相关 `docs/capability-map/`、对应 `labs/.../README.md` 与 `PRD.md`
- 发现同一事实在多个文档重复出现时，优先压缩重复表达，减少多处维护
- 如果某条规则只对单个项目成立，必须显式写出适用对象，避免被上升成整个主题或整个仓库的共识

## 当前路线
1. 先搭好简洁的项目骨架。
2. 定义 Web Server 演化主线。
3. 随着主线推进，挑出当前真实涉及的专题。
4. 由你基于 `references/` 和当前 stage 自己整理专题文档。
5. 用实验代码验证关键方案行为。
6. 随着学习推进持续迭代结构。

## 工作规则
先启动并迭代，不等待完美设计。
