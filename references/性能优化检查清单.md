# 性能优化 · 面试专题补充 Checklist

## 第一层：性能优化方法论 —— 面试中的思考框架
> 面试官问"你怎么做性能优化"时，最忌讳上来就说"加缓存、加索引"。你需要展示一个系统化的思路。
>

- [ ] **性能优化的完整链路思维** ⭐**本质**：性能优化不是"哪里慢改哪里"，而是一个严格的闭环——先量化（用数据定义"慢"）→ 定位瓶颈（找到真正的卡点，而不是猜）→ 针对性优化（只优化瓶颈处）→ 验证效果（用优化前后的数据证明有效）。这四步缺一不可。**为什么需要它**：没有这个框架，你很容易犯两个错——①优化了不是瓶颈的地方（比如数据库才是瓶颈，你却在优化代码逻辑），②优化了但说不出效果（面试时说"我优化了"但没有数据支撑，等于没说）。**关联**：这个链路思维贯穿了所有性能相关的工作——JVM 调优、SQL 优化、缓存设计、接口优化全部遵循同一个流程。参见核心原理图谱的「量化先行，数据驱动」。
    - 过关标准：拿到任何优化问题，能按这个链路逐层分析而不是乱猜：
        1. 先量化（慢在哪？有没有监控数据？）
        2. 定位瓶颈（是 CPU / 内存 / IO / 网络？是数据库 / 应用层 / 外部依赖？）
        3. 针对性优化（不要没找到瓶颈就开始优化）
        4. 验证效果（优化前后对比数据）
- [ ] **性能指标体系** ⭐**本质**：衡量系统性能的核心指标有四个——RT（Response Time，响应时间）、QPS/TPS（每秒请求/事务量，衡量吞吐能力）、P99/P95（百分位延迟，99% 的请求在这个时间内完成）、错误率（请求失败的比例）。**为什么需要它**：平均 RT 会被少量快请求拉低，掩盖真实问题。比如平均 RT 是 100ms，但 P99 是 3s——意味着每 100 个请求就有 1 个用户等了 3 秒。面试时说"P99 从 2s 降到 200ms"比说"平均响应时间变快了"专业得多，也可信得多。**关联**：P99 偏高通常指向长尾问题——GC 停顿、偶发的锁竞争、缓存未命中等。定位长尾延迟的能力连接了并发 checklist 的 GC 分析和锁排查。
    - 过关标准：能说出关键指标及含义——RT（响应时间）、QPS/TPS（吞吐量）、P99/P95（长尾延迟）、错误率，知道为什么看 P99 比看平均 RT 更有意义
- [ ] **能说出具体数字****本质**：面试中性能优化的可信度完全取决于数据。"我优化了缓存策略"是空话，"缓存命中率从 60% 提升到 95%，接口 P99 从 800ms 降到 120ms"是有说服力的回答。**为什么需要它**：面试官见过太多"背方案"的候选人，能说出具体数字的人意味着真正做过。即使是自己的练习项目（如 Side Project 中的压测数据），只要有具体数字，面试官也会认可。**关联**：这就是为什么实践练习计划中每个实验和 Side Project 每个阶段都要求记录优化前后数据——它们就是你面试时的弹药。
    - 过关标准：面试中能用数据说话，比如"接口 P99 从 800ms 优化到 150ms"、"数据库 QPS 从 2000 提升到 8000"。即使是自己练习的项目，也要养成记录优化前后数据的习惯

### 面试场景题
- [ ] "你做过哪些性能优化？效果怎么样？" —— 这是最常见的开放题，能不能说出具体数字和优化思路决定了回答质量
- [ ] "一个接口响应很慢，你怎么排查？" —— 考你的排查链路是否系统化，而不是上来就猜

## 第二层：JVM 调优 —— 三份 checklist 未覆盖的重点
> 前三份 checklist 覆盖了并发、缓存、消息队列等应用层优化，但 JVM 层面的优化是一个独立且重要的面试考点。
>

### 2.1 JVM 内存结构
- [ ] **运行时数据区** ⭐**本质**：JVM 将内存划分为几个区域——堆（存对象实例，所有线程共享，GC 的主要工作区）、方法区/元空间（存类信息、常量池，线程共享）、虚拟机栈（每个线程私有，存方法调用的栈帧）、本地方法栈（调用 native 方法用）、程序计数器（记录当前线程执行到哪条指令）。

**为什么需要它**：不同区域的 OOM 表现不同，排查方向也不同。堆 OOM（Java heap space）说明对象太多或有内存泄漏。元空间 OOM（Metaspace）说明类加载过多。栈溢出（StackOverflowError）说明递归太深或方法调用链过长。知道内存结构才能快速定位 OOM 类型。

**关联**：堆内存是 GC 的管辖范围，堆的分代模型（新生代 + 老年代）直接决定了 GC 的行为方式。虚拟机栈的大小（-Xss）和线程数相关——每个线程一个栈，线程越多消耗的栈内存越大。

    - 过关标准：能画出 JVM 内存结构图（堆、方法区/元空间、虚拟机栈、本地方法栈、程序计数器），能说出每个区域存什么、哪些会 OOM
- [ ] **堆内存分代模型** ⭐**本质**：堆分为新生代和老年代。新生代又分为 Eden 区和两个 Survivor 区（S0、S1）。大部分对象在 Eden 中创建，经过 Minor GC 存活后进入 Survivor，多次存活后晋升到老年代。默认比例是 Eden:S0:S1 = 8:1:1。**为什么需要它**：分代的设计基于一个经验观察——"大部分对象朝生夕死"。新生代用复制算法（快，但需要额外空间），老年代用标记-整理（慢，但空间利用率高）。理解了分代，你就能解释为什么 Minor GC 很快而 Full GC 很慢，以及为什么要避免大量对象直接进入老年代。**关联**：对象晋升到老年代的条件（年龄阈值、大对象直接进入老年代、Survivor 放不下）直接影响 GC 行为。如果老年代增长过快，Full GC 频率会升高——这连接了性能排查中"频繁 Full GC"的分析思路。
    - 过关标准：能说出新生代（Eden + S0 + S1）和老年代的划分，知道对象从新生代到老年代的晋升条件
- [ ] **元空间（Metaspace）****本质**：JDK 8 用元空间取代了永久代。元空间不在 JVM 堆中，而是使用本地内存（Native Memory）。存储的是类元信息、方法信息、常量池等。默认没有大小限制（受限于系统物理内存），但可以通过 -XX:MaxMetaspaceSize 设置上限。**为什么需要它**：如果应用大量动态生成类（如 CGLIB 代理、反射、JSP 编译），可能导致元空间持续增长直到 OOM。这是一种不太常见但排查时容易被忽略的 OOM 类型。**关联**：Spring AOP 使用 CGLIB 创建代理类，热部署工具（如 Spring DevTools）反复加载类——这些都可能导致元空间膨胀。了解元空间可以在遇到 Metaspace OOM 时快速定位方向。
    - 过关标准：知道 JDK8 用元空间取代了永久代，知道元空间使用本地内存，了解类加载过多导致元空间 OOM 的场景

### 2.2 垃圾回收（GC）
- [ ] **GC 算法基础** ⭐**本质**：三种基础算法——标记-清除（标记存活对象，清除未标记的，会产生内存碎片）、标记-复制（将存活对象复制到另一块区域，没有碎片但浪费一半空间）、标记-整理（标记存活对象并向一端移动，没有碎片也不浪费空间但移动开销大）。**为什么需要它**：每种 GC 回收器底层都是这三种算法的组合。新生代用复制算法（因为存活对象少，复制成本低），老年代用标记-清除或标记-整理（因为存活对象多，复制成本太高）。理解了算法才能理解回收器的设计取舍。**关联**：为什么新生代分 Eden + 两个 Survivor？因为复制算法需要一块"空闲区域"来接收存活对象。两个 Survivor 轮流充当这个角色，Eden 的存活对象复制到其中一个 Survivor，另一个 Survivor 保持空闲——这就是复制算法在分代模型中的具体应用。
    - 过关标准：能说出标记-清除、标记-复制、标记-整理三种算法的原理和优缺点，知道新生代和老年代分别用哪种
- [ ] **常见垃圾回收器** ⭐**本质**：CMS（Concurrent Mark-Sweep）追求低停顿，用标记-清除算法并发回收老年代，但会产生碎片，已在 JDK 14 移除。G1（Garbage First）是目前主流的回收器，把堆划分为多个 Region，优先回收垃圾最多的 Region，兼顾吞吐和停顿。ZGC 是最新一代回收器，停顿时间控制在 10ms 以内，适合超大堆和低延迟场景。**为什么需要它**：面试中最常问的是 G1——因为它是 JDK 9 以后的默认回收器，也是大部分生产环境的选择。你需要知道 G1 的 Region 分区思想、Mixed GC（同时回收新生代和部分老年代 Region）、以及它如何通过预测模型来控制停顿时间。**关联**：G1 的 Region 分区是"分而治之"思想在 GC 层面的体现——不再把堆严格分为新生代和老年代，而是灵活地选择垃圾最多的 Region 优先回收。ZGC 的着色指针和读屏障则代表了 GC 技术的最新方向。
    - 过关标准：能说出 CMS、G1、ZGC 各自的特点和适用场景。重点掌握 G1（目前主流），知道它的 Region 分区、Mixed GC 等核心概念
- [ ] **GC 日志分析** ⭐**本质**：GC 日志记录了每次 GC 事件的关键信息——GC 类型（Minor GC / Mixed GC / Full GC）、回收前后各区域的内存大小、GC 耗时。通过分析日志可以判断 GC 是否健康——Minor GC 频率是否正常、Full GC 是否过于频繁、单次 GC 停顿是否过长。**为什么需要它**：GC 问题是生产环境性能问题的高频原因。频繁 Full GC 会导致接口卡顿甚至服务不可用。能看懂 GC 日志是定位这类问题的第一步——你需要从日志中判断是内存不够、内存泄漏、还是对象分配速率过高。**关联**：GC 日志分析往往是排查链路的起点。发现 Full GC 频繁 → 用 jstat 看内存使用率 → 用 jmap dump 堆快照 → 用 MAT 分析大对象。这是一个完整的排查链条。
    - 过关标准：能看懂 GC 日志的关键信息——GC 类型（Minor / Major / Full）、回收前后内存大小、耗时，能判断 GC 是否异常
- [ ] **常见 GC 调优参数****本质**：核心参数包括——-Xms（初始堆大小）、-Xmx（最大堆大小）、-Xmn（新生代大小）、-XX:MaxMetaspaceSize（元空间上限）、-XX:+UseG1GC（选择 G1 回收器）、-XX:MaxGCPauseMillis（G1 的目标停顿时间）。生产环境通常建议 Xms = Xmx（避免堆动态扩缩容的开销）。**为什么需要它**：面试中如果你说"用默认参数"，面试官会追问"为什么不调？"。反过来如果你过度调参（比如手动设了一堆 -XX 参数但说不出原因），也会被质疑。合理的回答是：理解常用参数的含义，知道什么情况需要调整，大部分情况下 G1 的默认配置就够好了。**关联**：Xms = Xmx 避免了堆扩容时的 Full GC。-XX:MaxGCPauseMillis 设置过小可能导致 GC 回收不彻底，反而更频繁地触发 GC——这是一个典型的取舍问题，参见核心原理图谱的「权衡」。
    - 过关标准：知道 -Xms / -Xmx / -Xmn / -XX:MaxMetaspaceSize 等常用参数的含义，知道为什么建议 Xms = Xmx

### 2.3 JVM 问题排查工具
- [ ] **jstat —— GC 监控****本质**：jstat 是 JDK 内置的命令行工具，能实时查看 JVM 各内存区域的使用率和 GC 统计。最常用的命令是 `jstat -gcutil <pid> 1000`——每隔 1 秒打印一次各区内存使用百分比和 GC 次数/耗时。**为什么需要它**：它是最轻量、最快速的 GC 监控手段——不需要额外安装，不需要重启服务，一条命令就能看到当前 GC 状态。当你怀疑性能问题和 GC 有关时，jstat 是第一个使用的工具。**关联**：jstat 看到的是实时数据，如果要分析历史趋势（比如 Full GC 频率是否在逐渐增加），需要把 GC 日志接入监控系统（如 Prometheus + Grafana）。
    - 过关标准：能用 jstat -gcutil 查看各区内存使用率和 GC 次数/耗时
- [ ] **jmap —— 内存快照****本质**：jmap 可以导出 JVM 的堆快照（Heap Dump），生成 .hprof 文件。这个文件包含了堆中所有对象的信息，可以用分析工具打开查看哪些对象占用了最多内存。**为什么需要它**：当你确认是内存问题（比如 jstat 看到老年代持续增长），下一步就是 dump 堆快照来找到"谁在占内存"。但注意：jmap -dump 会触发 Full GC 并暂停应用（STW），在生产环境要谨慎使用，最好在流量低峰期操作。**关联**：更安全的做法是在启动参数中加 -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/path/to/dump——OOM 时自动导出堆快照，不需要手动 jmap。
    - 过关标准：能用 jmap -dump 导出堆快照，知道在线上要谨慎使用（会触发 STW）
- [ ] **MAT / VisualVM —— 堆分析****本质**：MAT（Memory Analyzer Tool）和 VisualVM 是堆快照的分析工具。MAT 的核心能力是 Dominator Tree（按对象占用内存大小排序）和 Leak Suspects（自动分析可能的内存泄漏点）。VisualVM 更轻量，适合快速浏览。**为什么需要它**：堆快照文件通常几百 MB 到几 GB，不可能人工阅读。MAT 能自动找出占用内存最大的对象链，告诉你"某个 HashMap 占了 800MB，它被哪个变量引用，引用链是什么"——这就是定位内存泄漏的核心手段。**关联**：MAT 发现的大对象往往指向——未关闭的资源（数据库连接、流）、缓存没有淘汰策略无限增长、ThreadLocal 泄漏、静态集合不断添加但不清理。这些原因连接了并发 checklist 中的资源管理和 ThreadLocal 知识。
    - 过关标准：能用工具分析堆快照，找到内存占用最大的对象，定位内存泄漏
- [ ] **Arthas —— 线上诊断神器** ⭐**本质**：Arthas 是阿里开源的 Java 在线诊断工具，核心能力是不重启服务就能排查问题。常用命令：dashboard（全局概览 CPU、内存、线程）、thread（查看线程状态和堆栈，支持找最忙线程）、trace（追踪方法调用链耗时）、watch（实时观察方法的入参和返回值）。**为什么需要它**：传统排查工具（jstack、jmap）需要登录服务器执行命令，而且 jmap 有 STW 风险。Arthas 可以远程连接，而且 trace 命令能精确到方法级别地定位"慢在哪一步"——这在排查接口性能问题时极其高效。**关联**：Arthas 的 trace 命令可以替代在代码中手动埋点计时的方式来排查性能瓶颈。它的 thread 命令是 jstack 的增强版，dashboard 可以替代多个监控命令的组合。
    - 过关标准：了解 Arthas 的核心命令（dashboard、thread、trace、watch），知道它能在不重启服务的情况下排查问题

### 面试场景题
- [ ] "线上服务频繁 Full GC，你怎么排查？" —— 考 GC 日志分析 → jstat 监控 → jmap dump → MAT 分析的完整链路
- [ ] "你的服务 OOM 了，错误信息是 java.lang.OutOfMemoryError: Java heap space，怎么排查？" —— 考堆内存分析能力
- [ ] "你们 JVM 参数是怎么配的？为什么这样配？" —— 面试官想看你是理解了在配，还是抄的默认值

## 第三层：数据库性能优化深入 —— 补充系统设计 checklist 未覆盖的细节
> 系统设计 checklist 已经覆盖了索引设计和分库分表的策略层面，这里补充实操层面的排查和优化技能。
>

### 3.1 慢 SQL 排查与优化
- [ ] **慢查询日志** ⭐**本质**：MySQL 的慢查询日志记录了执行时间超过阈值（long_query_time）的所有 SQL 语句。开启后可以精确找到哪些 SQL 拖慢了系统，而不是靠猜测。**为什么需要它**：数据库性能问题 90% 都来自少数几条烂 SQL。不开慢查询日志就排查慢 SQL，就像不看体检报告就开药——纯靠猜。它是数据库性能优化的起点。**关联**：找到慢 SQL 后，下一步就是用 EXPLAIN 分析执行计划——这两个工具是一条排查链上的前后步骤。
    - 过关标准：知道如何开启 MySQL 慢查询日志（slow_query_log），能根据日志找到慢 SQL
- [ ] **EXPLAIN 执行计划分析** ⭐**本质**：EXPLAIN 告诉你 MySQL 打算怎么执行一条 SQL——用不用索引、扫描多少行、有没有临时表或文件排序。关键字段：type（访问类型，从好到差：const > ref > range > index > ALL）、key（实际用到的索引）、rows（预估扫描行数）、Extra（额外信息，如 Using filesort、Using temporary 表示有优化空间）。**为什么需要它**：一条 SQL 慢不慢，不是看 SQL 写得多复杂，而是看执行计划。同样一条 SQL，有索引时 type=ref 扫描 10 行，没索引时 type=ALL 扫描 500 万行——差距是天壤之别。EXPLAIN 是唯一能在执行前预判 SQL 性能的工具。**关联**：EXPLAIN 中的 type=ALL（全表扫描）往往意味着索引失效或缺少索引，直接连接了系统设计 checklist 中的索引设计知识。Extra 中的 Using filesort 说明排序没用上索引，Using temporary 说明产生了临时表——这些都是优化的信号。
    - 过关标准：能看懂 EXPLAIN 输出的关键字段——type（ALL / index / range / ref / const）、key（实际使用的索引）、rows（扫描行数）、Extra（Using filesort / Using temporary），能据此判断 SQL 性能问题
- [ ] **索引失效的常见场景** ⭐**本质**：虽然加了索引，但以下操作会导致 MySQL 放弃使用索引——对索引列使用函数（如 WHERE YEAR(create_time) = 2024）、隐式类型转换（如 varchar 列用 int 查询）、LIKE 以通配符开头（如 LIKE '%abc'）、OR 条件中有非索引列、联合索引不满足最左前缀原则。**为什么需要它**：面试中"为什么加了索引还是慢"是经典追问。如果你只知道"加索引"但不知道索引什么时候失效，说明你的数据库优化能力停留在表面。理解失效场景才能写出真正能命中索引的 SQL。**关联**：最左前缀原则的本质是联合索引在 B+ 树中的排列方式——先按第一列排序，第一列相同时再按第二列排序。所以跳过第一列直接查第二列，B+ 树无法利用有序性。这连接了系统设计 checklist 中 B+ 树索引的原理。
    - 过关标准：能列出导致索引失效的常见操作——对索引列使用函数、隐式类型转换、LIKE '%xxx'、OR 条件、不满足最左前缀等
- [ ] **SQL 优化常见手段****本质**：常见的优化手段包括——避免 SELECT *（减少不必要的数据传输和内存占用）、使用覆盖索引（查询的字段全在索引中，不需要回表）、减少子查询改用 JOIN（子查询可能每行都执行一次）、深分页优化（LIMIT 100000, 10 需要扫描前 100010 行然后丢弃前 100000 行）。**为什么需要它**：这些手段不是"高级技巧"，而是日常开发中应该遵守的基本规范。尤其是深分页问题——当数据量上来后，LIMIT offset, size 的性能会急剧下降，这是分页接口性能问题的头号元凶。**关联**：深分页的优化思路有两种——①子查询优化：先用覆盖索引查出 ID，再根据 ID 查详细数据；②游标分页：用 WHERE id > last_id LIMIT size 替代 LIMIT offset, size。两种方式都避免了扫描大量无用行。
    - 过关标准：能说出避免 SELECT *、合理使用覆盖索引、减少子查询改用 JOIN、分页优化（深分页问题）等手段

### 3.2 连接池优化
- [ ] **数据库连接池（如 HikariCP / Druid）****本质**：数据库连接的创建涉及 TCP 三次握手 + 数据库认证，非常耗时（几十毫秒级别）。连接池预先创建一批连接保持存活，应用需要时从池中借出，用完归还，避免反复创建销毁的开销。核心参数是最大连接数、最小空闲连接数、连接获取超时时间。**为什么需要它**：不用连接池，每个请求都创建新连接，在高并发下连接创建的延迟会成为瓶颈，甚至耗尽数据库的最大连接数。连接池是几乎所有生产应用的标配，但参数配置不当（过大或过小）会带来不同的问题。**关联**：连接池的本质是"池化与复用"思想——和线程池完全一样的设计原理。连接池配大了会占满数据库连接数影响其他服务，配小了请求排队等连接。连接泄漏（借出不归还）的排查思路和线程泄漏类似。参见核心原理图谱的「池化与复用」。
    - 过关标准：知道连接池的核心参数（最大连接数、最小空闲、获取超时），能根据业务场景调整参数，知道连接泄漏的常见原因和排查方式

## 第四层：接口性能优化实战套路 —— 面试万能答题模板
> 面试中"怎么优化一个慢接口"是最常见的问题之一。这一层帮你建立一个系统化的回答模板。
>

- [ ] **接口优化的分层排查思路** ⭐**本质**：一个请求从发出到响应，经过了多个层——网络层 → 应用层 → 数据层 → 外部依赖 → 基础设施。性能瓶颈可能出在任何一层。系统化排查的方法是逐层排除，而不是猜测。**为什么需要它**：面试官问"怎么优化慢接口"时，如果你上来就说"加缓存"，面试官会追问"你怎么知道是数据库慢而不是网络慢？"分层排查思路能展示你的系统性思维——先定位瓶颈在哪一层，再针对性优化。**关联**：这个分层思路把四份 checklist 的知识点串起来了——应用层的问题找并发 checklist（线程池、异步），数据层的问题找本文档第三层（慢 SQL、连接池），缓存和消息队列找系统设计和分布式 checklist，基础设施的问题找 JVM 调优。
    - 过关标准：能按以下层次逐步分析，而不是一上来就说"加缓存"：
        1. **网络层**：DNS 解析、网络延迟、带宽瓶颈
        2. **应用层**：代码逻辑是否有冗余计算、N+1 查询、同步可改异步
        3. **数据层**：慢 SQL、缺索引、数据量过大
        4. **外部依赖**：下游服务慢、第三方 API 超时
        5. **基础设施**：CPU / 内存 / 磁盘 IO 瓶颈
- [ ] **常见优化手段速查表**（在对应 checklist 中有详细知识点，这里做串联）

| 优化方向 | 核心手段 | 详细知识点位置 |
| --- | --- | --- |
| 减少计算 | 缓存（多级缓存、本地缓存 + Redis） | 系统设计 2.2 |
| 减少等待 | 异步化（MQ 解耦、CompletableFuture 并行） | 并发 3.2 + 系统设计 2.3 |
| 减少 IO | 索引优化、批量查询替代循环查询、连接池 | 本文档 3.1 + 3.2 |
| 减少传输 | 精简返回字段、压缩、分页 | 本文档 3.1 |
| 提升吞吐 | 线程池调优、水平扩容 | 并发 3.1 + 系统设计 3.2 |
| 兜底防护 | 限流、熔断、降级 | 系统设计 3.1 |
