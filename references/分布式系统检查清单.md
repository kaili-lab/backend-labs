## 第一层：分布式核心理论 —— 能说出取舍逻辑
> 面试不会直接问"什么是 CAP"，而是问"你的系统在一致性和可用性之间怎么选的？为什么？"你需要能结合场景回答。
>

- [ ] **CAP 定理** ⭐**本质**：一个分布式系统在发生网络分区（P，Partition）时，不可能同时保证一致性（C，所有节点看到的数据一样）和可用性（A，每个请求都能得到响应）。你只能选 CP（保一致性，牺牲部分可用性）或 AP（保可用性，允许数据暂时不一致）。注意：P 不是一个可选项——网络分区在分布式系统中迟早会发生，所以本质上是在 C 和 A 之间选。**为什么需要它**：CAP 不是用来背的，而是帮你做技术选型的思维工具。面试中你需要说出"我在什么场景选了 CP / AP，为什么"。比如：订单和支付选 CP（不能容忍超卖或重复扣款），商品详情页选 AP（短暂展示旧数据比直接不可用好）。**关联**：CAP 的取舍贯穿了整个分布式系统的设计——Nacos 支持 AP 和 CP 两种模式，Redis 主从切换可能丢数据（AP 倾向），Zookeeper 保证强一致但不可用时拒绝服务（CP 倾向）。参见核心原理图谱的「可见性与一致性」和「权衡」。
    - 过关标准：不是背定义，而是能结合具体系统说出取舍。比如："我们的订单系统选了 CP，因为不能容忍超卖；商品详情页选了 AP，因为短暂数据不一致可接受"
- [ ] **BASE 理论****本质**：BASE 是 Basically Available（基本可用）、Soft State（软状态）、Eventually Consistent（最终一致）的缩写。它是对 CAP 中 AP 方向的补充——承认分布式系统做不到强一致，但可以在一定时间窗口后达到一致，这对大部分业务是可接受的。**为什么需要它**：强一致代价太高（需要分布式锁或共识协议，性能差），而很多业务场景其实不需要强一致。比如发了一条微博，粉丝晚几秒看到是可接受的。BASE 给了你一个理论基础来为"最终一致性方案"辩护。**关联**：你日常用的很多技术本质上都是 BASE——Redis 主从复制是最终一致的，消息队列异步处理是最终一致的，缓存和 DB 的同步也是最终一致的。面试时说"我们用了最终一致性方案"需要接着解释"一致性窗口大概多长、业务能不能接受"。
    - 过关标准：能解释 BASE 是对 CAP 中 AP 方向的补充，能举出你见过的"最终一致性"的实际案例（如缓存更新、异步消息通知）
- [ ] **一致性模型：强一致 / 最终一致 / 因果一致****本质**：强一致（写入成功后所有节点立刻看到新值）、最终一致（写入后过一段时间所有节点看到新值，但中间可能读到旧值）、因果一致（有因果关系的操作保证顺序，没有因果关系的操作不保证）。**为什么需要它**：不同业务场景需要不同级别的一致性。金融交易需要强一致，社交动态用最终一致就行，而因果一致在聊天场景中很重要（你看到了消息才能回复，这个因果顺序不能乱）。面试中能区分这三种一致性并匹配到业务场景，说明你理解了而不是在背。**关联**：强一致的典型实现是共识协议（Raft / Paxos），如 etcd、Zookeeper。最终一致的典型实现是异步复制 + 消息队列。因果一致在数据库领域是一个活跃的研究方向（如 CockroachDB 的混合逻辑时钟）。
    - 过关标准：能说出每种一致性的含义，并能举出对应的技术实现（如 Raft → 强一致，消息队列 → 最终一致）
- [ ] **脑裂问题****本质**：分布式集群中，由于网络分区导致一个集群分成了两个"子集群"，每个子集群都认为自己是完整的集群，各自独立对外提供服务——这就是脑裂。后果是两个子集群可能各自接受写请求，导致数据冲突且无法自动合并。**为什么需要它**：脑裂是分布式系统最危险的故障之一，因为它会产生数据不一致且很难自动恢复。面试中如果你提到了集群方案，面试官可能会追问"脑裂了怎么办"。**关联**：防止脑裂的标准方案是"过半机制"（Quorum）——任何决策需要超过半数节点同意才生效，这样两个子集群中最多只有一个能达到过半，另一个自动停止服务。Raft、Zookeeper、Redis Sentinel 都使用了这个思想。Fencing Token（隔离令牌）是另一种防护手段——持有旧 Token 的节点无法执行写操作。
    - 过关标准：能解释什么是脑裂、为什么会发生、常见的应对方案（过半机制、fencing token）

### 2.1 注册中心：Nacos
- [ ] **Nacos 的核心功能** ⭐**本质**：Nacos 同时充当注册中心和配置中心两个角色。作为注册中心——服务启动时向 Nacos 注册自己的地址，其他服务通过 Nacos 发现它的地址。作为配置中心——集中管理配置信息，配置变更时实时推送给服务，无需重启。**为什么需要它**：微服务架构中服务实例的 IP 和端口是动态变化的（扩容、重启、迁移），不能硬编码。注册中心解决了"服务在哪里"的问题。配置中心解决了"配置散落在各个服务中难以管理"的问题。两者合一减少了运维复杂度。**关联**：注册中心是微服务架构的基础设施——没有它，服务发现、负载均衡、健康检查都无法工作。你的 Spring Cloud 项目中 Feign 调用的底层就是从 Nacos 获取服务实例列表，再通过 Ribbon 做负载均衡。
    - 过关标准：能说出 Nacos 作为注册中心和配置中心的双重角色，知道服务注册、发现、健康检查的流程
- [ ] **Nacos 的 AP 模式 vs CP 模式** ⭐**本质**：Nacos 同时支持两种模式——临时实例（默认）使用 AP 模式（Distro 协议，优先保可用性，节点间最终一致），持久实例使用 CP 模式（Raft 协议，优先保一致性，需要过半节点存活才能写入）。**为什么需要它**：大部分微服务场景用临时实例（AP 模式）就够了——服务实例本来就是动态的，短暂的注册信息不一致（如某个节点还没看到新实例）影响不大。但如果你需要用 Nacos 管理一些"必须精确"的信息（如 DNS 记录、路由规则），就需要 CP 模式保证不会读到错误数据。**关联**：这是 CAP 定理在注册中心层面的直接体现。Eureka 只支持 AP（没有 CP 选项），Zookeeper 只支持 CP（不可用时拒绝服务），而 Nacos 两种都支持——面试时这是一个展示你理解 CAP 实际应用的好机会。
    - 过关标准：知道 Nacos 同时支持 AP（临时实例，Distro 协议）和 CP（持久实例，Raft 协议），能说出你的项目用的是哪种以及为什么
- [ ] **Nacos 健康检查机制****本质**：两种方式——临时实例由客户端主动向 Nacos 发送心跳（默认 5 秒一次），超过 15 秒没有心跳标记为不健康，超过 30 秒移除实例。持久实例由 Nacos 服务端主动探测（TCP 连接或 HTTP 请求），实例不健康时标记但不移除。**为什么需要它**：如果一个服务实例已经 OOM 挂了，但注册中心还记着它的地址，其他服务调用它就会失败。健康检查的作用是及时发现和剔除不健康的实例，保证服务发现返回的都是可用实例。**关联**：健康检查的时间窗口（15 秒标记不健康、30 秒移除）意味着在这段时间内可能有请求被路由到已经挂掉的实例。这就是为什么客户端还需要超时和重试机制——注册中心的健康检查不是实时的，不能完全依赖它。
    - 过关标准：能区分客户端主动上报（临时实例）和服务端主动探测（持久实例），知道不健康实例会被怎么处理
- [ ] **Nacos 挂了会怎样？****本质**：Nacos 客户端有本地缓存——即使 Nacos 集群完全不可用，已经注册的服务之间仍然可以通过本地缓存的服务列表继续通信。但新的服务无法注册，服务列表无法更新（新增/下线的实例无法感知），配置变更无法推送。**为什么需要它**：面试官问这个是想看你是否理解注册中心在架构中的角色和影响范围。如果你答"Nacos 挂了整个系统就瘫了"，说明你不理解客户端缓存机制；如果你答"完全没影响"，说明你忽略了注册和配置更新的影响。**关联**：本地缓存机制是一种"降级"策略——注册中心不可用时降级为使用缓存数据。这连接了系统设计 checklist 中的高可用设计思想。Nacos 自身也可以做集群部署（3 个节点以上）来提高可用性。
    - 过关标准：能说出客户端有本地缓存，短时间内服务间调用不受影响，但新服务无法注册、无法感知实例变化

### 2.2 分布式缓存：Redis
- [ ] **Redis 数据结构及业务场景** ⭐**本质**：Redis 提供 5 种基础数据结构，每种都有最适合的业务场景——String（缓存、计数器、分布式锁）、Hash（对象存储，如用户信息）、List（消息队列、最新动态列表）、Set（去重、共同好友、标签）、ZSet/Sorted Set（排行榜、延迟队列）。**为什么需要它**：面试中问"Redis 的数据结构"不是让你列举，而是看你能不能根据业务场景选对数据结构。选错了可能导致性能差或功能不满足——比如排行榜用 List 就不如 ZSet（ZSet 天然有序且支持按分数范围查询）。**关联**：每种数据结构的底层实现决定了它的性能特征——ZSet 底层用跳表（Skip List），支持 O(logN) 的插入和范围查询。String 用 SDS（Simple Dynamic String），支持 O(1) 的长度获取。了解底层实现在面试中是加分项。
    - 过关标准：不是列举 5 种数据结构，而是能说出"排行榜用 ZSet、分布式锁用 String + Lua、计数器用 INCR、Feed 流用 List/Stream"这种场景对应关系
- [ ] **Redis 持久化：RDB vs AOF** ⭐**本质**：RDB（Redis Database）是定时对内存做全量快照保存到磁盘，恢复快但可能丢失两次快照之间的数据。AOF（Append Only File）是记录每条写命令到日志文件，数据更安全但文件更大、恢复更慢。生产环境通常 RDB + AOF 混合使用——用 AOF 保证数据安全性，用 RDB 加速重启恢复。**为什么需要它**：Redis 是内存数据库，进程重启或宕机数据就丢了。持久化是让 Redis 具备数据恢复能力的机制。面试中考的不是你知不知道两种方式，而是你能不能说出各自的优缺点以及为什么混合使用。**关联**：AOF 的 appendfsync 策略有三种——always（每条命令都刷盘，最安全但最慢）、everysec（每秒刷一次，折中方案，最多丢 1 秒数据）、no（交给操作系统决定，最快但最不安全）。这又是一个性能和安全性之间的取舍。
    - 过关标准：能说出两者的原理、优缺点、以及生产环境通常 RDB + AOF 混合使用的原因
- [ ] **Redis 集群方案** ⭐**本质**：三种方案解决不同问题——主从复制（数据备份 + 读写分离，不解决高可用）、Sentinel 哨兵（在主从基础上加自动故障转移，解决高可用但不解决容量问题）、Cluster（数据分片到多个节点，同时解决高可用和容量问题，16384 个哈希槽分布在多个主节点上）。**为什么需要它**：单节点 Redis 有两个瓶颈——内存容量（单机内存有限）和可用性（单点故障）。哨兵解决了可用性问题但数据仍在一个主节点上，Cluster 同时解决了两个问题。面试中你需要能说出每种方案"解决了什么、没解决什么"，而不是只列举名字。**关联**：Cluster 的哈希槽（Hash Slot）分片本质上是「分而治之」——把 16384 个槽分配到 N 个主节点，每个节点负责一部分槽。客户端通过 CRC16(key) % 16384 计算 Key 属于哪个槽。如果请求到了错误的节点，节点会返回 MOVED 重定向。
    - 过关标准：能说出 主从复制、Sentinel 哨兵、Cluster 三种方案各自解决什么问题、适用什么规模
- [ ] **Redis 分布式锁** ⭐**本质**：基本实现是 `SET key value NX EX timeout`——NX 保证只有一个客户端能设置成功（互斥），EX 设置过期时间防止死锁（客户端崩溃后锁自动释放）。释放锁时用 Lua 脚本保证"先检查再删除"的原子性（防止误删别人的锁）。**为什么需要它**：单机锁（synchronized / ReentrantLock）只在一个 JVM 内有效。微服务架构中多个实例同时运行，需要跨实例的互斥——比如多个实例都在执行定时任务，只能让一个执行。分布式锁是最直接的解决方案。**关联**：分布式锁有几个经典问题——锁续期（业务执行时间超过锁过期时间，Redisson 的 Watch Dog 每 10 秒自动续期）、主从切换丢锁（主节点设了锁还没同步到从节点就宕机了，从节点升为主但没有锁信息，另一个客户端也能加锁成功）、RedLock 方案（向 N 个独立 Redis 实例加锁，过半成功才算加锁成功，但 Martin Kleppmann 对其安全性提出了质疑）。这些问题连接了 CAP 中 AP vs CP 的取舍。
    - 过关标准：能说出 SETNX + 过期时间的基本实现、锁续期问题（Redisson 的 Watch Dog）、Redis 主从切换导致锁丢失的问题、以及 RedLock 的争议
- [ ] **Redis 内存淘汰策略****本质**：当 Redis 内存使用达到 maxmemory 上限时，需要淘汰部分数据。常见策略——allkeys-lru（从所有 Key 中淘汰最近最少使用的）、volatile-lru（只从设了过期时间的 Key 中淘汰 LRU）、allkeys-random（随机淘汰）、noeviction（不淘汰，新写入直接报错）。**为什么需要它**：Redis 内存是有限的，不可能无限存数据。不设淘汰策略（noeviction），内存满了之后所有写操作都会失败。选择哪种策略取决于你的数据特点——如果所有数据都可能被访问，用 allkeys-lru；如果热数据没设过期时间（不想被淘汰），冷数据设了过期时间，用 volatile-lru。**关联**：内存淘汰策略的选择和你的缓存设计紧密相关——如果你依赖 Redis 做分布式锁或计数器（这些 Key 不能被淘汰），就不能用 allkeys-lru（可能把锁 Key 淘汰掉），而应该用 volatile-lru 并确保重要 Key 不设过期时间。
    - 过关标准：能说出 allkeys-lru、volatile-lru 等策略的含义和选择依据
- [ ] **Redis 大 Key / 热 Key 问题** ⭐**本质**：大 Key 是指 Value 特别大的 Key（如一个 List 有几百万元素、一个 String 有几百 MB）。热 Key 是指访问频率极高的 Key（如热搜对应的缓存）。大 Key 的危害——删除时阻塞 Redis（del 是同步操作，大 Key 删除耗时可达秒级）、网络带宽打满、主从同步延迟。热 Key 的危害——单节点过载。**为什么需要它**：这两个问题在生产环境中很常见且影响大——一个大 Key 的 del 操作可能阻塞 Redis 3 秒，期间所有请求都超时。面试中提到你知道并能排查这些问题，说明你有生产环境意识。**关联**：大 Key 的解决方案——拆分（把一个大 List 拆成多个小 List）、用 UNLINK 替代 DEL（异步删除，不阻塞主线程）、用 SCAN 替代 KEYS（渐进式遍历，不阻塞）。热 Key 的解决方案和系统设计 checklist 中的热点 Key 问题一致——本地缓存、Key 打散。发现方式包括 redis-cli --hotkeys、监控工具、业务埋点。
    - 过关标准：能说出大 Key 和热 Key 的危害、发现方式和处理方案

### 2.3 消息队列：RabbitMQ
- [ ] **RabbitMQ 核心模型** ⭐**本质**：RabbitMQ 的消息流转路径是 Producer → Exchange → Binding → Queue → Consumer。Exchange 根据类型和 Routing Key 决定消息路由到哪个 Queue——Direct（精确匹配 Routing Key）、Topic（通配符匹配，如 order.*）、Fanout（广播到所有绑定的 Queue）。**为什么需要它**：Exchange 是 RabbitMQ 区别于简单消息队列的核心设计——它让消息路由变得灵活。一条消息可以根据类型被路由到不同的 Queue 处理（Direct），也可以广播给所有订阅者（Fanout），还可以按模式匹配（Topic）。不同的 Exchange 类型对应不同的业务需求。**关联**：Fanout Exchange 适合事件广播（如"用户注册成功"通知短信服务、邮件服务、积分服务），Topic Exchange 适合按类型分发（如"order.created" 和 "order.paid" 路由到不同处理器）。面试时能说出你的项目用了哪种 Exchange 以及为什么，是加分项。
    - 过关标准：能画出 Producer → Exchange → Binding → Queue → Consumer 的完整流程图，能说出 Direct / Topic / Fanout Exchange 的区别和适用场景
- [ ] **消息可靠性：从生产到消费全链路** ⭐**本质**：保证消息不丢的三个环节——生产端开启 Confirm 模式（Broker 成功接收后回调确认，失败可重发）、Broker 持久化（Exchange、Queue、Message 都设置 durable/persistent，保证 Broker 重启不丢消息）、消费端手动 ACK（消费逻辑执行成功后才确认，失败则 NACK 可以重新入队或进死信队列）。**为什么需要它**：消息队列的价值之一是"保证消息不丢"，但这不是默认行为——不开 Confirm 生产端不知道消息是否到达 Broker，不持久化 Broker 重启就丢消息，不手动 ACK 消息可能在消费端丢失。每个环节都需要显式配置。**关联**：这三个环节对应了消息从"生产 → 存储 → 消费"的完整生命周期。面试官经常会沿着这条链路逐个追问"如果这一步出问题了怎么办"，你需要每个环节都说得清楚。
    - 过关标准：能按链路分析——生产端（Confirm 机制）、Broker（持久化 Exchange + Queue + Message）、消费端（手动 ACK / NACK / Reject），并知道每个环节不处理的后果
- [ ] **死信队列（DLX）****本质**：死信队列是一个普通的 Queue，只不过专门用来接收"死信"。消息变成死信有三种情况——被消费者拒绝（basic.reject / basic.nack 且 requeue=false）、消息 TTL 过期、队列达到最大长度。死信通过 Dead Letter Exchange 路由到死信队列。**为什么需要它**：两个实际用途——①异常消息处理：消费失败的消息进入死信队列，可以后续人工排查或自动重试，避免一条坏消息反复消费阻塞整个队列。②延迟队列实现：利用 TTL + DLX，消息在普通队列等待到过期后自动进入死信队列被消费——变相实现了延迟消费（如 30 分钟未支付自动取消订单）。**关联**：延迟队列在 RabbitMQ 中的实现是"TTL + 死信"这种间接方式。如果需要更精确的延迟消息，RabbitMQ 3.8+ 提供了官方的 Delayed Message Exchange 插件。Kafka 没有原生的延迟队列，通常用定时任务轮询或时间轮实现。
    - 过关标准：能说出消息进入死信的三种条件（被拒绝、TTL 过期、队列满），知道死信队列的实际用途（延迟队列、异常消息处理）
- [ ] **消息幂等性** ⭐**本质**：消息可能被重复投递（网络抖动导致 ACK 丢失，Broker 重新投递），所以消费端必须保证"处理多次和处理一次效果一样"。常见方案——业务唯一 ID + 去重表（消费前查数据库/Redis 看是否已处理过）、数据库唯一约束（重复插入直接失败，不产生副作用）、状态机（只有特定状态才允许操作，已处理的状态不会重复执行）。**为什么需要它**：不做幂等的后果是——重复扣库存、重复扣款、重复发短信。在分布式环境中"恰好一次"（exactly once）几乎不可能实现，实际做法是"至少一次投递 + 消费端幂等"来等效达到恰好一次的效果。**关联**：幂等不只是消息队列的问题，接口级别的幂等同样重要（第四层的接口幂等性设计）。同样的幂等方案可以复用——唯一 ID、数据库约束、状态机在消息消费和 HTTP 接口中都适用。
    - 过关标准：能说出为什么消息会重复（网络抖动、Consumer 处理后 ACK 失败），以及你会用什么方案保证幂等（业务唯一 ID + 去重表 / Redis Set）
- [ ] **RabbitMQ 高可用：镜像队列 / Quorum Queue****本质**：镜像队列（Classic Mirrored Queue）将 Queue 的数据复制到集群中的多个节点，主节点挂了可以从镜像节点接管。Quorum Queue 是 RabbitMQ 3.8+ 引入的新方案，基于 Raft 协议实现多副本一致性，比镜像队列更可靠、更高效。**为什么需要它**：单节点 RabbitMQ 是单点故障——节点挂了队列中未消费的消息全部丢失。高可用方案保证即使部分节点宕机，消息也不会丢且服务可以继续运行。Quorum Queue 是目前推荐的方案，镜像队列正在逐步被取代。**关联**：Quorum Queue 使用 Raft 协议——这和 Nacos 的 CP 模式（也用 Raft）以及 etcd 是同一个共识协议。Raft 保证过半节点写入成功才算成功，所以 3 个节点最多容忍 1 个宕机，5 个节点最多容忍 2 个宕机。
    - 过关标准：知道镜像队列的基本原理，了解 Quorum Queue 是更新的高可用方案（基于 Raft）

### 2.4 微服务调用：Spring Cloud 相关
- [ ] **服务间调用方式：Feign / RestTemplate / gRPC****本质**：RestTemplate 是手动构造 HTTP 请求（指定 URL、参数、序列化方式），灵活但代码冗长。Feign 是声明式 HTTP 客户端（写一个接口加注解就能调用远程服务），底层通过动态代理生成实现类，自动整合 Ribbon 做负载均衡。gRPC 是基于 HTTP/2 + Protobuf 的高性能 RPC 框架，比 JSON over HTTP 快很多但使用复杂度更高。**为什么需要它**：微服务之间需要通信，通信方式的选择影响开发效率和性能。Feign 最常用（开发效率高、和 Spring Cloud 集成好），gRPC 在高性能和跨语言场景下更合适（如 Go 服务和 Java 服务之间通信）。**关联**：Feign 的底层其实是 RestTemplate + Ribbon（负载均衡）+ Hystrix/Sentinel（容错）的封装。理解这一点后，Feign 调用慢或出问题时你就知道该排查哪一层——是 HTTP 调用本身慢？还是负载均衡策略有问题？还是触发了熔断？
    - 过关标准：能说出 Feign 的底层原理（动态代理 + Ribbon 负载均衡），知道 gRPC 在什么场景下更合适（高性能、跨语言）
- [ ] **服务雪崩与容错** ⭐**本质**：微服务调用链中，下游服务变慢会导致上游线程被阻塞在等待响应上——上游线程池耗尽后自己也变慢——再上游也受影响——整个调用链逐级崩溃，这就是雪崩效应。**为什么需要它**：微服务的调用链可能很长（A → B → C → D），只要链条中有一个环节出问题，就可能拖垮整条链路。雪崩效应是微服务架构最大的稳定性风险，必须有主动防护机制。**关联**：防护手段包括——超时（不无限等待）、熔断（连续失败后停止调用）、线程池隔离（每个下游服务用独立的线程池，一个服务慢不影响其他服务）、限流（从源头控制流量）。Spring Cloud Sentinel 和 Resilience4j 提供了这些能力。这连接了系统设计 checklist 中的高可用设计模式。
    - 过关标准：能解释调用链雪崩的过程（A → B → C，C 慢了导致 B 线程池满，进而 A 也挂），以及 Sentinel / Resilience4j 如何防护
- [ ] **分布式链路追踪****本质**：在微服务架构中，一个用户请求可能经过十几个服务。链路追踪的核心是给每个请求分配一个全局唯一的 TraceId，每个服务内部的处理分配一个 SpanId。TraceId 在所有服务间传播，最终可以把一个请求经过的所有服务、每段耗时串联起来。**为什么需要它**：当一个请求变慢或出错时，你需要知道"慢在哪个服务、哪个环节"。没有链路追踪，你只能在每个服务的日志里搜索，效率极低。有了链路追踪，一个 TraceId 就能看到整条调用链路和每段的耗时。**关联**：SkyWalking（自动探针、对代码无侵入）和 Zipkin（需要手动埋点或整合 Sleuth）是最常用的链路追踪系统。链路追踪的 TraceId 传播机制和 ThreadLocal 相关——在异步调用中 TraceId 可能丢失，需要用 TransmittableThreadLocal 或框架的上下文传播机制来解决。
    - 过关标准：知道 TraceId / SpanId 的传播机制，了解 SkyWalking / Zipkin 等工具的作用

## 第三层：分布式事务 —— 理论 + 方案选型
> 你说理论有了解但没实际用过。面试中这块是高频考点，至少需要能说清楚方案和取舍。
>

- [ ] **2PC（两阶段提交）****本质**：由一个协调者管理——第一阶段（Prepare）协调者询问所有参与者"能不能提交"，参与者执行事务但不提交，回复 Yes/No。第二阶段（Commit）如果所有参与者都回复 Yes，协调者通知所有参与者提交；任何一个回复 No 或超时，协调者通知所有参与者回滚。**为什么需要它**：2PC 是分布式事务最基础的协议，理解它才能理解后续方案为什么要改进它。但 2PC 本身问题很大——同步阻塞（Prepare 阶段所有参与者锁定资源等待 Commit，期间资源不可用）、单点故障（协调者挂了所有参与者卡在等待状态）、数据不一致（Commit 阶段部分参与者收到了提交请求部分没收到）。**关联**：2PC 的问题催生了 TCC、本地消息表等替代方案。MySQL 的 XA 事务就是 2PC 的实现，但因为性能问题在互联网场景中很少使用。
    - 过关标准：能画出 Prepare → Commit 的流程，能说出它的致命问题（同步阻塞、单点故障、数据不一致风险）
- [ ] **TCC（Try-Confirm-Cancel）** ⭐**本质**：把事务拆成三步——Try（预留资源，如冻结库存、冻结余额）、Confirm（确认执行，如扣除冻结的库存和余额）、Cancel（取消预留，如解冻库存和余额）。每一步都是独立的本地事务。**为什么需要它**：TCC 相比 2PC 最大的优势是性能——Try 阶段不锁定数据库行，而是用业务层面的"冻结"来预留资源，其他请求仍然可以操作未冻结的资源。但代价是业务侵入大——你需要为每个操作实现 Try/Confirm/Cancel 三个方法。**关联**：TCC 有两个经典问题——空回滚（Try 没执行但 Cancel 被调用了，Cancel 需要判断并跳过）和悬挂（Cancel 先于 Try 执行到达了，后来的 Try 不应该再执行）。这些问题的解决方案通常是在数据库中记录事务状态。
    - 过关标准：能用一个业务场景（如转账）说出 Try / Confirm / Cancel 各阶段做什么，知道 TCC 的优势（性能好）和劣势（业务侵入大、空回滚、悬挂问题）
- [ ] **本地消息表 / 事务消息** ⭐**本质**：核心思路是——在本地事务中同时写入业务数据和一条消息记录（放在同一个数据库的同一个事务中，保证原子性）。然后用定时任务或消息队列把消息投递给下游服务。下游消费成功后更新消息状态为"已消费"。**为什么需要它**：这是最实用、最常用的分布式事务方案。它不需要引入复杂的框架，只需要一张"消息表"和一个定时任务就能实现。本质上是把分布式事务拆解为"本地事务 + 最终一致性"——上游保证本地数据和消息的原子性，下游通过重试保证最终消费成功。**关联**：RocketMQ 的事务消息是这个方案的升级版——把"本地消息表 + 定时任务"封装成了 MQ 自身的能力（发送半消息 → 执行本地事务 → 提交/回滚消息）。RabbitMQ 没有原生事务消息，所以你的项目如果用 RabbitMQ 做分布式事务，本地消息表是最自然的选择。
    - 过关标准：能画出本地消息表的流程（本地事务写业务数据 + 消息表 → 定时任务/MQ 投递 → 下游消费），能说出它是最终一致性方案
- [ ] **Saga 模式****本质**：把一个长事务拆成多个本地事务，每个本地事务都有一个对应的补偿操作。按顺序依次执行各个本地事务，如果某一步失败，则逆序执行之前步骤的补偿操作。编排方式有两种——编排式（由一个协调器统一调度所有步骤）和协调式/事件式（每个步骤完成后发事件触发下一步）。**为什么需要它**：Saga 适合长事务场景（如：旅行预订 = 订机票 + 订酒店 + 订租车，每个都是独立的服务）。TCC 需要每个操作都能"冻结资源"，但有些外部服务（如第三方支付）你没法要求它实现 Try/Cancel 接口。Saga 只要求每个步骤有"正向操作"和"补偿操作"，更通用。**关联**：Saga 的补偿操作必须是幂等的（可能被执行多次），这又连接了幂等性设计。编排式 Saga 适合流程简单的场景（不超过 5 步），流程复杂时用协调式更灵活（但调试困难）。
    - 过关标准：能说出 Saga 的正向操作和补偿操作的编排方式（编排式 vs 协调式），知道它适合长事务场景
- [ ] **Seata 框架（了解即可）****本质**：Seata 是阿里开源的分布式事务框架，支持 AT / TCC / Saga 三种模式。AT 模式最简单——对业务无侵入，在执行 SQL 前自动生成回滚 SQL（undo log），如果事务需要回滚就执行 undo log 还原数据。**为什么需要它**：AT 模式是"低成本接入分布式事务"的方案——不需要改业务代码，只需要加注解。但 AT 模式依赖数据库锁实现隔离，高并发下性能不如 TCC。了解 Seata 在面试中说明你知道有现成框架，但也要能说出它的局限性。**关联**：Seata AT 模式的 undo log 本质上是"记录修改前的数据，失败时回放"——和数据库的事务回滚机制思路一样，只是把它扩展到了分布式场景。
    - 过关标准：知道 Seata 的 AT / TCC / Saga 三种模式，能说出 AT 模式的基本原理（自动生成回滚 SQL）

## 第四层：分布式环境下的经典问题
> 这些是面试中常见的横切问题，不属于某个组件但经常被问到。
>

- [ ] **分布式 ID 生成** ⭐**本质**：在分布式系统中需要全局唯一的 ID。几种方案——UUID（简单但无序、太长、不利于索引）、数据库自增（简单但有单点瓶颈）、号段模式（每次从数据库取一段 ID 缓存在内存中，减少数据库访问）、Snowflake 雪花算法（64 位 ID = 时间戳 + 机器 ID + 序列号，有序且高性能）。**为什么需要它**：分库分表后不能用数据库自增 ID（多个库各自自增会重复），需要一个全局唯一的 ID 生成方案。ID 是否有序影响了数据库的写入性能——有序 ID 顺序写入 B+ 树效率高，无序 ID（如 UUID）会导致大量页分裂。**关联**：Snowflake 的时间戳部分保证了 ID 大致有序，机器 ID 部分保证了多节点不冲突。但 Snowflake 依赖机器时钟——如果时钟回拨可能生成重复 ID。美团的 Leaf 和百度的 UidGenerator 是 Snowflake 的改进版本，面试中可以提及。
    - 过关标准：能说出 UUID、Snowflake、数据库自增、号段模式各自的优缺点，能根据场景选择方案（如：订单号用 Snowflake，短链用自增 + 62进制编码）
- [ ] **分布式 Session****本质**：单机应用的 Session 存在 Tomcat 内存中，多实例部署后请求可能被路由到不同实例，导致 Session 丢失。两种主流方案——Redis 集中存储 Session（所有实例从 Redis 读写 Session，Spring Session 提供了开箱即用的方案）、JWT 无状态化（把用户信息编码在 Token 中，服务端不存 Session）。**为什么需要它**：微服务和多实例部署是常态，Session 问题必须解决。Redis 方案实现简单但 Redis 是额外的依赖和故障点。JWT 方案无状态但 Token 无法主动失效（用户改密码后旧 Token 仍有效）、Token 过大增加网络开销。**关联**：JWT 的无状态性是「水平扩展」的前提——不依赖服务端存储，任何实例都能处理任何请求。Redis Session 方案本质上是把状态"外置"到了共享存储——和无状态服务 + 外部存储的架构思想一致。
    - 过关标准：能说出 Session 共享的几种方案（Redis 集中存储、JWT 无状态化），知道各自的优缺点
- [ ] **接口幂等性设计** ⭐**本质**：幂等是指同一个操作执行多次和执行一次效果一样。在分布式环境中，网络抖动、重试、消息重复等都可能导致同一个请求被执行多次。写操作（创建订单、扣款、扣库存）必须做幂等处理。**为什么需要它**：用户双击提交按钮、前端重试、网关重试、消息重复投递——这些在生产环境中经常发生。如果扣款接口不幂等，用户可能被多扣一次钱；如果下单接口不幂等，用户可能重复下单。**关联**：常见幂等方案——Token 机制（请求前先获取唯一 Token，提交时带上 Token，服务端验证 Token 只能用一次）、唯一索引（数据库层面防止重复插入）、状态机（只有特定状态才能执行操作）。这些方案和消息队列的消费幂等是同一套思路（分布式 checklist 2.3 的消息幂等性），可以复用。
    - 过关标准：能说出哪些接口需要幂等（支付、扣库存等写操作），能说出 Token 机制、唯一索引、状态机等方案
- [ ] **分布式定时任务****本质**：单机 Cron 定时任务在多实例部署时会被所有实例同时执行——导致任务重复执行。分布式定时任务框架解决两个问题——①保证同一时刻只有一个实例执行任务 ②支持任务分片（大任务拆成多个子任务分配给不同实例并行执行）。**为什么需要它**：很多业务有定时任务需求（定时对账、定时清理过期数据、定时发报告），多实例下不能重复执行。最简单的方案是用分布式锁（Redis 锁）让只有一个实例执行，但更完善的方案是用专门的分布式调度框架。**关联**：XXL-Job 和 ElasticJob 是最常用的两个框架。XXL-Job 有管理后台，支持任务管理、日志查看、失败重试。分布式锁方案更轻量但缺少管理和监控能力。选择取决于你的场景复杂度。
    - 过关标准：知道单机 Cron 的问题（多实例重复执行），了解 XXL-Job / ElasticJob 等分布式调度方案
