# 系统设计（System Design）· 面试导向 Checklist
> 系统设计面试考的不是标准答案，而是你的思考过程和权衡能力。面试官想看到的是：你能不能在约束条件下做出合理的取舍，并说清楚为什么。  
带 ⭐ 的是面试高频考点。每个模块末尾附有经典面试场景题。  
每个知识点包含「本质」「为什么需要它」「关联」三个维度，帮助你从根上理解而不是死记硬背。
>

---

## 第一层：系统设计方法论 —— 面试中的思考框架
> 很多人系统设计答不好，不是知识不够，而是没有结构化的思考方式。面试官给你一个开放题，你要有一套清晰的推进流程。
>

- [ ] **系统设计面试的标准流程** ⭐**本质**：系统设计面试是一个结构化的讨论过程，不是让你一上来就画架构图。标准流程是五步——①需求澄清（功能需求 + 非功能需求）②容量估算（QPS、存储、带宽）③高层架构设计（画核心组件和数据流）④核心模块详细设计 ⑤瓶颈分析与扩展方案。**为什么需要它**：面试官给的是模糊需求（如"设计一个短链接服务"），第一步不是设计而是提问——日活多少？读写比？URL 有效期？不澄清就开始设计，大概率方向跑偏。这个流程让你在 40 分钟内有节奏地推进，不遗漏关键环节。**关联**：这五步中每一步都对应后面的具体知识——容量估算需要你知道各组件的性能基线，高层架构需要你懂组件选型，详细设计需要你懂数据库/缓存/消息队列，瓶颈分析需要你懂高可用和高性能模式。
    - 过关标准：拿到任何设计题，能按这个流程推进而不慌：
        1. 需求澄清（功能需求 + 非功能需求）
        2. 容量估算（QPS、存储、带宽）
        3. 高层架构设计（画出核心组件和数据流）
        4. 核心模块详细设计
        5. 瓶颈分析与扩展方案
- [ ] **容量估算（Back-of-the-envelope Estimation）** ⭐**本质**：用简单的数学快速估算系统需要承受的负载——从 DAU（日活用户数）推算出 QPS（每秒请求数）→ 峰值 QPS（通常是平均值的 2-5 倍）→ 存储量（每条数据多大 × 每天多少条 × 保留多久）→ 带宽（QPS × 单次响应大小）。**为什么需要它**：容量估算决定了你的技术选型。QPS 1000 和 QPS 100 万是完全不同的架构——前者一台 MySQL 就够了，后者需要缓存、分库分表、消息队列等一系列方案。不做估算就设计，要么过度设计浪费资源，要么设计不足撑不住。**关联**：你需要记住一些常见的性能基线数字——单机 MySQL 约 1K-5K QPS（看复杂度），单机 Redis 约 10W QPS，单机 Nginx 约 5W+ 并发连接。这些数字帮你快速判断"需不需要加缓存""需不需要分库"。
    - 过关标准：能快速估算 DAU → QPS → 峰值 QPS → 存储量 → 带宽，熟悉常见数量级（如单机 MySQL 约 1K-5K QPS，Redis 约 10W QPS）
- [ ] **非功能需求的关键维度****本质**：功能需求定义"系统做什么"，非功能需求定义"系统做到什么程度"——高可用（99.9% 还是 99.99%？）、高性能（RT 要求多少？）、可扩展性（未来流量翻 10 倍架构还能撑吗？）、一致性（能容忍多长时间的数据不一致？）、数据持久性（数据绝对不能丢？还是可以接受少量丢失？）。**为什么需要它**：面试中主动提出非功能需求，展示的是你的"全局视野"。大部分候选人只关心功能怎么实现，忽略了可用性、一致性等约束。而恰恰是这些约束决定了架构的复杂度和取舍方向。**关联**：非功能需求直接对应核心原理图谱中的「权衡」——高可用和一致性之间的取舍就是 CAP，高性能和复杂度之间的取舍决定了你是否需要引入缓存和消息队列。
    - 过关标准：面对任何系统设计题，能主动提出并讨论：高可用、高性能、可扩展性、一致性要求、数据持久性

### 面试场景练习
- [ ] 拿到"设计一个 XX 系统"后，能在 5 分钟内完成需求澄清和容量估算，不被面试官催促

## 第二层：核心组件设计能力
> 系统设计的本质是组合这些组件来解决问题。你需要知道每个组件的能力边界和适用场景。
>

### 2.1 数据库设计与优化
- [ ] **Schema 设计原则** ⭐**本质**：Schema 设计的核心决策是范式化 vs 反范式化。范式化（拆表、消除冗余）保证数据一致性但查询需要 JOIN；反范式化（允许冗余、合并表）查询快但更新时需要同步多处数据。选择取决于读写比和查询模式。**为什么需要它**：数据库是大部分系统的核心，Schema 设计决定了后续所有查询的性能上限。一旦上线数据量上来后再改 Schema，成本极高（需要迁移数据）。面试中考的不是你会不会建表，而是你能不能根据业务特点做出合理的设计取舍。**关联**：读多写少的场景（如商品详情）倾向反范式化减少 JOIN；写多的场景（如交易流水）倾向范式化保证一致性。这连接了核心原理图谱的「权衡」——没有绝对好的设计，只有适合场景的设计。
    - 过关标准：能根据读写比例、查询模式来设计表结构，知道什么时候该反范式化
- [ ] **索引设计与优化** ⭐**本质**：MySQL 默认使用 B+ 树索引。B+ 树的特点是——所有数据存在叶子节点、叶子节点之间有链表连接（方便范围查询）、树高通常 3-4 层（千万级数据也只需 3-4 次磁盘 IO）。联合索引按最左前缀原则生效——索引 (a, b, c) 可以服务 a、(a,b)、(a,b,c) 的查询，但不能服务 b 或 (b,c) 的查询。**为什么需要它**：索引是数据库优化最直接、收益最高的手段。一个合适的索引可以把查询从全表扫描（几秒）变成索引查找（几毫秒）。但索引不是越多越好——每个索引在写入时都需要维护，过多索引会拖慢写性能。**关联**：索引设计需要结合 EXPLAIN 来验证（性能优化 checklist 3.1），索引失效的场景也需要了解。覆盖索引（查询字段都在索引中）可以避免回表，进一步提升查询性能。
    - 过关标准：能解释 B+ 树索引原理，能根据慢查询分析决定加什么索引，理解联合索引的最左前缀原则，知道什么情况下索引会失效
- [ ] **读写分离****本质**：将数据库的读请求和写请求分发到不同的实例——主库负责写，从库负责读。主库通过 binlog 将数据变更同步到从库。这样读操作可以被多个从库分担，提升整体读性能。**为什么需要它**：大部分业务系统是读多写少（读写比 10:1 甚至更高）。单库既要写又要读，读请求会抢占写操作的资源。读写分离让读操作分散到多个从库，写操作不受影响。**关联**：读写分离引入了主从延迟问题——刚写入的数据可能在从库上还查不到。这就是「可见性与一致性」问题在数据库层面的表现。应对方案包括强制读主（对刚写入的数据查主库）、半同步复制（主库等至少一个从库确认后才返回成功）。
    - 过关标准：能说出主从复制的原理、延迟问题及应对策略（强制读主、半同步复制）
- [ ] **分库分表** ⭐**本质**：当单表数据量过大（通常千万级以上），索引优化和读写分离都无法解决查询变慢的问题时，需要把数据拆分到多个库或表中。垂直拆分按业务维度拆（如订单库、用户库分开），水平拆分按数据维度拆（如按 user_id 取模分到不同表）。**为什么需要它**：MySQL 单表超过几千万行后，B+ 树层高增加、索引维护开销增大，查询性能会显著下降。分库分表是应对数据量增长的最终手段，但它引入的问题也最多——所以不到万不得已不要分。**关联**：分库分表是「分而治之」思想在数据层的应用。但拆分后你会遇到一系列新问题——分片键怎么选（决定数据分布是否均匀）、跨分片查询怎么办（比如按 user_id 分表后要按手机号查）、全局唯一 ID 怎么生成（分布式 checklist 的分布式 ID）、扩容怎么做（从 16 张表扩到 32 张表要重新分配数据）。
    - 过关标准：能说出什么时候需要分（数据量级参考），水平拆分 vs 垂直拆分的区别，分片键怎么选，分了之后带来的问题（跨分片查询、全局 ID、扩容）

### 面试场景题
- [ ] "你的订单表数据量到了 5000 万，查询变慢了，怎么优化？" —— 能从索引优化 → 读写分离 → 分库分表逐步展开
- [ ] "用户表按 user_id 分了 16 张表，现在要按手机号查用户怎么办？" —— 能说出映射表、ES 等方案

### 2.2 缓存设计
- [ ] **缓存使用场景与模式** ⭐**本质**：缓存的四种经典模式——Cache Aside（应用自己管理缓存：读时先查缓存未命中再查 DB 并写缓存，写时先更新 DB 再删缓存）、Read Through（缓存层自动从 DB 加载）、Write Through（写操作同时写缓存和 DB）、Write Behind（写操作只写缓存，异步刷到 DB）。**为什么需要它**：缓存不是"把数据扔到 Redis 里"这么简单。不同模式在一致性、性能、复杂度之间有不同的取舍。Cache Aside 最常用（简单、可控），Write Behind 性能最好但一致性最差（缓存挂了数据可能丢失）。面试中你需要说出"我选了 XX 模式，因为我的场景需要 XX"。**关联**：四种模式本质上是在一致性和性能之间做取舍——Write Behind 最快但最不一致，Write Through 最一致但最慢。参见核心原理图谱的「可见性与一致性」和「权衡」。
    - 过关标准：能说出 Cache Aside / Read Through / Write Through / Write Behind 四种模式的区别和适用场景
- [ ] **缓存三大问题：击穿、穿透、雪崩** ⭐**本质**：缓存击穿——某个热点 Key 过期的瞬间，大量请求同时打到 DB（一个热点的"击穿"）。缓存穿透——请求的数据在缓存和 DB 中都不存在，每次都穿过缓存打到 DB（恶意攻击或参数错误）。缓存雪崩——大量 Key 同时过期，或者缓存服务宕机，导致请求全部涌向 DB（大面积的"雪崩"）。**为什么需要它**：这三个问题是缓存使用中最常见的"反模式"，一旦发生可能直接打挂数据库，导致全站不可用。面试中几乎必问，而且面试官不要你背定义——他要你说出每种问题的具体解决方案和取舍。**关联**：击穿的解决方案包括互斥锁重建缓存（setnx）、热点数据永不过期（后台异步更新）。穿透的方案包括缓存空值、布隆过滤器。雪崩的方案包括过期时间加随机值、多级缓存、缓存集群高可用。每种方案都有取舍，比如布隆过滤器有误判率、缓存空值会占用内存。
    - 过关标准：不是背定义，而是能说出每种问题的触发条件、后果、以及 2-3 种解决方案，并能说清楚方案的取舍
- [ ] **缓存与数据库的一致性** ⭐**本质**：Cache Aside 模式下的经典问题——"先更新 DB 再删缓存"可能出现短暂的不一致（删缓存前有请求读到旧数据）但概率低；"先删缓存再更新 DB"可能出现严重不一致（删缓存后另一个请求读到 DB 旧值并写回缓存，此后新值再也不会被读到）。**为什么需要它**：缓存一致性是面试最常追问的话题之一。面试官想看的是你能不能分析出不同方案在并发场景下的时序问题，而不是只会背"先更新 DB 再删缓存"。**关联**：这本质上是「可见性与一致性」在缓存层的表现——和 JMM 中线程看到旧值的问题是同构的。延迟双删（先删缓存 → 更新 DB → 等一段时间再删缓存）是一种兼顾方案。要求强一致的场景可以通过 Canal 监听 binlog 异步更新缓存，实现最终一致。
    - 过关标准：能分析"先更新 DB 再删缓存"和"先删缓存再更新 DB"各自的问题，知道延迟双删的思路，能说出最终一致性方案
- [ ] **热点 Key 问题****本质**：当某个 Key 的访问量极高（如热搜、爆款商品），所有请求都打到同一个 Redis 节点上，导致该节点成为瓶颈甚至崩溃。本质是"单点过热"问题。**为什么需要它**：热点 Key 在突发流量（如明星热搜、秒杀活动）时很容易出现。单个 Redis 节点通常能扛 10W QPS，但一个热点 Key 可能瞬间带来 50W+ 的请求。如果不处理，Redis 节点崩溃后缓存击穿和雪崩会接踵而至。**关联**：解决方案是「分而治之」——本地缓存（Caffeine / Guava Cache）挡掉大部分读请求、Key 打散（把一个 Key 拆成 key_1, key_2, ... key_N 分散到多个节点）、限流保护。这和 LongAdder 分段思想、分库分表的分片思想是同一个原理。
    - 过关标准：能说出热点 Key 的发现方式和应对方案（本地缓存、Key 打散、限流）

### 2.3 消息队列设计
- [ ] **消息队列的核心价值** ⭐**本质**：消息队列的三大核心价值——异步（把不需要立即完成的操作放到后面执行，降低主流程 RT）、解耦（上游不需要知道下游有谁，只管发消息，下游各自消费）、削峰（流量高峰时消息在队列中缓冲，消费者按自己的能力匀速处理）。**为什么需要它**：没有消息队列的系统，上下游强耦合——下单要同步调用短信服务、库存服务、物流服务，任何一个挂了订单就失败。引入消息队列后，下单只需写库 + 发消息，后续操作异步执行，既快又稳。**关联**：异步化提升了性能但引入了最终一致性问题（参见核心原理图谱「异步化与解耦」）。消息队列的可靠性保障（不丢消息、不重复消费）是使用消息队列后必须解决的配套问题。
    - 过关标准：能说出三大价值（异步、解耦、削峰）并各举一个业务场景
- [ ] **消息可靠性保障** ⭐**本质**：消息在三个环节可能丢失——生产端发送后 Broker 没收到（网络问题）、Broker 收到后未持久化就崩溃（宕机丢数据）、消费端消费后未确认（消费者崩溃）。要保证"不丢消息"，每个环节都需要有对应的机制。**为什么需要它**：消息丢失在业务上可能意味着——用户支付成功但没有发货通知、库存扣了但订单没创建。这些问题比系统崩溃更严重，因为数据不一致往往很难发现和修复。**关联**：生产端的 Confirm 机制（RabbitMQ）或 acks=all（Kafka）保证 Broker 收到了消息。Broker 的持久化（durable queue + persistent message）保证宕机不丢。消费端的手动 ACK 保证消息处理完了才确认。三个环节对应消息生命周期的三个阶段。
    - 过关标准：能沿着消息生命周期分析丢失风险——生产端（确认机制）、Broker（持久化）、消费端（手动 ACK），并说出每个环节的保障方案
- [ ] **重复消费与幂等性** ⭐**本质**：在保障"不丢消息"的前提下，消息可能被重复投递——比如消费者处理完消息后 ACK 失败，Broker 认为没消费会重新投递。所以消费端必须做幂等处理——同一条消息处理多次和处理一次的结果一样。**为什么需要它**：不做幂等会导致——库存多扣一次、用户被多收一次钱、重复发短信。在分布式环境中，"恰好一次"（exactly once）几乎不可能实现，但"至少一次 + 幂等"可以等效达到同样效果。**关联**：幂等方案包括——唯一消息 ID + 去重表（消费前查是否处理过）、数据库唯一约束（重复插入直接报错）、状态机（只有特定状态才能执行操作，已处理的状态不会重复执行）。这些方案也适用于接口级别的幂等设计（分布式 checklist 第四层）。
    - 过关标准：能解释为什么消息可能重复（网络抖动、重试），能说出 2-3 种幂等方案（唯一 ID、数据库唯一约束、状态机）
- [ ] **消息顺序性****本质**：全局有序要求所有消息严格按发送顺序消费，代价极高（只能单分区单消费者）。分区有序只保证同一分区内有序，不同分区之间无序——通常够用（如同一个订单的消息发到同一个分区，保证该订单内的操作有序）。**为什么需要它**：某些业务场景要求消息顺序——比如同一个订单的"创建 → 支付 → 发货"消息，如果乱序消费会导致状态机混乱。但全局有序性能太差，大部分场景用分区有序就足够了。**关联**：RabbitMQ 通过单个 Queue 单个 Consumer 保证顺序（但牺牲并发）。Kafka 通过 Partition 保证分区内有序，用 Key（如订单 ID）决定消息进哪个 Partition——同一个 Key 的消息一定在同一个 Partition 内。
    - 过关标准：知道全局有序和分区有序的区别，能说出 RabbitMQ 和 Kafka 分别如何保证顺序
- [ ] **消息积压处理****本质**：当消费速度跟不上生产速度时，消息在队列中不断堆积。如果不处理，队列占满磁盘可能导致 Broker 崩溃，或者消息超过 TTL 被丢弃。**为什么需要它**：消息积压是生产环境的常见问题——消费者出 Bug、下游服务变慢、突发流量超过消费能力都可能导致积压。你需要有排查思路和应急手段。**关联**：应急方案包括——临时增加消费者实例（水平扩容，前提是 Partition 数够）、创建临时队列分流消息、对非关键消息降级丢弃。根本解决需要找到消费慢的原因——是消费逻辑有 Bug？是下游服务慢？还是单条消息处理时间过长？
    - 过关标准：能说出消息积压的排查思路和应急方案（扩消费者、临时队列转移、降级丢弃）

### 面试场景题
- [ ] "用户下单后要发短信、扣库存、通知物流，你怎么设计？如果消息丢了怎么办？"
- [ ] "消费者处理消息太慢，队列积压了 100 万条消息，怎么紧急处理？"

### 2.4 负载均衡与网关
- [ ] **负载均衡策略****本质**：负载均衡的核心目的是把请求均匀分散到多个服务实例上。常见策略——轮询（最简单，逐个分发）、加权轮询（性能好的实例分配更多请求）、一致性哈希（相同用户/请求路由到相同实例，适合有状态场景或利用本地缓存）、最少连接（优先分给当前连接数最少的实例）。**为什么需要它**：多实例部署后必须有负载均衡，否则单个实例过载而其他实例空闲。策略的选择要根据场景——无状态服务用轮询就够了，需要会话保持的用一致性哈希，实例性能不一致的用加权轮询。**关联**：一致性哈希在缓存场景中也大量使用——Redis Cluster 的数据分片就是一致性哈希的变种（哈希槽）。理解了一致性哈希，就同时理解了负载均衡和缓存分片两个场景。
    - 过关标准：能说出轮询、加权轮询、一致性哈希等策略的适用场景
- [ ] **API 网关的职责****本质**：API 网关是微服务架构中所有外部请求的统一入口。核心职责包括——路由（根据 URL 路由到不同的后端服务）、鉴权（统一的身份认证和权限检查）、限流（在入口处控制流量）、熔断（某个下游服务出问题时快速失败）、日志和监控（统一的请求记录和指标采集）。**为什么需要它**：没有网关的话，每个微服务都要自己实现鉴权、限流、日志——重复工作且不一致。网关把这些横切关注点统一处理，后端服务只需关注业务逻辑。同时网关也是系统安全的第一道防线。**关联**：网关的限流和熔断能力连接了高可用设计（第三层）。Spring Cloud Gateway 是你的技术栈中最自然的网关选型，底层基于 Netty 非阻塞 IO。
    - 过关标准：能说出网关的核心功能（路由、鉴权、限流、熔断）及为什么微服务架构需要网关

## 第三层：高可用与高性能设计模式
> 面试中面试官追问最多的方向："如果挂了怎么办？""如果流量翻10倍怎么办？"
>

### 3.1 高可用设计
- [ ] **冗余与故障转移** ⭐**本质**：高可用的第一原则是冗余——任何单点都可能挂，所以关键组件都要有备份。服务冗余（多实例部署）、数据冗余（主从复制、多副本）、跨机房冗余（异地容灾）。故障转移是指当主节点挂了后自动切换到备份节点继续服务。**为什么需要它**：单点故障（Single Point of Failure）是系统可用性的最大威胁。任何"只有一份"的东西都是风险——一个 MySQL 实例、一个 Redis 节点、一个部署实例。冗余和故障转移是消除单点故障的基本手段。**关联**：冗余引入了数据一致性问题——多个副本之间怎么保持同步？这又回到了核心原理图谱的「可见性与一致性」。Redis Sentinel 的故障转移、MySQL 主从切换、Nacos 的健康检查都是故障转移的具体实现。
    - 过关标准：能说出服务冗余（多实例）、数据冗余（主从/多副本）的基本思路，知道故障转移的触发机制
- [ ] **限流** ⭐**本质**：限流是在系统入口处控制请求速率，超出阈值的请求直接拒绝。两种主流算法——令牌桶（按固定速率生成令牌，请求需要获取令牌才能执行，允许一定程度的突发流量）和滑动窗口（统计时间窗口内的请求数，超出阈值拒绝，更精确但不允许突发）。**为什么需要它**：系统的处理能力是有上限的。如果不限流，超过能力的请求会导致系统过载——RT 急剧增加、错误率飙升、最终整个系统不可用。限流的本质是"牺牲部分请求来保全整体系统"。**关联**：限流应该加在哪一层？入口网关做全局限流保护整体系统，服务级别做局部限流保护单个服务。限流粒度可以按 IP、按用户、按接口——粒度选择取决于业务需求。分布式限流（多实例共享计数器）通常用 Redis + Lua 实现。
    - 过关标准：能说出令牌桶和滑动窗口算法的思路，知道限流应该加在哪一层（网关 vs 服务），能根据场景选择合适的限流粒度
- [ ] **熔断与降级** ⭐**本质**：熔断器有三个状态——Closed（正常通过）、Open（直接拒绝，不再调用下游）、Half-Open（放过少量请求试探下游是否恢复）。当对下游的调用失败率超过阈值时，熔断器打开，快速返回降级结果。降级是熔断后的具体处理策略——返回默认值、走缓存、功能裁剪。**为什么需要它**：如果下游服务挂了，你的服务还一直调它，线程会被阻塞在等待超时上——线程池被耗尽后你的服务也挂了，这就是服务雪崩。熔断的作用是"止损"——及早发现下游不可用，快速失败，不让故障扩散。**关联**：熔断和降级是「快速失败与兜底」的核心实现。在你的技术栈中，Spring Cloud Sentinel 或 Resilience4j 提供了开箱即用的熔断器。降级方案需要提前设计——面试官会追问"你的降级方案是什么？返回什么给用户？"
    - 过关标准：能解释熔断器的三态（Closed → Open → Half-Open），知道什么时候触发降级，降级方案怎么设计（返回默认值、走缓存、功能裁剪）
- [ ] **超时与重试设计****本质**：每个远程调用都必须设置超时时间——不设超时意味着调用方可能无限期等待，最终耗尽资源。重试是超时或失败后自动再试一次，但重试必须配合幂等（否则重复执行有副作用）和退避策略（重试间隔递增，避免同时重试打崩下游）。**为什么需要它**：分布式系统中网络是不可靠的，偶发的超时和失败是常态。合理的超时和重试可以大幅提升成功率。但不合理的重试（无限重试、同时重试）反而会加剧问题——下游本来就扛不住，你还不停重试，等于 DDoS 自己的系统。**关联**：重试必须配合幂等（分布式 checklist 第四层），退避策略常用指数退避（如 100ms → 200ms → 400ms）。超时 + 重试 + 幂等 + 熔断组合起来形成了完整的容错链路。
    - 过关标准：能解释为什么要设超时、重试为什么要配合幂等、为什么要有重试上限和退避策略

### 3.2 高性能设计
- [ ] **水平扩展 vs 垂直扩展****本质**：垂直扩展（Scale Up）是加 CPU、加内存——简单但有上限。水平扩展（Scale Out）是加机器——理论上无上限但前提是服务无状态。**为什么需要它**：面试官问"流量翻 10 倍怎么办"，核心答案就是水平扩展。但水平扩展有前提——你的服务必须是无状态的（Session 不能存在本地、文件不能存在本地）。如果服务有状态，要么先改造成无状态，要么把状态外置（Redis 存 Session、对象存储存文件）。**关联**：水平扩展的前提"无状态"连接了分布式 checklist 中的分布式 Session 问题。状态外置（Redis、对象存储）本质上是「分而治之」——把计算和存储分离。
    - 过关标准：能说出什么场景下选择哪种，以及水平扩展的前提（无状态服务）
- [ ] **多级缓存架构****本质**：在请求链路上设置多层缓存，每一层拦截一部分请求——浏览器缓存（HTTP Cache-Control）→ CDN（静态资源、就近访问）→ 网关缓存（接口级别）→ 本地缓存（Caffeine / Guava Cache，最快但容量小）→ 分布式缓存（Redis，容量大但有网络开销）→ 数据库。**为什么需要它**：单层缓存有局限——Redis 虽然快但有网络往返开销（约 1ms），本地缓存虽然零网络开销但容量有限且多实例间不一致。多级缓存让每一层处理最适合的请求，层层过滤后到达数据库的请求只是极少数。**关联**：多级缓存每一层的一致性问题不同——本地缓存和 Redis 之间可能不一致，Redis 和 DB 之间可能不一致。层数越多一致性越难保证，这又是「权衡」的体现。
    - 过关标准：能设计 浏览器缓存 → CDN → 网关缓存 → 本地缓存 → Redis → DB 的多级缓存方案
- [ ] **异步化设计****本质**：识别主流程中不需要同步完成的环节，改为异步执行。比如下单的主流程是"创建订单 + 扣库存"，而"发短信、发邮件、通知物流"可以异步。异步化的手段包括消息队列和 CompletableFuture。**为什么需要它**：同步链路中每增加一个环节，RT 就增加一截。异步化让主流程只做必要的事，非核心操作放到后台处理，大幅降低用户感知的响应时间。**关联**：异步化的代价是引入了最终一致性——用户下单后立即查订单，可能异步操作还没完成。这需要在产品层面给用户合理的预期（如"短信可能延迟几分钟到达"）。参见核心原理图谱的「异步化与解耦」。
    - 过关标准：能识别同步链路中可以异步化的环节，知道异步化带来的一致性问题及应对方式
- [ ] **池化技术****本质**：对创建和销毁成本高的资源（线程、数据库连接、HTTP 连接），预先创建一批放在"池"中复用，而不是每次使用都创建/销毁。**为什么需要它**：创建线程涉及操作系统调用（微秒级），创建数据库连接涉及 TCP 握手 + 认证（毫秒级）。高并发下每个请求都创建新资源，光创建的开销就可能耗尽系统能力。池化把这个开销摊到了系统启动时。**关联**：池化的核心挑战是参数配置——参见并发 checklist 的线程池参数设计和性能优化 checklist 的连接池优化。所有池化资源都面临同样的问题：配大了浪费资源，配小了不够用。参见核心原理图谱的「池化与复用」。
    - 过关标准：理解连接池、线程池、对象池的本质（复用昂贵资源），知道池化参数如何配置

### 面试场景题
- [ ] "你的服务依赖的下游接口响应变慢，从 50ms 变成 5s，你的系统会怎样？你怎么防护？" —— 考超时 + 熔断 + 降级
- [ ] "双十一流量是平时的 20 倍，你怎么让系统扛住？" —— 考限流 + 缓存 + 弹性扩容 + 降级的组合

## 第四层：经典系统设计题练习
> 面试中最常出的系统设计题。建议每道题都自己先设计，再对照优秀方案查漏补缺。
>

- [ ] **设计一个短链接服务（TinyURL）** ⭐
    - 考查重点：ID 生成、哈希/编码方案、读多写少的缓存策略、301 vs 302 的选择
- [ ] **设计一个秒杀系统** ⭐
    - 考查重点：流量层层拦截、库存扣减方案（Redis 预扣 + 异步落库）、防超卖、限流
- [ ] **设计一个消息推送系统（Feed 流）**
    - 考查重点：推模式 vs 拉模式 vs 推拉结合、大V问题、Timeline 存储
- [ ] **设计一个分布式限流器**
    - 考查重点：单机限流 vs 分布式限流、Redis + Lua 实现滑动窗口
- [ ] **设计一个订单系统**
    - 考查重点：订单状态机、分库分表、超时关单、幂等支付

### 练习方式建议
每道题用这个流程练习：

1. 给自己 40 分钟，按第一层的方法论框架独立设计
2. 画出架构图（纸上或白板工具）
3. 自己扮演面试官追问："如果 XX 挂了怎么办？""如果流量翻 10 倍呢？""数据不一致了怎么办？"
4. 对照网上优秀文章/视频查漏补缺
5. 过几天再设计一遍，看是否有提升
