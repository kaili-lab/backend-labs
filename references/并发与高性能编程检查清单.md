## 第一层：Java 内存模型（JMM）—— 地基
> 为什么从这里开始？理解了 JMM，锁、volatile、happens-before 就不再需要死记硬背，而是能自然推导出来。这一层是面试中"为什么"类追问的根基。
>

- [ ] **主内存与工作内存模型** ⭐**本质**：每个线程不是直接操作主内存，而是先把变量拷贝到自己的工作内存中操作，操作完再写回去。这就像每个人各自拿了一份文档副本去改，改完再合并——自然可能冲突。**为什么需要它**：因为 CPU 和主内存之间的速度差距太大（约 100 倍），所以硬件设计了多级缓存。JMM 是 Java 对这个硬件现实的抽象规范——它告诉你：多线程环境下，你不能假设一个线程的修改对另一个线程立刻可见。**关联**：这是理解 volatile（强制刷回主内存）、synchronized（加锁时同步主内存）、happens-before（定义什么操作之间保证可见性）的根基。
    - 过关标准：能画出线程、工作内存、主内存三者的关系图，能解释为什么两个线程修改同一个变量可能互相看不到
- [ ] **happens-before 原则** ⭐**本质**：JMM 定义的一组规则，告诉你"在什么条件下，操作 A 的结果一定对操作 B 可见"。它不是说时间上 A 先发生，而是说 A 的效果对 B 保证可见。**为什么需要它**：因为编译器和 CPU 会重排序指令来优化性能，你写的代码顺序不一定是实际执行顺序。没有 happens-before，你根本无法推理多线程代码的正确性——只能靠运气。**关联**：volatile 写-读、synchronized 解锁-加锁、线程 start/join 都建立了 happens-before 关系。理解了这个，你判断任何多线程代码是否安全都有了统一的分析工具。
    - 过关标准：能列出 JMM 定义的几条 happens-before 规则，并能用它来判断一段代码是否有可见性问题
- [ ] **volatile 关键字的语义** ⭐**本质**：volatile 做了两件事——①保证可见性（修改后立刻刷回主内存，读取时从主内存拿最新值）；②禁止指令重排序。但它不保证原子性——i++ 仍然不安全。**为什么需要它**：当你只需要一个轻量级的可见性保证（比如一个状态标志位），synchronized 太重了。volatile 是一个不加锁就能保证可见性的轻量方案。**关联**：volatile 的禁止重排序能力是 DCL 单例必须加 volatile 的原因。volatile 不保证原子性这一点，解释了为什么计数器场景要用 AtomicInteger 而不是 volatile int。
    - 过关标准：能说清楚 volatile 解决了什么问题（可见性 + 禁止重排序）、不能解决什么问题（原子性），能举出 volatile 适用和不适用的具体场景
- [ ] **指令重排序与内存屏障****本质**：编译器和 CPU 为了优化性能，会在不影响单线程语义的前提下调整指令执行顺序。内存屏障（Memory Barrier）是一种特殊指令，强制屏障前后的操作不能跨越屏障重排。**为什么需要它**：重排序在单线程下无害，但在多线程下可能导致一个线程看到另一个线程的操作以"不合逻辑"的顺序发生。典型案例就是 DCL 中对象半初始化问题。理解这个才能真正明白 volatile 和 synchronized 的底层保障机制。**关联**：volatile 写操作后插入 StoreStore + StoreLoad 屏障，读操作前插入 LoadLoad + LoadStore 屏障——这就是 volatile 禁止重排序的底层实现。synchronized 在进入和退出时也有对应的屏障。
    - 过关标准：能解释编译器和 CPU 为什么要重排序，volatile 和 synchronized 是如何通过内存屏障阻止重排序的
- [ ] **JMM 与实际硬件的关系****本质**：JMM 是一个抽象规范，它屏蔽了不同硬件（x86、ARM 等）的内存模型差异，给 Java 程序员提供统一的并发编程规则。底层对应的是 CPU 的 L1/L2/L3 缓存和缓存一致性协议（如 MESI）。**为什么需要它**：让你理解 JMM 不是凭空设计的，而是对真实硬件问题的抽象。比如"工作内存"对应的就是 CPU 缓存，"主内存"对应的就是物理内存。知道这个之后，JMM 的规则就不再抽象了。**关联**：MESI 协议保证缓存一致性的方式（Invalidate 消息）和 volatile 的实现高度相关。x86 架构天然保证较强的内存序，而 ARM 更弱——这也是为什么某些并发 Bug 只在特定平台出现。
    - 过关标准：知道 JMM 是一个抽象规范，能大致说出它和 CPU 缓存（L1/L2/L3）、缓存一致性协议（MESI）的对应关系

## 第二层：锁机制与线程协作 —— 工具箱
> 在 JMM 基础上重新理解锁——不是背"区别"，而是理解"为什么这样设计"。面试中锁相关的追问最密集。
>

### 2.1 synchronized 深入
- [ ] **对象头与 Mark Word** ⭐**本质**：每个 Java 对象在内存中除了实例数据，还有一个对象头。对象头中的 Mark Word 是一个动态变化的区域，存储了对象的哈希码、GC 分代年龄、以及锁状态信息。synchronized 加锁就是在修改 Mark Word。**为什么需要它**：理解了 Mark Word，你才能明白 synchronized 的锁信息"存在哪里"，以及为什么 Java 中任何对象都能当锁——因为每个对象头里都有 Mark Word。**关联**：Mark Word 中的锁标志位（2 bit）标识了当前锁的状态（无锁 / 偏向锁 / 轻量级锁 / 重量级锁），这是锁升级的物理基础。
    - 过关标准：能说出 Java 对象头的结构，知道锁信息存储在哪里
- [ ] **锁升级：偏向锁 → 轻量级锁 → 重量级锁** ⭐**本质**：JVM 根据竞争程度自动选择锁的"重量级别"——没竞争时用最轻的偏向锁（几乎零开销），有少量竞争时升级到轻量级锁（CAS 自旋），竞争激烈时升级到重量级锁（OS 互斥量，线程挂起）。锁只能升级不能降级。**为什么需要它**：因为大部分 synchronized 代码在实际运行时根本没有竞争，或者竞争很少。如果一上来就用重量级锁（操作系统层面的互斥量），性能浪费太大。锁升级是 JVM 对"大部分场景不需要重锁"这个现实的优化。**关联**：偏向锁的核心是在 Mark Word 中记录线程 ID，轻量级锁的核心是 CAS 替换 Mark Word，重量级锁的核心是 OS 的 Monitor。这三者和 CAS、操作系统线程调度直接相关。
    - 过关标准：能描述每种锁的适用场景和升级触发条件，知道为什么 JVM 要设计这个升级过程（性能优化）
- [ ] **synchronized 的 happens-before 语义****本质**：一个线程在 synchronized 块中做的所有修改，在它释放锁之后，对下一个获得同一把锁的线程一定可见。这就是 synchronized 的 happens-before 规则。**为什么需要它**：synchronized 不只是互斥（同一时刻只有一个线程执行），它还保证可见性。很多人只知道互斥，不知道可见性，这会导致在分析并发安全时遗漏问题。**关联**：回到第一层——synchronized 释放锁时，会把工作内存中的变量刷回主内存；获取锁时，会从主内存重新读取。这就是它保证可见性的 JMM 基础。
    - 过关标准：能用第一层的 JMM 知识解释为什么 synchronized 能保证可见性

### 2.2 AQS 与 ReentrantLock
- [ ] **AQS（AbstractQueuedSynchronizer）核心原理** ⭐**本质**：AQS 是 Java 并发包（java.util.concurrent）的基石框架。它用一个 int 类型的 state 变量表示资源状态，用一个 CLH 双向队列管理排队等待的线程。获取资源就是 CAS 修改 state，获取失败就进队列排队。**为什么需要它**：synchronized 是 JVM 内置的，不够灵活（不能中断等待、不能设超时、不能实现公平锁）。Java 需要一个灵活的、可扩展的锁框架来构建各种并发工具——AQS 就是这个框架。ReentrantLock、CountDownLatch、Semaphore、ReadWriteLock 底层全是 AQS。**关联**：AQS 的 state 用 volatile 修饰（保证可见性），修改 state 用 CAS（保证原子性）——直接回到了第一层的 JMM 和 CAS 知识。理解了 AQS，Java 并发包里一大半的工具你都能看懂源码了。
    - 过关标准：能说出 AQS 的 state + CLH 队列的设计思路，理解为什么它是 Java 并发包的基石
- [ ] **ReentrantLock vs synchronized** ⭐**本质**：两者都能实现互斥和可见性，但 ReentrantLock 基于 AQS 实现，提供了更多能力——可中断等待（lockInterruptibly）、可超时尝试（tryLock）、可实现公平锁、支持多个 Condition。**为什么需要它**：synchronized 能覆盖 80% 的场景，但在一些特殊需求下力不从心。比如：你想等锁但最多等 3 秒就放弃（tryLock），或者你需要多个等待条件（Condition），这时只能用 ReentrantLock。**关联**：ReentrantLock 的"可重入"是指同一个线程可以重复获取同一把锁而不死锁，这和 synchronized 一样。区别在于附加能力，而不是互斥的本质。
    - 过关标准：不是背区别表，而是能说出在什么业务场景下选择 ReentrantLock（如需要 tryLock、公平锁、Condition）
- [ ] **公平锁 vs 非公平锁****本质**：公平锁严格按照请求顺序（FIFO）分配锁；非公平锁允许新来的线程"插队"——如果锁恰好可用，新线程可以直接获取，不用排队。**为什么需要它**：非公平锁性能更好，因为减少了线程挂起和唤醒的开销（线程切换很昂贵）。大部分场景用非公平锁即可。但如果你的场景不能容忍"饥饿"（某些线程长期拿不到锁），就需要公平锁。**关联**：ReentrantLock 默认是非公平锁，通过构造函数 `new ReentrantLock(true)` 切换为公平锁。synchronized 是非公平的，无法切换——这也是选择 ReentrantLock 的场景之一。
    - 过关标准：能解释非公平锁为什么性能更好，公平锁在什么场景下必须使用
- [ ] **Condition 机制****本质**：Condition 是 ReentrantLock 的配套工具，相当于 Object.wait/notify 的增强版。一把锁可以创建多个 Condition，每个 Condition 有独立的等待队列，实现更精细的线程协作。**为什么需要它**：Object.wait/notify 只有一个等待队列，无法区分"等待不同条件"的线程。经典例子：生产者-消费者模型中，生产者等"队列不满"，消费者等"队列不空"——用两个 Condition 就能精确唤醒对应的线程，而不是 notifyAll 全部唤醒。**关联**：BlockingQueue 的底层实现（如 ArrayBlockingQueue）就是 ReentrantLock + 两个 Condition（notFull 和 notEmpty）。理解了 Condition，你就理解了 BlockingQueue 的原理。
    - 过关标准：能用 Condition 实现一个生产者-消费者模型，理解它和 Object.wait/notify 的区别

### 2.3 CAS 与原子类
- [ ] **CAS 原理（Compare And Swap）** ⭐**本质**：CAS 是一个原子操作——比较内存中的值是否等于预期值，如果是就更新为新值，否则不做任何事。整个操作在硬件层面是一条 CPU 指令（cmpxchg），不可被中断。**为什么需要它**：加锁虽然安全，但线程挂起和唤醒的开销很大。CAS 是一种"乐观"策略——假设没有竞争，直接尝试修改，失败了再重试。在低竞争场景下，CAS 比加锁快得多。**关联**：CAS 是 AQS 修改 state 的方式，是原子类（AtomicInteger）的底层实现，也是轻量级锁的核心操作。但 CAS 有 ABA 问题——值从 A 变成 B 再变回 A，CAS 会误认为没有变化。AtomicStampedReference 通过加版本号解决。
    - 过关标准：能说出 CAS 的三个操作数，知道它在硬件层面是一条 CPU 指令（cmpxchg），理解 ABA 问题及 AtomicStampedReference 的解决思路
- [ ] **Atomic 系列类（AtomicInteger、AtomicReference 等）****本质**：Java 对 CAS 的封装。内部是 volatile 变量 + Unsafe 类的 CAS 操作。提供了 incrementAndGet、compareAndSet 等原子方法，让你不用加锁就能安全地修改共享变量。**为什么需要它**：对于简单的计数、标志位等场景，用 synchronized 或 ReentrantLock 太重了。原子类用无锁的 CAS 实现，开销更小。它填补了"volatile 不能保证原子性"和"锁太重"之间的空白。**关联**：底层的 volatile 保证可见性，CAS 保证原子性——正好对应第一层 JMM 的两个核心概念。原子类是 volatile + CAS 的组合应用。
    - 过关标准：能说出它们底层用的是 CAS + volatile，能在合适场景用原子类替代锁
- [ ] **LongAdder vs AtomicLong** ⭐**本质**：AtomicLong 用单个变量做 CAS，高竞争下大量线程 CAS 失败重试，浪费 CPU。LongAdder 把一个计数器拆成多个 Cell，每个线程在自己的 Cell 上累加，最后汇总——用"分而治之"降低竞争。**为什么需要它**：在高竞争场景（比如全局请求计数器、metrics 统计），AtomicLong 会成为瓶颈。LongAdder 牺牲了精确读取（sum 不是原子的）换来了极大的写性能提升。**关联**：LongAdder 的分段思想和 ConcurrentHashMap 的分段锁、Redis Cluster 的分片是同一个核心原理——"一个热点扛不住，就拆成多个"。参见核心原理图谱的「分而治之」。
    - 过关标准：能解释高竞争下 LongAdder 为什么更快（分段思想），知道什么时候用哪个

### 2.4 常用并发工具类
- [ ] **CountDownLatch / CyclicBarrier / Semaphore** ⭐**本质**：三种不同的线程协作工具。CountDownLatch 是倒计数门——一个线程等多个线程完成（计数减到 0 后放行，一次性）。CyclicBarrier 是集合点——多个线程互相等对方到齐后一起出发（可重复使用）。Semaphore 是信号量——控制同时访问某个资源的线程数量。**为什么需要它**：多线程编程中经常需要协调线程之间的执行顺序或限制并发度。比如：主线程等 3 个子线程都查完数据再合并（CountDownLatch），或者限制最多 10 个线程同时访问数据库连接（Semaphore）。这些需求用 wait/notify 实现很复杂，用这些工具一行代码搞定。**关联**：三者底层全是 AQS。CountDownLatch 的 state 就是计数值，countDown() 就是 CAS 将 state 减 1，await() 就是检查 state 是否为 0。理解了 AQS，这三个工具的源码都能看懂。
    - 过关标准：不是背 API，而是能说出各自的典型业务场景（如：等多个数据源返回后合并、控制并发数），能说出 CountDownLatch 和 CyclicBarrier 的核心区别（一次性 vs 可重用）
- [ ] **ReadWriteLock 与 StampedLock****本质**：ReadWriteLock 允许读读并行、读写互斥、写写互斥——在读多写少场景中比独占锁性能好得多。StampedLock 更进一步，提供了"乐观读"模式——读操作不加锁，只在读完后验证期间是否有写操作，如果没有就直接用，有就升级为悲观读锁。**为什么需要它**：很多业务场景是读多写少（如配置信息、缓存数据）。如果用 synchronized，读和读之间也互斥，白白浪费了并发度。ReadWriteLock 让读操作可以并行，StampedLock 的乐观读更进一步减少了锁的开销。**关联**：ReadWriteLock 底层也是 AQS，用 state 的高 16 位记录读锁计数，低 16 位记录写锁计数——一个 int 变量同时管理两种锁。StampedLock 不基于 AQS，是独立实现，性能更好但使用更复杂（乐观读需要手动验证）。
    - 过关标准：能解释读写锁的适用场景（读多写少），知道 StampedLock 的乐观读模式及其优势

## 第三层：线程池与异步编程 —— 实战场景
> 你已经在用了，这一层帮你从"能用"升级到"能调优、能设计"。线程池是面试必考项，CompletableFuture 考察频率也在逐年增加。
>

### 3.1 线程池深入
- [ ] **ThreadPoolExecutor 7 个参数** ⭐**本质**：7 个参数定义了线程池的完整行为——corePoolSize（核心线程数，常驻）、maximumPoolSize（最大线程数，高峰期扩容用）、keepAliveTime（非核心线程空闲多久被回收）、unit（时间单位）、workQueue（任务排队的队列）、threadFactory（创建线程的方式）、handler（队列满了之后的拒绝策略）。**为什么需要它**：线程的创建和销毁很昂贵（涉及系统调用），线程池复用线程来避免这个开销。但线程池的参数配置直接决定了系统在不同负载下的表现——配错了要么资源浪费，要么 OOM，要么请求排队超时。**关联**：核心线程数的设计和任务类型（CPU 密集 / IO 密集）强相关：CPU 密集型一般设 N+1（N 为核数），IO 密集型可以设更大（如 2N），因为线程大部分时间在等 IO。这连接了操作系统的线程调度知识。
    - 过关标准：不是背参数名，而是能根据业务场景（CPU 密集 / IO 密集 / 混合型）设计合理的线程池配置，并能说出为什么这样配
- [ ] **线程池的工作流程** ⭐**本质**：任务提交后的处理顺序是——核心线程还没满 → 创建核心线程执行；核心线程满了 → 放入队列排队；队列满了 → 创建非核心线程执行；非核心线程也满了 → 执行拒绝策略。**为什么需要它**：很多人以为"队列满了才创建新线程"是个缺陷，其实这是刻意设计——优先排队而不是创建线程，因为线程数过多会导致上下文切换开销增大。理解这个流程才能正确配置参数，而不是机械地设一个大线程数。**关联**：这个流程解释了为什么 newFixedThreadPool 用无界队列（永远不会走到"创建非核心线程"这一步），以及为什么这样做在任务过多时会 OOM。
    - 过关标准：能说出任务提交后的完整流程：核心线程 → 队列 → 最大线程 → 拒绝策略，知道为什么是这个顺序而不是先扩线程再入队列
- [ ] **4 种拒绝策略及自定义策略** ⭐**本质**：当线程池的线程和队列都满了，新任务需要被拒绝。4 种内置策略是——AbortPolicy（抛异常，默认）、CallerRunsPolicy（让提交任务的线程自己执行）、DiscardPolicy（静默丢弃）、DiscardOldestPolicy（丢弃队列中最早的任务）。**为什么需要它**：不同业务场景对"扛不住了怎么办"有不同要求。比如日志采集可以丢弃（DiscardPolicy），核心业务不能丢弃需要降级处理（自定义策略：记日志 + 持久化 + 告警）。CallerRunsPolicy 看似优雅（不丢任务），但会阻塞提交线程，可能导致连锁反应。**关联**：拒绝策略是系统"兜底防护"思想的一种体现——和限流、熔断、降级是同一类设计思路。参见核心原理图谱的「快速失败与兜底」。
    - 过关标准：能说出每种策略的行为和适用场景，能写一个自定义拒绝策略（如记录日志 + 降级处理）
- [ ] **为什么不推荐 Executors 工厂方法** ⭐**本质**：newFixedThreadPool 用的是无界队列（LinkedBlockingQueue 默认 Integer.MAX_VALUE），任务堆积时队列无限增长导致 OOM。newCachedThreadPool 的最大线程数是 Integer.MAX_VALUE，短时间大量任务会创建海量线程导致 OOM。**为什么需要它（需要知道这个坑）**：阿里巴巴 Java 开发规范明确禁止使用 Executors 创建线程池，要求用 ThreadPoolExecutor 手动指定参数。面试中这是送分题，但你需要说出具体原因（OOM 风险），而不只是说"规范不让用"。**关联**：本质问题是"资源不设上限就可能耗尽"——和分布式系统中连接数不设限、请求不限流是同一类问题。
    - 过关标准：能说出 newFixedThreadPool 和 newCachedThreadPool 各自的隐患（OOM 风险），知道阿里巴巴规范为什么禁止使用
- [ ] **线程池监控与调优****本质**：ThreadPoolExecutor 暴露了运行时指标——getActiveCount()（活跃线程数）、getQueue().size()（队列中等待的任务数）、getCompletedTaskCount()（已完成任务数）、getRejectedExecutionCount()（被拒绝的任务数，需自定义拒绝策略统计）。**为什么需要它**：线程池参数不是配完就不管了。生产环境中负载是波动的，你需要通过监控数据来判断线程池是否健康——活跃线程长期等于最大线程数说明可能需要扩容，队列长期积压说明处理能力不足，拒绝次数增加说明要告警。**关联**：这连接了核心原理图谱的「量化先行」——没有监控就无法调优。生产中通常会把这些指标接入 Prometheus + Grafana 做可视化。
    - 过关标准：知道如何获取线程池运行时指标（activeCount、queueSize、completedTaskCount），能根据指标判断线程池是否需要调整
- [ ] **动态线程池的思路****本质**：ThreadPoolExecutor 的 setCorePoolSize() 和 setMaximumPoolSize() 允许在运行时动态修改参数。结合配置中心（如 Nacos），可以实现不重启服务就调整线程池配置。**为什么需要它**：传统方式改线程池参数需要改代码重新部署，而线上问题往往需要快速响应。动态线程池让你能在流量高峰时临时扩大线程数，流量低谷时缩回来，实现弹性调整。**关联**：美团的动态线程池方案是面试热点话题。它的核心思路是：线程池参数存在配置中心 → 配置变更时推送通知 → 应用接收通知后调用 set 方法动态修改。你已经在用 Nacos，这个思路可以直接落地。
    - 过关标准：了解如何在运行时动态修改核心线程数和队列大小（如美团的动态线程池方案），知道为什么需要动态调整

### 3.2 CompletableFuture 深入
- [ ] **常用组合操作** ⭐**本质**：CompletableFuture 是 Java 8 引入的异步编程工具，核心能力是链式组合多个异步操作。thenApply 对结果做转换（类比 Stream 的 map），thenCompose 对结果做扁平化转换（类比 flatMap），thenCombine 合并两个独立的异步结果，allOf 等所有完成，anyOf 等任一完成。**为什么需要它**：Future.get() 是阻塞的，多个异步任务只能逐个 get()，无法优雅地组合。CompletableFuture 让你能声明式地描述"A 完成后做 B，同时 C 也在跑，B 和 C 都完成后合并结果"——这在微服务调用多个下游时非常常见。**关联**：thenApply vs thenCompose 的区别和函数式编程中的 map vs flatMap 完全一致。如果 thenApply 的函数返回的也是 CompletableFuture，你会得到嵌套的 CompletableFuture<CompletableFuture>，而 thenCompose 会自动展平。
    - 过关标准：熟练使用 thenApply / thenCompose / thenCombine / allOf / anyOf，能说出 thenApply 和 thenCompose 的区别（类比 map vs flatMap）
- [ ] **异常处理** ⭐**本质**：CompletableFuture 链路中如果某一步抛异常，后续的 thenApply 等操作会被跳过。exceptionally 只处理异常并返回降级值，handle 同时处理正常结果和异常，whenComplete 只做副作用（如记日志）不改变结果。**为什么需要它**：异步调用最大的坑就是异常被静默吞掉——不用 get() 或不加异常处理，你可能完全不知道任务失败了。在生产环境中，这意味着数据丢失或业务逻辑错误且无任何告警。**关联**：这和消息队列中"消费异常处理"是同一类问题——异步操作的异常处理必须显式设计，不能指望框架自动处理。
    - 过关标准：知道 exceptionally / handle / whenComplete 的区别，能正确处理异步链路中的异常，知道异常不处理会怎样（静默吞掉）
- [ ] **线程切换与自定义线程池****本质**：CompletableFuture 的 supplyAsync 默认使用 ForkJoinPool.commonPool()，这是一个被所有 CompletableFuture 和并行 Stream 共享的全局线程池。如果你的异步任务是 IO 密集型（如 HTTP 调用、数据库查询），会长时间占用 commonPool 的线程，影响其他使用 commonPool 的代码。**为什么需要它**：生产环境中必须为不同类型的任务使用不同的线程池（隔离原则）。IO 密集型用专门的大线程池，CPU 密集型用小线程池，互不影响。如果全挤在 commonPool 里，一个慢任务会拖垮所有异步操作。**关联**：这和微服务中的"线程池隔离"（如 Hystrix / Resilience4j 的 bulkhead 模式）是同一个原理——隔离不同类型的工作负载，防止互相影响。参见核心原理图谱的「分而治之」。
    - 过关标准：知道 CompletableFuture 默认用 ForkJoinPool.commonPool()，知道什么时候要传入自定义线程池（IO 密集型操作），能解释为什么共享 commonPool 可能有问题

### 3.3 异步编程模式
- [ ] **Future vs CompletableFuture vs Reactive（WebFlux）****本质**：三者代表异步编程的三个演进阶段。Future 只能阻塞等结果（get），不能组合。CompletableFuture 支持链式组合和回调，适合大部分场景。Reactive（WebFlux / Project Reactor）是完全的响应式编程模型，适合超高并发的 IO 密集型场景，但学习曲线陡峭，调试困难。**为什么需要它（知道区别）**：面试中问这个，考的不是让你用 WebFlux，而是看你能不能做技术选型的权衡——大部分业务用 CompletableFuture 够了，WebFlux 在高并发网关或推送场景更合适，不要为了"看起来高级"而引入不必要的复杂度。**关联**：WebFlux 的底层是 Netty 的事件循环模型（Event Loop），和 Node.js 的异步模型类似。如果你有 Node.js 经验，可以类比理解。
    - 过关标准：能说出三者的适用场景和取舍，不盲目追新
- [ ] **异步编程的常见陷阱****本质**：异步引入了三个典型问题——①异常可能被静默吞掉（没加 exceptionally/handle），②ThreadLocal 在异步线程中失效（因为换了线程），③调试困难（堆栈断裂，看不到完整调用链）。**为什么需要它**：这些坑在开发阶段很难发现，往往在生产环境才暴露。比如 ThreadLocal 存的用户 ID 在异步线程中丢失，导致鉴权失败或数据串台——这是真实生产事故的高发区。**关联**：ThreadLocal 在线程池中的问题和第四层的"ThreadLocal 内存泄漏"直接相关。解决方案包括手动传递上下文、使用阿里的 TransmittableThreadLocal、或在框架层面做上下文传播。
    - 过关标准：知道异步调用中的异常丢失、ThreadLocal 失效、调试困难等问题及应对方式

### 4.1 死锁
- [ ] **死锁的四个必要条件** ⭐**本质**：死锁发生需要同时满足四个条件——①互斥（资源不可共享）②持有并等待（持有一个资源的同时等另一个）③不可抢占（已持有的资源不会被强制释放）④循环等待（A 等 B，B 等 A，形成环）。**为什么需要它**：知道四个条件，就知道预防死锁的方法——打破任何一个条件即可。实际中最常用的是打破"循环等待"（对资源排序，所有线程按相同顺序获取锁）和"持有并等待"（一次性申请所有资源，或使用 tryLock 超时放弃）。**关联**：这和分布式系统中的死锁（服务 A 等服务 B 的锁，服务 B 等服务 A 的锁）是同一个模型。tryLock 的超时机制对应了分布式场景中的超时 + 重试思路。
    - 过关标准：能说出四个条件（互斥、持有并等待、不可抢占、循环等待），并能说明打破哪个条件来预防死锁
- [ ] **死锁检测与排查** ⭐**本质**：JVM 内置了死锁检测能力。用 jstack 导出线程 dump 时，如果存在死锁，会在输出末尾明确标识 "Found one Java-level deadlock"，并列出每个线程持有了哪把锁、在等待哪把锁。**为什么需要它**：死锁在开发和测试阶段可能不出现（因为时序恰好没触发），但上线后在高并发下暴露。当服务"卡住了但不报错"时，死锁是首要怀疑对象。会用 jstack 定位死锁是一个必备的排查技能。**关联**：jstack 不只能查死锁，还能分析线程状态（BLOCKED、WAITING 等）。同一个工具覆盖了死锁排查和线程 dump 分析两个技能。
    - 过关标准：能用 jstack 导出线程 dump，找到 "Found one Java-level deadlock" 信息，看懂锁的持有和等待关系

### 4.2 线程 Dump 分析
- [ ] **线程状态：RUNNABLE / BLOCKED / WAITING / TIMED_WAITING** ⭐**本质**：RUNNABLE 表示线程正在执行或等待 CPU 调度；BLOCKED 表示线程在等待获取 synchronized 锁；WAITING 表示线程在无限期等待某个条件（如 Object.wait()、LockSupport.park()）；TIMED_WAITING 表示有超时的等待（如 Thread.sleep()、wait(timeout)）。**为什么需要它**：看到一个线程 dump，如果大量线程是 BLOCKED 状态且都在等同一把锁，说明锁竞争严重。如果线程是 WAITING 在某个队列上，可能是线程池任务不够或等待的条件永远不会满足。能读懂线程状态就能快速定位问题方向。**关联**：BLOCKED 对应 synchronized 的重量级锁等待，WAITING 对应 Condition.await() 或 LockSupport.park()——回到了第二层的锁机制。
    - 过关标准：看到一个线程 dump，能快速判断线程在干什么、等什么、被谁阻塞
- [ ] **线程 dump 分析工具****本质**：jstack 是 JDK 内置的命令行工具，直接导出线程快照。fastthread.io 是在线分析工具，可以把 jstack 的输出上传后得到可视化分析（如线程状态分布图、锁等待链图）。**为什么需要它**：生产环境出问题时，你不可能一行行读几百个线程的 dump。工具帮你快速定位——哪些线程阻塞了、等的是什么锁、哪个线程持有了那把锁。这是从"会用 jstack"到"能快速排查问题"的效率差距。**关联**：Arthas 的 `thread` 命令也能看线程状态，而且可以直接查看最忙的线程堆栈。在生产环境中 Arthas 比 jstack 更方便（不需要登录服务器执行命令）。
    - 过关标准：能使用 jstack，了解 fastthread.io 等在线分析工具

### 4.3 CPU 飙高排查
- [ ] **经典排查流程** ⭐**本质**：一套标准化的排查步骤——用 top 找到 CPU 最高的 Java 进程 → 用 top -Hp 找到进程内 CPU 最高的线程 → 把线程 ID 转成十六进制 → 用 jstack 导出线程 dump → 搜索十六进制线程 ID 找到对应堆栈 → 分析代码定位问题。**为什么需要它**：这是 Java 后端开发最经典的排查流程，面试中几乎是必考题。它考的不是你知不知道，而是你能不能一步步说清楚，因为这代表你有没有实际操作过。**关联**：定位到的堆栈信息可能指向死循环、频繁 GC（看到 GC 线程占 CPU）、锁竞争（大量线程 BLOCKED）——每种原因对应不同的解决方向，串联了本层的所有其他知识点。
    - 过关标准：能背出并实操这个流程：top 找进程 → top -Hp 找线程 → printf '%x' 转十六进制 → jstack 找线程堆栈
- [ ] **常见原因****本质**：CPU 飙高的几个典型原因——①死循环或热循环（代码逻辑问题）②频繁 Full GC（内存问题导致 GC 线程长时间占用 CPU）③锁竞争激烈（大量线程自旋等锁）④正则回溯（不良正则表达式导致指数级回溯）。**为什么需要它**：定位到线程堆栈后，你需要能快速归类问题类型。如果堆栈在你的业务代码里循环——是代码 Bug。如果堆栈在 GC 线程——是内存问题。如果大量线程堆栈在同一个 synchronized 处——是锁竞争。不同原因解决方向完全不同。**关联**：频繁 GC 连接了性能优化 checklist 的 JVM 调优部分，锁竞争连接了本 checklist 第二层的锁机制，正则回溯则是一个常被忽略但在生产中偶尔出现的坑。
    - 过关标准：能列出常见原因（死循环、频繁 GC、锁竞争激烈、正则回溯）并知道对应的排查方向

### 4.4 内存与 GC 相关并发问题
- [ ] **线程泄漏****本质**：线程被创建后没有被正确回收，持续占用系统资源。常见原因包括——每个请求 new Thread() 而不是用线程池、线程池使用不当（如 newCachedThreadPool 无上限创建线程）、线程卡在某个阻塞操作上无法退出。**为什么需要它**：线程泄漏不像内存泄漏那样有明显的 OOM 报错，它是一个缓慢的过程——线程数缓慢增长，直到系统资源耗尽或 OS 拒绝创建新线程（报 "unable to create new native thread"）。监控线程数是发现这类问题的关键。**关联**：连接了第三层的线程池知识——使用线程池而不是手动创建线程，以及正确关闭线程池（shutdown / shutdownNow），是预防线程泄漏的基本手段。
    - 过关标准：知道线程泄漏的常见原因（线程池使用不当、未关闭的线程），能通过线程数监控发现问题
- [ ] **ThreadLocal 内存泄漏** ⭐**本质**：ThreadLocal 的实现是每个线程内部有一个 ThreadLocalMap，Key 是 ThreadLocal 的弱引用，Value 是强引用。当 ThreadLocal 变量被回收后，Key 变成 null，但 Value 仍然被线程引用无法回收——这就是泄漏。在线程池中尤其严重，因为线程不会销毁，泄漏的 Value 永远不会被回收。**为什么需要它**：ThreadLocal 在 Web 开发中非常常用（存用户信息、请求上下文等）。配合线程池使用时，如果忘记 remove()，不仅内存泄漏，还会导致数据串台——用户 A 的请求在线程池中被复用的线程处理，读到了上一个请求（用户 B）遗留的 ThreadLocal 数据。**关联**：数据串台问题连接了异步编程的陷阱（第三层），也是生产环境真实事故的高发区。解决方案除了手动 remove()，还可以用阿里的 TransmittableThreadLocal 来自动传播和清理。
    - 过关标准：能解释为什么 ThreadLocal 配合线程池容易泄漏（线程复用导致 Entry 不被回收），知道必须在 finally 中手动 remove()
