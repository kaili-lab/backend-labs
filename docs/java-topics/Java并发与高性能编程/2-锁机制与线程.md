# 锁机制与线程协作 —— 工具箱

在 JMM 基础上重新理解锁——不是背"区别"，而是理解"为什么这样设计"。面试中锁相关的追问最密集。

## synchronized 深入
### 对象头与 Mark Word ⭐
1. 本质：每个 Java 对象在内存中除了实例数据，还有一个对象头。对象头中的 Mark Word 是一个动态变化的区域，存储了对象的哈希码、GC 分代年龄、以及锁状态信息。synchronized 加锁就是在修改 Mark Word。
2. 为什么需要它：理解了 Mark Word，你才能明白 synchronized 的锁信息"存在哪里"，以及为什么 Java 中任何对象都能当锁——因为每个对象头里都有 Mark Word。
3. 关联：Mark Word 中的锁标志位（2 bit）标识了当前锁的状态（无锁 / 偏向锁 / 轻量级锁 / 重量级锁），这是锁升级的物理基础。
4. 过关标准：能说出 Java 对象头的结构，知道锁信息存储在哪里

### 锁升级：偏向锁 → 轻量级锁 → 重量级锁 ⭐
1. 本质：JVM 根据竞争程度自动选择锁的"重量级别"
   1. 没竞争时用最轻的偏向锁（几乎零开销），有少量竞争时升级到轻量级锁（CAS 自旋），竞争激烈时升级到重量级锁（OS 互斥量，线程挂起）。
   2. 锁只能升级不能降级。
2. 为什么需要它：因为大部分 synchronized 代码在实际运行时根本没有竞争，或者竞争很少。如果一上来就用重量级锁（操作系统层面的互斥量），性能浪费太大。锁升级是 JVM 对"大部分场景不需要重锁"这个现实的优化。
3. 关联：偏向锁的核心是在 Mark Word 中记录线程 ID，轻量级锁的核心是 CAS 替换 Mark Word，重量级锁的核心是 OS 的 Monitor。这三者和 CAS、操作系统线程调度直接相关。
4. 过关标准：能描述每种锁的适用场景和升级触发条件，知道为什么 JVM 要设计这个升级过程（性能优化）

### synchronized 的 happens-before 语义
1. 本质：一个线程在 synchronized 块中做的所有修改，在它释放锁之后，对下一个获得同一把锁的线程一定可见。这就是 synchronized 的 happens-before 规则。
2. 为什么需要它：synchronized 不只是互斥（同一时刻只有一个线程执行），它还保证可见性。很多人只知道互斥，不知道可见性，这会导致在分析并发安全时遗漏问题。
3. 关联：回到第一层——synchronized 释放锁时，会把工作内存中的变量刷回主内存；获取锁时，会从主内存重新读取。这就是它保证可见性的 JMM 基础。
4. 过关标准：能用第一层的 JMM 知识解释为什么 synchronized 能保证可见性


## AQS 与 ReentrantLock

### AQS（AbstractQueuedSynchronizer）核心原理 ⭐
1. 本质：AQS 是 Java 并发包（java.util.concurrent）的基石框架。它用一个 int 类型的 state 变量表示资源状态，用一个 CLH 双向队列管理排队等待的线程。获取资源就是 CAS 修改 state，获取失败就进队列排队。

2. 为什么需要它：
   1. synchronized 是 JVM 内置的，不够灵活（不能中断等待、不能设超时、不能实现公平锁）。
   2. Java 需要一个灵活的、可扩展的锁框架来构建各种并发工具——AQS 就是这个框架。ReentrantLock、CountDownLatch、Semaphore、ReadWriteLock 底层全是 AQS。

3. 关联：AQS 的 state 用 volatile 修饰（保证可见性），修改 state 用 CAS（保证原子性）——直接回到了第一层的 JMM 和 CAS 知识。理解了 AQS，Java 并发包里一大半的工具你都能看懂源码了。

4. 过关标准：能说出 AQS 的 state + CLH 队列的设计思路，理解为什么它是 Java 并发包的基石

### ReentrantLock vs synchronized ⭐
1. 本质：
   1. 两者都能实现互斥和可见性，
   2. ReentrantLock 基于 AQS 实现，提供了更多能力——可中断等待、可超时尝试（tryLock）、可实现公平锁、支持多个 Condition。

2. 为什么需要它：synchronized 能覆盖 80% 的场景，但在一些特殊需求下力不从心。比如：你想等锁但最多等 3 秒就放弃（tryLock），或者你需要多个等待条件（Condition），这时只能用 ReentrantLock。

3. 关联：ReentrantLock 的"可重入"是指同一个线程可以重复获取同一把锁而不死锁，这和 synchronized 一样。区别在于附加能力，而不是互斥的本质。

4. 性能比较与选型：
   1. 在现代 JDK 下，不能简单说 `synchronized` 或 `ReentrantLock` 谁一定更快。低竞争、普通业务场景、临界区较短时，两者性能通常差距不大，很多时候感知不到明显区别。
   2. 如果只是单纯比较性能，应该按场景来看：低竞争下两者通常接近；中等竞争下非公平 `ReentrantLock` 有时会略有优势，但不稳定；高竞争下非公平 `ReentrantLock` 往往更容易保持吞吐，而公平锁通常最慢。
   3. 如果已经出现明显高竞争，不应只纠结两种锁谁快一点，而要先怀疑是不是共享热点设计有问题。这时往往更值得考虑减少共享状态、分段、CAS、LongAdder、读写分离等方案。
   4. 实际选型时，性能不是唯一标准。普通互斥优先 `synchronized`；需要可中断等待、超时尝试、公平锁、多 `Condition` 这类控制能力时，再优先选择 `ReentrantLock`。

5. 过关标准：不是背区别表，而是能说出在什么业务场景下选择 ReentrantLock（如需要 tryLock、公平锁、Condition），也知道多数普通互斥场景直接用 `synchronized` 就够了。

6. 补充
   1. JDK15偏向锁被废弃，JDK 21 已经彻底移除。
   2. 偏向锁的原理是，当线程开始加锁的时候，如果发现对象头中的锁位置是空的，那么它把它的线程 ID 写入进去。下一次当线程尝试加锁的时候，如果发现线程 ID 相同，那么就不用加锁。如果线程 ID 不同，那么会升级为轻量级锁。
   3. 在轻量级锁的层级上，synchronized 和 ReentrantLock 的性能几乎相同，底层原理都是CAS
   4. 简单场景下使用 synchronized 的优势：可以自动释放锁，就算有异常也可以自动释放，而 ReentrantLock 锁则必须使用 finally以确保异常情况下也能释放锁。
   5. 在可重入场景下，ReentrantLock 需要配对的加锁和释放锁，否则会发生锁泄露。

### 公平锁 vs 非公平锁
1. 本质：
   1. 公平锁严格按照请求顺序（FIFO）分配锁；
   2. 非公平锁允许新来的线程"插队"——如果锁恰好可用，新线程可以直接获取，不用排队。

2. 为什么需要它：
   1. 非公平锁性能更好，因为减少了线程挂起和唤醒的开销（线程切换很昂贵）。大部分场景用非公平锁即可。
   2. 但如果你的场景不能容忍"饥饿"（某些线程长期拿不到锁），就需要公平锁。

3. 关联：
   1. ReentrantLock 默认是非公平锁，通过构造函数 new ReentrantLock(true) 切换为公平锁。
   .2. synchronized 是非公平的，无法切换——这也是选择 ReentrantLock 的场景之一。

4. 过关标准：能解释非公平锁为什么性能更好，公平锁在什么场景下必须使用

### Condition 机制
1. 本质：Condition 是 ReentrantLock 的配套工具，相当于 Object.wait/notify 的增强版。一把锁可以创建多个 Condition，每个 Condition 有独立的等待队列，实现更精细的线程协作。

```java
public class SimpleBuffer {
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();
    private final Queue<Integer> queue = new ArrayDeque<>();
    private final int capacity = 10;

    public void put(int value) throws InterruptedException {
        lock.lock();
        try {
            while (queue.size() == capacity) {
                notFull.await();
            }
            queue.offer(value);
            notEmpty.signal();
        } finally {
            lock.unlock();
        }
    }

    public int take() throws InterruptedException {
        lock.lock();
        try {
            while (queue.isEmpty()) {
                notEmpty.await();
            }
            int value = queue.poll();
            notFull.signal();
            return value;
        } finally {
            lock.unlock();
        }
    }
}
```

上面这段代码里：
- `notFull` 表示“队列不满”这个条件，生产者在这里等待；
- `notEmpty` 表示“队列不空”这个条件，消费者在这里等待；
- 生产者放入元素后唤醒等待数据的消费者；
- 消费者取走元素后唤醒等待空间的生产者。

什么时候需要用 `Condition`：
- 同一把锁下，存在多个不同的等待条件；
- 你希望精确唤醒某一类线程，而不是把所有等待线程都叫醒；
- 你正在实现生产者-消费者、有界队列、资源池这类“条件不满足就等待，条件满足再继续”的协作逻辑。

什么时候通常不需要用 `Condition`：
- 只是做简单互斥保护，没有“等待条件成立再继续”的需求；
- 一个 `synchronized` + `wait/notify` 就已经足够，没必要为了写法统一强行换成 `ReentrantLock + Condition`；
- 更高层工具已经更合适，比如等多个任务完成用 `CountDownLatch`，多线程到齐再继续用 `CyclicBarrier`，生产者-消费者直接用 `BlockingQueue`。


2. 关联：BlockingQueue 的底层实现（如 ArrayBlockingQueue）就是 ReentrantLock + 两个 Condition（notFull 和 notEmpty）。理解了 Condition，你就理解了 BlockingQueue 的原理。

3. 过关标准：能用 Condition 实现一个生产者-消费者模型，理解它和 Object.wait/notify 的区别


## CAS 与原子类
### CAS 原理（Compare And Swap） ⭐
1. 本质：CAS 是一个原子操作——比较内存中的值是否等于预期值，如果是就更新为新值，否则不做任何事。整个操作在硬件层面是一条 CPU 指令（cmpxchg），不可被中断。
   1. 可以进一步理解为：CAS 是 Java 利用底层硬件原子指令，去解决并发问题中某一类特定问题的机制。
   2. 它最擅长解决的，不是所有并发问题，而是“多个线程同时更新同一个共享值时，如何避免基于旧值写回而产生丢失更新”这类问题。
   3. 所以 CAS 更适合单个共享变量的原子更新，比如计数器、状态位、引用替换；但它不能直接替代锁去解决多个变量一致性、复杂临界区、复杂业务步骤编排这类问题。

2. 为什么需要它：加锁虽然安全，但线程挂起和唤醒的开销很大。CAS 是一种"乐观"策略——假设没有竞争，直接尝试修改，失败了再重试。在低竞争场景下，CAS 比加锁快得多。
   1. CAS 失败后之所以通常要“自旋重试”，是因为它只能告诉你“这次比较并更新失败了”，却不会帮你自动完成后续逻辑。程序需要重新读取最新值，再次计算并尝试更新。
   2. 这也是 `AtomicInteger.incrementAndGet()` 这类方法底层常见的模式：读旧值 -> 算新值 -> CAS 更新 -> 失败就继续循环。
   3. 在低竞争场景下，这种失败重试成本很低，因为通常一两次就成功；但在高竞争场景下，很多线程会同时反复失败、反复重试，虽然线程没有阻塞，却会持续占用 CPU 做无效循环，这就是 CAS 为什么会“浪费 CPU”。
   4. 所以 CAS 的关键特点是：它用“忙等重试”换掉了“线程挂起与唤醒”。低竞争时这很划算，高竞争时就可能得不偿失，这也是为什么高并发热点计数场景会进一步引出 `LongAdder`。

3. 关联：CAS 是 AQS 修改 state 的方式，是原子类（AtomicInteger）的底层实现，也是轻量级锁的核心操作。但 CAS 有 ABA 问题——值从 A 变成 B 再变回 A，CAS 会误认为没有变化。AtomicStampedReference 通过加版本号解决。

4. 过关标准：能说出 CAS 的三个操作数，知道它在硬件层面是一条 CPU 指令（cmpxchg），理解 ABA 问题及 AtomicStampedReference 的解决思路

### ABA 问题
1. 本质：ABA 指的是一个值从 `A` 变成 `B`，又变回 `A`。这时 CAS 只看“当前值是不是预期值”，会误以为这个值从来没变过，于是错误地认为更新是安全的。

```java
AtomicReference<String> ref = new AtomicReference<>("A");

Thread t1 = new Thread(() -> {
    ref.compareAndSet("A", "B");
    ref.compareAndSet("B", "A");
});

Thread t2 = new Thread(() -> {
    boolean success = ref.compareAndSet("A", "C");
    System.out.println(success);
});
```

上面这个例子里，`t2` 看到的仍然是 `A`，所以 CAS 可能成功；但它不知道这个值其实已经被别的线程改过两次。对“只关心当前值”的场景，这可能没问题；但对“中间过程不能被人动过”的场景，这就是错误。

2. 为什么需要它：理解 ABA，才能真正明白 CAS 不是万能的。CAS 只能比较“值是否相等”，不能天然感知“这个值在中间有没有被改动过”。

3. 怎么解决：常见做法是给值加上版本号或时间戳，把“值相同”升级成“值相同且版本也相同”。JDK 里对应的是 `AtomicStampedReference`。

```java
AtomicStampedReference<String> ref = new AtomicStampedReference<>("A", 1);
int stamp = ref.getStamp();
boolean success = ref.compareAndSet("A", "C", stamp, stamp + 1);
```

这样即使值又变回了 `A`，只要版本号变了，CAS 就会失败。

4. 关联：这也是为什么文档前面说，CAS 更适合简单共享变量更新。只要业务开始关心“中间是否发生过变化”，就要考虑版本号、时间戳，甚至进一步升级为锁或更完整的一致性方案。

5. 过关标准：能解释什么是 ABA，知道它为什么会让 CAS 误判，能说出 `AtomicStampedReference` 的基本解决思路。

### Atomic 系列类（AtomicInteger、AtomicReference 等）
1. 本质：Java 对 CAS 的封装。内部是 volatile 变量 + Unsafe 类的 CAS 操作。提供了 incrementAndGet、compareAndSet 等原子方法，让你不用加锁就能安全地修改共享变量。

```java
AtomicInteger counter = new AtomicInteger(0);

boolean success = counter.compareAndSet(0, 1);
System.out.println(success); // true
System.out.println(counter.get()); // 1
```

这段代码演示了 CAS 的核心语义：
- 先比较当前值是否等于预期值 `0`；
- 如果相等，就原子地更新成 `1`；
- 如果不相等，说明别的线程已经先改过了，这次更新失败。

`incrementAndGet()` 底层思路也类似，只是它把“读取旧值 -> 计算新值 -> CAS 更新 -> 失败重试”封装起来了。可以近似理解为：

```java
for (;;) {
    int oldValue = counter.get();
    int newValue = oldValue + 1;
    if (counter.compareAndSet(oldValue, newValue)) {
        return newValue;
    }
}
```

这说明 AtomicInteger 解决的核心问题是：多个线程同时修改同一个值时，避免发生“读到同一个旧值、最后互相覆盖结果”的丢失更新问题。

2. 为什么需要它：对于简单的计数、标志位等场景，用 synchronized 或 ReentrantLock 太重了。原子类用无锁的 CAS 实现，开销更小。它填补了"volatile 不能保证原子性"和"锁太重"之间的空白。

3. 关联：底层的 volatile 保证可见性，CAS 保证原子性——正好对应第一层 JMM 的两个核心概念。原子类是 volatile + CAS 的组合应用。

4. 过关标准：能说出它们底层用的是 CAS + volatile，能解释 `compareAndSet` 和 `incrementAndGet` 的基本工作方式，知道它们适合简单共享变量更新而不是复杂复合业务逻辑。

### LongAdder vs AtomicLong ⭐
1. 本质：`AtomicLong` 只有一个共享值，所有线程都对这一个值做 CAS；`LongAdder` 会把一个计数器拆成多个槽位分散写入，读取时再把这些槽位求和。核心区别是：前者强调单点精确，后者强调高并发写入性能。

2. 为什么需要它：`AtomicLong` 在高竞争下容易变成单点热点，很多线程会同时 CAS 失败并重试，浪费 CPU。`LongAdder` 通过分散写入热点，显著减少冲突，所以在高并发计数场景下通常更快。

3. 核心代价：`LongAdder` 牺牲的是“读取时的严格瞬时精确性”。因为 `sum()` 需要把多个槽位加起来，而求和过程中其他线程可能还在继续写入，所以它通常不是一个全局原子快照。这意味着它适合统计类场景，不适合库存、余额这类要求每次读取都绝对精确的业务。

4. 选型结论：
   1. 需要精确值、竞争不高时，用 `AtomicLong`；
   2. 写非常频繁、竞争很高、读主要用于统计时，用 `LongAdder`；
   3. 可以把 `AtomicLong` 理解成“精确计数器”，把 `LongAdder` 理解成“高并发统计计数器”。

5. 局限性：`AtomicLong` 的局限是高竞争下容易成为热点；`LongAdder` 的局限是读取不是严格原子快照，占用空间也更多。

6. 过关标准：能解释为什么 `LongAdder` 在高竞争下更快，知道它牺牲了什么，以及什么时候必须坚持用 `AtomicLong`。


## 常用并发工具类
### CountDownLatch / CyclicBarrier / Semaphore ⭐
1. 本质：三种不同的线程协作工具。CountDownLatch 是倒计数门——一个线程等多个线程完成（计数减到 0 后放行，一次性）。CyclicBarrier 是集合点——多个线程互相等对方到齐后一起出发（可重复使用）。Semaphore 是信号量——控制同时访问某个资源的线程数量。

2. 为什么需要它：多线程编程中经常需要协调线程之间的执行顺序或限制并发度。比如：主线程等 3 个子线程都查完数据再合并（CountDownLatch），或者限制最多 10 个线程同时访问数据库连接（Semaphore）。这些需求用 wait/notify 实现很复杂，用这些工具一行代码搞定。

CountDownLatch 的经典场景：商品详情页聚合。一个接口要并发查询商品基础信息、库存、营销信息，主线程必须等所有子任务都完成后，才能组装响应并返回。这类“一个线程等多个任务完成”的一次性协作，非常适合 CountDownLatch。

```java
CountDownLatch latch = new CountDownLatch(3);

executor.submit(() -> {
    loadProductInfo();
    latch.countDown();
});

executor.submit(() -> {
    loadStockInfo();
    latch.countDown();
});

executor.submit(() -> {
    loadPromotionInfo();
    latch.countDown();
});

latch.await();
return buildProductDetail();
```

CyclicBarrier 的经典场景：并行批处理任务分阶段推进。比如 3 个线程分别处理不同分片的数据，第一阶段都先做预处理，必须等所有线程都完成这一阶段，才能统一进入第二阶段汇总或校验。这类“多个线程彼此等待，到齐再一起继续”的场景，非常适合 CyclicBarrier。

```java
CyclicBarrier barrier = new CyclicBarrier(3, () -> {
    System.out.println("第一阶段全部完成，进入下一阶段");
});

for (int i = 0; i < 3; i++) {
    executor.submit(() -> {
        doPhaseOne();
        barrier.await();
        doPhaseTwo();
    });
}
```

3. 关联：三者底层全是 AQS。CountDownLatch 的 state 就是计数值，countDown() 就是 CAS 将 state 减 1，await() 就是检查 state 是否为 0。理解了 AQS，这三个工具的源码都能看懂。

4. 过关标准：不是背 API，而是能说出各自的典型业务场景（如：等多个数据源返回后合并、控制并发数），能说出 CountDownLatch 和 CyclicBarrier 的核心区别（一个线程等大家完成 vs 大家彼此等待到齐）。


### ReadWriteLock 与 StampedLock
1. 本质：ReadWriteLock 允许读读并行、读写互斥、写写互斥——在读多写少场景中比独占锁性能好得多。StampedLock 更进一步，提供了"乐观读"模式——读操作不加锁，只在读完后验证期间是否有写操作，如果没有就直接用，有就升级为悲观读锁。
   1. ReentrantLock则是独占锁，即，哪怕是多个线程都是读，也只能排队；
   2. 而 ReadWriteLock 则是分离锁，即它可以让读和写分离，多个线程可以并行读，而不用排队；

2. 为什么需要它：很多业务场景是读多写少（如配置信息、缓存数据）。如果用 synchronized，读和读之间也互斥，白白浪费了并发度。ReadWriteLock 让读操作可以并行，StampedLock 的乐观读更进一步减少了锁的开销。

ReadWriteLock 的典型场景：本地配置中心或规则缓存。配置更新很少，但读取非常频繁。此时适合让多个读取线程并行，而更新线程独占写锁，避免“读也互斥”带来的吞吐浪费。

```java
public class ConfigCache {
    private final ReadWriteLock lock = new ReentrantReadWriteLock();
    private Map<String, String> config = new HashMap<>();

    public String get(String key) {
        lock.readLock().lock();
        try {
            return config.get(key);
        } finally {
            lock.readLock().unlock();
        }
    }

    public void refresh(Map<String, String> newConfig) {
        lock.writeLock().lock();
        try {
            config = new HashMap<>(newConfig);
        } finally {
            lock.writeLock().unlock();
        }
    }
}
```

这个场景适合 ReadWriteLock，因为：
- 读很多，写很少；
- 读操作之间不应该互相阻塞；
- 写操作需要独占，避免读到更新一半的数据。

StampedLock 的典型场景：热点统计对象或缓存元数据读取。大部分时候只是读取坐标、状态、统计值，写入偶尔发生；这时可以先走乐观读，只有检测到写冲突时再退化为悲观读。

```java
public class PointStore {
    private final StampedLock lock = new StampedLock();
    private double x;
    private double y;

    public double distanceFromOrigin() {
        long stamp = lock.tryOptimisticRead();
        double currentX = x;
        double currentY = y;

        if (!lock.validate(stamp)) {
            stamp = lock.readLock();
            try {
                currentX = x;
                currentY = y;
            } finally {
                lock.unlockRead(stamp);
            }
        }

        return Math.sqrt(currentX * currentX + currentY * currentY);
    }

    public void move(double deltaX, double deltaY) {
        long stamp = lock.writeLock();
        try {
            x += deltaX;
            y += deltaY;
        } finally {
            lock.unlockWrite(stamp);
        }
    }
}
```

这个场景适合 StampedLock，因为：
- 读非常频繁，写相对少；
- 很多读请求只是获取一个快照结果；
- 如果读期间没有写发生，就不用真正加读锁，可以进一步减少开销。

3. 使用场景总结：
   1. 读多写少，且希望多个读线程并行时，优先考虑 ReadWriteLock；
   2. 读非常频繁、写更少，且读逻辑可以接受“先乐观读取、失败再回退”的模式时，可以考虑 StampedLock；
   3. 如果业务逻辑已经很复杂，或者团队不熟悉乐观读验证，优先用 ReadWriteLock，代码更稳、更直观。

4. 关联：ReadWriteLock 底层也是 AQS，用 state 的高 16 位记录读锁计数，低 16 位记录写锁计数——一个 int 变量同时管理两种锁。StampedLock 不基于 AQS，是独立实现，性能更好但使用更复杂（乐观读需要手动验证）。

5. 过关标准：能解释读写锁的适用场景（读多写少），能看懂 ReadWriteLock 和 StampedLock 的典型代码，知道 StampedLock 的乐观读模式及其优势与使用前提。


## 并发集合与容器
### 为什么要有并发集合
1. 本质：普通集合（如 `HashMap`、`ArrayList`、`HashSet`）默认不是线程安全的。多个线程同时读写时，可能出现数据覆盖、结构损坏、死循环、读到脏数据等问题。并发集合的目标不是“让所有操作都加一把大锁”，而是在保证线程安全的前提下，尽量保留并发度。

2. 为什么需要它：如果每次都自己在外层加 `synchronized`，虽然也能保护集合，但粒度太粗，吞吐量往往很差。并发集合把常见场景内建进容器：有的偏向高并发读写，有的偏向读多写少，有的偏向线程协作与排队。

3. 关联：这部分是“锁、CAS、AQS”在容器层面的落地。理解并发集合，本质上是在理解：不同数据结构会选择不同同步策略。

4. 过关标准：能根据“读写比例、是否需要阻塞、是否需要排序、是否允许最终一致”选择合适的并发集合，而不是无脑用 `ConcurrentHashMap`。

### ConcurrentHashMap ⭐
1. 本质：高并发场景下的线程安全 Map。JDK 8 以后核心思路是“数组 + 链表 / 红黑树 + CAS + 局部同步”，目标是在保证线程安全的同时，避免整张表只靠一把大锁。

2. 为什么需要它：它适合“多线程同时读写 key-value”的场景，比如本地缓存、会话表、去重表、统计表。相比 `Hashtable` 或 `Collections.synchronizedMap`，它的并发度更高。

3. 使用场景：
   - 本地缓存：如用户信息缓存、配置缓存；
   - 并发统计：如接口调用次数、按 key 聚合计数；
   - 请求去重 / 幂等控制：如根据业务 key 记录处理中状态。

4. 注意点：
   - 复合操作仍然要小心，例如“先判断再更新”不一定天然安全；
   - 优先使用 `computeIfAbsent`、`compute`、`putIfAbsent` 这类原子方法；
   - 不要把它误解成“所有业务逻辑都自动线程安全”。

### CopyOnWriteArrayList / CopyOnWriteArraySet
1. 本质：写时复制容器。每次写操作都会复制底层数组，再替换旧数组；读操作直接读旧数组快照，因此读完全无锁或接近无锁。

2. 为什么需要它：适合“读远多于写，且可以接受写成本较高”的场景。它用写放大换取读性能和遍历稳定性。

3. 使用场景：
   - 监听器列表、观察者列表；
   - 配置快照、白名单快照；
   - 很少变更、但频繁遍历的集合。

4. 注意点：
   - 写多时性能会很差，因为每次写都复制数组；
   - 占内存更多；
   - 迭代看到的是快照，不一定反映最新写入。

### ConcurrentLinkedQueue
1. 本质：基于 CAS 的无界非阻塞队列，适合多线程并发入队、出队。

2. 为什么需要它：当你只需要“线程安全地排队”，但不需要阻塞等待时，它比外部加锁的普通队列更自然。

3. 使用场景：
   - 异步任务暂存；
   - 事件收集；
   - 生产快、消费也快，但不需要背压控制的队列。

4. 注意点：
   - 它是无界的，生产速度远大于消费速度时，可能导致内存压力；
   - 它不负责阻塞等待，不适合天然需要“没数据就等”的生产者-消费者模型。

### BlockingQueue 家族 ⭐
1. 本质：带阻塞语义的队列，是生产者-消费者模型最常用的并发容器。队列满时生产者可阻塞，队列空时消费者可阻塞。

2. 为什么需要它：它把“排队 + 等待 + 唤醒”这套线程协作机制内建好了，比手写 `wait/notify` 或 `Condition` 简单且稳定得多。

3. 常见类型与场景：
   - `ArrayBlockingQueue`：有界数组队列，容量固定，适合需要明确限流和背压的场景；
   - `LinkedBlockingQueue`：链表队列，可选有界或近似无界，吞吐较好，线程池默认常见；
   - `PriorityBlockingQueue`：按优先级出队，适合任务优先级调度；
   - `DelayQueue`：元素到期后才能取出，适合延迟任务、过期检查；
   - `SynchronousQueue`：不存元素，生产者必须直接把任务交给消费者，适合直接移交型场景。

4. 注意点：
   - 无界队列容易掩盖消费能力不足，最后把压力变成内存问题；
   - 有界队列更适合生产系统，因为它能把“处理不过来”尽早暴露出来；
   - 线程池和消息积压问题，很多时候本质上就是队列选型问题。

### ConcurrentSkipListMap / ConcurrentSkipListSet
1. 本质：线程安全且有序的 Map / Set，底层基于跳表，支持并发读写下的有序访问。

2. 为什么需要它：`ConcurrentHashMap` 虽快，但不保证有序；如果业务同时需要线程安全和有序范围查询，就要考虑这类容器。

3. 使用场景：
   - 按时间、分数、排名维护有序数据；
   - 需要 `floor`、`ceiling`、区间查询的场景；
   - 并发环境下的有序索引。

4. 注意点：
   - 它通常比 `ConcurrentHashMap` 更重；
   - 如果业务不需要排序，就不要为了“高级”而使用它。

### Collections.synchronizedXxx 的定位
1. 本质：JDK 提供的同步包装器，例如 `Collections.synchronizedList`、`Collections.synchronizedMap`。它们通常是在普通集合外层套一把锁。

2. 为什么需要它：适合简单、低并发、历史代码改造成本低的场景。

3. 使用场景：
   - 低并发管理后台；
   - 简单线程安全改造；
   - 对性能要求不高，只求尽快正确的场景。

4. 注意点：
   - 并发度通常不如专门的并发集合；
   - 遍历时往往仍需手动在外层加锁；
   - 更适合作为过渡方案，而不是高并发系统的首选。

### 选择思路
1. 如果你需要线程安全的 key-value 容器，优先考虑 `ConcurrentHashMap`。

2. 如果你是“读很多、写很少”的列表或集合，优先考虑 `CopyOnWriteArrayList` / `CopyOnWriteArraySet`。

3. 如果你需要无锁化的普通并发队列，考虑 `ConcurrentLinkedQueue`。

4. 如果你需要生产者-消费者、限流、背压、延迟执行，优先从 `BlockingQueue` 家族中选。

5. 如果你需要线程安全 + 有序访问 / 范围查询，考虑 `ConcurrentSkipListMap` / `ConcurrentSkipListSet`。

6. 如果只是简单改造旧代码、并发不高，可以用 `Collections.synchronizedXxx`，但不要把它当成高并发场景的默认方案。
