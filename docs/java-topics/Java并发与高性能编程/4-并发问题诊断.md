# 并发问题诊断 —— 排查与定位能力

这一层关注的是：当并发代码在线上出问题时，你如何判断、定位、归因，而不是只停留在“我知道有这些概念”。高级后端能力的一个分水岭，就是能不能把问题一步步查到根因。

## 死锁排查
### 死锁的四个必要条件 ⭐
1. 本质：死锁发生通常需要同时满足四个条件：互斥、持有并等待、不可抢占、循环等待。
2. 为什么需要它：理解这四个条件，才能知道预防思路。真正重要的不是背定义，而是知道只要打破其中任意一个条件，死锁就能被避免。
3. 使用场景：
   1. 多把锁并存时，统一加锁顺序来打破循环等待；
   2. 用 `tryLock` + 超时放弃，避免永久等待；
   3. 尽量缩小持锁范围，减少“持有并等待”的概率。
4. 关联：这和分布式系统里服务 A 等服务 B、服务 B 又反等服务 A 的问题是同一个模型，只是资源从本地锁换成了远程资源。
5. 过关标准：能说出四个条件，并能从工程实践角度说明如何破坏它们。

### 死锁检测与 `jstack`
1. 本质：JVM 自带死锁检测能力。线程 dump 中如果存在 Java 层死锁，`jstack` 通常会直接给出 `Found one Java-level deadlock` 之类的提示，并列出线程持有和等待的锁关系。
2. 为什么需要它：死锁最麻烦的地方是“服务卡住了，但日志可能并不报错”。这时会不会第一时间想到导线程 dump，是排查能力的重要区别。
3. 经典场景：两个线程分别先拿锁 A 再等锁 B，另一个线程先拿锁 B 再等锁 A，最终互相等待。
4. 关联：`jstack` 不只用于死锁，也能分析线程状态、锁等待链和热点线程，是并发排查最基础的工具之一。
5. 过关标准：知道遇到“服务卡死不报错”时优先看线程 dump，会用 `jstack` 查死锁。

## 线程 Dump 分析
### 常见线程状态：RUNNABLE / BLOCKED / WAITING / TIMED_WAITING ⭐
1. 本质：线程 dump 里最常见的状态包括：`RUNNABLE`（正在运行或等待 CPU）、`BLOCKED`（等 monitor 锁）、`WAITING`（无限期等待某个条件）、`TIMED_WAITING`（带超时的等待）。
2. 为什么需要它：排查线程问题时，第一步不是看代码，而是先看线程在干什么。大量 `BLOCKED` 往往意味着锁竞争，大量 `WAITING` 可能意味着条件没满足或线程池空等。
3. 使用场景：
   1. 大量线程 `BLOCKED` 在同一行，多半是某把锁竞争严重；
   2. 大量线程 `WAITING` 在 `Condition.await()`，要怀疑是否没人唤醒；
   3. 大量线程 `TIMED_WAITING` 在 `sleep()` 或网络调用，要怀疑外部依赖慢。
4. 关联：线程状态分析会把你重新带回锁、Condition、线程池这些基础机制上。
5. 过关标准：看到线程 dump 时，能先根据状态把问题大类分出来。

### `jstack`、Arthas 与线程分析工具
1. 本质：`jstack` 是 JDK 自带的线程快照工具，Arthas 的 `thread` 命令可以更方便地在线查看线程状态和热点线程，`fastthread.io` 这类工具可以把 dump 可视化。
2. 为什么需要它：真实生产环境里线程往往很多，纯靠人工逐行读 dump 效率很低。工具的价值不在“替你分析”，而在帮你快速缩小范围。
3. 使用场景：
   1. `jstack <pid>`：导出线程 dump；
   2. `thread -n 5`：看最占 CPU 的线程；
   3. 可视化工具：看状态分布、锁等待关系。
4. 关联：工具只是入口，真正的能力还是你能否把线程状态、锁等待、调用栈和业务代码对上。
5. 过关标准：知道常用工具各自适合什么场景，不会只停留在“听说过”。

## CPU 飙高排查
### 经典排查流程 ⭐
1. 本质：Java 线上 CPU 飙高有一套非常经典的定位路径：先找到进程，再找到最忙线程，再把线程 ID 转成十六进制，再用线程 dump 找对应栈帧，最后回到代码。
2. 为什么需要它：这是一条可以落地执行的排查链路，而不是抽象思路。面试问这个，考的是你有没有真正做过定位。
3. 典型步骤：
   1. `top` 找到 CPU 高的 Java 进程；
   2. `top -Hp <pid>` 找到进程内最忙线程；
   3. `printf '%x\n' <tid>` 转成十六进制；
   4. `jstack <pid>` 导出线程 dump；
   5. 搜索对应十六进制线程 ID，找到堆栈；
   6. 根据堆栈回到代码或框架层判断原因。
4. 关联：这条流程最终会把问题归到死循环、频繁 GC、锁竞争、外部依赖阻塞等具体方向上。
5. 过关标准：能不看资料把这条流程顺着说下来。

### CPU 飙高的常见原因
1. 本质：CPU 高不是结论，只是现象。真正常见的原因包括：死循环或热循环、频繁 Full GC、锁竞争和自旋、正则回溯、热点 JSON 序列化等。
2. 为什么需要它：找到最忙线程之后，你还要会分类。不然即使看到了栈，也不知道问题落在哪一类。
3. 使用场景：
   1. 业务代码循环不退出：代码 Bug；
   2. GC 线程占大量 CPU：内存或对象分配问题；
   3. 大量线程卡在同一把锁附近：锁竞争问题；
   4. 正则或解析逻辑耗时异常：算法或输入数据问题。
4. 关联：CPU 问题通常只是入口，最终又会回到锁、GC、线程池、业务代码设计这些更底层的原因。
5. 过关标准：能列出几类典型原因，并知道每类要往哪里继续查。

## 线程泄漏与上下文问题
### 线程泄漏
1. 本质：线程泄漏不是“Java 对象没回收”，而是线程持续存在、资源一直被占着，最终把系统线程数、内存、调度资源一点点耗尽。
2. 为什么需要它：线程泄漏往往不会立刻报错，而是先表现为线程数慢慢上涨、上下文切换变多、系统越来越慢，最后才报 `unable to create new native thread`。
3. 常见场景：
   1. 每个请求都手动 `new Thread()`；
   2. 线程池不受控，线程数无限膨胀；
   3. 线程长时间阻塞却没有退出机制；
   4. 应用关闭时没有正确关闭线程池。
4. 关联：这和线程池设计直接相关——很多所谓“线程泄漏”，本质上是线程池参数或生命周期管理有问题。
5. 过关标准：知道线程泄漏的典型来源，并知道要监控线程数变化。

### ThreadLocal 内存泄漏与数据串台 ⭐
1. 本质：`ThreadLocalMap` 的 key 是弱引用，value 是强引用。在线程长期存活、尤其是线程池线程被复用时，如果 `ThreadLocal` 用完不 `remove()`，value 可能长期残留在线程里，既造成内存泄漏，也可能造成数据串台。
2. 为什么需要它：这是 Java Web 开发里非常真实的线上问题。很多人只知道 `ThreadLocal` 好用，却忽略了它在线程池和异步环境下的生命周期问题。
3. 经典场景：
   1. 请求 A 在线程池线程里放入用户信息；
   2. 请求结束后忘记 `remove()`；
   3. 请求 B 复用了同一线程，结果读到了 A 遗留的数据。
4. 使用建议：
   1. `ThreadLocal` 用完务必在 `finally` 里 `remove()`；
   2. 不要假设异步线程会自动继承上下文；
   3. 需要跨线程传播上下文时，要显式设计传播机制。
5. 关联：这和上一层“异步编程陷阱”直接相连——线程一旦切换，`ThreadLocal` 就不再是你以为的“请求级变量”。
6. 过关标准：能解释为什么线程池环境下 `ThreadLocal` 容易出问题，知道 `finally + remove()` 是最低要求。

## 面试场景题
1. “线上服务 CPU 100%，你会怎么一步步排查？” —— 考 CPU 排查流程
2. “服务卡住了但没有报错，你首先怀疑什么？怎么验证？” —— 考死锁与线程 dump
3. “为什么 `ThreadLocal` 在线程池里容易出事故？” —— 考上下文与线程复用
4. “大量线程都是 `BLOCKED` 状态，你会优先想到什么？” —— 考线程状态与锁竞争
5. “线程数持续上涨但没有 OOM，你会怀疑什么？” —— 考线程泄漏意识
