# 线程池与异步编程 —— 实战工具箱

这一层关注的是：当你已经知道“可以并发做事”之后，如何把并发做得更稳定、可控、可观测。线程池决定系统如何承接任务，异步编程决定多个任务如何组织与编排。

## 线程池基础与选型
### ThreadPoolExecutor 的 7 个参数 ⭐
1. 本质：`ThreadPoolExecutor` 用 7 个参数定义了线程池的完整行为：`corePoolSize`、`maximumPoolSize`、`keepAliveTime`、`unit`、`workQueue`、`threadFactory`、`handler`。它们共同决定线程池在平峰、高峰、积压、过载时分别怎么表现。
2. 为什么需要它：线程池不是“开几个线程那么简单”，而是一种资源控制器。参数配置错了，轻则任务排队过久，重则线程数失控、队列堆积、内存被打满。
3. 如何理解参数：
   1. `corePoolSize`：常驻线程数，适合承接日常稳定流量；
   2. `maximumPoolSize`：高峰期允许扩到多大；
   3. `workQueue`：承接暂时处理不过来的任务；
   4. `handler`：线程和队列都满了以后，系统如何兜底。
4. 关联：线程池参数设计和任务类型直接相关。CPU 密集型更接近 `N` 或 `N + 1`，IO 密集型可以更大，因为线程很多时间在等外部资源。
5. 过关标准：不是背参数名，而是能根据业务类型说出线程池为什么要这样配。

### `workQueue`：线程池真正的缓冲区
1. 本质：`workQueue` 决定了“核心线程忙不过来时，任务先堆在哪里、堆多少、以什么方式堆”。它不只是一个实现细节，而是线程池承接突发流量的第一道缓冲层。
2. 真实开发场景里怎么用：
   1. Web 服务里处理请求时，如果任务执行时间短、突发流量明显，常会用有界队列做短时缓冲，避免线程数无限上涨；
   2. 批处理、异步通知、日志落盘这类允许排队的任务，通常可以给更大的有界队列，但也要有上限；
   3. 如果希望尽量不排队、让线程池更积极扩容，可以考虑 `SynchronousQueue`，因为它本质上不存任务，只做直接移交。
3. 常见类型和取舍：
   1. `ArrayBlockingQueue`：有界、内存可控，适合生产环境显式限流；
   2. `LinkedBlockingQueue`：如果不指定容量，默认近似无界，容易把问题藏成任务堆积和内存风险；
   3. `SynchronousQueue`：不缓存任务，适合希望快速扩线程、快速暴露过载的场景，但配置不当也容易把线程数冲高。
4. 需要注意的坑点：
   1. 无界队列会让 `maximumPoolSize` 基本失去意义，因为任务还没轮到扩线程就已经一直进队列了；
   2. 队列太大不等于更安全，它只是把处理不过来的压力往后拖，最后可能变成延迟暴涨、OOM、请求超时；
   3. 队列里堆的是“待执行任务对象”，如果任务里还持有大对象、请求上下文、文件句柄，积压的代价会比想象中更大。
5. 电商场景案例：
   1. 订单支付成功后，系统通常会异步做多件事：发优惠券、发站内信、同步积分、推送物流准备信息。如果这些任务都扔进一个线程池，而 `workQueue` 用的是默认近似无界的 `LinkedBlockingQueue`，大促高峰时任务表面上“都提交成功了”，但实际上只是不断堆在队列里。结果可能是用户付款成功后很久才收到优惠券，积分半小时后才到账，最后排查发现不是消费者挂了，而是线程池一直没扩容，压力全藏在队列里。
   2. 另一个常见场景是商品详情页的异步聚合查询。假设一个接口会并行查询商品、价格、营销、库存 4 个下游，如果线程池队列过大，请求高峰时大量聚合任务会先排队，接口看起来不是“直接失败”，而是 RT 持续抬高，最终把上游网关超时打满。这类场景反而更适合小一些的有界队列，尽早暴露系统扛不住，而不是靠堆积把问题拖成雪崩。

### `handler`：过载时系统怎么失败
1. 本质：`handler` 就是拒绝策略。它定义了当线程池和队列都满了以后，新任务是报错、回退到调用方执行、静默丢弃，还是丢掉旧任务。它本质上决定了系统在过载时如何暴露压力。
2. 真实开发场景里怎么用：
   1. 核心链路任务通常不能静默丢失，更适合显式失败，方便上层降级或告警；
   2. 非核心异步任务，比如部分埋点、低价值通知，可以接受丢弃，但要在业务上确认“丢了也没关系”；
   3. `CallerRunsPolicy` 常用于给上游施加背压，让提交任务的线程自己执行任务，从而自然放慢提交速度。
3. 需要注意的坑点：
   1. `DiscardPolicy` 和 `DiscardOldestPolicy` 最危险的地方不是功能本身，而是容易让任务悄悄丢失却没人发现；
   2. `CallerRunsPolicy` 虽然能传递背压，但如果提交线程本身是 Tomcat、Netty 或消息消费线程，可能把上游入口也一起拖慢；
   3. `AbortPolicy` 会直接抛异常，如果上层没处理好，就会把线程池过载变成接口报错或任务链路中断。
4. 电商场景案例：
   1. 订单创建链路里的“创建订单、冻结库存、写支付单”这类核心任务，一旦线程池过载，通常更适合 `AbortPolicy` 或自定义拒绝后显式返回“系统繁忙，请稍后重试”。原因不是它更优雅，而是核心交易任务不能悄悄丢，否则用户以为下单成功，后台其实根本没处理。
   2. 营销埋点、搜索曝光日志、推荐打点这类非核心异步任务，某些场景可以接受丢弃，但前提是业务明确知道“少量丢日志不会影响交易主流程”。这种任务即使使用丢弃策略，也最好补监控计数，否则线上很容易长期丢数据却没人发现。
   3. `CallerRunsPolicy` 在电商里也很常见，比如导出订单报表、批量发货通知这类后台管理任务。它的价值是线程池满了以后让提交方自己慢下来，起到天然背压作用。但如果你把它直接用在用户下单入口线程上，高峰期就会把 Web 请求线程拖住，结果变成整个接口层一起变慢。
5. 额外建议：生产里不要把拒绝策略当“最后再看”的参数，而要提前想清楚业务在过载时是要失败、降级、延迟，还是允许少量丢弃。这个选择本质上是业务决策，不只是技术参数。
6. 过关标准：能把拒绝策略和业务容错要求对应起来，而不是只会背 4 个策略名字。

### 线程池的工作流程 ⭐
1. 本质：任务提交后的典型流程是：核心线程未满先建核心线程执行；核心线程满了先进队列；队列满了再扩到最大线程数；最大线程数也满了才触发拒绝策略。
2. 为什么需要它：理解这个流程，才能真正理解为什么线程池是“资源控制”而不是“任务越多线程越多”。它的设计重点不是让任务尽快启动，而是避免线程无限膨胀。
3. 使用场景：
   1. 流量平稳时，大部分请求由核心线程处理；
   2. 短时突发流量由队列承接；
   3. 再高的峰值由非核心线程兜住；
   4. 真正扛不住时，通过拒绝策略把过载显式暴露出来。
4. 关联：这也是为什么无界队列会隐藏问题——它会让线程池很难扩容，却会把压力悄悄转成任务堆积和内存风险。
5. 过关标准：能完整说出“核心线程 → 队列 → 最大线程 → 拒绝策略”这条路径，并知道为什么顺序是这样。

### 拒绝策略与过载保护 ⭐
1. 本质：当线程池和队列都满了，新任务必须被拒绝。JDK 内置了 4 种策略：`AbortPolicy`、`CallerRunsPolicy`、`DiscardPolicy`、`DiscardOldestPolicy`。
2. 为什么需要它：拒绝策略不是“异常情况才考虑的东西”，而是线程池设计的一部分。系统总会遇到处理不过来的时刻，你必须提前决定是报错、降级、回退还是丢弃。
3. 使用场景：
   1. 核心业务通常不能静默丢弃，优先显式失败或降级；
   2. 非关键日志、埋点类任务可以考虑丢弃；
   3. `CallerRunsPolicy` 适合让上游感知背压，但要注意它会拖慢提交线程。
4. 关联：拒绝策略和限流、熔断、降级本质上是同一类问题——系统扛不住时，如何优雅失败。
5. 过关标准：能说出每种策略的行为差异，并能根据业务重要性选择合理兜底方式。

### 为什么不推荐 `Executors` 工厂方法 ⭐
1. 本质：`Executors` 提供的几个快捷工厂方法很方便，但默认参数往往不安全。`newFixedThreadPool` 常见问题是无界队列，`newCachedThreadPool` 常见问题是线程数几乎无上限。
2. 为什么需要它：生产环境里最危险的不是“立刻报错”，而是“问题被参数默认值悄悄藏起来”。无界队列会把压力变成任务堆积和内存风险，无界线程数会把压力变成线程爆炸和系统资源耗尽。
3. 使用建议：
   1. 生产环境优先显式 new `ThreadPoolExecutor`；
   2. 把线程数、队列大小、拒绝策略都配清楚；
   3. 不要把“方便”放在“可控”前面。
4. 关联：这和数据库连接池、消息队列消费者数、网关限流是同一个原则——关键资源必须设上限。
5. 过关标准：能说出 `newFixedThreadPool` 和 `newCachedThreadPool` 的典型隐患，以及为什么生产上更推荐显式配置。

### 线程池监控与动态调优
1. 本质：线程池监控就是持续观察线程池当前有没有“忙不过来”、有没有“任务堆住”、有没有“已经开始拒绝任务”。它本质上是在回答一个问题：这个线程池现在是健康运行，还是已经快把系统拖垮了。
2. 为什么需要它：很多线程池问题平时不明显，只有流量一上来才暴露。最危险的不是“线程池立刻报错”，而是它表面还在工作，但任务已经越堆越多、延迟越来越高，最后把接口超时、数据库压力、内存占用一起拖起来。所以线程池不能只创建，不观察。
3. 可以先把线程池想成一个餐厅后厨：
   1. 线程 = 厨师；
   2. 队列 = 后厨待做订单；
   3. 拒绝策略 = 厨房接不下单时怎么处理；
   4. 监控 = 店长一直盯着：现在有多少厨师在忙、后面堆了多少单、有没有开始退单。
4. 初学时最值得先看这 6 个指标：
   1. `poolSize`：当前线程池里一共有多少线程。它告诉你线程池现在开到了多大。
   2. `activeCount`：当前真正正在干活的线程数。如果它长期接近 `poolSize`，说明线程池很忙。
   3. `corePoolSize` / `maximumPoolSize`：这是你配置的上下限。监控时不是只看配置值本身，而是看运行状态有没有长期贴着上限。
   4. `queueSize`：队列里还有多少任务在排队。它是最重要的积压信号之一。持续上涨通常说明“生产速度大于消费速度”。
   5. `completedTaskCount`：已经完成的任务总数。单看它意义不大，但结合时间窗口看增长速度，可以判断线程池处理能力是否稳定。
   6. 拒绝次数：这个指标通常要自己在拒绝策略里打点统计。它表示线程池已经正式扛不住了，是非常关键的过载信号。
5. 这些指标分别说明什么：
   1. `activeCount` 很高，但 `queueSize` 不高：说明线程池忙，但暂时还能跟上；
   2. `activeCount` 很高，`queueSize` 还在涨：说明线程池已经开始处理不过来，任务在堆积；
   3. `poolSize` 已经接近 `maximumPoolSize`，`queueSize` 继续增长：说明扩线程和排队都快到极限了；
   4. 拒绝次数开始出现：说明线程池已经进入明确过载，不是“快不行了”，而是“已经不行了”；
   5. `queueSize` 长期很高，但 CPU 并不高：往往说明任务不是 CPU 忙，而是在等数据库、远程调用、磁盘、网络这类外部资源。
6. 真实开发里怎么监控：
   1. 最基础的做法，是定时把线程池指标打印日志或上报监控系统；
   2. 如果项目里用了 Spring Boot + Micrometer，通常会把这些指标接到 Prometheus，再用 Grafana 画图看趋势；
   3. 如果暂时没有完整监控平台，至少也应该在关键线程池上定期输出：线程数、活跃线程数、队列长度、完成任务数、拒绝次数。
7. 一个简单的落地方向：

```java
ThreadPoolExecutor executor = ...;

int poolSize = executor.getPoolSize();
int activeCount = executor.getActiveCount();
int queueSize = executor.getQueue().size();
long completedTaskCount = executor.getCompletedTaskCount();
```

这几个方法就是最基础的观测入口。先能看见，再谈调优。

8. 电商场景下怎么看这些指标：
   1. 例如“订单支付成功后的异步通知线程池”，如果你发现 `queueSize` 在大促期间不断上涨，而 `activeCount` 长期打满，说明发券、发消息、积分同步这些任务处理不过来了；
   2. 如果订单主链路线程池开始出现拒绝次数，说明系统已经影响到核心交易，不能只调线程数，通常还要考虑限流、降级、拆分线程池；
   3. 如果商品详情页聚合查询线程池队列变长，但 CPU 负载不高，往往不是线程太少，而是某个下游服务慢，把线程都阻塞住了。
9. 动态调优到底在调什么：
   1. 调 `corePoolSize`：适合平峰和常态流量变化，比如业务长期增长后，常驻线程明显不够；
   2. 调 `maximumPoolSize`：适合应对短时高峰，但前提是机器资源扛得住，也确认任务确实值得并发处理；
   3. 调 `workQueue` 容量：适合重新定义“是让任务先排队，还是更早暴露过载”；
   4. 调拒绝策略：适合重新定义系统在过载时是报错、降级、背压还是允许丢弃。
10. 需要特别注意的误区：
   1. 线程数不是越大越好。线程多了会带来上下文切换、内存占用、数据库连接争抢，可能让系统更慢；
   2. 队列不是越大越好。队列越大，只是把问题藏得更深；
   3. 拒绝任务不一定是坏事。很多时候“及时拒绝”比“堆到最后整体雪崩”更健康；
   4. 看到线程池忙，不要条件反射只会加线程，也要怀疑是不是下游慢、任务太重、线程池没做隔离。
11. 关联：线程池监控的核心不是背 API，而是建立一个判断链路：线程在忙什么，任务有没有堆住，系统是在短时高峰还是已经持续过载。只有先看清现象，调优才不是瞎调。
12. 过关标准：能说清楚线程池最基本该看哪些指标，知道每个指标异常时大概意味着什么，也知道调优不只是“把线程数调大”。

## CompletableFuture 与异步编排
### 常用组合操作 ⭐
1. 本质：`CompletableFuture` 解决的是“多个异步任务如何组合”的问题。`thenApply` 做结果转换，`thenCompose` 做异步链式展开，`thenCombine` 合并两个独立结果，`allOf` 等全部完成，`anyOf` 等任意一个先完成。
2. 为什么需要它：传统 `Future.get()` 只能阻塞等待，很难优雅表达“先并行调用多个下游，再在结果都回来后组装”。而这正是后端聚合接口最常见的需求。
3. 经典场景：商品详情页同时查商品、库存、营销信息，最后统一合并。

```java
CompletableFuture<Product> productFuture =
        CompletableFuture.supplyAsync(this::loadProduct, executor);
CompletableFuture<Stock> stockFuture =
        CompletableFuture.supplyAsync(this::loadStock, executor);
CompletableFuture<Promotion> promotionFuture =
        CompletableFuture.supplyAsync(this::loadPromotion, executor);

CompletableFuture<ProductDetail> detailFuture = CompletableFuture
        .allOf(productFuture, stockFuture, promotionFuture)
        .thenApply(ignored -> new ProductDetail(
                productFuture.join(),
                stockFuture.join(),
                promotionFuture.join()
        ));
```

4. 关联：`thenApply` 和 `thenCompose` 的区别，本质上和函数式编程里的 `map` / `flatMap` 一样。
5. 过关标准：能说出常见组合操作分别适合什么场景，能写出一个真实的并发聚合案例。

### 异常处理
1. 本质：异步链路里的异常不会像同步代码那样自动沿调用栈冒出来。如果不显式处理，异常可能被延迟感知，甚至在某些场景下被忽略。
2. 为什么需要它：异步最危险的不是失败，而是“失败了但没人知道”。这会让系统表现成结果缺失、降级未触发、数据不完整，却没有明显报错。
3. 使用场景：
   1. `exceptionally`：失败时给默认值；
   2. `handle`：同时处理成功和异常；
   3. `whenComplete`：记录日志、埋点、清理上下文。

```java
CompletableFuture<String> future = CompletableFuture
        .supplyAsync(this::callRemoteService, executor)
        .exceptionally(ex -> "fallback-result");
```

4. 关联：这和消息队列消费失败处理很像——异步链路的异常必须显式设计，不能指望框架自动帮你兜底。
5. 过关标准：知道异常不处理会带来什么后果，能正确区分 `exceptionally`、`handle`、`whenComplete` 的职责。

### 默认线程池与线程池隔离 ⭐
1. 本质：`CompletableFuture.supplyAsync()` 默认使用 `ForkJoinPool.commonPool()`。这意味着如果你不显式传线程池，不同异步任务可能会共享同一个全局公共池。
2. 为什么需要它：公共线程池适合轻量、短时任务，但不适合混合大量 IO 阻塞任务。一旦慢任务把公共线程池占满，其他异步逻辑也会被拖慢。
3. 使用场景：
   1. CPU 密集型短任务可以使用专门的小线程池；
   2. IO 密集型任务必须显式传入自己的线程池；
   3. 不同业务类型应尽量线程池隔离，避免互相影响。

```java
CompletableFuture<String> future = CompletableFuture
        .supplyAsync(this::callRemoteService, ioExecutor);
```

4. 关联：这和微服务里的舱壁隔离（bulkhead）是同一个思想——不要让所有任务挤在一个共享资源池里。
5. 过关标准：知道 `CompletableFuture` 默认线程池是什么，知道什么时候必须显式传自定义线程池。

## 异步编程模式与边界
### Future vs CompletableFuture vs Reactive
1. 本质：它们代表异步编程的不同层级。`Future` 只能拿结果，组合能力很弱；`CompletableFuture` 适合大多数业务异步编排；Reactive 更适合超高并发 IO 流式处理。
2. 为什么需要它：这不是为了追新，而是为了做技术选型。很多业务并不需要响应式全家桶，用 `CompletableFuture` 就够了；只有在网关、流式推送、极高并发 IO 场景下，Reactive 才更有优势。
3. 使用场景：
   1. 业务服务层异步聚合：优先 `CompletableFuture`；
   2. 只需简单提交任务并等待：`Future` 即可；
   3. 大量非阻塞 IO 与流式数据处理：再考虑 Reactive。
4. 关联：不要把“更复杂的模型”误认为“更高级的答案”，选型的关键始终是问题和代价是否匹配。
5. 过关标准：能说出三者各自适合解决什么问题，而不是只会背定义。

### 异步编程的常见陷阱 ⭐
1. 本质：异步会引入新的问题：异常可能不被感知、`ThreadLocal` 上下文丢失、调用链断裂导致调试困难。
2. 为什么需要它：很多异步 Bug 在开发阶段并不明显，上线后才暴露成“数据偶发缺失”“上下文串台”“链路日志断掉”这类很难定位的问题。
3. 常见陷阱：
   1. 没有处理异步异常，导致任务失败却没有告警；
   2. 在线程池中依赖 `ThreadLocal`，结果上下文没有自动传递；
   3. 过度链式编排，结果可读性和可调试性很差。
4. 关联：`ThreadLocal` 在线程池中的问题，会直接连接到下一层“并发问题诊断”里的内存泄漏和数据串台。
5. 过关标准：知道异步不是“只要并发起来就行”，而是必须同步考虑异常、上下文传播和可观测性。
