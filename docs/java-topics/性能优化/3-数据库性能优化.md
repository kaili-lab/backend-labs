# 数据库性能优化 —— 从 SQL 到连接池

数据库优化的重点不是背术语，而是：当接口慢时，你能不能判断是不是数据库导致的；如果是，问题在 SQL、索引、连接池，还是整体架构。

## SQL 与索引
### 慢查询日志与 EXPLAIN ⭐
1. 本质：慢查询日志是发现问题 SQL 的入口，`EXPLAIN` 是理解 SQL 执行路径的基础工具。
2. 为什么需要它：数据库慢，第一步不是直接加索引，而是先确认到底哪条 SQL 慢、慢在哪里。
3. 使用场景：
   1. 慢日志中先找到高频或高耗时 SQL；
   2. 用 `EXPLAIN` 看是否走索引、扫描行数、回表情况；
   3. 结合业务过滤条件判断 SQL 是否写得合理。
4. 关联：真正的数据库优化常常是“SQL 写法 + 索引设计 + 业务分页策略”一起调，而不是单点优化。
5. 过关标准：知道慢 SQL 的基本定位路径，而不是只会说“建索引”。

### 索引失效与 SQL 优化常见手段
1. 本质：索引不是建了就一定生效，很多写法会让优化器放弃索引或让索引效果大打折扣。
2. 为什么需要它：很多线上慢 SQL，根本不是“没建索引”，而是“明明建了，但写法让它没用起来”。
3. 常见场景：
   1. 模糊查询、函数操作、类型转换导致索引失效；
   2. 条件顺序和联合索引不匹配；
   3. 查得太多，回表成本过高。
4. 关联：数据库优化最终还是服务于业务请求链路，所以常要结合缓存、异步化、读写分离一起看。
5. 过关标准：知道索引失效常见原因，也知道 SQL 优化不只是“再建一个索引”。

## 连接池与数据库承压
### 数据库连接池优化
1. 本质：数据库连接很昂贵，所以通常用连接池复用。连接池参数决定了高峰期数据库请求如何排队、超时和恢复。
2. 为什么需要它：接口慢不一定是 SQL 慢，也可能是连接池拿连接太慢，或者连接数配置和数据库承载能力不匹配。
3. 使用场景：
   1. 高并发时看连接池等待时间；
   2. 连接数过大导致数据库上下文切换上升；
   3. 连接数过小导致请求大量排队。
4. 关联：连接池问题和线程池问题非常像，本质上都是有限资源池的参数调优。
5. 过关标准：知道数据库连接池不是越大越好，也知道接口慢可能卡在“拿连接”这一步。
