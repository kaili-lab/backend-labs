# JVM 调优与问题定位 —— 运行时视角

JVM 这一层的重点，不是把参数背下来，而是知道：对象在哪里分配、为什么会频繁 GC、出了问题用什么工具定位。

## JVM 基础
### 运行时数据区与堆内存分代模型 ⭐
1. 本质：对象大多分配在堆中，堆再按代划分，目的是让“朝生夕死”的对象和“活得久”的对象分开处理，降低回收成本。
2. 为什么需要它：如果你不知道对象主要待在哪里，就很难理解 GC 为什么频繁、为什么晋升、为什么会 Full GC。
3. 使用场景：
   1. 大量短生命周期对象：更容易引起 Young GC；
   2. 老年代对象积压：更容易引起 Full GC；
   3. 类加载过多：要关注元空间。
4. 关联：堆、元空间、GC 日志分析、内存泄漏定位，都是同一条链路上的内容。
5. 过关标准：知道堆分代的基本意义，知道元空间主要放什么。

### GC 算法与常见垃圾回收器
1. 本质：GC 的核心问题是“如何判断对象没用了”和“如何高效回收”。不同回收器在吞吐、停顿时间和资源消耗之间做不同取舍。
2. 为什么需要它：调优不是死记参数，而是先知道当前系统更怕长停顿，还是更怕吞吐下降。
3. 使用场景：
   1. 面向普通服务时，更关注停顿时间和稳定性；
   2. 批处理类任务更关注整体吞吐；
   3. 高堆内存场景下更要关注 Full GC 代价。
4. 关联：GC 问题很多时候不是 GC 本身的问题，而是对象分配、缓存策略、集合膨胀等业务行为引发的结果。
5. 过关标准：知道垃圾回收器的目标是不同的，不会把“换 GC”当成默认优化方案。

### GC 日志分析与常用工具 ⭐
1. 本质：JVM 问题定位高度依赖工具。`jstat` 看 GC 变化趋势，`jmap` 导出堆信息，MAT / VisualVM 看对象占用和引用链，Arthas 更适合线上动态诊断。
2. 为什么需要它：JVM 调优如果没有工具支撑，很容易变成猜测。你需要先看证据，再决定调什么。
3. 使用场景：
   1. GC 频繁：先看 GC 日志和 `jstat`；
   2. 内存涨不回去：导堆分析；
   3. 线上偶发卡顿：结合 Arthas、线程状态和 GC 现象一起看。
4. 关联：JVM 问题常常和线程池、缓存、数据库连接池等上层组件联动，而不是孤立出现。
5. 过关标准：知道不同工具分别适合看什么，不会只记一个 `jstack`。
