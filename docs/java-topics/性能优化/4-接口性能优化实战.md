# 接口性能优化实战 —— 从链路到方案

真正的接口优化，往往不是某一个点突然神奇提速，而是从请求链路出发，把不必要的等待、重复计算和资源争抢一层层拿掉。

## 接口优化套路
### 接口优化的分层排查思路 ⭐
1. 本质：接口慢时，要按链路拆解：应用层、线程池、缓存、数据库、外部依赖、序列化、网络传输，逐层看谁最耗时。
2. 为什么需要它：接口优化最怕“盲改”。有时你以为是数据库慢，实际上可能是下游 HTTP 调用慢；你以为是代码慢，实际上是线程池满了。
3. 使用场景：
   1. 先看整体 RT 和长尾；
   2. 再看每个子步骤耗时；
   3. 找出最主要的那一段；
   4. 最后决定是缓存、并发、异步、批量还是索引。
4. 关联：这会把前面所有主题真正串起来——线程池、数据库、缓存、JVM、异步，最终都在接口链路里体现。
5. 过关标准：知道接口慢时不要一上来就加缓存，而是先拆链路定位主要瓶颈。

### 常见优化手段速查表
1. 本质：常见优化手段包括缓存、并行化、异步化、批量化、索引优化、连接池调优、序列化优化、减少无效远程调用等。
2. 为什么需要它：很多问题本质不同，对应手段也不同。把手段背成“武器库”，前提是你知道什么时候该用哪一把。
3. 使用场景：
   1. 重复读取多：优先缓存；
   2. 请求内多个独立耗时步骤：优先并行化；
   3. 非核心链路：优先异步化；
   4. 大量小请求：优先批量化；
   5. 数据访问慢：优先 SQL / 索引优化。
4. 关联：优化手段之间常常可以组合，但越组合越要注意一致性、可观测性和复杂度。
5. 过关标准：能根据不同瓶颈匹配不同优化方案，而不是把所有优化都归结成“上缓存”。
