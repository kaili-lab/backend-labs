# 性能优化方法论 —— 先会定位，再谈优化

性能优化最容易走偏的地方，是一上来就背“优化技巧”。真正的起点应该是：先定义目标、先量化瓶颈、先知道你在优化哪一层。

## 优化思维框架
### 性能优化的完整链路思维 ⭐
1. 本质：性能优化不是“哪里慢就改哪里”，而是从现象到指标、从指标到瓶颈、从瓶颈到方案的链路化过程。
2. 为什么需要它：如果没有方法论，优化很容易退化成拍脑袋调参数，最后甚至优化错方向。
3. 常见步骤：
   1. 先明确问题：是吞吐、延迟、错误率还是资源占用；
   2. 再量化：到底慢多少、在哪个时间段慢；
   3. 再定位：CPU、内存、磁盘、网络、数据库、缓存、锁、线程池，究竟谁是主要瓶颈；
   4. 最后再选方案。
4. 关联：这套思路后面会贯穿 JVM、数据库、接口、缓存、线程池等所有具体优化动作。
5. 过关标准：能先说定位过程，再谈优化手段，而不是上来就说“加缓存”。

### 性能指标体系
1. 本质：优化之前必须先知道你在看什么。最常见的指标包括 RT、TP99、QPS、吞吐、CPU、内存、GC、错误率、线程池积压、数据库慢查询等。
2. 为什么需要它：同样叫“性能差”，有的人是 RT 高，有的人是吞吐低，有的人是 CPU 打满；这些问题的解决方向完全不同。
3. 使用场景：
   1. 接口慢：优先看 TP95 / TP99 和下游耗时；
   2. 系统扛不住：优先看吞吐和资源占用；
   3. 偶发超时：优先看长尾请求和依赖抖动。
4. 关联：高并发系统里，平均值往往意义不大，长尾指标更重要。
5. 过关标准：知道至少一组常见性能指标，并知道它们分别对应什么问题。

### 能说出具体数字
1. 本质：面试里说“性能很好”“做了优化”都不够，必须尽量说出前后对比和量化结果。
2. 为什么需要它：数据能体现你是真的做过，而不是只会讲概念。
3. 使用场景：
   1. “接口 RT 从 800ms 降到 250ms”；
   2. “慢 SQL 从 2s 降到 100ms”；
   3. “GC 次数从每分钟 20 次降到 3 次”。
4. 关联：量化表达本身也是一种工程能力，它决定你后面能不能做持续调优。
5. 过关标准：知道优化结论必须尽量用指标和前后对比来表达。

## 面试场景题
1. “性能优化一般从哪里开始，不会先做什么？”
2. “为什么平均 RT 下降了，但用户还是觉得慢？”
3. “你做过一次性能优化，怎么证明它真的有效？”
