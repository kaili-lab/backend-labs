# 高可用与高性能设计 —— 让系统扛得住

系统设计走到这里，重点已经不只是“功能能跑通”，而是“故障来了能不能稳住，流量上来了能不能扛住”。

## 高可用设计
### 冗余与故障转移
1. 本质：高可用的核心是避免单点。只要一个节点挂掉就全局不可用，说明系统还没有真正具备可用性设计。
2. 为什么需要它：真实线上环境里，机器宕机、网络抖动、依赖异常都是常态，而不是例外。
3. 使用场景：多实例部署、主从切换、跨机房容灾、冗余副本。
4. 关联：冗余提升可用性的同时，也会带来一致性和成本问题。
5. 过关标准：能说出“高可用不是不出故障，而是故障来了还能继续服务”。

### 限流、熔断、降级 ⭐
1. 本质：这三者都是过载和故障场景下的保护手段。限流控制流量入口，熔断阻断故障扩散，降级给用户一个可接受但较弱的结果。
2. 为什么需要它：系统扛不住时，最危险的不是“丢一部分请求”，而是全链路一起崩。
3. 使用场景：
   1. 网关层做全局限流；
   2. 下游不稳定时做熔断；
   3. 非核心能力返回兜底结果或临时关闭。
4. 关联：这几者经常一起出现，真正体现的是“快速失败与保核心链路”的思想。
5. 过关标准：知道三者分别解决什么问题，也知道为什么它们通常是组合使用。

### 超时与重试设计
1. 本质：所有远程调用都必须设超时；重试必须有边界、有退避，并且配合幂等。
2. 为什么需要它：不设超时等于资源可以被无限占住；盲目重试等于在下游已经扛不住时继续补刀。
3. 使用场景：
   1. 每个远程调用都设超时；
   2. 只对可重试的失败做有限次重试；
   3. 配合指数退避和熔断。
4. 关联：超时、重试、幂等、熔断组合起来，才是一条完整的容错链。
5. 过关标准：能说出为什么“重试不是越多越好”。

## 高性能设计
### 水平扩展 vs 垂直扩展
1. 本质：垂直扩展是升级单机资源，水平扩展是增加机器实例。前者简单但有上限，后者更适合持续扩展。
2. 为什么需要它：当流量翻倍时，你必须明确系统是靠“更强单机”还是“更多副本”来承载。
3. 使用场景：
   1. 早期先垂直扩展，成本低、改造少；
   2. 流量持续增长后，逐步转向水平扩展；
   3. 水平扩展的前提通常是服务无状态。
4. 关联：无状态、Session 外置、文件外置，都是为水平扩展服务。
5. 过关标准：知道水平扩展不是一句口号，而是要以无状态设计为前提。

### 多级缓存、异步化与池化技术 ⭐
1. 本质：高性能的常见三板斧是：用缓存挡住重复读，用异步拆掉非核心同步步骤，用池化复用昂贵资源。
2. 为什么需要它：很多系统的瓶颈不是代码不够快，而是做了太多不该同步做的事，或者重复访问了太多昂贵资源。
3. 使用场景：
   1. 多级缓存降低数据库压力；
   2. 异步化缩短主链路 RT；
   3. 线程池、连接池、对象池降低创建成本。
4. 关联：这几种手段都不是白赚性能，它们会分别引入一致性、可观测性和参数调优问题。
5. 过关标准：能说出高性能设计不是只有缓存，而是多种手段协同配合。
