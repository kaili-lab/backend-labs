# 数据库与缓存设计 —— 数据层的核心取舍

很多系统设计题的核心都落在数据层：数据怎么建模、读写压力怎么分担、热点怎么处理、一致性怎么接受。数据库和缓存往往是一组一起讨论的问题。

## 数据库设计
### Schema 设计原则与索引设计 ⭐
1. 本质：数据表设计的目标不是“字段列出来”，而是让核心查询和更新路径足够稳定、足够高效。
2. 为什么需要它：表结构和索引一旦定得不好，后面很多问题都会被放大成性能问题。
3. 使用场景：
   1. 先围绕核心查询和更新路径设计表；
   2. 索引要服务于真实访问模式；
   3. 避免为了“以后可能用到”加过多索引。
4. 关联：索引不是越多越好，因为写入成本、存储和维护复杂度都会上涨。
5. 过关标准：知道数据库设计要围绕访问模式，而不是只围绕实体关系图。

### 读写分离与分库分表
1. 本质：当单库或单实例扛不住时，常见扩展路径是先读写分离，再按业务或数据规模做分库分表。
2. 为什么需要它：数据库往往是最早暴露瓶颈的部分，不提前想清楚扩展路径，系统会很快触顶。
3. 使用场景：
   1. 读压力大：优先考虑读写分离；
   2. 数据量和写压力都过大：再考虑分库分表；
   3. 分库后要重新面对全局 ID、跨库查询、事务和分页问题。
4. 关联：分库分表从来不是“升级版数据库”，而是一种用复杂度换容量的方案。
5. 过关标准：知道什么时候还不该分库分表，也知道它一旦上了会引入哪些新问题。

## 缓存设计
### 缓存使用场景与一致性 ⭐
1. 本质：缓存的目标不是“让所有东西都变快”，而是把高频、可复用、可接受短暂不一致的数据挡在数据库前面。
2. 为什么需要它：数据库通常是最贵、最脆弱的共享资源，而缓存是最直接的减压手段。
3. 使用场景：
   1. 读多写少的数据；
   2. 热点数据；
   3. 可接受短时间不一致的数据。
4. 关联：缓存一旦引入，就必须回答一致性问题，而一致性没有零成本答案。
5. 过关标准：能说出什么时候加缓存，什么时候即使慢也不应该轻易加缓存。

### 缓存三大问题：击穿 / 穿透 / 雪崩
1. 本质：缓存不是只会“命中”或“未命中”，还会出现热点 Key 失效、大量无效请求绕过缓存、大量 Key 同时过期等特殊风险。
2. 为什么需要它：这些问题往往会把缓存层的优势反过来放大成数据库风险。
3. 常见应对：
   1. 击穿：热点 Key 加互斥、预热、永不过期配异步刷新；
   2. 穿透：布隆过滤器、空值缓存、请求校验；
   3. 雪崩：过期时间打散、限流、多级缓存、降级。
4. 关联：缓存问题本质上也是系统设计问题，因为它最终还是会回到高可用与容量保护。
5. 过关标准：知道三者的区别和最常见的处理思路。

### 热点 Key 与多级缓存
1. 本质：热点问题不是“有没有缓存”，而是“缓存本身会不会被打成新的热点单点”。
2. 为什么需要它：流量极端集中时，单层 Redis 也可能成为瓶颈。
3. 使用场景：
   1. 本地缓存 + Redis 组合；
   2. 热点数据预热；
   3. 分层缓存降低单点压力。
4. 关联：多级缓存提升性能的同时，也会加重一致性和失效策略的复杂度。
5. 过关标准：知道热点问题不是只靠一个 Redis 集群就能完全解决。
