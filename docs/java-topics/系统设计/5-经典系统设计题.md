# 经典系统设计题 —— 练习入口与复盘方向

前面的内容更多是在积累组件和方法论，这一份把它们落回到经典题型上。重点不是立刻做出“完美架构图”，而是把问题拆开、把关键权衡说出来。

## 高频题型
### 短链接服务（TinyURL） ⭐
1. 重点：ID 生成、短码映射存储、重定向策略、读多写少下的缓存设计。
2. 为什么常考：它体量不大，但能同时考到数据库、缓存、ID、热点和可扩展性。
3. 复盘时要问自己：热点短链怎么处理？短码怎么生成？301 和 302 为什么选其一？

### 秒杀系统 ⭐
1. 重点：流量削峰、库存控制、防超卖、限流、异步落库。
2. 为什么常考：它能集中考你对高并发、高可用、最终一致性的理解。
3. 复盘时要问自己：高峰流量先挡在哪一层？库存在哪里扣？失败怎么补偿？

### Feed 流 / 推送系统
1. 重点：推模式、拉模式、推拉结合、大 V 问题、时间线存储。
2. 为什么常考：它能考到读写模型和热点用户问题。
3. 复盘时要问自己：大 V 发一条内容，为什么不能直接全量推？

### 分布式限流器
1. 重点：单机限流 vs 分布式限流、令牌桶 / 漏桶 / 滑动窗口、Redis + Lua。
2. 为什么常考：它看起来小，但能很好测试你是否理解网关治理和共享状态。
3. 复盘时要问自己：为什么分布式限流需要原子性？为什么常常用 Lua？

### 订单系统
1. 重点：订单状态机、支付回调、幂等、分库分表、超时关单。
2. 为什么常考：它是最典型的综合题，几乎能把数据库、缓存、MQ、事务、幂等全部串起来。
3. 复盘时要问自己：订单状态怎么设计？支付回调重复怎么办？库存和订单一致性怎么保证？

## 练习方式建议
1. 先按“需求澄清 -> 容量估算 -> 主流程 -> 关键组件 -> 高可用与高性能”的顺序答一遍
2. 每次只重点练一两个核心权衡，不追求第一次就把所有点都答满
3. 练习后一定复盘：哪些地方只是列术语，哪些地方能真正说出为什么
4. 过几天重复设计同一道题，看自己有没有形成稳定框架
