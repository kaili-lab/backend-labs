# 系统设计方法论 —— 先搭框架，再选组件

系统设计最怕一上来就开始画组件图。真正好的回答，通常是先把问题抽象清楚：需求是什么、规模多大、什么最重要、约束在哪里，然后再选技术。

## 面试回答框架
### 系统设计面试的标准流程 ⭐
1. 本质：系统设计题本质上是在考“结构化思考”。标准流程通常是：先明确需求，再做规模估算，再确定核心数据流，最后才补高可用和扩展性。
2. 为什么需要它：没有方法论，回答会很散；有了框架，即使细节没答全，整体也会显得清楚可靠。
3. 常见步骤：
   1. 先澄清功能需求和非功能需求；
   2. 粗略做容量估算；
   3. 设计核心数据模型与主流程；
   4. 再补缓存、消息队列、分库分表、限流、高可用等。
4. 关联：后面所有组件设计，其实都是在这个流程框架下往里填答案。
5. 过关标准：知道系统设计题不是“想到什么说什么”，而是有固定答题顺序。

### 容量估算（Back-of-the-envelope Estimation）
1. 本质：容量估算不是要算得特别准，而是快速判断系统量级和关键瓶颈大概会落在哪里。
2. 为什么需要它：没有量级，很多设计选择都没有意义。比如 1 万 QPS 和 10 QPS 的系统，不可能用同一套成本结构看待。
3. 使用场景：
   1. 估算 QPS、带宽、存储量、每日新增数据；
   2. 判断数据库单机能不能扛住；
   3. 判断需不需要缓存、MQ、分库分表。
4. 关联：容量估算不是数学题，而是帮你筛掉明显不合理方案。
5. 过关标准：能快速给出一个粗量级，并用它支撑后面的设计选择。

### 非功能需求的关键维度
1. 本质：系统设计不只看功能，还要看一致性、可用性、延迟、吞吐、扩展性、成本、可观测性这些非功能维度。
2. 为什么需要它：很多设计题没有唯一正确答案，真正区分优劣的是你优先保护哪几个维度。
3. 使用场景：
   1. 金融系统更偏一致性；
   2. 内容分发更偏可用性和吞吐；
   3. 高并发接口更重视长尾延迟。
4. 关联：这会直接决定你后面选择 CP / AP、缓存 / 直查、同步 / 异步等方案。
5. 过关标准：能在开头先说清楚“这道题我最优先保障什么”。

## 面试场景练习
1. “如果系统流量翻 10 倍，你先问哪些问题，再决定方案？”
2. “为什么系统设计题要先做量级估算，而不是先画架构图？”
3. “什么叫非功能需求，它为什么会影响技术选型？”
