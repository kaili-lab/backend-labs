# 框架中的单例与多实例 —— 不只是语法问题，更是生命周期设计

很多人第一次接触 Spring Boot 时，会觉得有点矛盾：  
一边是 `Controller`、`Service`、`Repository` 这类对象经常被说成“默认单例”；  
另一边是请求参数对象、表单对象、`@RequestBody` 对象、局部变量，又明显不是全局只存在一份。

这个理解方向是对的，但还可以再精确一点：

- Spring Boot 里，**确实同时存在单例和多实例**
- NestJS 里，**也同时存在单例和多实例**
- Hono 里，**也同时存在共享对象和按请求创建的对象**，只是它默认没有像 Spring / Nest 那样强势的 IoC 容器来统一管理“类实例生命周期”

所以真正要理解的，不是“某个框架是不是单例框架”，而是：

1. 哪些对象应该被长期复用
2. 哪些对象必须和一次请求绑定
3. 框架靠什么机制来区分这两类对象

---

## 先给结论

### 1. 你的理解对吗

对，但要补一句：

- 在 Spring Boot 里，`Controller`、`Service`、`Repository` 之所以常见为单例，不是因为这些类天生只能单例，而是因为它们是 **Spring 容器管理的 Bean**，而 Spring Bean 的默认作用域就是 `singleton`
- 请求参数对象、`@RequestBody` 对象、`@ModelAttribute` 对象，通常**不是容器里的单例 Bean**，而是在一次请求处理过程中由 MVC 参数解析、反序列化、数据绑定等机制临时创建出来的对象

也就是说，**单例还是多实例，不取决于“类名长什么样”，而取决于“这个对象由谁管理、活多久、是否需要隔离状态”**。

### 2. Node.js 框架里有没有同样的问题

有，而且很常见。

- 在 `NestJS` 里，这个问题和 Spring 非常像，因为它也有 IoC 容器和显式的 scope
- 在 `Hono` 里，没有那么重的容器概念，但依然会同时存在：
  - 启动期创建、被长期复用的共享对象
  - 每次请求单独生成的 `Context`、请求数据对象、局部状态

所以从工程视角看，**Node.js 框架并不是“天然全是单例”或者“天然全是多实例”**，而是同样在做生命周期分层。

## 一、Spring Boot：为什么 `Controller` 常常是单例，而请求参数对象不是

### Spring Boot 里的单例对象

Spring Boot 底层还是 Spring 容器。只要一个对象被注册成 Spring Bean，而你又没有改 scope，它默认就是单例。

常见例子：

- `@Controller` / `@RestController`
- `@Service`
- `@Repository`
- `@Component`
- `@Configuration`
- 通过 `@Bean` 注册的大量基础设施对象
- 很多框架组件，比如拦截器、消息转换器、配置绑定后的某些共享组件

这些对象的共同特点是：

- 需要被很多请求反复使用
- 创建成本通常比普通数据对象高
- 更适合承载“能力”，不适合承载“某一次请求的临时状态”

### Spring Boot 里的多实例对象

Spring Boot 项目里同样有大量多实例对象，只是它们不一定都以“Bean scope”这个名义出现。

常见例子：

- `@RequestBody` 反序列化出来的 DTO / VO
- `@ModelAttribute` 绑定出来的表单对象
- 方法参数里的分页对象、查询对象、过滤条件对象
- 方法内 `new` 出来的临时对象
- 每个请求独立的异常对象、校验结果对象、返回值包装对象
- 显式声明为 `@RequestScope` 的 Bean
- 显式声明为 `prototype` 的 Bean

这里要特别区分两类“多实例”：

#### 第一类：不是 Spring 管的 Bean，但每次请求都会新建

例如：

- 控制器方法参数上的 `@RequestBody UserCreateRequest`
- `@ModelAttribute OrderQuery`

这类对象通常是一次请求一次新建，因为请求体、查询参数、表单字段都不一样。  
它们的职责是**承接这次请求的数据**，而不是作为全局共享组件存在。

#### 第二类：是 Spring Bean，但作用域不是单例

例如：

- `@RequestScope`
- `@SessionScope`
- `@ApplicationScope`
- `prototype`

这类对象仍然是 Bean，只是 Spring 给它们配置了不同的生命周期。

### 为什么 Spring 这样设计

核心原因其实很朴素：**把“能力对象”和“请求数据对象”分开管理**。

#### 适合单例的原因

`Controller`、`Service`、`Repository` 这类对象更像“柜台”“流程编排器”“访问数据库的入口”。  
它们本身通常不应该保存某个用户这一次请求的临时状态，所以单例更划算：

- 只创建一次，启动后反复复用
- 节省内存和对象创建成本
- 依赖关系更稳定，容器更容易管理
- 更容易统一织入事务、AOP、日志、监控等能力

但前提是：**它们最好是无状态的**。

如果把下面这种数据塞进单例 Bean 的字段里，就很危险：

- 当前用户 ID
- 当前订单号
- 当前请求参数
- 上一次接口调用算出来的临时结果

因为 Spring Boot 常见运行方式是多线程处理请求，多个线程会同时访问同一个单例 Bean。  
这时只要 Bean 内部保存了会变化的请求态数据，就会出现线程安全问题、数据串线问题。

#### 适合多实例的原因

请求参数对象、表单对象、请求上下文对象之所以适合多实例，是因为它们天然带有“本次请求专属数据”：

- 每个请求内容不同
- 生命周期很短
- 不应该跨请求共享
- 就算创建很多个，成本通常也很低

这类对象的重点不是“复用”，而是“隔离”。

### Spring Boot 里的实际权衡

#### 什么时候优先单例

适合放单例的，通常是下面这类对象：

- 业务规则编排类
- DAO / Repository
- 配置读取与封装类
- 工具协调类
- 连接池、客户端、序列化器、缓存访问器等基础设施对象

它们的关键词是：**共享能力、无请求态、创建成本相对高、希望集中管理**。

#### 什么时候要多实例

适合多实例的，通常是下面这类对象：

- 请求 DTO
- 查询条件对象
- 本次请求的用户上下文
- 需要和 HTTP request 强绑定的状态对象
- 明确带有可变会话状态的对象

它们的关键词是：**状态隔离、生命周期短、与请求强绑定**。

#### Spring 的一个典型提醒

Spring 官方对 `prototype` 有一个很重要的提醒：  
它适合有状态对象，而单例更适合无状态对象。

这句话很值得记住，因为它直接点出了生命周期选择背后的核心标准：  
**你管理的是“共享能力”，还是“瞬时状态”**。

## 二、NestJS：和 Spring 很像，但背后的运行模型不同

### NestJS 里的单例对象

NestJS 也有 IoC 容器，默认也是把大部分 provider 当成单例来管理。

常见例子：

- `@Injectable()` 标记的 service/provider
- 默认 scope 的 controller
- 数据库连接相关 provider
- 配置服务
- 大部分 guard、interceptor、pipe、filter 的共享实例

Nest 官方对这一点说得很直接：  
在 Nest 中，几乎所有东西默认都是跨请求共享的，单例是默认推荐方案。

### NestJS 里的多实例对象

NestJS 里常见的多实例对象，主要有三类：

#### 1. `REQUEST` scope provider

如果一个 provider 标记成 `Scope.REQUEST`，那么每个请求都会得到一个新实例。

常见场景：

- 多租户中按请求决定租户上下文
- 每个请求独立收集 trace / request id
- 每个请求独立的缓存
- 需要直接访问原始 request 对象的服务

#### 2. `TRANSIENT` scope provider

`TRANSIENT` 不是“每个请求一个”，而是“每个消费者注入时给一份新实例”。

这更像“每次依赖注入都单独给你一份工具对象”。  
适合：

- 带调用链上下文的 logger
- 希望不同消费者拿到不同实例的轻量对象

#### 3. 请求处理过程中产生的数据对象

例如：

- `@Body()` 接收的 DTO
- `@Query()` 解析出的查询参数对象
- pipe 转换后的参数对象
- 方法内部创建的局部对象

这些对象往往不是“全局共享 provider”，而是请求进入控制器方法前后生成的数据承载体。

### 为什么 NestJS 默认更偏单例

Nest 官方解释过一个很重要的背景：  
Node.js 不是“一个请求一个线程”的模型，因此让 provider 跨请求共享在大多数场景下是安全且高效的。

这里要注意，“安全”不是说可以随便在单例里塞请求态数据，而是说：

- 共享实例本身没有问题
- 真正危险的是**在共享实例里保存可变的请求专属状态**

所以 NestJS 的推荐思路和 Spring 很像：

- 默认尽量单例
- 只在确实需要请求隔离时才用 request scope

### NestJS 里的实际权衡

#### 为什么默认单例

因为单例 provider 有明显优势：

- 启动时创建一次即可
- 依赖图更稳定
- 内存占用更可控
- 平均响应成本更低

Nest 官方也明确提醒：  
request-scoped provider 会带来性能影响，因为每个请求都要重新创建实例；如果依赖链里某个 provider 是 request scope，还会把上层 controller 一起“带成” request scope。

也就是说，在 NestJS 中，请求作用域不是不能用，而是**要用在真有必要的地方**。

#### 什么时候该用 request scope

下面这些情况用 request scope 是合理的：

- 需要把 request 对象注入服务
- 需要每个请求独立的 tenant context
- 需要每个请求独立统计、追踪、缓存
- 某个对象的正确性依赖“这次请求的边界”

#### 什么时候不该用 request scope

下面这些对象通常不值得做成 request scope：

- 普通业务 service
- 数据库访问 service
- 配置服务
- 大部分纯计算逻辑
- 无状态的工具类

原因很简单：它们本身不依赖请求边界，做成 request scope 只会增加实例创建和 GC 压力。

## 三、Hono：没有重型容器，但照样有共享对象和请求态对象

### 先说一个容易误解的点

如果你从 Spring 或 NestJS 过来，容易下意识地问：

- Hono 里的哪个 class 是单例？
- 哪个 provider 是 request scope？

但 Hono 默认不是这套思路。  
它没有像 Spring / NestJS 那样默认替你管理一大批 class 实例生命周期的 IoC 容器。

所以在 Hono 里，更准确的问法通常是：

- 哪些对象是在应用启动时创建并长期复用的？
- 哪些对象是在每次请求处理时创建的？

### Hono 里常见的“共享对象”

常见例子：

- `const app = new Hono()` 创建出来的应用实例
- 启动阶段注册好的路由定义
- 启动阶段挂载的 middleware
- 模块级创建的数据库客户端、连接池、配置对象
- middleware 工厂在启动时生成并复用的闭包

这类对象本质上就相当于“单例式共享对象”。

比如：

- 你在模块顶层创建一个数据库客户端
- 然后在 handler 里反复使用它

这就是典型的共享实例。

### Hono 里常见的“多实例对象”

Hono 官方文档对 `Context` 说得非常明确：  
`Context` 会为每个请求实例化一次，并一直保留到响应返回。

所以这类对象天然是按请求创建的：

- `Context`
- `c.req`
- 解析出来的 body / query / param 数据
- `c.set()` 放进去的本次请求变量
- handler / middleware 里的局部变量
- 你在 handler 里手动 `new` 出来的临时对象

它们的特点和 Spring / NestJS 里的请求对象很像：  
**生命周期短，只服务这一次请求。**

### 为什么 Hono 里也必须区分这两类对象

虽然 Hono 没有重型 Bean 容器，但后端应用的基本矛盾并没有消失：

- 有些资源很贵，应该复用
- 有些数据很短命，必须隔离

比如：

#### 适合共享

- 数据库连接池
- Redis 客户端
- 配置对象
- 预编译模板
- 纯函数式工具服务

这些东西如果每个请求都重建，成本很高，没有必要。

#### 适合按请求生成

- 当前请求用户信息
- 当前请求解析后的参数
- 追踪 ID
- 权限判断结果
- 当前请求里的临时缓存

这些东西如果做成模块级共享状态，就很容易串请求。

### Hono 的一个关键工程习惯

在 Hono 里，最常见的问题不是“框架把东西错误地做成单例”，而是**开发者自己把请求态数据放进了模块级共享变量**。

例如：

- 在模块顶层定义一个可变对象保存“当前用户”
- 在 middleware 外面保存“本次请求的临时数据”

这类写法在并发请求下会互相污染。  
所以 Hono 的经验法则通常是：

- 共享资源放模块级
- 请求态数据放 `Context`、局部变量或按请求创建的对象里

## 四、把三个框架放在一起看

| 框架 | 常见单例 / 共享对象 | 常见多实例对象 | 默认倾向 |
| --- | --- | --- | --- |
| Spring Boot | `Controller`、`Service`、`Repository`、`@Configuration`、基础设施 Bean | `@RequestBody` DTO、`@ModelAttribute` 对象、`@RequestScope` Bean、局部变量 | 默认单例 Bean |
| NestJS | 默认 provider、默认 controller、配置服务、数据库 provider | `REQUEST` scope provider、`TRANSIENT` provider、`@Body()` DTO、局部对象 | 默认单例 provider |
| Hono | `Hono` app、路由定义、模块级客户端、启动时创建的 middleware 闭包 | `Context`、`c.req`、请求数据对象、局部状态、handler 内临时对象 | 默认共享应用对象 + 请求态对象手动隔离 |

这张表背后其实是同一个逻辑：

- **能力**尽量复用
- **状态**尽量隔离

## 五、为什么有些类适合单例，有些更适合多实例

这个问题不分框架，底层判断标准是一样的。

### 1. 是否持有可变状态

如果一个对象会保存本次请求特有的数据，它就不适合做全局共享单例。  
否则多个请求一进来，状态就会互相覆盖。

适合单例：

- 无状态 service
- 仅读配置的组件
- 线程安全 / 并发安全的共享客户端

适合多实例：

- 当前请求上下文
- 会话态对象
- 带请求级缓存的对象

### 2. 创建成本高不高

如果一个对象创建代价高，而且逻辑上可以安全复用，就应该尽量共享。

典型例子：

- 数据库连接池
- HTTP 客户端
- 序列化器
- 配置解析器

如果每次请求都重建，纯属浪费。

### 3. 是否需要请求隔离

如果一个对象的正确性依赖这次请求的边界，那它就应该是多实例或者 request-scoped。

典型例子：

- 当前租户
- 当前用户
- trace id
- request-level cache

这类信息一旦共享，语义马上错掉。

### 4. 并发模型会影响风险暴露方式，但不改变结论

Spring Boot 常见是多线程并发，请求共享单例 Bean 时，线程安全问题会更直观地暴露出来。  
Node.js 常见是事件循环模型，不是每个请求一个线程，但这并不等于“共享可变状态就没事”。

真正该记住的是：

- Java 里常见风险是多线程同时改同一份状态
- Node.js 里常见风险是异步交错导致共享状态被后来的请求覆盖

表面形式不同，工程结论一样：  
**请求态数据不要塞进长期共享对象里。**

### 5. 容器能力越强，生命周期越显式；容器越轻，越靠开发习惯守边界

这也是 Spring / NestJS / Hono 的一个很有意思的对比：

- Spring：作用域体系最完整，生命周期管理最强
- NestJS：也有显式 scope，但整体比 Spring 轻一些
- Hono：更轻，更自由，但也更依赖开发者自觉区分共享对象和请求态对象

所以不是谁先进谁落后，而是：

- 你要更多自动化管理，还是更多自由度
- 你愿不愿意用框架约束来换一致性

## 六、几个最容易踩的误区

### 误区 1：`Controller` 是单例，所以它里面什么都不能变

不完全对。

更准确地说：

- 单例对象里**可以有字段**
- 但这些字段不应该保存“请求专属、会变化、会互相覆盖”的状态

像注入的只读依赖、配置项、线程安全客户端，这些放在字段里都没问题。

### 误区 2：Node.js 是单线程，所以共享状态天然安全

这是常见误解。

Node.js 不是“没有并发”，而是“并发模型和 Java 不一样”。  
多个请求的异步流程照样会交错执行，所以共享可变状态照样可能串。

### 误区 3：只要做成 request scope 就一定更合理

也不对。

request scope 是用隔离换正确性，但也会带来：

- 更多对象创建
- 更复杂的依赖传播
- 更高的 GC 压力
- 更差的平均性能

所以它适合“必须和请求绑定”的对象，不适合滥用。

### 误区 4：请求参数对象是“多实例 Bean”

很多时候不是。

在 Spring MVC 和 NestJS 里，控制器方法参数里的 DTO / body 对象，往往只是请求处理流程中临时构造出的数据对象，不一定是 IoC 容器长期管理的 Bean / provider。

这点如果不分清，很容易把“对象生命周期”和“DI 容器管理范围”混成一件事。

## 七、落到工程实践，应该怎么判断

如果你在写一个类，先问自己四个问题：

### 1. 这个对象表示的是“能力”还是“状态”

- 表示能力：优先考虑单例 / 共享
- 表示状态：优先考虑多实例 / 请求隔离

### 2. 这个对象里会不会保存某次请求独有的数据

- 会：不要轻易做共享单例
- 不会：更适合长期复用

### 3. 它创建起来贵不贵

- 很贵：优先想办法复用
- 很轻：按请求创建也没太大问题

### 4. 一旦共享错了，代价是什么

- 如果只是多占点内存，问题还小
- 如果会导致串用户、串租户、串权限，那就必须隔离

这四个问题，比死记“哪些类默认是单例”更重要。  
因为默认规则只是起点，真正决定设计质量的是你有没有把生命周期和状态边界想清楚。

---

## 八、最后用一句话收住

单例和多实例，表面看是在讨论“对象创建几次”，本质上是在讨论：

- **哪些能力应该复用**
- **哪些状态必须隔离**
- **框架帮你管理到什么程度**
- **你自己要为生命周期边界负责到什么程度**

Spring Boot、NestJS、Hono 只是做法不同，但它们回答的其实是同一个工程问题。
