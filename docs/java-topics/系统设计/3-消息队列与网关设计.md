# 消息队列与网关设计 —— 解耦、削峰与入口治理

这一层放的是系统设计题里非常常见的两个组件：消息队列负责把同步链路拆开，网关负责站在系统最入口处做统一治理。

## 消息队列
### 消息队列的核心价值 ⭐
1. 本质：消息队列最核心的价值通常是三件事：异步解耦、流量削峰、系统间编排。
2. 为什么需要它：如果所有动作都强行同步完成，请求链路会越来越长，系统耦合也会越来越重。
3. 使用场景：
   1. 下单后异步发券、发通知；
   2. 秒杀流量先入队，后端慢慢消费；
   3. 多系统之间通过事件协作。
4. 关联：MQ 引入后，问题会从同步 RT 转移成消息可靠性、幂等性和消费积压。
5. 过关标准：能说出为什么这一步不直接同步调用，而要改成 MQ。

### 消息可靠性、幂等性与顺序性
1. 本质：消息队列上线后，真正难的是“不丢、不乱、不重复出错”。
2. 为什么需要它：只要引入 MQ，就必须接受“至少一次投递”这类现实，所以消费端幂等通常是默认要求。
3. 使用场景：
   1. 生产端确认与重试；
   2. 消费端 ack 与失败重试；
   3. 幂等处理；
   4. 需要顺序时限定分区或单 Key 串行。
4. 关联：这会和接口幂等、订单状态机、补偿机制直接连起来。
5. 过关标准：知道 MQ 的难点不在“发出去”，而在“最终正确处理”。

### 消息积压处理
1. 本质：消息积压说明生产速度已经显著超过消费能力。
2. 为什么需要它：积压不是简单告警，而是系统已经开始滞后，如果不及时处理，最终会拖垮更多下游。
3. 常见应对：扩容消费者、提升单消费者效率、限流上游、丢弃非关键消息、做优先级区分。
4. 关联：这和线程池队列积压是同类问题，本质上都是“处理速度赶不上进入速度”。
5. 过关标准：知道消息积压时不能只盯着 MQ，还要回到上游和下游一起看。

## 负载均衡与网关
### 负载均衡策略
1. 本质：负载均衡解决的是请求如何在多个实例之间分配，不同策略会影响流量均匀性、热点风险和会话黏性。
2. 为什么需要它：系统开始多实例部署后，负载均衡就是最基础的一层流量分发能力。
3. 使用场景：
   1. 轮询：简单、均衡；
   2. 加权：适合不同实例规格不一致；
   3. 一致性哈希：适合缓存命中或会话相关场景。
4. 关联：负载均衡不是“转发请求”这么简单，它会和缓存、会话、灰度发布一起影响系统行为。
5. 过关标准：知道常见策略分别适合什么情况。

### API 网关的职责
1. 本质：API 网关不是普通转发层，而是系统入口治理层。它常常负责鉴权、路由、限流、灰度、聚合、日志和统一错误处理。
2. 为什么需要它：没有网关，很多横切能力就会散落到各个服务里，最终重复建设且难以统一治理。
3. 使用场景：
   1. 做统一鉴权和签名校验；
   2. 对热点接口限流；
   3. 实现灰度发布和路由控制；
   4. 聚合多个下游接口。
4. 关联：网关不是“万能层”，过重的业务逻辑不应该堆到网关里。
5. 过关标准：知道网关该做什么、不该做什么。
