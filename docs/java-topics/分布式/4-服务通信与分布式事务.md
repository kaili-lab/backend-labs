# 服务通信与分布式事务 —— 从调用链到一致性方案

当系统拆成多个服务后，问题就不再只是“接口怎么调”，而是“调用失败怎么办”“跨服务修改数据怎么保证一致”。这一层把服务通信和事务方案放在一起看，会更接近真实业务。

## 服务通信
### 服务间调用方式：Feign / RestTemplate / gRPC
1. 本质：服务间调用本质都是远程调用，差别主要在于调用方式、协议、开发体验和性能表现。
2. 为什么需要它：面试官问“为什么这里用 Feign，不用 gRPC”，想听的不是框架功能清单，而是场景与代价。
3. 使用场景：
   1. `Feign`：Spring Cloud 体系内、开发效率优先；
   2. `RestTemplate` / `WebClient`：更显式地控制调用细节；
   3. `gRPC`：高性能、强契约、内部 RPC 场景更常见。
4. 关联：真正决定线上稳定性的，常常不是调用框架本身，而是超时、重试、熔断、线程池隔离这些周边策略。
5. 过关标准：能说出为什么你的项目当前更适合哪一种调用方式。

### 服务雪崩与容错 ⭐
1. 本质：一个下游服务变慢或不可用，如果调用方没有防护，线程会不断堆积，最终把上游也拖挂，这就是雪崩。
2. 为什么需要它：微服务最大的风险之一，就是小故障沿调用链放大成大故障。
3. 常见应对：
   1. 每个远程调用必须有超时；
   2. 失败率升高时要熔断；
   3. 关键链路做降级或走缓存；
   4. 不同下游尽量线程池隔离。
4. 关联：这和线程池隔离、限流、舱壁模式本质一致，都是防止一个依赖把整条链拖死。
5. 过关标准：能解释“下游慢为什么会拖垮上游”，并说出至少两层防护。

### 分布式链路追踪
1. 本质：链路追踪解决的是“一个请求经过多个服务后，问题到底出在哪一跳”。
2. 为什么需要它：系统一旦拆成多个服务，单靠本地日志已经无法看清全链路。
3. 使用场景：
   1. 统计每一跳耗时，定位慢点；
   2. 通过 traceId 串联日志；
   3. 分析失败传播路径。
4. 关联：链路追踪不直接解决问题，但它决定你能不能快速定位问题。
5. 过关标准：能说出 traceId 的作用，以及为什么微服务里必须有链路追踪。

## 分布式事务
### 2PC / TCC / 本地消息表 / Saga / Seata ⭐
1. 本质：分布式事务不是一个方案，而是一组不同的一致性方案。它们在一致性、性能、侵入性、补偿难度之间做了不同取舍。
2. 为什么需要它：只要一个业务操作要跨多个服务、多个存储，就会遇到“前一步成功，后一步失败怎么办”的问题。
3. 方案理解：
   1. `2PC`：更强一致，但阻塞重、性能差、对可用性不友好；
   2. `TCC`：业务侵入强，但控制力强；
   3. 本地消息表 / 事务消息：更偏最终一致，工程上更常见；
   4. `Saga`：长事务拆步骤，靠补偿回滚；
   5. `Seata`：把常见模式框架化，但你仍然要理解背后的代价。
4. 关联：真正的面试重点通常不是“背方案名字”，而是为什么这里不用强一致，而改用最终一致。
5. 过关标准：能对比至少 2 到 3 种方案，并说明当前业务为什么更适合哪一种。

### 如何选择事务方案
1. 本质：事务方案选择的核心不是技术偏好，而是业务容错边界。
2. 为什么需要它：很多场景其实不需要“所有步骤同时成功或同时失败”，而是需要“最终不要出错，并且用户能接受恢复过程”。
3. 选型思路：
   1. 强一致要求极高：再考虑更重的事务方案；
   2. 可接受短时间不一致：优先最终一致；
   3. 业务可补偿：优先 Saga / 消息方案；
   4. 链路特别核心且步骤少：才考虑 TCC。
4. 关联：你后续做系统设计题时，事务选型常常就是系统设计取舍的关键一问。
5. 过关标准：知道“分布式事务不是默认就上”的原因，也知道为什么很多公司更偏消息最终一致。

## 面试场景题
1. “下单要扣库存和扣余额，分别在不同服务，你怎么保证一致性？”
2. “为什么很多业务最终没有上 TCC，而是选本地消息表或事务消息？”
3. “一个下游服务变慢，为什么会把上游也拖垮？”
