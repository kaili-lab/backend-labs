# 分布式经典问题 —— 横切能力与高频追问

这一层放的是不属于某个单独组件、但又几乎一定会被追问的横切问题。它们往往最能体现你有没有把分布式系统真正串起来。

## 常见横切问题
### 分布式 ID 生成 ⭐
1. 本质：分布式系统需要全局唯一 ID，而且很多时候还希望它大致有序、利于索引写入。
2. 为什么需要它：数据库自增在单机里简单，但分库分表后就不再适用。
3. 常见方案：UUID、数据库自增、号段模式、Snowflake。
4. 关联：ID 是否有序，会直接影响数据库写放大和索引页分裂。
5. 过关标准：知道常见方案的优缺点，能说出为什么很多订单号更偏向 Snowflake 一类方案。

### 分布式 Session
1. 本质：服务多实例部署后，用户请求可能被路由到不同节点，单机内存里的 Session 不再可靠。
2. 为什么需要它：认证态如果只存在本地内存里，一旦切实例，请求就会“突然掉登录”。
3. 常见方案：
   1. Redis 集中存 Session；
   2. JWT 无状态化；
   3. 尽量状态外置。
4. 关联：这和水平扩展的前提“无状态服务”直接相关。
5. 过关标准：知道为什么多实例会带来 Session 问题，也知道 Redis Session 和 JWT 各自的代价。

### 接口幂等性设计 ⭐
1. 本质：同一个请求执行多次，结果仍然只算一次，这就是幂等。
2. 为什么需要它：在分布式环境里，用户重复点击、网关重试、消息重复投递都很常见。如果不做幂等，就会出现重复扣款、重复下单、重复发货。
3. 常见方案：幂等 token、唯一索引、状态机、去重表。
4. 关联：接口幂等和消息幂等本质是同一类问题，都是“一个动作可能被重复执行”。
5. 过关标准：知道哪些接口必须做幂等，也知道常见实现手段。

### 分布式定时任务
1. 本质：单机定时任务在多实例环境下会重复执行，所以分布式环境里必须保证“同一时刻谁来执行、是否需要分片执行”。
2. 为什么需要它：定时对账、过期清理、定时推送这类任务在多实例部署下非常常见，重复执行会直接造成业务错误。
3. 常见方案：
   1. 分布式锁保证同一时刻只有一个实例执行；
   2. 使用 XXL-Job、ElasticJob 这类专门框架；
   3. 大任务做分片并行。
4. 关联：它和分布式锁、调度中心、高可用设计是连在一起的。
5. 过关标准：知道单机 Cron 为什么到了分布式环境就不够用了。

## 推荐学习资源
1. 《凤凰架构》—— 中文分布式系统很好的入门与整体视角
2. 《数据密集型应用系统设计》—— 分布式理论与工程权衡的核心书
3. 小林 coding 的 Redis / 消息队列相关内容 —— 面试导向强
4. RabbitMQ 官方教程 —— 帮助建立更扎实的消息模型理解
