# 注册中心与服务发现

在微服务架构里，这一专题主要解决“服务实例在哪里、调用方怎么找到它、实例是否还健康”这类问题。配置中心经常和注册中心一起出现，但它们解决的不是完全同一个问题。`Nacos` 可以作为常见实现例子来理解这套机制，但专题本身不等于某个具体产品。

## 以 Nacos 为例理解实现
### Nacos 的核心功能 ⭐
1. 本质：Nacos 同时扮演注册中心和配置中心两个角色。注册中心负责服务注册、发现和健康管理；配置中心负责集中配置与动态推送。
2. 为什么需要它：微服务实例的地址是动态变化的，不能硬编码；配置分散在各个服务中也很难统一管理。
3. 使用场景：
   1. 服务启动时向 Nacos 注册自己的实例地址；
   2. 调用方从 Nacos 拉取实例列表，再做负载均衡；
   3. 配置修改后动态推送到服务，无需重启。
4. 关联：Feign、Spring Cloud LoadBalancer 这类调用链路，本质上都依赖注册中心提供的实例列表；但“服务发现”和“配置下发”仍然是两个相邻能力，不要混成一个概念。
5. 过关标准：能把“注册”“发现”“健康检查”串成完整流程，也知道为什么很多产品会顺带把配置中心一起做进去。

### Nacos 的 AP 模式 vs CP 模式 ⭐
1. 本质：Nacos 对临时实例更偏 AP，对持久实例更偏 CP。它不是永远只选一种，而是把 CAP 取舍暴露给使用者。
2. 为什么需要它：服务实例本身往往是动态的，短时间注册信息不一致通常可以接受；但某些元数据或关键路由信息不能乱，必须更偏一致。
3. 使用场景：
   1. 普通微服务实例注册，多数场景用默认 AP 即可；
   2. 对必须精确的配置或持久实例信息，才考虑更偏 CP 的模式。
4. 关联：这正是 CAP 在真实注册中心里的落地，不同注册中心的差异，很多时候就差在这里。
5. 过关标准：知道 Nacos 为什么同时支持 AP 和 CP，也知道多数服务注册场景为什么通常更偏 AP。

### 健康检查与实例剔除
1. 本质：注册中心不是只存一张地址表，它还要持续判断实例是不是活着。只有这样，调用方拿到的服务列表才有意义。
2. 为什么需要它：实例已经宕机但还留在注册表里，调用就会不断打到坏节点，放大故障。
3. 使用场景：
   1. 临时实例通过心跳上报维持健康；
   2. 超时未上报则标记不健康，再进一步剔除；
   3. 持久实例更多依赖服务端主动探测。
4. 关联：健康检查永远不是绝对实时的，所以客户端仍然需要超时、重试、熔断来补上最后一层防线。
5. 过关标准：能解释“为什么有注册中心了，还要做超时和重试”。

### Nacos 挂了会怎样
1. 本质：注册中心挂掉，不等于服务之间立刻完全失联。很多客户端会保留本地缓存，短时间内还能继续按旧实例列表通信。
2. 为什么需要它：这是典型的高可用思维考题。真正要说清的是“哪些能力还能用，哪些能力会失效”。
3. 影响范围：
   1. 已有实例之间短时间调用可能还能继续；
   2. 新服务无法注册；
   3. 实例上下线变化无法及时感知；
   4. 配置变更无法下发。
4. 关联：这类“依赖挂了但核心链路短时间还能跑”的设计，本质上就是缓存与降级思路。
5. 过关标准：能把“短期可用”和“长期失真”这两个层面区分清楚。

## 面试场景题
1. “Nacos 集群挂了 2 个节点，还能不能继续写入？为什么？”
2. “实例已经 OOM 了，注册中心为什么不会立刻摘掉它？”
3. “注册中心挂了，为什么系统不一定马上全挂？”
