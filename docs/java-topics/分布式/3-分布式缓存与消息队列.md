# 分布式缓存与消息队列 —— Redis 与 RabbitMQ

这一层关注两个最常见的基础设施：Redis 负责承接热点与状态共享，RabbitMQ 负责异步解耦与流量削峰。它们几乎是后端面试的高频组合。

## Redis
### Redis 数据结构与业务场景 ⭐
1. 本质：Redis 不是单一 KV，而是一组面向不同业务模型的数据结构。结构选对了，代码和性能都会更顺。
2. 为什么需要它：面试官问 Redis 数据结构，真正想看的是你能不能把结构和业务场景对应起来。
3. 使用场景：
   1. `String`：缓存对象、计数器、分布式锁；
   2. `Hash`：存用户、商品这类对象字段；
   3. `List`：简单消息流、最新列表；
   4. `Set`：去重、标签、共同关系；
   5. `ZSet`：排行榜、延迟任务、按分值排序的数据。
4. 关联：数据结构的选择会直接影响后续的热点 Key、内存占用、命令复杂度和扩展方式。
5. 过关标准：不是背 5 种结构，而是能说出“什么场景为什么选它”。

### 持久化：RDB vs AOF
1. 本质：RDB 更像周期性快照，AOF 更像操作日志。一个偏恢复速度和体积，一个偏数据保留粒度。
2. 为什么需要它：Redis 是内存数据库，不做持久化就意味着重启后数据全丢。做持久化，就必须接受性能、恢复速度和丢失窗口之间的取舍。
3. 使用场景：
   1. 对恢复速度要求高、可接受少量数据丢失时，更偏 RDB；
   2. 对数据保留要求更高时，更偏 AOF；
   3. 生产里常常是两者结合。
4. 关联：这和数据库 binlog、快照思路本质一致，都是“全量 + 增量”的组合取舍。
5. 过关标准：知道 RDB / AOF 的核心差异，知道为什么生产里常常会组合使用。

### 集群、分布式锁与热点问题 ⭐
1. 本质：Redis 一旦进入生产，就不只是“单机缓存”。你需要考虑主从、哨兵、Cluster、分布式锁、大 Key / 热 Key 等问题。
2. 为什么需要它：缓存问题经常不是“有没有缓存”，而是“缓存本身会不会成为瓶颈或风险点”。
3. 使用场景：
   1. 主从 / 哨兵：提高可用性；
   2. Cluster：提高容量和水平扩展能力；
   3. 分布式锁：控制跨实例并发修改；
   4. 热 Key：热点流量全部打到单点；
   5. 大 Key：迁移、删除、网络传输成本异常高。
4. 关联：热 Key 和大 Key 问题，本质上都和“热点集中”有关，后续常会引出本地缓存、多级缓存、异步化、分片等优化手段。
5. 过关标准：知道 Redis 不只是“快”，也会有容量、热点、一致性和可用性问题。

## RabbitMQ
### RabbitMQ 核心模型 ⭐
1. 本质：RabbitMQ 的核心是“生产者 -> 交换机 -> 队列 -> 消费者”。交换机决定消息怎么路由，队列决定消息怎么被消费。
2. 为什么需要它：面试官问 MQ 往往不是让你背组件名，而是看你是否理解消息从生产到消费中间发生了什么。
3. 使用场景：
   1. 异步解耦：下单后发通知、发券、记日志；
   2. 流量削峰：短时间大量请求先入队；
   3. 业务编排：多个系统通过消息协作。
4. 关联：异步化带来的不是“白赚性能”，而是把同步链路压力转成最终一致性和消息可靠性问题。
5. 过关标准：能画出消息流转路径，并解释交换机、队列、消费者各自负责什么。

### 消息可靠性、死信队列与幂等性 ⭐
1. 本质：真正的 MQ 生产问题不在于“能不能发出去”，而在于“会不会丢”“会不会重复”“失败了怎么处理”。
2. 为什么需要它：只要引入 MQ，就必须处理重复消费、消费失败、消息积压和异常补偿，否则只是把问题从同步接口挪到了异步链路。
3. 使用场景：
   1. 生产端确认消息是否成功到 Broker；
   2. 消费端显式 ack，避免处理失败却被当成成功；
   3. 用死信队列承接多次失败的消息；
   4. 用唯一索引、去重表、状态机做消费幂等。
4. 关联：消息幂等和接口幂等本质上是同一类问题：一个动作可能被执行多次，如何保证最终结果只算一次。
5. 过关标准：能说出一条消息从生产到消费要保证可靠，至少要考虑哪些环节。

### 高可用与队列模型
1. 本质：消息队列自己也可能出故障，所以你还要考虑主从、高可用队列、恢复能力和消费模型。
2. 为什么需要它：如果 MQ 是系统异步链路的中枢，它挂了往往不是局部故障，而是整条业务链都被卡住。
3. 使用场景：
   1. Quorum Queue：更适合新的高可用场景；
   2. 消费者限流：避免消费太快把下游打挂；
   3. 消息积压处理：扩容消费者、加快消费、分流热点。
4. 关联：这会进一步连接到服务雪崩、熔断降级、线程池隔离等问题。
5. 过关标准：知道 MQ 不只是“发消息”，还要考虑高可用、堆积和恢复。
