# 分布式核心理论 —— 取舍与边界

分布式专题的第一步，不是背组件，而是先建立一套“为什么这样选”的判断框架。很多面试题表面上在问技术名词，本质上是在问你是否理解一致性、可用性和故障边界之间的取舍。

## 一致性与可用性
### CAP 定理 ⭐
1. 本质：当网络分区发生时，分布式系统不可能同时完全保证一致性和可用性。因为网络分区不可避免，所以真正的取舍是：在分区出现时，更偏向一致性，还是更偏向可用性。
2. 为什么需要它：CAP 不是背概念，而是帮你做业务取舍。订单、支付、库存更偏 CP；商品详情、推荐、动态流更偏 AP。
3. 使用场景：
   1. 订单支付系统更重视数据正确，宁愿部分请求失败，也不能错扣钱；
   2. 商品详情页更重视服务可用，短时间读到旧数据往往可以接受。
4. 关联：Nacos 的 AP / CP、Redis 主从复制的最终一致、Zookeeper 的强一致，本质上都在体现 CAP 的不同取向。
5. 过关标准：能结合具体业务说明为什么这里更偏 CP 或 AP，而不是只会背定义。

### BASE 理论
1. 本质：BASE 强调的是“基本可用、软状态、最终一致”，它是对 AP 思路的工程化表达：允许短时间不一致，但希望系统最终回到一致。
2. 为什么需要它：很多业务不值得为强一致支付过高成本。相比一上来就上分布式事务，很多场景用最终一致更实际。
3. 使用场景：
   1. 缓存和数据库的数据同步；
   2. 下单后异步发券、发通知；
   3. 社交动态和消息推送这类允许短时间延迟的业务。
4. 关联：消息队列、异步补偿、缓存延迟双删，本质上都属于 BASE 思路下的方案。
5. 过关标准：知道 BASE 不是“不要一致性”，而是“接受短时间不一致，最终回到一致”。

### 一致性模型：强一致 / 最终一致 / 因果一致
1. 本质：强一致要求写入后所有节点立刻看到同样结果；最终一致允许短时间读到旧值；因果一致强调有因果关系的操作必须保持顺序。
2. 为什么需要它：不同业务对一致性的要求完全不同，不能一刀切。技术选型前，先把一致性要求说清楚。
3. 使用场景：
   1. 金融扣款、库存扣减：更接近强一致；
   2. 缓存更新、异步通知：通常接受最终一致；
   3. 聊天消息、事件流：往往需要因果顺序不能乱。
4. 关联：共识协议更接近强一致，消息队列和异步复制更接近最终一致。
5. 过关标准：能把一致性模型和业务场景对上，而不是只会列术语。

## 故障与隔离
### 脑裂问题 ⭐
1. 本质：由于网络分区，一个集群可能被分成两个都认为自己还活着的小集群，各自继续提供服务，最终导致数据冲突和状态分裂。
2. 为什么需要它：脑裂是分布式系统里最危险的故障之一，因为它不是简单不可用，而是“错误地继续可用”。
3. 常见应对：
   1. 过半机制（Quorum），确保同一时刻只有一个分区能继续决策；
   2. Fencing Token，防止旧节点拿着过期身份继续写入；
   3. 明确主从切换条件，避免双主并存。
4. 关联：Redis Sentinel、Raft、Zookeeper 背后都在解决同类问题。
5. 过关标准：能解释脑裂为什么危险，知道常见防护思路。

## 面试场景题

2. “为什么说 P 不是可选项？”
3. “脑裂为什么比简单宕机更危险？”
4. “你说用了最终一致性，那一致性窗口能接受到什么程度？”
