# 第一层：Java 内存模型（JMM）
> 为什么从这里开始？理解了 JMM，锁、volatile、happens-before 就不再需要死记硬背，而是能自然推导出来。这一层是面试中"为什么"类追问的根基。
>

## 主内存与工作内存模型 ⭐
### 本质
每个线程不是直接操作主内存，而是先把变量拷贝到自己的工作内存中操作，操作完再写回去。这就像每个人各自拿了一份文档副本去改，改完再合并——自然可能冲突。

### 为什么需要它
因为 CPU 和主内存之间的速度差距太大（约 100 倍），所以硬件设计了多级缓存。JMM 是 Java 对这个硬件现实的抽象规范——它告诉你：多线程环境下，你不能假设一个线程的修改对另一个线程立刻可见。

### 关联
这是理解 volatile（强制刷回主内存）、synchronized（加锁时同步主内存）、happens-before（定义什么操作之间保证可见性）的根基。
### 过关标准
能画出线程、工作内存、主内存三者的关系图，能解释为什么两个线程修改同一个变量可能互相看不到

## happens-before 原则 ⭐
### 本质
JMM 定义的一组规则，告诉你"在什么条件下，操作 A 的结果一定对操作 B 可见"。它不是说时间上 A 先发生，而是说 A 的效果对 B 保证可见。

**程序次序规则**
- 规则内容：同一个线程内，前面的操作 happens-before 后面的操作。
- 它解决什么问题：解决“单线程语义不能被并发优化破坏”的问题。
- 为什么需要它：
    - JVM 和 CPU 会做指令重排优化。
    - 但如果连同一个线程里“先写 A，再写 B”都不能成立，那程序员根本没法写代码。
- 对线程安全的意义：
    - 它保证你在单线程内部看到的执行结果符合代码逻辑。
    - 这是其他跨线程推理的起点。
- 但它的边界也很重要：
    - 这个规则只约束同一个线程内部。
    - 它不能保证另一个线程能马上看到这些操作。

**管程锁定规则（unlock 先于后续 lock）**
- 规则内容：对同一把锁，线程 A 的 unlock，happens-before 线程 B
  后面对这把锁的 lock。
- 它解决什么问题：解决“临界区内修改的共享数据，对后来拿到同一把锁的线程不可见”的问题。
- 为什么需要它：
    - 如果没有这个规则，即使大家都用了 synchronized，后一个线程
      也可能看不到前一个线程在锁内的更新，那加锁就没意义了。
- 对线程安全的意义：
    - 它让锁不只是“互斥”，还带来“释放前写入的数据，对后续获得同一锁的线程可见”。
    - 这就是为什么 synchronized 不只是防同时进入，还能保证数据一致。
- 它实际解决的两层问题：
    - 互斥问题：同一时刻只有一个线程进临界区
    - 可见性问题：前一个线程改过的值，后一个线程能看见

**volatile 变量规则**

- 规则内容：对一个 volatile 变量的写，happens-before 后面对这个
  变量的读。
- 它解决什么问题：解决“一个线程修改了状态，另一个线程一直看不到”的问题。
- 为什么需要它：
    - 多线程下，每个线程可能从自己的工作内存、缓存里读数据。
    - 普通变量的更新，不保证别的线程立刻看见。
- 对线程安全的意义：
    - 它保证一个线程对 volatile 的写，对另一个线程后续读这个变量时是可见的。
    - 同时还会约束相关重排序，避免“标志位已经改了，但真实数据还没准备好”的问题。
- 最典型场景：
    - 停止标志 running
    - 配置开关
    - 状态发布
- 但它不能解决什么：
    - volatile 不能保证复合操作原子性
    - 比如 count++ 仍然不安全，因为它是读-改-写三步。

**传递性**
- 规则内容：如果 A happens-before B，B happens-before C，那么 A happens-before C。
- 它解决什么问题：解决“跨多个同步动作时，如何把可见性链条串起来”的问题。
- 为什么需要它：
    - 现实并发里，线程之间的关系经常不是一步直达，而是通过锁、volatile、start/join 一层层传递。
    - 没有传递性，很多线程安全结论都推不出来。
- 对线程安全的意义：
    - 它是你做并发推理的“演绎规则”。
    - 让你能从局部同步关系，推导出整体可见性。
- 典型理解方式：
    - 线程 A 在锁内写数据
    - 然后释放锁
    - 线程 B 获取锁后再写 volatile 标志
    - 线程 C 读到这个 volatile 标志
    - 那么 A 的写，最终也能推导到 C 可见

**线程启动规则**
- 规则内容：线程对象调用 start()，happens-before 新线程中的所有操作。
- 它解决什么问题：解决“主线程在线程启动前准备的数据，新线程是否一定看得见”的问题。
- 为什么需要它：
    - 启动线程时，通常会先设置参数、共享对象、初始状态。
    - 如果新线程看不到这些准备工作，那线程启动就不可靠了。
- 对线程安全的意义：
    - 它保证“先准备，再启动”这条发布路径是安全的。
    - 新线程能看到启动前由创建线程完成的初始化结果。
- 典型场景：
    - 把任务参数放进对象字段
    - 构造好共享资源
    - 然后再 start()

**线程中断规则**：
- 规则内容：一个线程对另一个线程调用 interrupt()，happens-before 被中断线程检测到中断事件。
- 它解决什么问题：解决“取消信号发出后，目标线程能不能可靠感知到”的问题。
- 为什么需要它：
    - 中断是 Java 里协作式停止的重要机制。
    - 如果发出中断但对方不一定看得见，那取消、超时、关闭流程都不可靠。
- 对线程安全的意义：
    - 它保证中断不是“随缘通知”，而是有可见性保障的线程协作信号。
    - 这对于任务取消、服务关闭、阻塞唤醒非常重要。
- 要注意：
    - 中断规则保证的是“中断信号可见”
    - 不等于“线程一定立刻停下”
    - 因为 Java 的中断是协作式的，不是强杀
- 类比：
    - 像你按下员工的“停止工作”呼叫器。
    - 规则保证对方能收到这个信号，但对方收到后怎么收尾，是另一层逻辑。


**线程终止规则**：
- 规则内容：线程中的所有操作，happens-before 其他线程检测到它终止，或者 join() 返回。
- 它解决什么问题：解决“一个线程执行完后，它的结果对等待它的线程是否可见”的问题。
- 为什么需要它：
    - 很多并发协作是“你先算，算完我再读结果”。
    - 如果 join() 返回了，主线程却还看不到子线程写入的数据，那等待就失去意义了。
- 对线程安全的意义：
    - 它保证“线程完成 -> 结果可读”这条协作路径。
    - 所以 join() 不只是等待时间点，也建立了可见性关系。
- 典型场景：
    - 主线程等待多个工作线程计算完成，再汇总结果join() 返回，或检测到它已经终止。
- 对象终结规则：对象构造完成，先于它的 finalize() 方法开始。


## volatile 关键字的语义 ⭐

### 本质
volatile 做了两件事——
1. 保证可见性（修改后立刻刷回主内存，读取时从主内存拿最新值）；
2. 禁止指令重排序。但它不保证原子性——i++ 仍然不安全。

### 适用场景
1. 更准确地说，volatile 适合“状态发布”这类场景：一个线程负责写入最新状态，其他线程只读取这个状态，并且读写都不是复合操作。典型例子是停止标记、配置开关、任务完成标记。

```java
public class VolatileVisibilityDemo {
    private static volatile boolean running = true;

    public static void main(String[] args) throws InterruptedException {
        Thread worker = new Thread(() -> {
            while (running) {
                // 模拟工作线程持续处理任务
            }
            System.out.println("worker stopped");
        });

        worker.start();

        Thread.sleep(1000);
        running = false;

        worker.join();
    }
}
```

上面这个例子里，`running` 只是一个状态位：写线程负责发布新状态，读线程只关心能否及时看到变化，因此 `volatile` 足够。

如果变成“多线程都写”，问题通常就不再只是可见性，而是原子性。例如 `count++` 会拆成“读 -> 改 -> 写”三步，多个线程可能互相覆盖结果。此时：
- `volatile` 只能保证每次读到的是新值；
- 但不能保证这三步是一个不可分割的整体；
- 所以通常要用 `synchronized`、`Lock` 或 `AtomicInteger` 这类同步手段。

```java
public class VolatileCounterDemo {
    private static volatile int count = 0;

    public static void main(String[] args) throws InterruptedException {
        Thread t1 = new Thread(() -> {
            for (int i = 0; i < 1000; i++) {
                count++;
            }
        });

        Thread t2 = new Thread(() -> {
            for (int i = 0; i < 1000; i++) {
                count++;
            }
        });

        t1.start();
        t2.start();
        t1.join();
        t2.join();

        System.out.println(count);
    }
}
```

这里结果不一定是 `2000`。原因不是“看不见”，而是两个线程可能都读到同一个旧值，再分别写回，产生丢失更新。

### 关联
1. DCL 不是“只靠锁保证可见性”，而是“锁负责互斥创建，volatile 负责安全发布和禁止重排序”。

```java
public class Singleton {
    private static volatile Singleton instance;

    private Singleton() {
    }

    public static Singleton getInstance() {
        // volatile的可见性，在这里发挥作用
        if (instance == null) {
            // 保证同一时间只有一个线程能进入内部，保证实例只被创建一次
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

这里的 `volatile` 不是为了互斥，而是为了防止 `instance = new Singleton()` 发生指令重排序，导致别的线程拿到一个“引用已经可见，但对象还没完全初始化”的实例。

DCL 里两种机制分工不同：
- `synchronized`：保证同一时刻只有一个线程进入初始化逻辑，避免创建多个实例；
- `volatile`：保证初始化完成后的实例对其他线程立即可见，并禁止“先把引用赋值、后执行构造”的重排序。

如果只有锁没有 `volatile`，外层无锁读取 `instance` 时，仍然可能读到一个“非 null 但未完全初始化”的对象引用。

2. volatile 不保证原子性这一点，解释了为什么计数器场景要用 AtomicInteger 而不是 volatile int。

这里的关键是区分“可见性”和“原子性”：
- `volatile int` 能保证线程读到的是最新值；
- 但 `count++` 不是一个原子动作，而是 “读 --> 加一 --> 写回” 三步；
- 多个线程同时执行时，可能都读到同一个旧值，再分别写回，最终产生丢失更新。

所以计数器场景的问题通常不是“看不见”，而是“多个线程同时修改时，更新被覆盖”。这也是为什么 `volatile` 适合状态标记，不适合高并发计数。

`AtomicInteger` 解决的是“安全更新”问题。它通过 CAS 这类原子操作，保证加一过程不会被其他线程无声覆盖，因此在并发计数场景下，应该优先使用 `AtomicInteger`，而不是 `volatile int`。

### 过关标准
能说清楚 volatile 解决了什么问题（可见性 + 禁止重排序）、不能解决什么问题（原子性），能举出 volatile 适用和不适用的具体场景


## 指令重排序与内存屏障
### 本质
编译器和 CPU 为了优化性能，会在不影响单线程语义的前提下调整指令执行顺序。内存屏障（Memory Barrier）是一种特殊指令，强制屏障前后的操作不能跨越屏障重排。

### 为什么需要它
重排序在单线程下无害，但在多线程下可能导致一个线程看到另一个线程的操作以"不合逻辑"的顺序发生。典型案例就是 DCL 中对象半初始化问题。理解这个才能真正明白 volatile 和 synchronized 的底层保障机制。

### 关联
volatile 写操作后插入 StoreStore + StoreLoad 屏障，读操作前插入 LoadLoad + LoadStore 屏障——这就是 volatile 禁止重排序的底层实现。synchronized 在进入和退出时也有对应的屏障。

### 过关标准
能解释编译器和 CPU 为什么要重排序，volatile 和 synchronized 是如何通过内存屏障阻止重排序的

## JMM 与实际硬件的关系
### 本质
JMM 是 Java 提供的一层并发语义规范，它不等于真实硬件，也不是 JVM 随便发明出来的一套规则，而是站在语言层面对底层硬件差异做出的统一抽象。程序员写的是 `volatile`、`synchronized`、`final`、`happens-before`，JVM 再把这些语义翻译成当前平台能执行的机器级约束。

从程序员视角看，可以先抓住一句话：JMM 不是在描述“CPU 里面真实发生了什么细节”，而是在承诺“只要你遵守这些并发规则，不管底层是 x86 还是 ARM，Java 程序都应该表现出一致的并发语义”。

### 为什么需要它
如果没有 JMM，Java 并发编程就会直接暴露在硬件差异面前，而现实中的硬件并不是“所有线程天然共享一个绝对实时一致的内存视图”：
- 每个 CPU 核心都有自己的缓存层级，如 L1 / L2 / L3；
- 编译器和 CPU 都会为了性能做重排序；
- 不同架构对读写顺序的默认保证强弱不同，x86 较强，ARM 通常更弱。

这意味着：同一段多线程代码，在一个平台上“看起来没问题”，换个平台却可能暴露 Bug。JMM 的作用，就是把这些底层差异收敛掉，让程序员不必直接按某一种 CPU 的规则写并发代码。

### 程序员应该怎么理解它
可以把 JMM 理解成“Java 并发世界里的交通规则”，而不是“硬件结构图”。它最重要的价值，不是让你背 CPU 细节，而是帮你建立下面这套判断方式：
- 一个线程写入的数据，另一个线程什么时候一定能看到；
- 哪些操作之间允许重排序，哪些不允许；
- 哪些同步手段能建立可靠的可见性和有序性；
- 出现并发 Bug 时，应该先从可见性、原子性、有序性哪一类问题入手分析。

因此，学习 JMM 的重点不是“工作内存是不是某一级缓存”的一一对应，而是知道：Java 里的共享变量并不是总能被所有线程立即、按直觉地看到，必须通过 JMM 规定的同步手段来建立正确关系。

### 和硬件的大致对应关系
可以做一个“够用但不过度机械”的映射：
- JMM 中的“主内存”可以粗略理解为共享内存视图，底层最终落到物理内存；
- JMM 中的“工作内存”可以粗略理解为线程执行时所依赖的本地副本，现实里常常对应寄存器、CPU Cache、写缓冲区等；
- `volatile`、锁、原子类等机制，最终都会借助 CPU 指令、缓存一致性协议和屏障语义来实现。

这里要特别注意：这只是帮助理解的映射，不是严格的硬件结构对照表。JMM 是语言层抽象，真实实现会受到 JVM、CPU 架构、操作系统的共同影响。

### 关联
MESI 这类缓存一致性协议解决的是“一个核心改了缓存行，别的核心怎么感知”这类硬件层问题；JMM 解决的是“Java 程序员通过什么规则可靠地表达并发意图”。两者不是同一层，但前者是后者落地的重要基础。

也因此：
- `volatile` 的可见性依赖底层缓存一致性和屏障语义实现；
- `synchronized` 的进入和退出，不只是加锁解锁，还隐含了内存可见性语义；
- x86 因为内存序较强，某些问题不容易暴露；ARM 因为更弱，更能暴露那些原本就写错的并发代码。

### 对程序员的实际意义
从工程角度，理解 JMM 主要是为了避免三类错觉：
1. “我已经改了变量，别的线程应该立刻看到。”——不一定，除非用了正确同步手段；
2. “这段代码在我电脑上一直没出问题，所以就是安全的。”——不一定，可能只是当前平台和时机没触发；
3. “加了 `volatile` 就线程安全了。”——不一定，它解决不了复合操作的原子性问题。

所以在写并发代码时，程序员真正要问的是：
- 这段共享状态是否需要可见性保证？
- 这段更新是否需要原子性？
- 这段初始化或发布是否需要禁止重排序？
- 我现在依赖的是 `volatile`、锁、原子类，还是线程启动/结束等 happens-before 关系？

### 过关标准
能站在程序员视角说明：JMM 是对底层硬件差异的统一抽象；它关心的是多线程下的可见性、原子性和有序性；理解它的目的不是背硬件细节，而是知道写并发代码时应该依赖哪些同步规则，而不是依赖“机器大概率会这样执行”。

1. DCL（双重检查锁）单例为什么要加 volatile？不加会怎样？" —— 考指令重排序 + happens-before
2. 两个线程同时对一个 int 变量做 i++，执行 1000 次，结果一定是 2000 吗？为什么？加了 volatile 就行了吗？" —— 考可见性 vs 原子性的区别
3. 你怎么判断一段多线程代码是否有线程安全问题？你的分析思路是什么？" —— 考你能不能用 happens-before 做系统性分析
3.1. 首先看是否存在共享的可变状态。如果多个线程会访问同一份可变数据，尤其是出现“一个线程写、其他线程读”或“多个线程同时写”的情况，就说明这里可能存在线程安全风险，下一步要继续分析是否有足够的同步手段来建立 happens-before 关系
