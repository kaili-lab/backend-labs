# 锁机制与线程协作 —— 工具箱

在 JMM 基础上重新理解锁——不是背"区别"，而是理解"为什么这样设计"。面试中锁相关的追问最密集。

## synchronized 深入
### 对象头与 Mark Word ⭐
1. 本质：每个 Java 对象在内存中除了实例数据，还有一个对象头。对象头中的 Mark Word 是一个动态变化的区域，存储了对象的哈希码、GC 分代年龄、以及锁状态信息。synchronized 加锁就是在修改 Mark Word。
2. 为什么需要它：理解了 Mark Word，你才能明白 synchronized 的锁信息"存在哪里"，以及为什么 Java 中任何对象都能当锁——因为每个对象头里都有 Mark Word。
3. 关联：Mark Word 中的锁标志位（2 bit）标识了当前锁的状态（无锁 / 偏向锁 / 轻量级锁 / 重量级锁），这是锁升级的物理基础。
4. 过关标准：能说出 Java 对象头的结构，知道锁信息存储在哪里

### 锁升级：偏向锁 → 轻量级锁 → 重量级锁 ⭐
1. 本质：JVM 根据竞争程度自动选择锁的"重量级别"
   1. 没竞争时用最轻的偏向锁（几乎零开销），有少量竞争时升级到轻量级锁（CAS 自旋），竞争激烈时升级到重量级锁（OS 互斥量，线程挂起）。
   2. 锁只能升级不能降级。
2. 为什么需要它：因为大部分 synchronized 代码在实际运行时根本没有竞争，或者竞争很少。如果一上来就用重量级锁（操作系统层面的互斥量），性能浪费太大。锁升级是 JVM 对"大部分场景不需要重锁"这个现实的优化。
3. 关联：偏向锁的核心是在 Mark Word 中记录线程 ID，轻量级锁的核心是 CAS 替换 Mark Word，重量级锁的核心是 OS 的 Monitor。这三者和 CAS、操作系统线程调度直接相关。
4. 过关标准：能描述每种锁的适用场景和升级触发条件，知道为什么 JVM 要设计这个升级过程（性能优化）

### synchronized 的 happens-before 语义
1. 本质：一个线程在 synchronized 块中做的所有修改，在它释放锁之后，对下一个获得同一把锁的线程一定可见。这就是 synchronized 的 happens-before 规则。
2. 为什么需要它：synchronized 不只是互斥（同一时刻只有一个线程执行），它还保证可见性。很多人只知道互斥，不知道可见性，这会导致在分析并发安全时遗漏问题。
3. 关联：回到第一层——synchronized 释放锁时，会把工作内存中的变量刷回主内存；获取锁时，会从主内存重新读取。这就是它保证可见性的 JMM 基础。
4. 过关标准：能用第一层的 JMM 知识解释为什么 synchronized 能保证可见性


## AQS 与 ReentrantLock

### AQS（AbstractQueuedSynchronizer）核心原理 ⭐
1. 本质：AQS 是 Java 并发包（java.util.concurrent）的基石框架。它用一个 int 类型的 state 变量表示资源状态，用一个 CLH 双向队列管理排队等待的线程。获取资源就是 CAS 修改 state，获取失败就进队列排队。
2. 为什么需要它：
   1. synchronized 是 JVM 内置的，不够灵活（不能中断等待、不能设超时、不能实现公平锁）。
   2. Java 需要一个灵活的、可扩展的锁框架来构建各种并发工具——AQS 就是这个框架。ReentrantLock、CountDownLatch、Semaphore、ReadWriteLock 底层全是 AQS。
3. 关联：AQS 的 state 用 volatile 修饰（保证可见性），修改 state 用 CAS（保证原子性）——直接回到了第一层的 JMM 和 CAS 知识。理解了 AQS，Java 并发包里一大半的工具你都能看懂源码了。
4. 过关标准：能说出 AQS 的 state + CLH 队列的设计思路，理解为什么它是 Java 并发包的基石

### ReentrantLock vs synchronized ⭐
1. 本质：两者都能实现互斥和可见性，但 ReentrantLock 基于 AQS 实现，提供了更多能力——可中断等待（lockInterruptibly）、可超时尝试（tryLock）、可实现公平锁、支持多个 Condition。
2. 为什么需要它：synchronized 能覆盖 80% 的场景，但在一些特殊需求下力不从心。比如：你想等锁但最多等 3 秒就放弃（tryLock），或者你需要多个等待条件（Condition），这时只能用 ReentrantLock。
3. 关联：ReentrantLock 的"可重入"是指同一个线程可以重复获取同一把锁而不死锁，这和 synchronized 一样。区别在于附加能力，而不是互斥的本质。
4. 过关标准：不是背区别表，而是能说出在什么业务场景下选择 ReentrantLock（如需要 tryLock、公平锁、Condition）

### 公平锁 vs 非公平锁
1. 本质：公平锁严格按照请求顺序（FIFO）分配锁；非公平锁允许新来的线程"插队"——如果锁恰好可用，新线程可以直接获取，不用排队。
2. 为什么需要它：非公平锁性能更好，因为减少了线程挂起和唤醒的开销（线程切换很昂贵）。大部分场景用非公平锁即可。但如果你的场景不能容忍"饥饿"（某些线程长期拿不到锁），就需要公平锁。
3. 关联：ReentrantLock 默认是非公平锁，通过构造函数 new ReentrantLock(true) 切换为公平锁。synchronized 是非公平的，无法切换——这也是选择 ReentrantLock 的场景之一。
4. 过关标准：能解释非公平锁为什么性能更好，公平锁在什么场景下必须使用

### Condition 机制
1. 本质：Condition 是 ReentrantLock 的配套工具，相当于 Object.wait/notify 的增强版。一把锁可以创建多个 Condition，每个 Condition 有独立的等待队列，实现更精细的线程协作。
2. 为什么需要它：Object.wait/notify 只有一个等待队列，无法区分"等待不同条件"的线程。经典例子：生产者-消费者模型中，生产者等"队列不满"，消费者等"队列不空"——用两个 Condition 就能精确唤醒对应的线程，而不是 notifyAll 全部唤醒。
3. 关联：BlockingQueue 的底层实现（如 ArrayBlockingQueue）就是 ReentrantLock + 两个 Condition（notFull 和 notEmpty）。理解了 Condition，你就理解了 BlockingQueue 的原理。
4. 过关标准：能用 Condition 实现一个生产者-消费者模型，理解它和 Object.wait/notify 的区别


## CAS 与原子类
### CAS 原理（Compare And Swap） ⭐
1. 本质：CAS 是一个原子操作——比较内存中的值是否等于预期值，如果是就更新为新值，否则不做任何事。整个操作在硬件层面是一条 CPU 指令（cmpxchg），不可被中断。
2. 为什么需要它：加锁虽然安全，但线程挂起和唤醒的开销很大。CAS 是一种"乐观"策略——假设没有竞争，直接尝试修改，失败了再重试。在低竞争场景下，CAS 比加锁快得多。
3. 关联：CAS 是 AQS 修改 state 的方式，是原子类（AtomicInteger）的底层实现，也是轻量级锁的核心操作。但 CAS 有 ABA 问题——值从 A 变成 B 再变回 A，CAS 会误认为没有变化。AtomicStampedReference 通过加版本号解决。
4. 过关标准：能说出 CAS 的三个操作数，知道它在硬件层面是一条 CPU 指令（cmpxchg），理解 ABA 问题及 AtomicStampedReference 的解决思路

### Atomic 系列类（AtomicInteger、AtomicReference 等）
1. 本质：Java 对 CAS 的封装。内部是 volatile 变量 + Unsafe 类的 CAS 操作。提供了 incrementAndGet、compareAndSet 等原子方法，让你不用加锁就能安全地修改共享变量。
2. 为什么需要它：对于简单的计数、标志位等场景，用 synchronized 或 ReentrantLock 太重了。原子类用无锁的 CAS 实现，开销更小。它填补了"volatile 不能保证原子性"和"锁太重"之间的空白。
3. 关联：底层的 volatile 保证可见性，CAS 保证原子性——正好对应第一层 JMM 的两个核心概念。原子类是 volatile + CAS 的组合应用。
4. 过关标准：能说出它们底层用的是 CAS + volatile，能在合适场景用原子类替代锁

### LongAdder vs AtomicLong ⭐
1. 本质：AtomicLong 用单个变量做 CAS，高竞争下大量线程 CAS 失败重试，浪费 CPU。LongAdder 把一个计数器拆成多个 Cell，每个线程在自己的 Cell 上累加，最后汇总——用"分而治之"降低竞争。
2. 为什么需要它：在高竞争场景（比如全局请求计数器、metrics 统计），AtomicLong 会成为瓶颈。LongAdder 牺牲了精确读取（sum 不是原子的）换来了极大的写性能提升。
3. 关联：LongAdder 的分段思想和 ConcurrentHashMap 的分段锁、Redis Cluster 的分片是同一个核心原理——"一个热点扛不住，就拆成多个"。参见核心原理图谱的「分而治之」。
4. 过关标准：能解释高竞争下 LongAdder 为什么更快（分段思想），知道什么时候用哪个


## 常用并发工具类
### CountDownLatch / CyclicBarrier / Semaphore ⭐
1. 本质：三种不同的线程协作工具。CountDownLatch 是倒计数门——一个线程等多个线程完成（计数减到 0 后放行，一次性）。CyclicBarrier 是集合点——多个线程互相等对方到齐后一起出发（可重复使用）。Semaphore 是信号量——控制同时访问某个资源的线程数量。
2. 为什么需要它：多线程编程中经常需要协调线程之间的执行顺序或限制并发度。比如：主线程等 3 个子线程都查完数据再合并（CountDownLatch），或者限制最多 10 个线程同时访问数据库连接（Semaphore）。这些需求用 wait/notify 实现很复杂，用这些工具一行代码搞定。
3. 关联：三者底层全是 AQS。CountDownLatch 的 state 就是计数值，countDown() 就是 CAS 将 state 减 1，await() 就是检查 state 是否为 0。理解了 AQS，这三个工具的源码都能看懂。
4. 过关标准：不是背 API，而是能说出各自的典型业务场景（如：等多个数据源返回后合并、控制并发数），能说出 CountDownLatch 和 CyclicBarrier 的核心区别（一次性 vs 可重用）

### ReadWriteLock 与 StampedLock
1. 本质：ReadWriteLock 允许读读并行、读写互斥、写写互斥——在读多写少场景中比独占锁性能好得多。StampedLock 更进一步，提供了"乐观读"模式——读操作不加锁，只在读完后验证期间是否有写操作，如果没有就直接用，有就升级为悲观读锁。
2. 为什么需要它：很多业务场景是读多写少（如配置信息、缓存数据）。如果用 synchronized，读和读之间也互斥，白白浪费了并发度。ReadWriteLock 让读操作可以并行，StampedLock 的乐观读更进一步减少了锁的开销。
3. 关联：ReadWriteLock 底层也是 AQS，用 state 的高 16 位记录读锁计数，低 16 位记录写锁计数——一个 int 变量同时管理两种锁。StampedLock 不基于 AQS，是独立实现，性能更好但使用更复杂（乐观读需要手动验证）。
4. 过关标准：能解释读写锁的适用场景（读多写少），知道 StampedLock 的乐观读模式及其优势


## 并发集合与容器
### 为什么要有并发集合
1. 本质：普通集合（如 `HashMap`、`ArrayList`、`HashSet`）默认不是线程安全的。多个线程同时读写时，可能出现数据覆盖、结构损坏、死循环、读到脏数据等问题。并发集合的目标不是“让所有操作都加一把大锁”，而是在保证线程安全的前提下，尽量保留并发度。
2. 为什么需要它：如果每次都自己在外层加 `synchronized`，虽然也能保护集合，但粒度太粗，吞吐量往往很差。并发集合把常见场景内建进容器：有的偏向高并发读写，有的偏向读多写少，有的偏向线程协作与排队。
3. 关联：这部分是“锁、CAS、AQS”在容器层面的落地。理解并发集合，本质上是在理解：不同数据结构会选择不同同步策略。
4. 过关标准：能根据“读写比例、是否需要阻塞、是否需要排序、是否允许最终一致”选择合适的并发集合，而不是无脑用 `ConcurrentHashMap`。

### ConcurrentHashMap ⭐
1. 本质：高并发场景下的线程安全 Map。JDK 8 以后核心思路是“数组 + 链表 / 红黑树 + CAS + 局部同步”，目标是在保证线程安全的同时，避免整张表只靠一把大锁。
2. 为什么需要它：它适合“多线程同时读写 key-value”的场景，比如本地缓存、会话表、去重表、统计表。相比 `Hashtable` 或 `Collections.synchronizedMap`，它的并发度更高。
3. 使用场景：
   - 本地缓存：如用户信息缓存、配置缓存；
   - 并发统计：如接口调用次数、按 key 聚合计数；
   - 请求去重 / 幂等控制：如根据业务 key 记录处理中状态。
4. 注意点：
   - 复合操作仍然要小心，例如“先判断再更新”不一定天然安全；
   - 优先使用 `computeIfAbsent`、`compute`、`putIfAbsent` 这类原子方法；
   - 不要把它误解成“所有业务逻辑都自动线程安全”。

### CopyOnWriteArrayList / CopyOnWriteArraySet
1. 本质：写时复制容器。每次写操作都会复制底层数组，再替换旧数组；读操作直接读旧数组快照，因此读完全无锁或接近无锁。
2. 为什么需要它：适合“读远多于写，且可以接受写成本较高”的场景。它用写放大换取读性能和遍历稳定性。
3. 使用场景：
   - 监听器列表、观察者列表；
   - 配置快照、白名单快照；
   - 很少变更、但频繁遍历的集合。
4. 注意点：
   - 写多时性能会很差，因为每次写都复制数组；
   - 占内存更多；
   - 迭代看到的是快照，不一定反映最新写入。

### ConcurrentLinkedQueue
1. 本质：基于 CAS 的无界非阻塞队列，适合多线程并发入队、出队。
2. 为什么需要它：当你只需要“线程安全地排队”，但不需要阻塞等待时，它比外部加锁的普通队列更自然。
3. 使用场景：
   - 异步任务暂存；
   - 事件收集；
   - 生产快、消费也快，但不需要背压控制的队列。
4. 注意点：
   - 它是无界的，生产速度远大于消费速度时，可能导致内存压力；
   - 它不负责阻塞等待，不适合天然需要“没数据就等”的生产者-消费者模型。

### BlockingQueue 家族 ⭐
1. 本质：带阻塞语义的队列，是生产者-消费者模型最常用的并发容器。队列满时生产者可阻塞，队列空时消费者可阻塞。
2. 为什么需要它：它把“排队 + 等待 + 唤醒”这套线程协作机制内建好了，比手写 `wait/notify` 或 `Condition` 简单且稳定得多。
3. 常见类型与场景：
   - `ArrayBlockingQueue`：有界数组队列，容量固定，适合需要明确限流和背压的场景；
   - `LinkedBlockingQueue`：链表队列，可选有界或近似无界，吞吐较好，线程池默认常见；
   - `PriorityBlockingQueue`：按优先级出队，适合任务优先级调度；
   - `DelayQueue`：元素到期后才能取出，适合延迟任务、过期检查；
   - `SynchronousQueue`：不存元素，生产者必须直接把任务交给消费者，适合直接移交型场景。
4. 注意点：
   - 无界队列容易掩盖消费能力不足，最后把压力变成内存问题；
   - 有界队列更适合生产系统，因为它能把“处理不过来”尽早暴露出来；
   - 线程池和消息积压问题，很多时候本质上就是队列选型问题。

### ConcurrentSkipListMap / ConcurrentSkipListSet
1. 本质：线程安全且有序的 Map / Set，底层基于跳表，支持并发读写下的有序访问。
2. 为什么需要它：`ConcurrentHashMap` 虽快，但不保证有序；如果业务同时需要线程安全和有序范围查询，就要考虑这类容器。
3. 使用场景：
   - 按时间、分数、排名维护有序数据；
   - 需要 `floor`、`ceiling`、区间查询的场景；
   - 并发环境下的有序索引。
4. 注意点：
   - 它通常比 `ConcurrentHashMap` 更重；
   - 如果业务不需要排序，就不要为了“高级”而使用它。

### Collections.synchronizedXxx 的定位
1. 本质：JDK 提供的同步包装器，例如 `Collections.synchronizedList`、`Collections.synchronizedMap`。它们通常是在普通集合外层套一把锁。
2. 为什么需要它：适合简单、低并发、历史代码改造成本低的场景。
3. 使用场景：
   - 低并发管理后台；
   - 简单线程安全改造；
   - 对性能要求不高，只求尽快正确的场景。
4. 注意点：
   - 并发度通常不如专门的并发集合；
   - 遍历时往往仍需手动在外层加锁；
   - 更适合作为过渡方案，而不是高并发系统的首选。

### 选择思路
1. 如果你需要线程安全的 key-value 容器，优先考虑 `ConcurrentHashMap`。
2. 如果你是“读很多、写很少”的列表或集合，优先考虑 `CopyOnWriteArrayList` / `CopyOnWriteArraySet`。
3. 如果你需要无锁化的普通并发队列，考虑 `ConcurrentLinkedQueue`。
4. 如果你需要生产者-消费者、限流、背压、延迟执行，优先从 `BlockingQueue` 家族中选。
5. 如果你需要线程安全 + 有序访问 / 范围查询，考虑 `ConcurrentSkipListMap` / `ConcurrentSkipListSet`。
6. 如果只是简单改造旧代码、并发不高，可以用 `Collections.synchronizedXxx`，但不要把它当成高并发场景的默认方案。

## 面试场景题
1. "你有一个商品库存扣减的操作，怎么保证线程安全？用 synchronized、ReentrantLock 还是 CAS？为什么？" —— 考你在具体场景下的选型能力
2. "AQS 了解吗？ReentrantLock 的 lock() 方法底层是怎么实现的？" —— 面试官常见的追问链：lock → AQS → CAS + CLH 队列
3. "CAS 有什么缺点？如果竞争很激烈，CAS 一直失败怎么办？" —— 考 CAS 自旋开销 → LongAdder 分段思想
4. "生产者-消费者模式你会怎么实现？能说出几种方式？" —— 考 wait/notify、Condition、BlockingQueue 等多种实现
5. "10 个线程同时查询 10 个不同数据源，全部完成后合并结果返回给用户，你怎么实现？" —— 考 CountDownLatch / CompletableFuture


## 自测实操
1. 手写一个简易的不可重入锁（基于 AQS 或 CAS）
2. 用 Condition 实现一个有界阻塞队列
3. 用 CountDownLatch 实现：主线程等待 3 个子线程分别查询不同数据源，全部完成后合并结果
