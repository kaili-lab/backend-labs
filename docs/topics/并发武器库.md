# 并发武器库

## 这篇文档要解决什么问题
当你在后端项目里遇到“当前线程做不过来，需要让别的线程帮忙做事”时，真正想知道的通常不是某一个 API 怎么写，而是：

- 我手里到底有哪些武器可以选
- 它们分别适合什么场景
- 它们之间的关系是什么
- 在 Spring Boot 项目里一般怎么落地
- 常见的坑是什么
- 对应代码实验应该去哪里看

这篇文档的目标，就是把这些问题放在一个文件里讲清楚。

## 先用一句话理解并发武器
如果把一次请求想象成一个人独自干活，那么并发武器的作用就是：

- 把原本一个人做的事拆给更多执行单元
- 决定这些执行单元如何被创建和管理
- 决定多个任务完成后如何汇总结果
- 在收益和成本之间做平衡

也就是说，并发不是单纯“开更多线程”，而是：
- 让任务更快完成
- 同时又不把系统资源用爆

## 先看一张简单地图
站在 Java 后端工程视角，可以把常见并发武器理解成 5 类：

1. 直接创建线程
2. 用 JDK 线程池管理线程
3. 用 `Future` / `CompletableFuture` 编排任务
4. 用 Spring Boot 的异步能力接入业务层
5. 用 JDK 21 虚拟线程降低线程使用成本

它们不是互斥关系，而是不同层次的工具。

简单理解：
- `Thread` 更像最底层执行单元
- 线程池更像资源管理工具
- `CompletableFuture` 更像任务编排工具
- Spring 异步更像框架层封装
- 虚拟线程更像新的线程使用模型

## 1. 直接创建线程
### 是什么
最原始的方式就是直接创建 `Thread`，然后调用 `start()`。

常见配套概念：
- `Thread`
- `Runnable`
- `Callable`
- `FutureTask`

### 适合什么场景
- 学习线程最基本的概念
- 做很小的实验
- 想最直观地观察线程行为

### 不适合什么场景
- 高并发 Web 服务
- 请求量较大的线上系统
- 需要统一治理线程资源的项目

### 怎么理解它的定位
它像“临时找一个人来帮你做事”。

优点是：
- 直观
- 容易理解

缺点是：
- 难管理
- 难复用
- 数量一多容易失控

### 需要注意
- `Runnable` 和 `Callable` 是任务描述，不是线程管理方案
- `start()` 才是真正启动线程，直接调用 `run()` 仍然是同步执行
- 线程一旦创建出来，就会占用真实系统资源，不是“免费”的

### 常见坑
- 在线上请求里频繁 `new Thread()`
- 没有统一命名线程，排查困难
- 线程里异常没有统一处理
- 业务逻辑和线程管理耦合在一起

## 2. JDK 线程池
### 是什么
线程池是 Java 并发里最核心的工程化武器之一。

常见接口和实现：
- `Executor`
- `ExecutorService`
- `ThreadPoolExecutor`
- `ScheduledExecutorService`
- `ForkJoinPool`

### 它解决什么问题
它解决的重点不是“能不能并发”，而是：
- 如何复用线程
- 如何控制并发规模
- 如何排队
- 如何在过载时拒绝任务
- 如何统一管理线程生命周期

### 典型适用场景
- 一个请求里有多个独立耗时任务
- 应用内部有后台批量任务
- 需要限制某类任务的并发数量
- 不同业务任务需要线程池隔离

### 最重要的具体武器
#### `ThreadPoolExecutor`
这是最常用、最重要的线程池实现。

适合：
- 大多数业务线程池场景
- 需要显式控制线程数、队列、拒绝策略的场景

#### `ScheduledExecutorService`
这是定时任务线程池。

适合：
- 定时扫描
- 延迟执行
- 周期性后台任务

#### `ForkJoinPool`
这也是一种线程池，但它不是普通业务里最常用的那种通用线程池。

它更适合：
- 分治任务
- 可递归拆分的大任务
- 计算型并行任务

它最核心的思路是：
- 先把一个大任务拆成很多小任务
- 小任务并行执行
- 最后再把结果汇总回来

它还有一个重要特性叫“工作窃取（work-stealing）”：
- 某个线程空闲后，会去拿别的线程队列里的任务来执行

这使它在大量可拆分的小任务场景里效率很好。

但在普通业务开发里，它通常不是第一优先级，因为很多业务问题更常见的是：
- IO 等待
- 请求内聚合
- 下游调用
- 后台任务调度

这些场景通常先想到的是 `ThreadPoolExecutor`，而不是 `ForkJoinPool`。

### 需要注意
- 线程池解决的是资源控制，不只是让任务“并发起来”
- 不同任务最好按类型隔离线程池，不要长期混用一个池
- CPU 密集型任务和 IO 密集型任务，配置思路通常不同
- 线程池参数没有放之四海而皆准的默认值，必须结合场景看

### 常见坑
- 只会用 `Executors` 工厂方法，却不了解底层参数含义
- 所有任务混用一个线程池
- 线程池和队列盲目配很大
- CPU 密集任务和 IO 密集任务用同一套配置
- 拒绝策略没想清楚，系统过载时行为失控

### `ThreadPoolExecutor` 特别要注意什么
这是最容易在真实项目里踩坑的地方，至少要记住下面几点：

- 不要默认使用无界队列
- 不要只会抄线程数配置，而不知道为什么这么配
- 不要忽略拒绝策略
- 不要忽略线程命名
- 不要忽略 `ThreadLocal`、上下文污染和任务残留问题

其中“不要默认使用无界队列”尤其重要。

因为无界队列很容易制造一种错觉：
- 线程池没有明显报错
- 线程数也没有无限增长
- 看起来系统很稳定

但真实情况可能是：
- 任务在队列里越堆越多
- 请求延迟越来越高
- 内存压力越来越大
- 最后问题以更隐蔽、更难排查的形式暴露出来

所以在理解线程池时，不要只看“有没有报错”，还要看：
- 队列是否在持续堆积
- 延迟是否在持续恶化
- 系统是否已经进入“看似活着、其实越来越慢”的状态

## 3. `Future` 与 `CompletableFuture`
### 是什么
这一类武器更偏“任务编排”，而不是“线程怎么创建”。

常见成员：
- `Future`
- `CompletableFuture`
- `CompletionStage`

### 它解决什么问题
- 任务先发出去，结果之后再拿
- 多个任务如何并发执行并汇总
- 一个任务完成后如何接下一个任务
- 异常如何沿异步链路传播

### 典型适用场景
- 聚合接口
- 多下游并发调用
- 请求内多个独立任务并行
- 异步链式处理

### `Future`
适合先理解“未来结果”这个概念。

但它的问题是：
- 多任务组合能力弱
- 编排不自然

### `CompletableFuture`
这是现代 Java 后端里非常重要的一种武器。

适合：
- 一个请求里并发做几件独立的事
- 汇总多个异步结果
- 组织较清晰的异步流程

### 需要注意
- `CompletableFuture` 更擅长组织任务，不等于它本身解决了资源控制
- 是否显式指定执行器，会直接影响线程资源是否可控
- 任务之间必须足够独立，否则强行并发只会增加复杂度

### 常见坑
- 没有显式指定线程池，默认执行器不可控
- 只写 happy path，不处理异常
- 链式调用太长后，可读性迅速下降
- 本来不适合并发的任务也硬拆成异步步骤

## 4. Spring Boot 的异步能力
### 是什么
在 Spring Boot 项目里，并发不一定要直接写底层线程 API，也可以通过 Spring 提供的抽象来接入。

最常见的是：
- `@Async`
- `TaskExecutor`
- `ThreadPoolTaskExecutor`
- `TaskScheduler`

### 它解决什么问题
它更像是“把并发能力接入 Spring 业务层”。

也就是说：
- 线程池配置可以交给 Spring 容器管理
- 业务代码可以用更接近业务的方式表达异步
- 线程命名、隔离、注入会更方便

### 典型适用场景
- 发邮件
- 发送通知
- 附加型异步任务
- 定时任务
- 需要在 Spring 项目里统一管理执行器的场景

### 最常用的具体武器
#### `@Async`
最常见的 Spring 异步入口。

适合：
- 把某些非关键路径工作切出去
- 让业务层写法更简单

#### `ThreadPoolTaskExecutor`
这是 Spring 项目里最常见的业务线程池封装。

适合：
- 定义具名线程池
- 为 `@Async` 提供执行器
- 做不同业务任务的线程池隔离

#### `TaskScheduler`
用于定时任务。

### 需要注意
- Spring 异步的底层仍然需要执行器，不是“加个注解就自动正确”
- 异步后，请求上下文、事务上下文、日志上下文不一定还能直接沿用
- 业务层异步不等于可靠异步，可靠性要求高时可能更适合消息队列

### 常见坑
- `@Async` 自调用不生效
- 没显式配置执行器，默认线程池不可控
- 异步线程里直接依赖请求上下文或事务上下文
- 明明应该用消息队列保证可靠性的任务，却用 `@Async` 硬做

## 5. JDK 21 虚拟线程
### 是什么
虚拟线程是 JDK 21 正式可用的重要能力。

常见入口：
- `Thread.ofVirtual()`
- `Executors.newVirtualThreadPerTaskExecutor()`

### 它解决什么问题
它的核心价值不是让单个任务跑得更快，而是：
- 降低线程的使用成本
- 让大量阻塞型任务更容易承受
- 让“一个任务一个线程”的写法重新变得有吸引力

### 适合什么场景
- 大量 IO 阻塞型任务
- 希望保留同步写法，但并发量较大
- 调用外部接口、数据库、网络服务时线程数可能很多的场景

### 需要注意
- 虚拟线程降低的是线程使用成本，不会自动消除下游资源上限
- 它更适合阻塞型 IO 场景，不是“任何任务都更快”的开关
- 在使用 `ThreadLocal`、上下文传递、旧依赖时，要重新评估行为

### 不要误解它
虚拟线程不等于：
- 系统自动高性能
- 下游资源无限承载
- 数据库连接数不再是问题
- 不需要限流和资源控制

### 常见坑
- 把虚拟线程当万能性能开关
- 忽略真正瓶颈在数据库、CPU、锁竞争或外部依赖限流
- 不评估上下文传递、ThreadLocal 使用方式

## 这些武器之间是什么关系
可以这样记：

- `Thread`：最原始的执行单元
- 线程池：把线程管理工程化
- `CompletableFuture`：把多个任务组织起来
- Spring 异步：把并发接到业务框架层
- 虚拟线程：提供更轻量的线程模型

也就是说：
- 线程池和虚拟线程更偏“谁来执行任务”
- `CompletableFuture` 更偏“任务如何编排”
- Spring 异步更偏“在业务项目里怎么落地”

## 选择时先问自己什么问题
在决定用哪种并发武器前，先问自己：

1. 任务之间是否彼此独立
2. 当前瓶颈是等待 IO，还是 CPU，还是共享资源竞争
3. 是否需要统一控制并发规模
4. 是否需要结果汇总或链式编排
5. 是否在 Spring Boot 业务层里使用
6. 是否需要更可靠的异步机制，而不是仅仅切线程

## 当前项目里的建议学习顺序
建议顺序是：

1. 先理解直接创建线程到底解决什么问题
2. 再理解为什么必须进入线程池和资源控制
3. 然后理解 `CompletableFuture` 如何做任务编排
4. 接着理解 Spring Boot 中如何接入业务线程池
5. 最后再比较虚拟线程和传统方案
