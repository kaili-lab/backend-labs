# Web Server 演化主线

## 目标
通过一个逐步扩展的 Web Server，把高级后端知识串成一条连续的演化主线。

这个主线的重点不是罗列技术，而是回答：
- 系统在当前阶段遇到了什么问题
- 为什么会在这个阶段遇到这个问题
- 有哪些可选思路
- 为什么当前先选择这个方案
- 这个方案又会带来什么新的复杂度

## 这份文档怎么使用
这份文档先承担“总览”的职责，帮助快速看清整条演化路线。
具体阶段的细化内容，后续可以继续补充在本文件中，或者拆成独立阶段文档。

## 当前阶段的执行原则
这条主线优先帮助建立后端开发的整体认知地图，而不是一开始就深入底层实现细节。

因此当前顺序应当是：
- 先通过项目演进理解系统如何变化
- 再根据主线中真实出现的问题补知识点
- 最后把更深的底层细节沉淀到 `docs/java-topics/` 对应专题中

例如对象头、锁升级、AQS、底层源码机制等内容，不应作为当前起步阶段的学习主线，而应在主线推进到并发、资源控制等阶段时再补充。

## 主线颗粒度原则
主线阶段需要保持清晰，但不设死的阶段数量上限。

当前更看重的是：
- 每个顶层阶段是否足够完整
- 阶段之间是否有清晰递进关系
- 是否避免把细小技术点过度拆成大量碎片化阶段

因此：
- 8~10 个主阶段是一个较自然的参考范围
- 如果后续自然长到 10 个以上，不必为了数字强行合并
- 真正需要避免的是出现 20+ 或 30+ 个过碎的顶层阶段
- 可以在单个主阶段内部继续使用小步骤或补充场景

## 路线概览

### `stage-00`：建立最小基线
先有一个最小可运行的 Web Server，作为后续所有讨论的起点和对照组。

### `stage-01`：请求内独立任务的串行瓶颈
当一个请求内部需要完成多个彼此独立的耗时任务时，串行执行会明显拉长总响应时间。
这一阶段重点理解“为什么并发会成为 Java 后端的第一批关键能力之一”。

### `stage-02`：资源控制与池化
当并发能力提升后，线程、连接、任务等资源的创建和调度成本开始明显上升。
这一阶段重点理解“为什么系统不仅要能处理更多请求，还要能稳定地控制资源”。

### `stage-03`：引入数据库后的新瓶颈
系统开始持久化数据后，瓶颈通常从应用层转移到数据库层。
这一阶段重点理解“数据库为什么会成为很多业务系统最早暴露出来的核心瓶颈”。

### `stage-04`：数据库压力上升后的优化路径
随着数据量和访问量增加，单靠最初的数据库接入方式已经不够。
这一阶段重点理解“应该如何定位数据库压力来源，以及有哪些典型优化方向”。

### `stage-05`：热点与缓存
并不是所有请求都平均分布，一些热点数据会重复打到数据库或重复计算。
这一阶段重点理解“缓存为什么出现、适合解决什么问题、又会引入什么代价”。

### `stage-06`：异步化与削峰
当同步请求链路承担过多工作时，接口响应和系统稳定性都会受影响。
这一阶段重点理解“哪些工作应该异步化，以及异步之后如何保证可靠性”。

### `stage-07`：单机到多实例
当单机资源上限逐步触顶后，系统必须考虑横向扩展。
这一阶段重点理解“多实例、负载均衡、无状态设计为什么会成为自然下一步”。

### `stage-08`：稳定性与可观测性
当系统组件越来越多时，问题不再只是“能不能跑”，而是“出问题时能不能稳住并快速定位”。
这一阶段重点理解“稳定性设计和可观测性为什么会成为后期系统的核心能力”。

## 阶段之间的推进逻辑
这条主线不是按技术名词推进，而是按“压力如何迁移”推进：

1. 先建立最小系统基线
2. 请求内部出现多个独立耗时任务，暴露串行瓶颈
3. 并发提升后，暴露资源管理压力
4. 接入数据库后，暴露数据访问压力
5. 热点出现后，暴露缓存和读优化需求
6. 同步链路过重后，暴露异步化需求
7. 单机触顶后，暴露多实例扩展需求
8. 系统复杂后，暴露稳定性和可观测性需求

## 阶段写作模板
从 `stage-01` 开始，每个阶段建议按下面结构展开。

### 1. 当前阶段系统形态
说明系统现在大概长什么样，处于什么规模和复杂度水平。

### 2. 出现的问题
描述这个阶段最典型的压力、瓶颈或异常现象。

### 3. 为什么会出现
解释问题背后的根因，而不仅仅是表面现象。

### 4. 可选思路
列出常见的解决方向，说明它们各自的适用前提。

### 5. 当前选择
说明为什么在这个阶段优先选择某个方案，而不是其他方案。

### 6. 怎么落地
简要说明工程实现方向，详细实现可以交给 `labs/`。

### 7. 新的代价
说明引入这个方案后，系统会增加哪些新的复杂度、限制或风险。

### 8. 关联知识点
链接到 `docs/java-topics/` 中对应的知识点文档。

### 9. 关联实验
链接到 `labs/` 中对应的实验目录或阶段目录。

## `stage-00` 的写法要求
`stage-00` 只负责建立起点，不需要像后续阶段那样展开得很重。

建议只写清楚：
- 当前系统是什么形态
- 请求处理的最基本流程是什么
- 当前先不解决哪些问题
- 为什么它只能作为起点，而不是终点

## 下一步建议
下一步可以优先细化以下 3 个阶段：
- `stage-01`：请求内独立任务的串行瓶颈
- `stage-02`：资源控制与池化
- `stage-03`：引入数据库后的新瓶颈

这 3 个阶段最容易把后端核心知识自然串起来。
