# Web Server 演化主线

## 目标
通过一个逐步演进的模块化单体项目，把后端知识串成一条连续的演化主线。

这个主线的重点不是罗列技术，而是回答：
- 系统在当前阶段遇到了什么问题
- 为什么会在这个阶段遇到这个问题
- 有哪些可选思路
- 为什么当前先选择这个方案
- 这个方案又会带来什么新的复杂度

## 实验方式

采用**一个模块化单体项目**（`labs/web-server-scaling/`），以商城下单链路为主线，逐步迭代。

- 文档驱动演进——每到一个阶段，用文档说清压力和因果
- 代码验证方案行为——用测试或 demo 展示关键方案的运行特征，不需要真打高并发流量
- 压力是推演出来的，不是压测出来的

详细的项目设计见 `labs/web-server-scaling/PRD.md`。
演进线的完整阐述见 `docs/capability-map/后端演进线.md`。

## 这份文档怎么使用
这份文档承担"总览"的职责，帮助快速看清整条演化路线。
具体的演进因果分析在 `docs/capability-map/后端演进线.md` 中展开。

## 当前阶段的执行原则
这条主线优先帮助建立后端开发的整体认知地图，而不是一开始就深入底层实现细节。

因此当前顺序应当是：
- 先通过项目演进理解系统如何变化
- 再根据主线中真实出现的问题补知识点
- 最后把更深的底层细节沉淀到 `docs/java-topics/` 对应专题中

## 路线概览

| 阶段 | 核心压力 | 引入的关键能力 |
|------|---------|-------------|
| 1. 同步模型 | 先让功能跑通 | 基础业务链路，全同步执行 |
| 2. 异步化 | 耗时操作阻塞主流程 | 主流程与附属流程分离 |
| 3. 线程池 | 异步任务无管理 | 统一资源调度 |
| 4. 性能优化 | 资源吃紧 | 池化、SQL 优化、指标体系 |
| 5. 数据库优化 | 数据库先扛不住 | 索引、读写分离 |
| 6. 缓存 | 热点数据重复查 | Redis 缓存，引入副本 |
| 7. 一致性 | 多副本状态分散 | 事务边界、缓存与 DB 同步 |
| 8. 稳定性 | 依赖失败传播 | 超时、熔断、降级、限流 |
| 9. MQ 解耦 | 同步链路过重 | 消息队列、事件驱动 |
| 10. 服务拆分 | 单体/单库失效 | 模块拆为独立服务 |
| 11. 可观测性 | 系统看不清 | 日志、指标、链路追踪 |

## 阶段之间的推进逻辑
这条主线不是按技术名词推进，而是按"压力如何迁移"推进：

1. 先建立最小同步模型
2. 慢操作出现，开始异步化
3. 异步变多，需要统一调度
4. 资源吃紧，需要池化和性能优化
5. 数据库先成为瓶颈
6. 热点出现，引入缓存
7. 副本变多，一致性问题显性化
8. 依赖链变长，稳定性治理进入主线
9. 同步链路过重，消息队列解耦
10. 单体承载不了，进入服务拆分
11. 系统复杂到看不清，必须靠可观测性驾驭

## 阶段写作模板
每个阶段建议按下面结构展开：

1. **当前阶段系统形态** — 系统现在大概长什么样
2. **出现的问题** — 最典型的压力、瓶颈或异常现象
3. **为什么会出现** — 问题背后的根因
4. **可选思路** — 常见解决方向及适用前提
5. **当前选择** — 为什么优先选择某个方案
6. **怎么落地** — 工程实现方向，详细实现交给 `labs/`
7. **新的代价** — 引入方案后增加的复杂度
8. **关联知识点** — 链接到 `docs/java-topics/` 对应文档
9. **关联实验** — 链接到 `labs/web-server-scaling/` 中对应的模块或功能
