# 项目方向与基础设施基线

## 目的
这份文档用于记录 `web-server-scaling` 主题在真正进入代码实现前的几个基础决定，避免后续反复摇摆。

## 当前主题结构
`web-server-scaling` 不是单个项目，而是一个主题目录。

当前已经明确：
- 第一个项目是 `mall-monolith/`，用于承载 Spring Boot 模块化单体能力
- 第二个项目是 `mall-microservices/`，用于承载基于单体拆分后的 Spring Cloud 微服务能力

这里的关键边界是：
- 演进线负责解释能力为什么出现
- 单体项目负责先把不依赖服务边界的能力做扎实
- 微服务项目再承载消息驱动、服务边界和分布式协调问题

## 业务载体选择
项目主线不是先做一个完整商城系统，而是做一个不断演进的通用 Web 系统。
商城系统作为主要业务语境，用来承载高并发、高可用、高扩展等后端问题。

这样设计的原因是：
- 商城场景天然会遇到读多写少、热点数据、订单写入、库存扣减、支付回调等典型问题
- 这些问题能自然引出缓存、异步化、一致性、限流、幂等等高级后端知识
- 如果一开始就追求“完整商城”，容易被过多业务细节分散注意力

## 当前业务切入建议
建议优先从以下几个子域定义单体项目的能力范围：
- 商品与库存：适合引出查询路径、缓存、热点和读压力
- 订单：适合引出写路径、事务、状态流转和分页查询
- 支付：适合引出幂等、状态分离和写链路一致性
- 通知：适合引出附属流程异步化、后台任务和失败重试

这里要注意两层区别：
- 在文档范围里，上面这些模块应一次性定义清楚边界
- 在代码实现里，可以按依赖关系分批推进

优惠券、运营后台、完整用户体系等内容不进入当前单体项目的首版范围。

## 主实验技术栈选择
主实验以 Java 为主。

## 为什么主用 Java
主用 Java 不只是因为性能，更重要的是它和这个项目要观察的问题非常契合：
- Java 在多线程、线程池、连接池、JVM 内存模型等方面，有更成熟也更典型的后端工程实践
- 这些能力和本项目的主线高度相关，便于把“并发 -> 资源控制 -> 数据访问 -> 稳定性”串起来
- Java 在传统后端、企业应用、数据库访问、线程模型和性能调优上有大量成熟经验，适合作为主实验载体

## 如何看待 Java 和 Node.js
如果只看后端常见场景下的原始性能和多核资源利用能力，Java 通常更强，尤其是在多线程和 CPU 压力更明显的场景里。
但这并不代表 Node.js 不适合做后端。
Node.js 在 IO 密集型场景、开发效率和轻量服务方面仍然有很强的价值。

因此当前决定是：
- 主实验以 Java 为主
- Node.js 保留为对照视角或后续支线，不作为第一阶段主实现

## 基础设施范围
当前主题下的基础设施按项目边界区分：

### `mall-monolith/` 当前基线
- 数据库
- Redis

### `mall-microservices/` 后续再讨论
- Kafka
- 服务注册发现相关中间件
- 网关和其他服务治理基础设施

## 基础设施接入原则
这里要同时避免两种误解：
- 不是把所有中间件一开始全部塞进单体项目
- 也不是说文档里不写到的模块，代码里就不能先规划边界

当前原则是：
- 单体项目先接入自己真实需要的基础设施，也就是数据库和 Redis
- 消息队列和微服务治理相关组件进入第二个项目再引入
- 能力边界先定义清楚，实现节奏按依赖关系推进

## 工程设计原则与可维护性
在整个主题推进过程中，需要持续关注设计原则和代码可维护性，但不应为了套用模式而提前复杂化。

当前建议遵循这些原则：
- 优先简单设计，不提前抽象
- 优先清晰边界，控制模块耦合
- 优先可演进，而不是一次性设计完美
- 设计模式按问题引入，不为模式而模式
- 每个阶段都考虑后续是否容易修改、测试和扩展

这部分的定位是：
- 设计原则作为贯穿主线的判断标准
- 设计模式作为具体问题出现后的局部解决手段
- 可维护性作为评估当前实现是否健康的重要维度

## 当前环境策略
当前优先采用 Docker / Docker Compose 来管理数据库、Redis，以及后续微服务项目可能需要的队列和治理依赖。
Windows 10 + WSL2 + Ubuntu 作为开发环境底座，Docker 基于这套环境运行。

这里还有两个额外原则：
- 能不装在 WSL Ubuntu 中的基础设施，就尽量不直接装在 WSL 中
- 宿主机优先负责系统级能力，WSL 优先负责开发命令和代码执行

## 网络与代理注意事项
当前宿主机使用 `v2rayN` 并开启 `Tun` 模式，WSL 网络会受到宿主机网络环境影响。
因此本项目在环境设计上应遵循：
- 先保证宿主机和 Docker Desktop 的网络可用
- 只有当 WSL 内开发命令实际出现网络问题时，再补充 WSL 侧代理配置
- 避免一开始就在 Windows、WSL、Docker 中同时叠加多层代理设置

## 后续需要继续讨论的问题
- Spring Boot 单体项目的模块接口粒度如何控制
- `mall-monolith/` 的 `docker-compose.yml` 与后续 `mall-microservices/` 的环境编排如何划分
- Redis 在商品读路径、幂等控制和限流中的使用边界
- 微服务项目的拆分顺序以及第一批服务边界怎么切
