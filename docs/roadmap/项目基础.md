# 项目方向与基础设施基线

## 目的
这份文档用于记录主实验在真正进入阶段演化之前的几个基础决定，避免后续在实现过程中频繁摇摆。

## 业务载体选择
项目主线不是先做一个完整商城系统，而是做一个不断演进的通用 Web 系统。
商城系统作为主要业务语境，用来承载高并发、高可用、高扩展等后端问题。

这样设计的原因是：
- 商城场景天然会遇到读多写少、热点数据、订单写入、库存扣减、支付回调等典型问题
- 这些问题能自然引出缓存、异步化、一致性、限流、幂等等高级后端知识
- 如果一开始就追求“完整商城”，容易被过多业务细节分散注意力

## 当前业务切入建议
建议优先从以下两个子域切入：
- 商品：适合引出查询路径、缓存、热点和读压力
- 订单：适合引出写路径、事务、异步化、幂等和一致性

库存、支付、优惠券、消息通知等主题可以随着主线推进再逐步引入。

## 主实验技术栈选择
主实验以 Java 为主。

## 为什么主用 Java
主用 Java 不只是因为性能，更重要的是它和这个项目要观察的问题非常契合：
- Java 在多线程、线程池、连接池、JVM 内存模型等方面，有更成熟也更典型的后端工程实践
- 这些能力和本项目的主线高度相关，便于把“并发 -> 资源控制 -> 数据访问 -> 稳定性”串起来
- Java 在传统后端、企业应用、数据库访问、线程模型和性能调优上有大量成熟经验，适合作为主实验载体

## 如何看待 Java 和 Node.js
如果只看后端常见场景下的原始性能和多核资源利用能力，Java 通常更强，尤其是在多线程和 CPU 压力更明显的场景里。
但这并不代表 Node.js 不适合做后端。
Node.js 在 IO 密集型场景、开发效率和轻量服务方面仍然有很强的价值。

因此当前决定是：
- 主实验以 Java 为主
- Node.js 保留为对照视角或后续支线，不作为第一阶段主实现

## 基础设施范围
后续主实验中可能逐步引入的基础设施包括：
- 数据库
- Redis
- 消息队列
- 其他与阶段目标相关的中间件

## 基础设施接入原则
这些依赖应按主线阶段逐步引入，而不是一开始全部接入。
只有当系统真正出现对应压力时，才把某项基础设施加入主实验。

## 工程设计原则与可维护性
在整个项目演进过程中，需要持续关注设计原则和代码可维护性，但不应为了套用模式而提前复杂化。

当前建议遵循这些原则：
- 优先简单设计，不提前抽象
- 优先清晰边界，控制模块耦合
- 优先可演进，而不是一次性设计完美
- 设计模式按问题引入，不为模式而模式
- 每个阶段都考虑后续是否容易修改、测试和扩展

这部分的定位是：
- 设计原则作为贯穿主线的判断标准
- 设计模式作为具体问题出现后的局部解决手段
- 可维护性作为评估当前实现是否健康的重要维度

## 当前环境策略
当前优先采用 Docker / Docker Compose 来管理数据库、Redis、消息队列等外部依赖。
Windows 10 + WSL2 + Ubuntu 作为开发环境底座，Docker 基于这套环境运行。

这里还有两个额外原则：
- 能不装在 WSL Ubuntu 中的基础设施，就尽量不直接装在 WSL 中
- 宿主机优先负责系统级能力，WSL 优先负责开发命令和代码执行

## 网络与代理注意事项
当前宿主机使用 `v2rayN` 并开启 `Tun` 模式，WSL 网络会受到宿主机网络环境影响。
因此本项目在环境设计上应遵循：
- 先保证宿主机和 Docker Desktop 的网络可用
- 只有当 WSL 内开发命令实际出现网络问题时，再补充 WSL 侧代理配置
- 避免一开始就在 Windows、WSL、Docker 中同时叠加多层代理设置
