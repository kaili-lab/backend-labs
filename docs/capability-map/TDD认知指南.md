# TDD 认知指南
**测试驱动开发的本质、边界与实战应用**

---

## 一、为什么顺序是反的：TDD 的核心本质

正常人的直觉是先实现再测试。TDD 强行把这个顺序反过来——先写测试，再写实现。这个「反常」的顺序背后有一个深层原因，理解它比记住步骤重要得多。

当你先写测试的时候，你被迫在实现任何东西之前就必须回答一个问题：**这个方法对外的行为契约是什么？**

- 它接受什么输入？
- 在什么条件下返回什么输出？
- 什么情况下应该报错，报什么错？

这些问题在「先实现」的模式下很容易被跳过——你脑子里有个模糊的想法就开始写代码了，边写边想。TDD 的顺序强迫你在动手之前把模糊变成清晰。

这个「强迫澄清」的过程会暴露两件事：一是你对需求的理解是否真的清楚；二是你设计的接口是否真的合理——如果一个方法很难为它写测试，往往说明这个方法承担的职责太多，或者依赖关系设计得有问题。

所以 TDD 被称为「测试驱动**设计**」不是夸张，先写测试这个动作本身就是在做设计决策。

## 二、什么时候 TDD 有价值，什么时候没有

TDD 不是万能的，盲目应用反而是浪费。能判断边界，才是真正掌握了 TDD。

### 适合 TDD 的场景

- **逻辑复杂、边界条件多的核心业务代码**：比如计费规则、权限校验、状态机流转。这类代码契约清晰，边界多，先写测试能帮你系统性地覆盖所有情况。
- **需求已经明确、接口设计相对稳定的模块**：TDD 在稳定的契约上效果最好。
- **对回归敏感的关键路径**：先写测试意味着这条路径从一开始就有保护网。

### 不适合 TDD 的场景

- **探索性代码**：你还不知道这个东西最终长什么样，边写边想。强行 TDD 会写出大量之后要扔掉的测试。
- **需求本身还没想清楚**：如果业务逻辑还在变，先写的测试会不断失效，维护成本高于收益。
- **UI 交互和视觉层**：这一层的「正确性」很难用测试精确描述，人眼判断反而更高效。
- **一次性脚本或工具代码**：生命周期短，投入写测试的成本无法回收。

> **实用原则：** 问自己一个问题——这段代码三个月后还会被人改吗？如果答案是会，TDD 的投入就值得。

## 三、Red-Green-Refactor：被忽略的第三步

TDD 的操作节奏通常被描述为三个阶段：

```
Red   → 写一个失败的测试（还没有实现，必然失败）
Green → 写刚好能让测试通过的最简实现
Refactor → 在测试保护下，改善代码结构
```

大多数人做到 Green 就停了，但 **Refactor 才是 TDD 驱动出好设计的关键环节**。

### 为什么 Refactor 这么重要

Green 阶段的目标是「让测试通过」，不是「写出好代码」。你被允许写出丑陋的、重复的、直白的实现——硬编码返回值、大量 if-else、复制粘贴——只要测试绿了就行。这是故意的。

这个设计的意图是把两件事分开：先证明逻辑是对的（Green），再改善代码结构（Refactor）。当你在 Refactor 阶段整理代码时，测试就是你的安全网——任何不小心破坏了行为的改动都会立刻被发现。

没有 Refactor 这一步，TDD 会让代码越来越乱，因为每个 Green 阶段留下的「刚好能跑」的代码在不断积累。

### Refactor 阶段做什么

- 消除重复逻辑
- 提取有意义的中间变量或私有方法
- 改善命名，让代码表达意图而不只是步骤
- 调整结构，让职责更清晰

**Refactor 阶段不做什么：** 不加新功能，不改行为，只改结构。如果想加新功能，回到 Red 阶段，先写新测试。

## 四、TDD 与测试分层的关系

TDD 通常在单元测试层实践，但它的思维方式可以往上渗透，理解这个边界对实际应用很有帮助。

### TDD 最自然的主场：单元测试层

单元测试层和 TDD 的契合度最高，原因是：

- 反馈速度极快，Red-Green-Refactor 一个循环可以在几秒内完成
- 完全隔离（Mock 掉外部依赖），测试稳定，不受环境影响
- 契约清晰，一个函数/方法的行为容易精确描述

在这一层做 TDD 的心智负担最低，也最容易建立节奏感。

### TDD 思维渗透到集成测试层

集成测试层可以借用 TDD 的思维方式，但不完全照搬节奏。

实际做法是：在开始实现一个功能之前，先写一个集成测试描述这个功能的核心业务行为（类似 BDD 的 Given-When-Then），让它失败，然后再去实现。这个测试充当「功能完成的验收标准」，驱动你从外到内地推进实现。

但集成测试的执行速度慢（秒级到分钟级），不适合像单元测试那样每几秒跑一次。所以实践中通常是：

- 集成测试层：写一个失败的验收测试，明确「完成」的定义
- 单元测试层：用 TDD 节奏驱动具体实现
- 集成测试变绿：确认功能整体符合预期

这个「外层验收测试 + 内层 TDD」的组合，是很多团队实际采用的工作流。

### E2E 层通常不做 TDD

E2E 测试执行太慢，反馈循环太长，不适合作为 TDD 的驱动层。它更多是功能完成后的最终验收，而不是开发过程中的驱动力。

## 五、遗留代码怎么办

现实中大多数项目不是从零开始的绿地项目，而是已有大量没有测试覆盖的存量代码。在这种情况下，TDD 的思维依然有价值，但需要调整策略。

### 不要试图一次性补全所有测试

给存量代码补测试是个无底洞，优先级不对、成本极高，而且存量代码的结构往往不友好（强耦合、难 Mock），强行补测试会非常痛苦。

### 实用策略：「童子军原则」式渗透

> 离开营地时，让它比你来时更干净一点。

每次你因为需求修改或 bug 修复而接触某段代码时：

1. **修 bug 之前先写一个测试复现这个 bug**，让它失败，然后修复，确认测试变绿。这个测试同时成为回归保护。
2. **加新功能时对新增部分用 TDD**，不强求改造周围的老代码。
3. **重构前先补测试**，只覆盖你即将改动的行为，作为重构的安全网，不求全覆盖。

这个策略的核心是：**不求一步到位，但每次接触都让测试覆盖向前推进一点**。几个月下来，被频繁改动的核心路径自然会积累出不错的测试覆盖，而那些从未被触碰的边角代码不补测试也无妨。

### 处理难以测试的遗留代码

遗留代码通常难以测试，因为它的结构本来就不是为测试设计的——强耦合、静态方法、全局状态、大类。这时候有几个常见手段：

- **提取方法**：把你真正想测的逻辑提取成独立的小方法，对这个小方法写测试，而不是试图测试整个大类。
- **接缝（Seam）**：找到代码中可以「插入」替换行为的点，通过接口或参数注入让依赖变得可替换，然后 Mock 掉它。
- **接受部分覆盖**：不是所有遗留代码都值得改造，判断改造成本和收益，不要为了覆盖率数字而过度投入。

## 六、与 AI 工具协作时的 TDD

Claude Code 等 AI 工具可以承担 TDD 中大量机械性的工作，但前提是你掌握了 TDD 的本质。

### AI 可以做的

- 根据你描述的行为契约生成测试骨架
- 补全边界 case 的测试
- 根据测试生成刚好能通过的最简实现
- 在 Refactor 阶段提供重构建议

### 你必须做的

- **定义行为契约**：这个方法该做什么、边界在哪，这个判断必须由你来做，AI 不了解你的业务逻辑
- **Review 生成的测试**：判断测试是否真的覆盖了重要场景，还是只是在凑覆盖率
- **决定什么时候用 TDD**：哪段代码值得用 TDD，哪段不值得，这是工程判断

**工具放大你的能力，但不能替代你的判断。** 不懂 TDD 的本质，你就没有能力 review AI 的输出，也没有能力告诉它真正重要的契约是什么。反过来，懂了 TDD 之后，AI 工具会让整个 TDD 工作流的效率大幅提升。

---

## 七、总结

TDD 的本质可以用三句话概括：

> **先想清楚契约，再动手实现。** 顺序的改变，驱动的是设计质量。

> **Red-Green-Refactor 缺一不可。** Green 证明逻辑正确，Refactor 才产生好设计。

> **TDD 是工具，不是教条。** 判断它在哪个场景有价值，比机械执行步骤更重要。
