# 后端工程能力地图
> 这份文档不是继续堆技术点，而是回答一个更实际的问题：
> 如果已经理解了“后端开发核心原则”，那么一个后端工程师接下来应该把能力建立在哪些方向上？
>

---

## 先说结论
**这份能力地图不属于第一层，而更像第二层。**

如果把整个后端认知结构分层，可以先这样理解：

1. **核心原则层**：解释后端世界为什么会复杂，底层矛盾是什么
2. **能力地图层**：解释工程师实际要具备哪些能力域，才能处理这些矛盾
3. **专题展开层**：解释某个具体方向的原理、实现、边界和坑
4. **主线 / 实验层**：解释这些能力在真实系统演化和代码行为里是怎么出现的

所以，这份文档的作用不是替代 `docs/java-topics/后端开发核心原则.md`，而是承接它：
从“理解世界为什么会这样”，走到“工程师需要会什么”。

---

## 这份文档回答什么
如果说“核心原则”回答的是：
- 后端问题为什么会出现
- 技术方案背后在对抗什么矛盾

那么“能力地图”回答的是：
- 一个后端工程师平时到底在处理哪些问题
- 这些问题为什么会自然落到几个固定能力域里
- 每个能力域都需要同时具备哪四类能力：
  - **怎么落地**
  - **细节怎么实现**
  - **出问题怎么排查**
  - **不同场景下怎么取舍**

换句话说，核心原则更像“物理定律”，能力地图更像“驾驶课程大纲”。

---

## 后端工程的 8 个核心能力域

### 1. 请求链路与应用执行
**一句话**：你要知道一个请求是怎么进来、怎么被处理、怎么返回的。

**为什么程序员一定会遇到它**：后端代码不是孤立函数，而是跑在 Web 容器、框架、中间件和线程模型上的。只要你写接口，就一定会接触路由、过滤器、参数绑定、事务边界、线程切换、上下文传递这些问题。

**这一类能力要会什么**：
- 落地：能把一个 HTTP / RPC 请求稳定接进系统并完成业务处理
- 实现：理解 Web 容器、框架调用链、Spring 请求处理链路、事务边界
- 排障：能定位请求为什么慢、为什么没进控制器、为什么事务没生效
- 取舍：知道同步调用、异步调用、MVC / Reactive、单体 / 微服务链路的选择边界

**典型专题入口**：JVM 与执行模型、Java Web / Tomcat 请求链路、Spring 核心机制

---

### 2. 数据与状态管理
**一句话**：你要知道数据放在哪里、怎么变化、谁说了算。

**为什么程序员一定会遇到它**：业务系统本质上就是在处理状态变化：下单、扣库存、支付、退款、发货、对账，本质都是对数据的读写和约束。一旦数据跨缓存、数据库、消息、搜索索引流动，就会出现一致性、事务、幂等、重复消费、状态编排这些问题。

**这一类能力要会什么**：
- 落地：能设计表结构、索引、事务边界、缓存读写策略
- 实现：理解 ORM、SQL、锁、事务隔离级别、缓存一致性、分布式事务思路
- 排障：能定位脏数据、慢 SQL、死锁、缓存不一致、重复扣减等问题
- 取舍：知道强一致和最终一致、数据库事务和消息事务、缓存和数据库之间怎么选

**典型专题入口**：持久层与事务、数据库与缓存设计、分布式事务、分布式经典问题

---

### 3. 并发、异步与任务编排
**一句话**：你要知道怎么让系统同时做更多事，又不把结果做错。

**为什么程序员一定会遇到它**：只要系统追求吞吐，就不会一直串行执行。多线程、线程池、异步任务、消息队列、定时任务、批处理任务都会让“多个执行单元同时推进一件事”成为常态，于是并发安全、任务协作、异常传播、重试补偿都会出现。

**这一类能力要会什么**：
- 落地：能用线程池、异步编排、消息队列、定时任务把工作拆开并跑起来
- 实现：理解 JMM、锁、CAS、线程池参数、CompletableFuture、消息投递与消费模型
- 排障：能定位线程池打满、任务堆积、消息重复、异步异常丢失、上下文丢失
- 取舍：知道同步转异步的收益和代价，知道何时该并发、何时该串行

**典型专题入口**：Java 内存模型、锁机制与线程、线程池与异步编程、消息队列

---

### 4. 资源、性能与容量规划
**一句话**：你要知道系统的瓶颈在哪里，以及还能扛多少。

**为什么程序员一定会遇到它**：资源总是有限的，流量却会增长。CPU、内存、线程、连接、磁盘 IO、网络带宽、数据库连接数都会成为上限。只要系统上线并被使用，性能和容量问题就不是“会不会来”，而是“什么时候来”。

**这一类能力要会什么**：
- 落地：能做缓存、池化、分页、批处理、削峰、热点治理、读写分离等优化
- 实现：理解线程池、连接池、缓存、多级缓存、索引、SQL 优化、JVM 参数
- 排障：能根据 RT、QPS、P99、GC、线程 dump、慢 SQL 找到瓶颈层
- 取舍：知道吞吐和延迟、内存和 CPU、缓存收益和一致性代价之间怎么平衡

**典型专题入口**：性能优化方法论、JVM 调优、数据库性能优化、接口性能优化实战

---

### 5. 稳定性、容错与恢复
**一句话**：你要知道系统出错时怎么止损，出事后怎么恢复。

**为什么程序员一定会遇到它**：线上系统一定会遇到超时、抖动、依赖失败、流量洪峰、节点重启、数据延迟、消息积压。系统越分布式，故障越不是偶发，而是日常背景噪声。

**这一类能力要会什么**：
- 落地：能设计超时、重试、熔断、降级、限流、幂等、补偿、回滚方案
- 实现：理解故障传播链路、雪崩、重试风暴、熔断状态机、降级策略设计
- 排障：能判断是入口被打爆、下游依赖故障、重试放大、消息堆积还是数据补偿失败
- 取舍：知道可用性、一致性、用户体验、系统复杂度之间怎么平衡

**典型专题入口**：高可用与高性能设计、分布式经典问题、服务通信与分布式事务

---

### 6. 可观测性与问题诊断
**一句话**：你要知道怎么让系统“开口说话”。

**为什么程序员一定会遇到它**：系统一复杂，问题就不会直接写在脸上。真正的线上排障，往往不是“看一眼代码就知道”，而是要通过日志、指标、链路、profiling 一层一层缩小范围。

**这一类能力要会什么**：
- 落地：能补齐日志、指标、链路追踪、告警和诊断工具
- 实现：理解日志规范、指标体系、Tracing、JVM 诊断工具、数据库分析工具
- 排障：能从现象倒推链路，从链路回到线程、SQL、GC、网络和依赖方
- 取舍：知道监控成本、日志量、采样精度、诊断深度之间怎么平衡

**典型专题入口**：可观测性与工程实践、并发问题诊断、JVM 调优与问题定位

---

### 7. 安全、权限与边界控制
**一句话**：你要知道什么能进来、谁能操作、哪些边界不能被越过。

**为什么程序员一定会遇到它**：只要系统对外提供能力，就会面对身份校验、权限控制、参数校验、数据越权、接口滥用、敏感信息保护这些问题。很多安全问题并不是黑客技术炫技，而是边界没定义清楚。

**这一类能力要会什么**：
- 落地：能设计认证、鉴权、参数校验、审计、接口防刷、数据脱敏方案
- 实现：理解会话、Token、权限模型、幂等 Token、签名、限流、审计链路
- 排障：能定位鉴权失效、越权访问、重复提交、恶意流量和敏感日志泄露
- 取舍：知道安全强度、用户体验、系统复杂度和开发成本之间怎么平衡

**典型专题入口**：当前项目后续需要补独立专题，与网关、认证、接口治理密切相关

---

### 8. 架构演进与技术取舍
**一句话**：你要知道什么时候该加东西，什么时候不该加。

**为什么程序员一定会遇到它**：后端工程师不是只写功能，还要随着业务规模和系统压力变化不断做选择：是否加缓存、要不要拆服务、是否上消息队列、是不是先做主从、什么时候需要分库分表。真正难的不是知道这些词，而是知道它们什么时候值得引入。

**这一类能力要会什么**：
- 落地：能针对当前压力提出可执行的演进方案，而不是空谈大架构
- 实现：理解缓存、队列、分布式事务、微服务、网关、分库分表的引入条件
- 排障：能复盘某个架构决策为什么在当前阶段失效，复杂度是在哪里爆出来的
- 取舍：能围绕当前约束，在性能、稳定性、一致性、成本、团队经验之间做判断

**典型专题入口**：系统设计方法论、数据库与缓存设计、消息队列与网关设计、服务扩展演进路线

## 这 8 个能力域和“核心原则”是什么关系
可以把它们理解成两张不同尺度的地图：

- **核心原则**回答“后端世界最底层的矛盾是什么”
- **能力地图**回答“工程师为了处理这些矛盾，需要在哪些工作面上形成能力”

一个非常粗略但有用的映射是：

- 可见性与一致性 -> 数据与状态管理、可观测性与问题诊断
- 互斥与原子性 -> 并发、异步与任务编排
- 权衡与取舍 -> 架构演进与技术取舍
- 分而治之 -> 架构演进、容量规划、分布式设计
- 异步化与解耦 -> 并发、异步与任务编排、稳定性设计
- 池化与复用 -> 资源、性能与容量规划
- 快速失败与兜底 -> 稳定性、容错与恢复
- 量化先行，数据驱动 -> 可观测性与问题诊断

所以，能力地图不是替代核心原则，而是把“原理层”翻译成“工程工作层”。

## 这份文档怎么用
如果你后面继续扩展这个目录，我建议每一篇能力地图文档都尽量遵守同一个套路：

1. 先说明这个能力域在解决什么问题
2. 再说明程序员为什么会自然遇到这个问题
3. 再说明这类能力通常包含哪四件事：落地、实现、排障、取舍
4. 最后再链接到具体专题和实验

这样做的好处是：
- 不会一上来就掉进实现细节
- 不会把 high level 文档写成技术名词堆
- 能持续保持“原理 -> 能力 -> 专题 -> 实验”的结构感

## 当前边界
这份文档的目标不是覆盖所有细节，而是让你先形成一个判断：

**一个后端工程师，平时真正要反复处理的，不是零散技术名词，而是这几类稳定重复出现的工程问题。**

当你有了这层地图，再去学 Redis、MQ、事务、线程池、链路追踪、分库分表，就不再是在学孤立知识点，而是在给某个能力域补工具和方法。
