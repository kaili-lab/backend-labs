# 后端演进线
> 这份文档不讲“后端世界为什么复杂”，也不讲“工程师能力应该分成哪些块”。
> 它只回答一个更贴近直觉的问题：
> 一个后端系统，通常是怎样从“小而简单”一步步演进到“复杂但可控”的？
>

---

## 先说这份文档怎么看
这份文档适合用一种“浅 -> 深 -> 浅”的方式读：

- **先看浅层**：先抓住每个阶段的一句话结论，知道当前主要压力是什么
- **再看深层**：再理解为什么旧方案不够了、新方案为什么会自然出现
- **最后回到浅层**：记住这个阶段最该带走的一句判断

这样做的原因很简单：
后端演进不是在背技术名词，而是在理解“系统压力变化时，工程师通常会被逼着做什么选择”。

这里要额外固定一个边界：
- 这份文档描述的是**知识上的常见演进顺序**
- 它不直接规定某个主题要按什么顺序写代码
- 同一条演进线，可以由一个项目承载，也可以由多个项目分段承载
- 放到 `web-server-scaling` 里时，它负责解释“为什么会出现这些能力”，而不是规定 `mall-monolith/` 和 `mall-microservices/` 的实施排期

---

## 总体路线
你可以先把整个过程想象成这样：

```plain
小系统能跑
  -> 某些操作开始变慢
  -> 想办法别阻塞主流程
  -> 异步任务越来越多，需要统一管理
  -> 资源开始吃紧，需要池化和性能优化
  -> 数据库先成为瓶颈
  -> 读流量继续增长，引入缓存
  -> 副本和缓存变多，一致性问题开始突出
  -> 服务依赖变长，故障传播和稳定性问题出现
  -> 业务进一步拆分，进入分布式协作
  -> 系统越来越大，必须靠监控、链路和指标来驾驭
```

这不是唯一演进路线，但它是很常见的一条主线。

## 阶段 1：系统很小，先让功能跑通

### 浅：先记一句话
**在系统还小的时候，简单往往比“先进”更重要。**

### 深：为什么这个阶段不需要太多技术
一个刚起步的 Spring Boot 项目，通常有这些特点：
- 用户不多，请求量不高
- 一个接口里做几次数据库查询也还能接受
- 少数慢操作虽然有点卡，但还没有卡到业务无法接受
- 代码主要压力不是性能，而是“先把需求做对”

这时候，最自然的实现方式通常是：
- Controller 收请求
- Service 写业务逻辑
- Repository / Mapper 操作数据库
- 所有事情同步执行
- 数据主要落在一个数据库里

为什么这时候不急着上多线程、缓存、消息队列？
因为这些技术虽然能解决后面的问题，但也会提前带来复杂度：
- 多线程会带来并发安全和上下文传递问题
- 缓存会带来一致性问题
- 消息队列会带来异步、重试、幂等、堆积问题

程序员在这个阶段最应该警惕的，不是“技术太少”，而是“系统还没长大，就先把自己设计复杂了”。

### 例子：放到一个小业务里
比如你在做一个小型后台管理系统，新增商品接口里只做三件事：校验参数、写数据库、返回结果。虽然你也知道线程池、缓存、MQ 很常见，但这个阶段其实没必要硬塞进去，因为它们解决的是还没真正出现的问题。

### 它对应什么
- **对应核心原则**：权衡——没有银弹，只有取舍
- **对应能力域**：请求链路与应用执行、数据与状态管理、架构演进与技术取舍
- **这一阶段最关键的程序员判断**：先把简单模型跑顺，不要过早为未来可能出现的问题支付复杂度成本

### 浅：最后带走一句话
**小系统的默认策略应该是：先同步、先单体、先单库，把简单模型用够。**

## 阶段 2：出现局部慢操作，开始考虑异步化

### 浅：先记一句话
**不是所有事情都要在主请求里做完。**

### 深：为什么异步通常是第一个被引入的变化
当业务逐渐多起来后，你会开始遇到一种很典型的情况：
- 主业务其实不复杂
- 但它后面挂着一些“耗时但不必立刻完成”的动作

比如：
- 下单成功后发短信、发邮件
- 用户注册后初始化资料
- 支付成功后做积分、通知、埋点
- 上传文件后做缩略图、转码、审核

这时候如果仍然全部同步执行，用户会感觉接口很慢。
程序员就会开始思考：
“这里面哪些动作是用户必须现在等到的，哪些动作可以晚一点完成？”

于是，异步化往往成为第一个自然引入的思路。
最开始甚至不一定要很重：
- 可能只是 `@Async`
- 可能只是自己起一个后台线程
- 可能只是简单的事件机制

这里的关键不是“用了什么异步工具”，而是思维发生了变化：
**开始把主流程和附属流程拆开。**

不过，异步一进来，复杂度也会跟着进来：
- 任务失败了怎么办
- 异步异常谁来接
- 用户现在看到的状态是不是最终状态
- 如果重复触发，会不会多执行一次

### 例子：放到一个小业务里
比如用户注册接口里，真正影响用户体验的是“账号创建成功”。而发欢迎邮件、初始化默认标签、记录埋点，其实都可以稍后做。这个时候你不是在“炫技上异步”，而是在保护主流程的响应时间。

### 它对应什么
- **对应核心原则**：异步化与解耦、权衡——没有银弹，只有取舍
- **对应能力域**：并发、异步与任务编排，请求链路与应用执行
- **这一阶段最关键的程序员判断**：先分清主流程和附属流程，再决定哪些事情值得异步化

### 浅：最后带走一句话
**异步的真正起点，不是技术升级，而是你开始区分“必须立刻完成”和“可以延后完成”。**

---

## 阶段 3：异步任务变多，线程池开始成为必需品

### 浅：先记一句话
**零散的异步能救急，但规模化的异步必须被管理。**

### 深：为什么会从“能异步”走到“要管理异步”
当系统里只有一两个耗时操作时，临时异步通常还能顶住。
但当类似场景越来越多时，问题就会慢慢暴露：
- 每个地方都自己开线程，数量不可控
- 高峰期线程暴涨，CPU 和内存压力上升
- 任务执行顺序和积压情况看不清
- 异常处理分散，出问题很难排查

这时候，程序员会意识到：
问题已经不只是“要不要异步”，而是“异步任务应该怎么统一调度”。

于是线程池变得自然：
- 线程数量可控
- 任务队列可控
- 拒绝策略可控
- 生命周期可控

这一步很重要，因为它代表系统开始从“写功能”进入“管理资源”。
而线程池本质上是在回答一个很工程化的问题：
**有限线程资源，应该如何分配给不断到来的任务？**

但线程池也不是用了就结束，新的问题会接着出现：
- 核心线程数怎么设
- 队列该大还是小
- 拒绝策略怎么选
- 线程池打满时，系统表现会怎样

### 例子：放到一个小业务里
比如订单成功后有发短信、记积分、写审计日志、同步用户画像四类异步任务。开始时每个地方都随手异步一下还能用，等活动一来，线程数暴涨、任务积压、错误难查，你就会发现必须把这些任务统一收口到线程池里管理。

### 它对应什么
- **对应核心原则**：池化与复用、异步化与解耦、资源有限
- **对应能力域**：并发、异步与任务编排，资源、性能与容量规划
- **这一阶段最关键的程序员判断**：异步任务一旦形成规模，问题就不再是“能不能并发”，而是“怎么有边界地并发”

### 浅：最后带走一句话
**线程池不是“异步版 for 循环”，而是系统第一次认真面对“资源调度”这件事。**

---

## 阶段 4：资源开始吃紧，性能优化进入视野

### 浅：先记一句话
**系统一上量，性能问题本质上就是资源分配问题。**

### 深：为什么这时候会开始关注池化、复用和指标
到了这一步，系统通常已经不是“功能能跑就行”了，而是开始出现这些信号：
- 请求 RT 上升
- 高峰期 CPU 飙高
- 数据库连接不够用
- GC 变频繁
- 某些接口一慢就连锁影响别的接口

程序员这时会越来越清楚：
很多慢，不是因为业务逻辑多复杂，而是资源被低效使用了。
于是会自然进入下面这些优化动作：
- 线程池参数调整
- 数据库连接池配置
- HTTP 连接复用
- SQL 和索引优化
- 批量处理、分页、削峰

这里最关键的认知转变是：
**优化不是“把代码写得更酷”，而是让有限资源承载更多有效工作。**

这个阶段还会引出一个很重要的方法论：
不要靠感觉优化，要开始看数据。
也就是：
- RT、QPS、P99
- CPU、内存、GC
- 慢 SQL
- 线程 dump

### 例子：放到一个小业务里
比如大促前你发现订单列表接口时快时慢。直觉上你怀疑是 SQL 问题，但一看监控才发现真正先打满的是数据库连接池，随后线程池排队、RT 被拖长。这种时候真正有用的不是拍脑袋，而是先看指标再动手。

### 它对应什么
- **对应核心原则**：池化与复用、量化先行，数据驱动、权衡——没有银弹，只有取舍
- **对应能力域**：资源、性能与容量规划，可观测性与问题诊断
- **这一阶段最关键的程序员判断**：优化前先定位真正的瓶颈层，不要把“感觉”当成证据

### 浅：最后带走一句话
**性能优化真正开始的标志，不是写了缓存，而是开始用指标理解系统。**

---

## 阶段 5：数据库先扛不住，读写分离与数据优化进入主线

### 浅：先记一句话
**大多数业务系统，数据库往往是最早暴露压力的核心部件。**

### 深：为什么数据库通常先成为瓶颈
在很多 Spring Boot 业务系统里，最终的数据真实来源仍然是数据库。
所以随着流量增长，数据库往往最先暴露问题：
- 查询越来越慢
- 热点表竞争激烈
- 连接池不够用
- 写入拖慢整体吞吐
- 复杂 SQL 把数据库拖重

程序员在这个阶段通常不会一上来就分库分表。
更自然的路径一般是：
1. 先做 SQL / 索引优化
2. 再做表结构和查询方式优化
3. 再考虑连接池和事务边界
4. 读压力很大时，再考虑读写分离

为什么读写分离会在这里出现？
因为很多系统天然就是“读多写少”。
如果写仍集中到主库，而读流量能被多个从库分摊，那么数据库整体承载能力就会明显提升。

但这一步一做，新的理解就必须补上：
- 主从同步有延迟
- 刚写完立刻读，可能读不到最新值
- 某些核心场景不能接受旧数据

这时你就会第一次非常具体地感受到：
**性能提升，往往是靠牺牲一部分简单性和一致性换来的。**

### 例子：放到一个小业务里
比如订单列表接口越来越慢，第一反应不应该是“立刻分库分表”，而应该先看 SQL 有没有走索引、查询是不是带了不必要的 join、分页方式是否合理、事务是否包得过大。只有这些都做得差不多了，读压力仍然顶不住，读写分离才开始变得值得。

### 它对应什么
- **对应核心原则**：权衡——没有银弹，只有取舍、分而治之、量化先行，数据驱动
- **对应能力域**：数据与状态管理，资源、性能与容量规划，架构演进与技术取舍
- **这一阶段最关键的程序员判断**：先把单库能力用深，再决定是否通过拆分换容量

### 浅：最后带走一句话
**数据库优化的常见顺序是：先把数据库用对，再考虑把数据库拆开。**

---

## 阶段 6：读流量继续增长，缓存成为自然选择

### 浅：先记一句话
**当大量请求都在重复读同一批数据时，缓存几乎一定会出现。**

### 深：为什么缓存通常是数据库之后的下一步
如果数据库已经被读请求压得很重，而这些读请求又有明显重复性，比如：
- 商品详情被反复查询
- 热门列表反复读取
- 用户基础信息高频访问
- 配置数据被大量读取

那么程序员很自然会想到：
“既然很多数据短时间内不会变，为什么每次都打到数据库？”

于是缓存成为非常自然的下一步。
它解决的问题很直接：
- 降低数据库读压力
- 降低访问延迟
- 提高系统整体吞吐

但缓存不是“加上就快了”这么简单。
它会立刻把系统从“单数据源思维”带到“多副本思维”：
- 数据到底以数据库为准，还是以缓存为准
- 更新时先改谁
- 删除缓存和更新数据库的顺序是什么
- 缓存失效时会不会击穿数据库

这一步是很多程序员第一次真正感受到：
**一个优化手段，为什么会把系统从简单直接拉进复杂治理。**

### 例子：放到一个小业务里
比如商品详情页被大量重复访问，数据库顶不住时，把商品基本信息放到 Redis 看起来很自然。可一旦商品价格或库存更新，就必须回答一个更难的问题：用户此刻看到的到底是不是最新值？

### 它对应什么
- **对应核心原则**：可见性与一致性、权衡——没有银弹，只有取舍
- **对应能力域**：数据与状态管理，资源、性能与容量规划
- **这一阶段最关键的程序员判断**：缓存提升的是读性能，但你同时引入了副本同步问题

### 浅：最后带走一句话
**缓存的本质，不只是把数据放快一点的地方，而是正式引入“副本”。**

---

## 阶段 7：副本变多，一致性问题开始成为显性主题

### 浅：先记一句话
**只要一个数据不止存在一个地方，一致性就迟早会变成主问题。**

### 深：为什么系统越快，越容易碰到一致性问题
在前面的阶段里，你已经逐渐引入了很多“为了快、为了稳、为了扛量”的手段：
- 线程池
- 读写分离
- 缓存
- 异步处理

这些方案有一个共同点：
它们几乎都会制造“多个地方存在状态”的局面。
比如：
- 数据库主库和从库
- 数据库和 Redis
- 用户看到的订单状态和异步任务里的订单状态
- 消息投递方和消费方对同一事件的理解

于是系统开始出现更难解释的问题：
- 为什么刚改完数据，下一次查询还是旧的
- 为什么用户看到了支付成功，但库存还没扣
- 为什么消息已经发出，但下游还没处理完成

这时程序员必须从“功能正确”提升到“状态一致性正确”。
也就是说，不再只是问“代码跑没跑”，而要问：
**在不同时间、不同节点、不同副本上，大家看到的是不是同一个现实。**

### 例子：放到一个小业务里
比如用户支付成功后，订单状态先写数据库，再由异步任务刷新缓存并通知库存系统。这时从数据库、缓存、消息消费者三个角度看，同一笔订单很可能在几秒钟内处于不同状态。用户问“我到底付没付成功”，你面对的已经不是单点功能问题，而是跨副本的一致性问题。

### 它对应什么
- **对应核心原则**：可见性与一致性、权衡——没有银弹，只有取舍
- **对应能力域**：数据与状态管理，可观测性与问题诊断，稳定性、容错与恢复
- **这一阶段最关键的程序员判断**：系统越追求快和扛量，就越要认真设计状态同步与一致性边界

### 浅：最后带走一句话
**一致性问题往往不是系统一开始就有，而是系统开始优化之后被“制造”出来的。**

---

## 阶段 8：还没拆服务，稳定性和容错就已经会成为主线

### 浅：先记一句话
**稳定性问题不是拆分之后才有，而是依赖一多就会先冒出来。**

### 深：为什么这一步通常早于服务拆分
很多人一提到稳定性，就容易先想到“微服务、分布式、服务治理”。
但真实项目里，稳定性问题往往在还没拆服务之前就已经出现了。

原因很简单：哪怕你现在还是一个 Spring Boot 单体应用，只要业务开始依赖更多外部组件，请求链路就已经变长了：
- 调数据库
- 调缓存
- 调搜索
- 调短信服务
- 调支付服务
- 调第三方接口

这时候虽然你还没有把系统拆成很多内部服务，但你已经处在一个“依赖很多别人”的环境里。
而只要依赖多了，现实世界里的噪声就会不断出现：
- 网络抖动
- 下游超时
- 实例重启
- 第三方接口偶发失败
- 高峰期失败被重试放大

于是程序员会先被迫面对一个非常现实的问题：
现在的问题已经不只是“功能能不能成功执行”，而是“某个依赖失败时，怎么别把整个入口请求和整台应用一起拖死”。

这就是为什么超时、重试、熔断、降级、限流会在这一步进入主线。
它们并不是拆分之后才需要的高级能力，而是系统一旦依赖变多，就必须尽早建立的防护边界。

后面如果真的再走到服务拆分，这些问题只会被进一步放大，而不会从那时才第一次出现。

### 例子：放到一个小业务里
比如下单接口还在一个单体应用里，但它已经要调库存、优惠券、支付风控、短信通知。只要其中一个下游超时，整条链路都可能被拖慢。如果你没有超时、重试边界、熔断和降级，下游的小问题就很容易变成入口的大故障。

### 它对应什么
- **对应核心原则**：快速失败与兜底、权衡——没有银弹，只有取舍
- **对应能力域**：稳定性、容错与恢复，可观测性与问题诊断，架构演进与技术取舍
- **这一阶段最关键的程序员判断**：稳定性不是让系统永不失败，而是让失败别无限扩散；而且这件事通常在拆分前就要开始做

### 浅：最后带走一句话
**稳定性建设的本质，是承认失败一定会发生，然后尽早为失败设计边界。**

---

## 阶段 9：同步链路太重，消息队列与事件解耦开始出现

### 浅：先记一句话
**消息队列常常不是因为“已经微服务化”才出现，而是因为同步链路已经太重了。**

### 深：为什么 MQ 往往在拆分前后都可能出现
随着业务扩展，一个“事件”往往会牵动很多后续动作。
比如用户下单成功后，可能还要：
- 扣库存
- 生成物流单
- 发优惠券
- 记积分
- 推送通知
- 做风控记录

如果这些都同步串起来：
- 主流程会越来越长
- 失败面会越来越大
- 某个下游变慢，会拖垮整个入口请求
- 上游系统会越来越像“总调度中心”

这时候，哪怕系统还是一个大单体，或者只是部分模块独立了，程序员也会自然想到：
“能不能不要每次都同步调到底？能不能先把事件发出去，后面的动作再各自处理？”

于是消息队列出现得就很自然。
它把“一个请求同步串很多动作”变成“主流程只完成核心动作，后续动作按事件异步展开”。

这一步和服务拆分有关系，但不是严格前后依赖关系：
- 有些项目是在单体阶段就先用了 MQ 做异步、削峰和事件通知
- 有些项目是在部分拆分之后，才更大规模地用 MQ 解决系统间解耦问题

所以更准确的理解不是“先拆分，后 MQ”，也不是“先 MQ，后拆分”一定固定，而是：
**只要同步链路已经太重，MQ 就会变得值得。**

但它一进来，又会把系统推进到更高复杂度：
- 消息会不会丢
- 会不会重复消费
- 消费失败怎么重试
- 堆积了怎么办
- 消息到了，但业务状态还没完全一致怎么办

### 例子：放到一个小业务里
比如“订单创建成功”这一个事件，后面要触发库存冻结、积分发放、优惠券回收、用户触达和数据分析。如果全同步串联，主流程会越来越长；如果改成发一个订单事件，让后续动作异步处理，主流程就会轻很多，但消息可靠性和幂等性随之成为新主题。

### 它对应什么
- **对应核心原则**：异步化与解耦、分而治之、权衡——没有银弹，只有取舍
- **对应能力域**：并发、异步与任务编排，稳定性、容错与恢复，架构演进与技术取舍
- **这一阶段最关键的程序员判断**：消息队列解决的是同步链路过重、系统耦合和流量峰值问题，但会把时序和一致性复杂度留到后面处理

### 浅：最后带走一句话
**消息队列不是“微服务专属工具”，而是在同步链路过重时出现的解耦手段。**

---

## 阶段 10：单体和单库模型开始失效，才真正进入分布式拆分

### 浅：先记一句话
**拆分往往不是前面问题的起点，而是前面问题被放大后的下一步。**

### 深：为什么服务拆分通常更晚、更重
前面的很多方案，本质上都还是在“尽量延长单体 / 单库模型的寿命”：
- 用异步减轻主流程阻塞
- 用线程池统一管理任务
- 用超时、熔断、降级保护依赖调用
- 用 MQ 让系统别什么都同步串起来
- 用缓存和读写分离继续榨出单体 / 单库的承载能力

但当业务继续增长，你终究可能会碰到这些信号：
- 单表数据量太大
- 单库写入和查询都扛不住
- 单体应用太臃肿，发布和协作成本太高
- 不同业务模块的扩容需求完全不同
- 某些模块必须独立演进，否则互相拖累

这时候，系统才会逐步进入更重的演进动作：
- 分库分表
- 服务拆分
- 独立部署
- 注册发现和服务治理
- 分布式事务和跨服务协作

这里最需要说清楚的一点是：
拆分并不是“从这里开始系统才有依赖链、稳定性和一致性问题”。
更准确地说，前面这些问题很多早就已经出现了；
而拆分会把它们进一步推到更高层级：
- 原来是应用调外部依赖，现在变成服务之间大量互调
- 原来是单库内事务，现在变成跨服务、跨库协作
- 原来是单体内排障，现在变成跨服务链路排障
- 原来是一个应用发布，现在变成多个服务协调发布

所以拆分从来不是免费午餐。
它解决的是容量和边界问题，但会把协调成本迅速抬高：
- 跨服务调用更复杂
- 跨库查询更麻烦
- 一致性更难保证
- 故障定位更难
- 测试和发布链路更长

所以程序员真正成熟的标志，不是“很早就会拆”，而是：
**知道什么时候单体 / 单库真的已经不值继续硬撑了，什么时候拆分才值得。**

### 例子：放到一个小业务里
比如订单、库存、支付原本都在一个应用里。前面你已经做过异步通知、线程池治理、缓存、读写分离，甚至部分流程也用 MQ 解耦了。但后来订单量继续上涨，库存热点明显、支付接口又要独立扩容、团队协作也越来越互相卡住，这时把它们拆成独立服务才会更自然。可一拆开，你立刻就要面对跨服务调用、跨库查询、分布式事务和发布协调问题。

### 它对应什么
- **对应核心原则**：分而治之、权衡——没有银弹，只有取舍、可见性与一致性
- **对应能力域**：架构演进与技术取舍，数据与状态管理，稳定性、容错与恢复
- **这一阶段最关键的程序员判断**：拆分不是炫耀架构能力，而是在单整体模型失效、且前面手段已经不足时做出的工程选择

### 浅：最后带走一句话
**拆分解决的是承载和边界问题，但会把原本已有的复杂度推到更高层级。**

---

## 阶段 11：系统已经看不清了，可观测性成为驾驶舱

### 浅：先记一句话
**系统一复杂，感觉就不可靠了，必须靠可观测性来驾驶。**

### 深：为什么监控、日志、链路追踪一定会补上来
在小系统里，很多问题还能靠经验猜中。
但到了分布式和多组件协作阶段，你会发现：
- 错误不一定出在报错的位置
- 慢请求不一定慢在你怀疑的地方
- 某个峰值问题可能是多个因素叠加

这时如果没有足够的日志、指标和链路，你甚至不知道问题从哪里开始追。
所以系统发展到一定程度后，可观测性不再是“锦上添花”，而是“没有就没法维护”。

程序员在这个阶段真正建立起来的能力，是：
- 用指标看健康度
- 用日志看上下文
- 用链路看跨服务传播
- 用 profiling / dump 看局部内部状态

也就是说，开始具备“解释系统行为”的能力，而不是只会“修改业务代码”的能力。

### 例子：放到一个小业务里
比如用户反馈“下单偶尔很慢”，但你本地根本复现不了。最后靠接口 RT 指标发现问题集中在大促时段，再靠链路追踪看到卡在库存服务，继续看日志和线程 dump 才定位到线程池排队。没有这些观测手段，这类问题几乎只能靠猜。

### 它对应什么
- **对应核心原则**：量化先行，数据驱动
- **对应能力域**：可观测性与问题诊断，资源、性能与容量规划，稳定性、容错与恢复
- **这一阶段最关键的程序员判断**：系统越复杂，排障越不能靠经验猜，而要靠数据逐层收缩范围

### 浅：最后带走一句话
**可观测性不是给系统加监控，而是给工程师补眼睛。**

---

## 阶段 12：最后回看，你会发现一切都连起来了

### 浅：先记一句话
**后端演进不是技术堆叠，而是压力驱动下的一连串因果。**

### 深：把前面的线重新串起来
如果回头看，你会发现这条演进线其实很自然：

- 一开始系统小，所以简单同步模型够用
- 慢操作出现，所以开始异步化
- 异步变多，所以需要线程池统一调度
- 资源吃紧，所以开始池化和性能优化
- 数据库先扛压，所以先做数据库优化和读写分离
- 读流量继续增加，所以引入缓存
- 副本变多，所以一致性问题开始显性化
- 依赖链变长，所以稳定性治理进入主线
- 业务协作变重，所以消息队列和解耦出现
- 单整体承载不了，所以进入拆分和分布式
- 系统越来越复杂，所以必须靠可观测性来驾驭

这条线里没有任何一步是“为了高级而高级”。
每一步几乎都是因为前一个阶段的简单模型开始失效。

这就是为什么真正理解后端，不是记住“有哪些技术”，而是理解：
- 它们通常在什么压力下出现
- 为什么不是更早，也不是更晚
- 它们解决了什么
- 又制造了什么新的复杂度

### 例子：回看一条完整业务线
如果把“用户下单”这件事从头到尾回看一遍，你会看到它几乎能串起整条演进线：最初只是同步写库；后来加异步通知；再后来异步任务进线程池；数据库变慢后做索引和读写分离；热点商品上缓存；跨系统协作上 MQ；规模再大后拆服务；最后靠日志、指标和链路追踪持续驾驭。它不是一组散点技术，而是一条有因果的工程路线。

### 它对应什么
- **对应核心原则**：前面 8 条几乎都会在这条线上出现
- **对应能力域**：前面 8 个能力域几乎都会被逐步触发
- **这一阶段最关键的程序员判断**：真正成熟的理解，不是记住每个技术点，而是能把它们放回系统演进的上下文里

### 浅：最后带走一句话
**成熟的后端认知，不是会背技术栈，而是能顺着系统压力推演出技术演进。**
