# V1：跑通核心交易链路

## 定位

这是 `mall-monolith/` 的第一阶段，目标是先搭好最小业务基础模型，再把核心交易 happy path 跑通，不追求一开始就把所有复杂情况都处理完。

当前切片只覆盖：

- `mall-monolith/`

当前切片明确不覆盖：

- `mall-microservices/`
- Kafka、事件总线、Saga、服务注册发现、网关

## 当前要解决的问题

先把最小业务基础和下面这条主链路真正跑起来：

- 分类与商品基础资料
- 商品读取
- 创建订单
- 扣减库存
- 创建支付记录
- 确认支付
- 创建并执行通知任务

如果这条主链路还没有稳定跑通，后面的幂等、缓存、可观测性和多实例讨论都会失去落点。

## 本切片必须完成的范围

### 业务流程

- 提供最小商品域：分类、商品、商品上下架状态
- 提供商品查询接口
- 提供创建订单接口
- 创建订单时完成库存扣减、订单落库、支付记录初始化
- 提供确认支付接口
- 确认支付后更新支付状态、更新订单状态、创建通知任务
- 通过调度器 + 线程池异步执行通知任务

### 最小业务基础模型

- 分类先做轻量版，只承担商品归类和读路径组织作用
- 商品先采用“一个记录就是一个可售卖项”的简化模型，不在 V1 引入完整 `SPU / SKU`
- 库存继续独立建模，不直接塞进商品表
- 基础资料优先通过 Flyway 初始化，不在 V1 实现完整后台管理能力
- 订单项必须保存商品快照，避免后续商品资料变化影响历史订单

### 工程骨架

- 建立 `mall-common`、`mall-inventory`、`mall-order`、`mall-payment`、`mall-notification`、`mall-app` 的基础结构
- 建立 Flyway 基础迁移
- 接入 PostgreSQL
- 保留 Redis 配置位，但不把缓存作为本切片重点
- 提供最小可用的 `/actuator/health`

### 最小验证

- 至少有一个集成级 smoke test 覆盖 happy path
- 可以通过手动接口调用验证订单、支付、通知任务的状态变化

## 本切片暂时不强调的内容

- 完整支付幂等
- 高并发下的库存竞争治理
- 通知失败后的复杂重试策略
- Redis 缓存命中率优化
- 多实例任务领取竞争
- 系统化指标和结构化日志治理

这些内容不是不重要，而是放到后续切片里在更真实的压力下再处理。

## 当前训练重点

- 怎么把一个完整业务链路按模块拆开
- 怎么把“商城基础资料”和“交易主链路”控制在最小可实现范围内
- 怎么定义第一条清晰的本地事务边界
- 为什么通知不应该阻塞支付确认主链路
- 为什么要先把任务持久化，再交给异步执行器

## 进入 V2 的触发压力

当 V1 跑通后，主动引入下面这些问题：

- 同一个订单被重复确认支付
- 同一商品被并发下单
- 通知执行失败或超时
- 同类通知任务被重复创建

这些压力会逼出幂等、状态约束、并发控制和异步可靠性问题。

## 验收口径

- 可以从商品查询一路跑到支付确认成功
- 订单状态和支付状态按预期变化
- 通知任务能被异步执行并看到结果
- 主链路相关表结构、模块边界和接口职责清晰
- 有最小 smoke test 证明 happy path 可运行
