# V4：稳定性与多实例友好

## 定位

这是 `mall-monolith/` 的第四阶段，目标是让系统在故障、堆积和扩容场景下更稳、更容易排障，也更适合多实例运行。

当前切片只覆盖：

- `mall-monolith/`

这里说的“多实例”指的是：

- 同一个单体应用以多个副本运行

这里明确不等于：

- 已经进入微服务拆分

当前切片明确不覆盖：

- 服务注册发现
- 网关
- Kafka / 事件驱动
- 跨服务链路追踪

## 当前要解决的问题

当 V3 之后系统开始更像一个真实服务，问题就不再只是“对不对、快不快”，还包括：

- 下游通知适配器超时或抖动
- 线程池和任务调度出现堆积
- 两个实例同时扫描任务时可能发生竞争
- 出问题时缺少足够的日志和指标来定位

## 本切片必须完成的范围

### 可观测性

- 为关键链路补齐结构化日志
- 日志中保留 `orderNo`、`productId`、`taskId` 等业务键
- 补齐 `Actuator` 健康检查和关键 Micrometer 指标

### 稳定性边界

- 在线程池、调度器和外部通知适配层明确超时与重试边界
- 区分核心写链路和附属异步链路的失败处理策略
- 为线程池明确队列、拒绝策略和观测方式

### 多实例友好

- 让通知任务领取在多实例下不重复执行
- 明确应用无状态化约束
- 在同库多实例场景下验证任务扫描和领取行为

## 本切片暂时不强调的内容

- 微服务熔断器与网关治理
- 服务级链路追踪平台
- 分布式配置中心
- 服务拆分后的容量规划

## 当前训练重点

- 怎么给系统补日志、指标和健康检查，而不是只靠肉眼看控制台
- 为什么超时、重试、线程池和队列必须一起设计
- 为什么多实例问题在单体阶段也值得提前处理
- 故障出现后怎样基于日志、指标和任务状态倒推问题

## 典型压力场景

- 通知适配器偶发超时
- 通知任务短时间内大量堆积
- 两个应用实例同时扫描待执行任务
- 线上报错时无法快速知道是库存、支付还是通知链路出了问题

## 验收口径

- 关键链路日志能串起订单、支付和通知任务
- 至少有一组关键指标可以帮助判断线程池和任务执行状态
- 在多实例场景下，通知任务不会被重复消费
- 出现失败、超时或堆积时，有可操作的排障入口
