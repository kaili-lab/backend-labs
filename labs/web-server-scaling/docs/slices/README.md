# 实施切片总览

这里存放 `mall-monolith/` 的训练路线与实施切片。

## 当前边界

- 当前 `V1 ~ V4` 只覆盖 `mall-monolith/`
- `mall-microservices/` 暂不进入这四个切片
- 微服务、Kafka、Saga、服务治理等内容，等单体四个切片完成后再单独规划新的切片序列

这样做的原因是：

- 单体已经足够承载事务、一致性、异步、缓存、稳定性和可观测性这些高级能力
- 先把“怎么做、为什么这么做、代价和边界是什么”在单体里讲清楚
- 避免一上来就被分布式基础设施淹没

## 四个切片的关系

| 切片 | 目标 | 主要引入的压力 | 主要训练能力 |
| --- | --- | --- | --- |
| `V1` | 最小业务基础模型 + 核心交易 happy path | 先不追求高压，只验证完整主链路 | 商品/分类/库存基础建模、模块边界、主事务链路、最小异步链路 |
| `V2` | 保证链路在重复调用和失败下不做错 | 重复支付、并发下单、通知失败 | 幂等、状态机、库存并发控制、异步任务可靠性 |
| `V3` | 解决热点读和查询变慢问题 | 热点商品读取、订单查询变慢 | 缓存、索引、分页、慢 SQL 观察 |
| `V4` | 让系统更稳、更容易排障、更适合多实例运行 | 下游超时、任务堆积、多实例竞争 | 可观测性、线程池治理、超时重试边界、多实例友好设计 |

## 推进原则

- 每个切片都要明确：做什么、为什么现在做、暂时不做什么
- 每个切片都要先定义验收口径，再开始实现
- 不是为了编号而编号，只有在出现新的系统压力时才进入下一切片
- 当前切片没有讲清楚之前，不提前跳到后面的技术名词

## 建议推进顺序

1. 先完成 `V1-跑通核心交易链路`
2. 再进入 `V2-一致性与异步可靠性`
3. 然后进入 `V3-读路径与性能优化`
4. 最后完成 `V4-稳定性与多实例友好`

完成这四个切片后，再决定是否进入 `mall-microservices/` 的新阶段。
