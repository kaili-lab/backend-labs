# V2：一致性与异步可靠性

## 定位

这是 `mall-monolith/` 的第二阶段，目标不是新增很多功能，而是让 V1 的主链路在重复调用、并发竞争和任务失败下也不做错。

当前切片只覆盖：

- `mall-monolith/`

当前切片明确不覆盖：

- 微服务拆分
- Kafka / outbox / 事件总线
- 跨服务分布式事务

## 当前要解决的问题

V1 能跑通 happy path 后，系统会自然暴露这些压力：

- 支付回调可能重复到达
- 用户可能重复提交或多个请求同时抢同一库存
- 通知发送可能失败，需要重试
- 同一种通知任务可能被重复创建

这一阶段要把“结果不能做错”放在第一位。

## 本切片必须完成的范围

### 状态与幂等

- 为支付确认建立幂等语义
- 明确订单状态与支付状态的合法流转
- 拒绝非法状态跳转和重复生效

### 并发与库存

- 为库存扣减选择一种明确策略并落实到代码
- 至少能解释当前为什么选择这条路，而不是另一条路
- 确保库存不会被扣成负数

### 异步任务可靠性

- 为通知任务增加唯一性约束或等价防重机制
- 增加失败重试、重试次数和下次执行时间控制
- 区分可重试失败和不可重试失败

## 本切片暂时不强调的内容

- 热点读优化
- Redis 缓存策略
- 订单查询性能优化
- 完整的指标体系和线程池治理
- 多实例部署下的任务竞争验证

## 当前训练重点

- 怎么实现幂等，而不是只停留在概念上
- 为什么支付状态和订单状态不能简单混成一个字段
- 并发库存控制的方案取舍是什么
- 为什么异步任务重试必须带状态，而不能靠“无限重跑”

## 典型压力场景

- 同一个 `orderNo` 连续调用两次确认支付
- 两个请求同时购买同一个库存紧张的商品
- 通知适配器第一次失败，第二次恢复
- 重复提交支付确认后，通知任务不能成倍增长

## 验收口径

- 重复确认支付不会重复推进业务状态
- 并发下单后库存结果正确，且不会出现负库存
- 通知失败后可以按规则重试，并保留失败信息
- 状态流转和失败恢复规则可以在文档和代码里讲清楚
