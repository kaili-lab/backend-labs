# V3：读路径与性能优化

## 定位

这是 `mall-monolith/` 的第三阶段，目标是在正确性的基础上，开始处理热点读和查询性能问题。

当前切片只覆盖：

- `mall-monolith/`

当前切片明确不覆盖：

- 分库分表
- 读写分离
- 搜索引擎
- 微服务阶段的跨服务缓存问题

## 当前要解决的问题

当 V2 先保证系统“做得对”之后，下一步就会遇到“做得够不够快”：

- 商品详情被反复读取
- 订单详情和列表查询开始变慢
- 数据量上涨后，SQL 和索引问题暴露

这一阶段要开始训练“量化先行”的性能思维。

## 本切片必须完成的范围

### 商品读路径优化

- 为商品读路径引入 Redis 缓存
- 采用明确的 Cache Aside 读写策略
- 说明哪些数据可以缓存，哪些数据必须回源数据库

### 订单查询优化

- 为订单详情和分页列表补齐合理索引
- 明确分页查询策略
- 对慢查询保留观察入口

### 性能验证

- 在优化前先记录基线
- 优化后对比关键指标或至少对比接口耗时变化
- 把“优化了什么、为什么有效、代价是什么”说清楚

## 本切片暂时不强调的内容

- 跨服务缓存一致性
- 复杂多级缓存
- 分布式限流
- 微服务读链路治理

## 当前训练重点

- 缓存不是为了“用了 Redis”，而是为了缓解真实热点读压力
- 索引不是凭感觉加，而是围绕查询路径设计
- 分页、缓存和一致性之间要做取舍
- 性能优化必须带量化对比，而不是只说“快了”

## 典型压力场景

- 同一个商品被高频查询
- 订单列表随着数据增长明显变慢
- 缓存 miss 时数据库读压力上升
- 商品信息更新后缓存可能短暂陈旧

## 验收口径

- 商品热点查询有明确缓存策略并实际生效
- 订单查询路径的索引和分页策略可解释
- 至少有一组前后对比数据证明优化效果
- 能清楚说明缓存收益、代价和边界
