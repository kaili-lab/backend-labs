# V1-03：支付确认与通知任务最小异步流程

## 本批目标

在 `V1-02` 的基础上，补齐支付确认和最小异步通知流程。

本批完成后，应达到：

- 可以确认支付
- 订单状态能推进到已支付
- 能创建通知任务
- 通知任务能被调度器和线程池异步执行

## 本批范围

- 定义通知任务表：
  - `ntf_notification_task`
- 提供确认支付接口
- 在一个事务中完成：
  - 支付状态更新
  - 订单状态更新
  - 通知任务入库
- 补调度器扫描逻辑
- 补通知执行线程池
- 跑通最小异步通知执行

## 本批明确不做

- 完整幂等语义
- 复杂失败重试策略
- 多实例任务领取竞争治理
- 外部通知适配器的完整稳定性策略

## 验收标准

### 结构验收

- 已新增并迁移通知任务表：
  - `ntf_notification_task`
- 已存在支付确认入口
- 已存在通知任务扫描与执行所需的最小调度器和线程池配置

### 运行验收

- 支付确认接口可访问
- 对已创建订单执行支付确认后，应用能返回成功结果
- 通知任务会被扫描并进入执行流程

### 数据验收

- 支付确认成功后：
  - `pay_payment` 状态更新为成功
  - `ord_order` 状态推进为已支付
  - `ntf_notification_task` 成功入库
- 通知任务执行后，任务状态能够发生变化，并可观察成功或失败结果

### 事务验收

- 支付状态更新、订单状态推进、通知任务入库三者必须同属一次事务
- 如果该事务失败，不应出现“支付已成功但通知任务未入库”的半成品状态

### 边界验收

- 本批不应实现完整幂等语义
- 本批不应实现复杂重试策略和多实例任务竞争治理
- 本批不应把通知执行放回支付确认同步主链路中

## 风险与注意点

- 通知任务入库必须和支付确认放在同一事务里
- 通知执行不能反向影响核心支付结果
- 本批只追求最小异步流程，不提前扩展成复杂任务系统
