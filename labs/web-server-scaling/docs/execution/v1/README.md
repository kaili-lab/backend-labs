# V1 执行目录

这里存放 `V1-跑通核心交易链路` 的执行清单与批次记录。

当前建议按 4 个批次推进：

- `01-最小业务模型与工程骨架.md`
- `02-订单创建与支付记录初始化.md`
- `03-支付确认与通知任务最小异步流程.md`
- `04-基础验证与收口.md`

## 这 4 个批次分别负责什么

### `01-最小业务模型与工程骨架.md`

目标：

- 把 `mall-monolith` 父工程和多模块骨架搭起来
- 接入 Spring Boot、PostgreSQL、Flyway、Actuator
- 落地分类、商品、库存的最小表结构与种子数据
- 让应用先启动起来，并能通过健康检查

### `02-订单创建与支付记录初始化.md`

目标：

- 基于最小业务模型打通“创建订单”主链路
- 在一个清晰事务里完成库存扣减、订单写入、订单项快照、支付记录初始化

### `03-支付确认与通知任务最小异步流程.md`

目标：

- 打通支付确认流程
- 在同一事务里完成支付状态更新、订单状态推进、通知任务入库
- 用调度器 + 线程池跑通最小异步通知链路

### `04-基础验证与收口.md`

目标：

- 补最小 smoke test
- 校验关键接口和关键状态流转
- 补齐 V1 范围内的必要文档、配置和运行说明

## V1 与“项目能否启动”的关系

这里要特别说明：

- **`V1-01` 完成后，项目就应该可以启动起来**
- 但这时只代表“工程骨架、基础模型和基础设施接通了”，还不代表完整交易链路已经跑通
- **整个 `V1` 完成后，才表示这个单体项目已经具备最小可运行的业务 happy path**

也就是说：

- `V1-01` 的完成标准是“应用能启动、基础表已落、健康检查可用”
- `V1` 的完成标准是“商品查询、创建订单、确认支付、通知任务最小异步流程都可运行”
