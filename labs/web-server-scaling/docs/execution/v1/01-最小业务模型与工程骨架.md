# V1-01：最小业务模型与工程骨架

## 本批目标

这是 `mall-monolith` 的第一批执行任务，目标不是打通完整交易链路，而是先把最小业务模型和可运行工程骨架搭起来。

本批完成后，应达到：

- 项目可以启动
- 数据库迁移可以执行
- 分类、商品、库存的最小表结构存在
- 有一批可供后续开发使用的种子数据
- `/actuator/health` 可访问

## 本批范围

### 工程结构

- 创建 `mall-monolith` 父工程
- 创建模块：
  - `mall-common`
  - `mall-inventory`
  - `mall-order`
  - `mall-payment`
  - `mall-notification`
  - `mall-app`
- 建立父子 `pom.xml` 关系
- 固定 `JDK 21`、Spring Boot、Maven 基线

### 基础设施接入

- `mall-app` 接入 Spring Boot 启动能力
- 接入 PostgreSQL
- 接入 Flyway
- 接入 Actuator
- 预留 Redis 配置位，但本批不把缓存作为重点能力

### 最小业务模型

- 落地分类表：`cat_category`
- 落地商品表：`prd_product`
- 落地库存表：`inv_inventory`
- 为这些表补齐最小种子数据

### 最小读能力

- 提供一个最小商品读取接口，能返回：
  - 商品基本信息
  - 商品状态
  - 当前库存

## 本批明确不做

- 创建订单
- 支付记录初始化
- 确认支付
- 通知任务
- 幂等、重试、并发控制细节
- Redis 缓存逻辑

## 涉及文件类型

- 父 `pom.xml`
- 各模块 `pom.xml`
- `MallApplication`
- `application.yml`
- Flyway 迁移脚本
- 分类、商品、库存的实体/仓储/查询接口
- 基础 DTO、异常和统一响应

## 验收标准

### 结构验收

- `mall-monolith` 父工程存在，且能聚合所有子模块
- 下列模块均已创建并能被 Maven 识别：
  - `mall-common`
  - `mall-inventory`
  - `mall-order`
  - `mall-payment`
  - `mall-notification`
  - `mall-app`
- `mall-app` 作为当前启动模块存在，且包含 Spring Boot 启动类
- Flyway 迁移脚本已建立，且至少覆盖分类、商品、库存三张表

### 运行验收

- 在本地环境下，项目可以完成基础构建
- `mall-app` 可以正常启动
- `/actuator/health` 可访问并返回健康状态
- 应用启动过程中没有因为数据库迁移失败而中断

### 数据验收

- 数据库中存在：
  - `cat_category`
  - `prd_product`
  - `inv_inventory`
- 至少有一组可用于后续开发的分类、商品、库存种子数据
- 最小商品读取接口可以返回：
  - 商品基本信息
  - 商品状态
  - 当前库存

### 边界验收

- 本批不应出现：
  - `ord_order`
  - `ord_order_item`
  - `pay_payment`
  - `ntf_notification_task`
- 本批不应提供创建订单、确认支付、通知执行相关接口
- 本批可以预留 Redis 配置，但不应出现实际缓存读写逻辑

## 风险与注意点

- 先保证模块依赖方向正确，不要为了快把所有代码都塞进 `mall-app`
- 先用最小业务模型，不要在本批提前引入 `SPU / SKU`
- 如果启动链路已经通了，也不要在本批顺手做订单逻辑，避免范围膨胀
