# V1-02：订单创建与支付记录初始化

## 本批目标

在 `V1-01` 的基础上，打通“创建订单”这条同步主写链路。

本批完成后，应达到：

- 可以创建订单
- 可以扣减库存
- 可以写入订单与订单项快照
- 可以初始化支付记录

## 本批范围

- 定义订单相关表结构：
  - `ord_order`
  - `ord_order_item`
- 定义支付记录表结构：
  - `pay_payment`
- 提供创建订单接口
- 在一个清晰事务中完成：
  - 商品状态校验
  - 库存校验
  - 库存扣减
  - 订单写入
  - 订单项快照写入
  - 支付记录初始化

## 本批明确不做

- 确认支付
- 通知任务
- 支付幂等
- 高并发库存竞争治理

## 验收标准

### 结构验收

- 已新增并迁移下列表：
  - `ord_order`
  - `ord_order_item`
  - `pay_payment`
- 订单相关读写能力已落在 `mall-order`
- 支付记录初始化能力已落在 `mall-payment`

### 运行验收

- 创建订单接口可访问
- 在商品和库存种子数据存在的前提下，可以成功创建订单
- 请求失败场景下，应用不会因为未处理异常直接崩溃

### 数据验收

- 创建订单成功后，数据库中应同时出现：
  - 订单记录
  - 订单项快照记录
  - 支付记录（`PENDING`）
- 库存数量会随订单创建减少
- 订单项中保留商品快照，而不是只保留商品引用

### 事务验收

- 当库存校验失败或任一步骤抛错时：
  - 库存扣减不应生效
  - 订单不应落库
  - 支付记录不应残留

### 边界验收

- 本批不应提供确认支付接口
- 本批不应引入通知任务表和调度逻辑
- 本批不应实现复杂并发库存治理

## 风险与注意点

- 订单项必须保留商品快照
- 库存扣减和订单写入必须处于同一事务边界
- 先不要为了“以后微服务”提前把事务拆散

## 完成记录（2026-03-23）

### Completed
- 已新增并迁移 `ord_order`、`ord_order_item`、`pay_payment` 三张表。
- 已在 `mall-order` 落地订单与订单项持久化能力（含商品快照）。
