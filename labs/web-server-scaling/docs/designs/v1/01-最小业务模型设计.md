# V1：最小业务模型设计

## 背景

V1 不是要先做一个“完整商城后台”，而是要先搭好足以支撑第一条交易链路的最小业务基础。

如果一开始就把分类树、品牌、属性组、`SPU / SKU`、营销、购物车全部展开，项目会迅速偏向“搭功能大全”，而不是围绕后端能力做训练。

但如果完全没有商品和分类，只做订单、库存、支付，又会让项目失去“商城语境”，后续很多缓存、查询和读路径优化也会缺少真实落点。

所以 V1 的目标不是“完整商城模型”，而是：

- 保留商城语境
- 支撑第一条交易链路
- 控制复杂度

## 当前边界

V1 当前只定义这些业务对象：

- 分类
- 商品
- 库存
- 订单
- 支付记录
- 通知任务

V1 当前明确不引入：

- 品牌
- 属性组 / 规格维度
- 完整 `SPU / SKU` 双层模型
- 营销活动
- 购物车
- 用户地址
- 履约、发货、售后
- 后台商品维护全套 CRUD

## 候选方案

### 方案 A：一开始就做完整商城基础资料体系

包括：

- 多级分类树
- 品牌
- 属性组
- `SPU / SKU`
- 商品媒体资源
- 后台管理接口

优点：

- 业务形态更像真实商城
- 后续扩展时字段更少返工

问题：

- 会明显挤占 V1 对交易链路的注意力
- 早期很难判断哪些复杂度是真需要，哪些只是“看起来完整”
- 当前阶段训练重点不是商品中后台，而是交易主链路

### 方案 B：完全不做分类，只保留商品和库存

优点：

- 最简单
- 可以更快进入订单和支付

问题：

- 商城语境太薄
- 后续商品查询、缓存和读路径优化缺少自然落点
- “商品域”会退化成纯测试数据容器

### 方案 C：轻量分类 + 简化商品 + 独立库存

也就是当前选择：

- 分类保留，但只承担归类作用
- 商品先按“一个记录就是一个可售卖项”建模
- 库存独立建模
- 基础资料优先通过迁移脚本和种子数据提供，不急着做后台管理

## 当前选择

V1 采用 **方案 C：轻量分类 + 简化商品 + 独立库存**。

### 1. 分类：轻量版

当前目标只保留“商品属于哪个分类”这个能力。

建议表：

- `cat_category`

建议字段：

- `id`
- `category_id`
- `name`
- `status`
- `sort_order`
- `created_at`
- `updated_at`

当前约束：

- V1 不要求完整分类树能力
- 如果需要为后续预留扩展，可以加 `parent_category_id`，但 V1 不把树操作当作重点能力
- 当前不提供分类管理后台接口

### 2. 商品：简化版可售卖项

V1 暂时不区分 `SPU / SKU`，一个商品记录就代表一个可直接下单的可售卖项。

建议表：

- `prd_product`

建议字段：

- `id`
- `product_id`
- `category_id`
- `name`
- `price`
- `status`
- `created_at`
- `updated_at`

当前约束：

- `status` 至少支持 `ON_SHELF / OFF_SHELF`
- 商品价格以当前商品表为准，但下单时必须写入订单快照
- V1 只需要支持查询，不要求商品维护后台

### 3. 库存：独立模型

建议继续保持库存和商品分表。

建议表：

- `inv_inventory`

建议字段：

- `id`
- `product_id`
- `stock`
- `created_at`
- `updated_at`

这样设计的原因是：

- 商品信息和库存变化频率不同
- 库存是交易并发热点，后续更容易单独治理
- 有利于后面讨论事务边界、并发扣减和缓存边界

### 4. 基础资料初始化方式

V1 当前建议：

- 分类、商品、库存的初始数据通过 Flyway 迁移或专用种子数据脚本初始化

当前不建议在 V1 做：

- 分类新增接口
- 商品新增 / 编辑接口
- 后台商品管理页面

这样设计的原因是：

- V1 当前目标是尽快让交易主链路落地
- 早期把基础资料“先准备好”比“先把后台都做完”更符合训练目标

### 5. 订单快照

订单项保留：

- `product_id`
- `product_name`
- `unit_price`
- `quantity`

这样设计的原因是：

- 商品名称、价格、分类都可能后续变化
- 历史订单必须保留交易发生当时的业务事实

## 为什么当前这样选

- 它比“没有分类的裸交易系统”更像商城
- 它比“完整商城中后台”更轻，能把精力留给交易链路
- 它给后续缓存、查询、索引优化留出了真实对象
- 它没有提前引入 `SPU / SKU`、属性系统这些当前还用不到的复杂度

## 对 V1 后续设计的直接影响

- `商品查询` 在 V1 只需要支持读，不需要支持完整维护
- `创建订单` 直接基于简化商品模型和独立库存模型推进
- `订单项快照` 是 V1 必须具备的能力，不后置
- 后续如果进入更完整商品域，再评估是否从“简化商品”演进到 `SPU / SKU`

## 当前结论

V1 先采用下面这个最小业务基础：

- 轻量分类
- 简化商品
- 独立库存
- 订单项商品快照
- 基础资料通过种子数据初始化

这足以支撑第一条交易链路，也不会把 V1 变成商品中后台项目。
