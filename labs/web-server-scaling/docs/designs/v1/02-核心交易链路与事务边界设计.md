# V1：核心交易链路与事务边界设计

## 背景

在 V1 里，商品、分类和库存这些基础模型只是起点，真正要跑通的是第一条完整交易链路。

这条链路的目标不是一上来就处理所有异常和复杂流量，而是先把最小 happy path 做对，并把后续 `V2` 会用到的一致性问题提前留出清晰边界。

本设计建立在 `01-最小业务模型设计.md` 之上。

## 当前边界

V1 当前只覆盖下面这条链路：

1. 查询商品
2. 创建订单
3. 扣减库存
4. 创建支付记录
5. 确认支付
6. 创建通知任务
7. 异步执行通知

V1 当前明确不覆盖：

- 支付失败处理
- 超时未支付自动取消
- 完整支付回调幂等细节
- 高并发库存竞争优化
- `Kafka / outbox / 事件总线`
- 分布式事务

## 候选方案

### 方案 A：核心状态同步提交，附属通知异步化

也就是：

- 下单阶段同步完成库存扣减、订单落库、支付记录创建
- 支付确认阶段同步完成支付成功、订单已支付、通知任务入库
- 通知发送由任务表 + 调度器 + 线程池异步执行

优点：

- 主链路简单清晰
- 本地事务边界容易观察
- 通知不阻塞支付确认
- 为后续 V2 的幂等、重试、任务可靠性保留自然演进空间

问题：

- 仍需要面对本地事务边界的设计
- 支付确认和通知入库之间的关系必须定义清楚

### 方案 B：支付确认后直接同步发送通知

优点：

- 实现最直接
- 不需要任务表和调度器

问题：

- 通知失败会拖慢甚至破坏主链路
- 后续不好讨论重试、补偿和多实例任务领取
- 不符合“附属流程尽量异步化”的主线目标

### 方案 C：V1 就引入事件总线或消息队列

优点：

- 更接近分布式事件驱动形态
- 通知链路天然异步

问题：

- 提前引入基础设施复杂度
- 会把“单体阶段的事务边界”讨论变得模糊
- 不符合当前主题对 `mall-monolith` 的边界约束

## 当前选择

V1 采用 **方案 A：核心状态同步提交，附属通知异步化**。

## 当前链路定义

### 1. 商品查询

目的：

- 支撑下单前的商品展示和可售状态读取

当前只要求：

- 能读取商品基本信息
- 能读取当前库存
- 能看到商品是否允许下单

### 2. 创建订单

当前建议链路：

```text
POST /api/orders
  -> OrderController
    -> OrderService.createOrder()
      -> 查询商品与库存
      -> 校验商品状态与库存数量
      -> 扣减库存
      -> 创建订单与订单项快照
      -> 创建支付记录(PENDING)
      -> 返回订单结果
```

当前事务边界：

- `OrderService.createOrder()` 在一个本地事务中完成：
  - 库存扣减
  - 订单写入
  - 订单项写入
  - 支付记录初始化

这样设计的原因是：

- 这四步共同构成“订单已创建、等待支付”的最小业务事实
- 如果其中一步失败，不应该留下半成品状态

### 3. 确认支付

当前建议链路：

```text
POST /api/payments/{orderNo}/confirm
  -> PaymentController
    -> PaymentFacade.confirmPayment()
      -> 更新支付记录为 SUCCESS
      -> 更新订单状态为 PAID
      -> 创建通知任务
      -> 返回成功结果
```

当前事务边界：

- `PaymentFacade.confirmPayment()` 在一个本地事务中完成：
  - 支付状态更新
  - 订单状态更新
  - 通知任务入库

这样设计的原因是：

- 支付成功后，订单状态和“待通知事实”必须一起成立
- 如果支付成功但通知任务没入库，后续附属流程会失联

### 4. 异步通知

当前建议链路：

```text
NotificationTaskScheduler 扫描待处理任务
  -> 领取任务
    -> 交给 notificationExecutor
      -> 调用 NotificationService
      -> 成功则标记 SUCCESS
      -> 失败则记录错误并等待后续重试
```

当前边界：

- 通知执行不在支付确认事务内
- 通知执行失败不回滚订单和支付状态

这样设计的原因是：

- 通知属于附属流程，不应反向影响核心交易结果
- 为后续 V2 的任务重试和可靠性治理保留空间

## 当前模块职责

### `mall-order`

- 创建订单
- 保存订单及订单项快照
- 负责订单状态流转

### `mall-inventory`

- 查询库存
- 校验库存
- 扣减库存

### `mall-payment`

- 创建支付记录
- 更新支付状态

### `mall-notification`

- 创建通知任务
- 扫描任务
- 执行通知

### `mall-app`

- 持有启动、配置、调度和支付确认这种跨模块编排入口

## 为什么当前先这样分

当前 V1 先沿用 PRD 里的主线分工：

- 下单由 `mall-order` 承担主用例
- 支付确认由 `mall-app` 中的编排入口承接

这样做的原因是：

- 下单本质上仍以订单事实建立为中心
- 支付确认天然横跨支付、订单和通知三个模块
- V1 先把链路跑通，比一开始统一所有编排风格更重要

当前边界也要说明：

- 这是一种 V1 阶段的当前选择
- 如果后续发现跨模块编排越来越多，再评估是否统一收敛到更清晰的 application service / facade 风格

## 代价与边界

当前选择的代价：

- V1 的用例编排风格不是完全对称的
- 支付确认与异步通知之间的可靠性还只是最小实现
- 还没有处理复杂失败恢复和重复调用问题

但这些代价是当前可接受的，因为：

- 它们正好是 V2 要引入的压力点
- V1 的目标是先把最小链路跑通，并让边界可见

## 对后续切片的影响

- `V2` 会在这条链路上补幂等、状态约束、库存并发控制和任务重试
- `V3` 会在商品读取和订单查询路径上补缓存与性能优化
- `V4` 会在任务执行、线程池和多实例上补稳定性与可观测性

## 当前结论

V1 当前采用下面这条最小交易设计：

- 下单阶段：同步完成库存扣减、订单创建、支付记录初始化
- 支付确认阶段：同步完成支付成功、订单已支付、通知任务入库
- 通知阶段：通过任务表 + 调度器 + 线程池异步执行

这条链路足够小，能先跑起来；同时也足够真实，能自然长出后续切片要解决的问题。
