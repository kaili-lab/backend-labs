# stage-03：引入数据库后的新瓶颈

## 阶段目标
到了 `stage-02`，我们已经知道：
- 请求内并发有价值
- 线程池可以帮助控制执行资源

但系统一旦真正开始持久化数据，压力的重心就会发生迁移。

这一阶段要理解的是：
- 为什么数据库会成为很多系统最早暴露出来的核心瓶颈
- 为什么应用层并发提升后，数据库反而更容易成为压力集中点
- 为什么数据库接入之后，后面的优化思路会和纯应用内优化明显不同

## 主场景
这一阶段继续沿用商品详情聚合接口，但现在不再使用纯模拟数据。

当前接口需要从数据库中读取：
- 商品基本信息
- 库存信息
- 推荐信息

这样做的目的不是把业务做复杂，而是让主线继续保持同一个业务语义，
同时把新的变化集中在“数据已经进入数据库”这一点上。

## 当前系统形态
基于 `stage-02` 继续演进。

当前阶段新增：
- PostgreSQL
- Flyway 数据库迁移
- Spring Data JPA + Hibernate
- 商品、库存、推荐信息的数据库表

当前阶段仍然保持：
- 单体应用
- 单实例运行
- 不接 Redis
- 不接消息队列

## 这一阶段要解决什么问题
一旦数据进入数据库，新的现实就出现了：
- 每次请求不再只是本地内存或本地模拟任务
- 请求需要通过连接访问外部数据库
- 数据查询、索引、连接数、事务、网络往返等因素开始影响整体性能

所以这一阶段要让你先建立一个直觉：
- 并发能力提升后，数据库通常会更快成为新的压力中心
- 应用层的“快”不代表整个系统都会一起快

## 当前实现
这一阶段提供两个入口：

1. 数据库版商品详情聚合接口
- `GET /products/{productId}/details/database`

这个接口会从 PostgreSQL 中读取：
- 商品基本信息
- 库存信息
- 推荐信息

2. 数据库批量读取实验接口
- `GET /experiments/database/repeated-read?productId=1&times=5`

这个接口会重复执行商品详情数据库读取，
用来观察：
- 同一个查询被重复执行时，整体耗时如何累计
- 数据库一旦进入链路，请求成本为什么会明显不同于纯内存模拟

## 为什么这一阶段还不急着做数据库优化
因为当前阶段首先要站住一个结论：
- 数据库已经成为系统中的独立压力源

只有这个事实先明确了，下一阶段再讨论：
- 如何分析数据库压力到底来自哪里
- 是索引问题、SQL 问题、连接问题，还是读写模式问题

也就是说：
- `stage-03` 重点是“数据库成为新的瓶颈中心”
- `stage-04` 重点是“数据库压力应该如何分析和优化”

## 外部依赖启动方式
当前阶段需要先启动 PostgreSQL。

在 `labs/web-server-scaling/stage-03/` 目录下执行：

```bash
docker compose up -d
```

默认会启动一个本地 PostgreSQL 17 容器，端口为 `5432`。

## 应用运行方式
在 `labs/web-server-scaling/stage-03/` 目录下执行：

```bash
./mvnw spring-boot:run
```

## 验证方式
服务启动后，可以执行：

```bash
curl http://localhost:8080/products/1/details/database
curl 'http://localhost:8080/experiments/database/repeated-read?productId=1&times=5'
```

## 当前阶段的观察重点
这一阶段重点观察：
- 数据已经正式进入数据库，不再只是应用内模拟
- 同一个业务接口现在需要依赖数据库访问
- 批量重复读取时，总耗时会随着数据库交互次数明显累积
- 为什么数据库会成为后续优化必须重点分析的对象

## 关联知识点
这一阶段后续可以逐步关联到：
- `docs/topics/数据库压力.md`
- 数据库连接为什么也是资源
- ORM 提升开发效率的同时，会把哪些细节先隐藏起来
- 为什么数据库问题常常要到真实压力出现后才更容易暴露

## 如何判断这个阶段达标
达到以下条件即可认为 `stage-03` 完成：
- 应用已经正式接入 PostgreSQL
- 数据库结构通过 Flyway 管理
- 商品详情接口能够从数据库读取真实数据
- 能通过实验接口感受到数据库已经成为请求链路中的独立成本来源
