# stage-01：请求内独立任务的串行瓶颈

## 阶段目标
这个阶段要解决的问题不是数据库、缓存或消息队列，而是一个更早出现的基础瓶颈：
当一个请求内部存在多个彼此独立、但都需要返回给用户的耗时任务时，串行执行会明显拉长总响应时间。

这一阶段的目标是：
- 理解为什么并发会成为 Java 后端开发中的第一批关键能力
- 观察串行执行和并发执行在总响应时间上的差异
- 为下一阶段的线程池和资源控制打下背景

## 主场景
当前阶段使用“商品详情聚合接口”作为主场景。

假设一个商品详情页接口需要同时获取：
- 商品基本信息
- 库存信息
- 推荐信息

这 3 项数据都属于当前请求返回所必需的内容，而且彼此独立。
如果每个操作都需要一定耗时，那么串行执行就会让总响应时间明显增长。

## 为什么选择这个场景
选择这个场景的原因是：
- 业务语义简单，容易理解
- 多个子任务之间天然独立
- 并发收益直观
- 不需要提前引入数据库或缓存
- 非常适合说明“并发为什么出现”而不是“并发实现细节是什么”

## 当前系统形态
基于 `stage-00` 的最小 Spring Boot 应用继续演进。

当前阶段仍然保持：
- 单体应用
- 单实例运行
- 不接 PostgreSQL
- 不接 Redis
- 不接消息队列

新增的只是：
- 一个商品详情聚合接口
- 该接口内部的多个模拟耗时子任务
- 用于对比串行与并发效果的最小实现

## 这一阶段应该包含什么
- 一个聚合接口，例如 `/products/{id}/details`
- 串行实现版本
- 并发实现版本
- 最基础的耗时观测方式
- 对响应时间差异的说明文档

## 这一阶段不应该包含什么
- 不引入线程池作为主角
- 不讨论复杂的线程池参数配置
- 不接入真实数据库
- 不接入 Redis
- 不接入 Kafka
- 不引入复杂业务建模
- 不深入对象头、锁升级、AQS 等底层机制

## 当前阶段的关键问题
当前阶段最核心的问题是：
- 一个请求里如果有多个独立耗时任务，为什么串行执行会成为瓶颈
- 当这些任务可以并发执行时，响应时间为什么会下降
- 并发带来的收益具体体现在哪里

## 为什么当前先不讲线程池
因为这一阶段首先要证明：
- 并发确实能带来收益
- 串行确实会成为瓶颈

只有这个结论先站住了，下一阶段再讨论：
- 为什么不能无限开线程
- 为什么需要资源控制
- 为什么线程池会成为下一步自然选择

也就是说：
- `stage-01` 重点是“并发为什么有价值”
- `stage-02` 重点是“并发为什么需要控制成本”

## 当前实现
当前阶段提供两个接口：
- `GET /products/{productId}/details/serial`
- `GET /products/{productId}/details/concurrent`

两者都会返回：
- 当前商品 ID
- 使用的策略名称
- 总耗时
- 商品信息、库存信息、推荐信息这 3 个子任务的结果

## 可以如何落地
当前阶段通过模拟耗时任务来构建最小实验：
- 商品信息查询：模拟耗时 300ms
- 库存信息查询：模拟耗时 300ms
- 推荐信息查询：模拟耗时 300ms

然后做两个版本：
- 串行版本：依次执行三个任务
- 并发版本：同时执行三个任务

通过比较总耗时，让这个阶段的问题和收益变得可见。

## 当前阶段的观察重点
这一阶段重点观察：
- 串行版本总耗时大约是多少
- 并发版本总耗时大约是多少
- 用户侧感知会有什么不同
- 为什么这个阶段还不应该引入更多基础设施

## 运行方式
在 `labs/web-server-scaling/stage-01/` 目录下执行：

```bash
./mvnw spring-boot:run
```

## 验证方式
服务启动后，可以执行：

```bash
curl http://localhost:8080/products/1/details/serial
curl http://localhost:8080/products/1/details/concurrent
```

通常会观察到：
- 串行版本耗时接近 900ms
- 并发版本耗时接近 300ms 到 400ms

## 如何判断这个阶段达标
达到以下条件即可认为 `stage-01` 完成：
- 能展示一个真实可理解的聚合接口场景
- 能同时提供串行和并发两种实现
- 能观察并说明两者在响应时间上的区别
- 能清楚说明为什么线程池问题应留给下一阶段
