# Web Server 主实验

## 当前定位

`labs/web-server-scaling/` 不再按 `stage-00`、`stage-01` 这种方式创建多个独立项目。

当前已经明确改为：
- `web-server-scaling` 作为一个主题目录，后续包含两个代码项目
- 第一个项目是 **Spring Boot 模块化单体项目**
- 第二个项目是基于这个单体项目拆出来的 **Spring Cloud 微服务项目**

这里要特别区分三件事：
- `docs/capability-map/后端演进线.md` 仍然负责解释“后端能力为什么会一步步出现”
- 单体项目不再按那条演进线逐阶段建多个 stage 项目
- 微服务相关问题也不直接塞进单体项目，而是在同一主题下进入第二个微服务项目

也就是说：
演进线现在是**知识解释顺序**，不是**代码实施顺序**。

## 为什么这样调整

- 多个独立项目的搭建成本高，容易陷入“搭基础设施”而不是“理解后端”的循环
- 先做一个能力完整的单体项目，更适合集中观察模块边界、事务、异步、缓存、稳定性、可观测性这些能力是怎么一起工作的
- 很多后端能力并不要求先拆成微服务，在线程池、缓存、数据库优化、多实例等问题上，单体就足够承载
- 当前为了控制第一个项目的复杂度，消息队列和服务边界问题统一放到第二个项目中演示
- 真正必须依赖服务边界、网络边界、独立部署边界的问题，再基于这个单体项目拆出 Spring Cloud 微服务项目会更清楚

## 当前方案

### 项目 1：Spring Boot 单体项目

这个项目采用：
- `Spring Boot`
- `Maven` 多模块
- 单应用启动
- 单体内清晰模块边界
- 在同一个项目中直接规划完整能力版图，而不是逐步新建 stage 项目
- 尽可能覆盖所有不依赖服务边界的后端能力点
- 支付后的附属异步流程先通过任务表、调度和线程池实现，不提前引入 `Kafka / outbox`

### 项目 2：Spring Cloud 微服务项目

后续会在同一个 `web-server-scaling` 主题下，基于单体项目拆出微服务项目，用来承载这些问题：
- Kafka / 事件驱动
- 服务注册与发现
- 服务间远程调用治理
- 网关
- 分布式事务 / Saga
- 配置中心
- 跨服务链路追踪
- 独立部署带来的协调成本

## 这个主题要覆盖什么

Spring Boot 单体项目重点覆盖：
- 请求链路与应用执行
- 数据库建模、事务和状态流转
- 异步化
- 线程池、后台任务与任务编排
- PostgreSQL 使用与优化
- Redis 缓存
- 稳定性设计
- 可观测性
- 多实例友好设计

后续 Spring Cloud 微服务项目补充覆盖：
- Kafka / 事件驱动
- 服务注册与发现
- 服务间远程调用治理
- 网关
- 分布式事务 / Saga
- 配置中心
- 跨服务链路追踪
- 独立部署带来的协调成本

当前不放进单体项目、而是交给后续微服务项目的内容：
- 必须建立服务边界后才有意义的问题
- 当前主题里刻意后置的消息队列与事件总线设计
- 只为了“看起来像微服务”而提前拆服务

## 原演进阶段与当前项目的关系

下面这张表保留“原演进阶段”的视角，但它现在表达的是**能力映射**，不是“代码实施顺序”。

| 原演进阶段 | 当前在主实验中的落位 |
|-----------|----------------------|
| 阶段 1：同步模型 | 订单、库存、支付的核心同步写链路 |
| 阶段 2：异步化 | 通知、后台任务从主链路拆出 |
| 阶段 3：线程池 | 统一异步任务执行器与后台任务调度 |
| 阶段 4：性能优化 | 连接池、分页、索引、慢 SQL 观察 |
| 阶段 5：数据库优化 | 表结构、事务边界、索引、读写路径治理 |
| 阶段 6：缓存 | Redis 缓存热点商品和读路径数据 |
| 阶段 7：一致性 | 订单、支付、通知任务、缓存之间的状态同步 |
| 阶段 8：稳定性 | 超时、重试、熔断、限流、降级 |
| 阶段 9：MQ 解耦 | 在后续 Spring Cloud 微服务项目中演示 Kafka、事件驱动与服务间解耦 |
| 阶段 10：服务拆分 | 在同一主题下转入后续 Spring Cloud 微服务项目 |
| 阶段 11：可观测性 | 日志、健康检查、指标、链路追踪入口 |

## docs 目录分层

`docs/` 用来承接 `PRD.md` 下面那一层的实施资料，避免把总蓝图和落地过程混写在同一份文档里。

- `docs/slices/`：训练路线与实施切片，当前定义的 `V1 ~ V4` 只覆盖 `mall-monolith/`
- `docs/designs/`：关键方案设计文档，按切片分目录存放
- `docs/execution/`：真正开始编码前的执行清单与批次记录，按切片分目录存放

这里要特别固定一个边界：

- 当前 `V1 ~ V4` 只服务于 `mall-monolith/`
- `mall-microservices/` 不进入这四个切片
- 微服务、Kafka、Saga、服务治理等内容，等单体四个切片稳定后再单独规划新的切片序列

## 关键原则

- **演进线负责解释因果，主实验负责集中验证能力。**
- **先把单体能力用深，再基于它拆出微服务项目。**
- **单体项目里的附属异步流程，先用任务表 + 调度 / 线程池，不提前引入 Kafka / outbox。**
- **当前主实验不再以“Stage 1 / Stage 2 的代码项目顺序”推进。**
- **模块边界要真实存在，但不为了微服务而提前引入分布式复杂度。**
