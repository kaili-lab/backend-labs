# Web Server 主实验

## 实验方案调整说明

原方案是按演进线的每个阶段创建独立项目（stage-00、stage-01、...），逐步演示系统从简单到复杂的演化过程。

经过讨论，决定调整为：**用一个模块化单体项目，覆盖整条演进线的关键能力点。**

### 为什么调整

- 演进线文档（`docs/capability-map/后端演进线.md`）已经把每一步的压力和因果说清楚了，代码实验的职责是"验证方案行为"，不是"重演整个故事"
- 多个独立项目的搭建成本高，容易陷入"搭基础设施"而不是"理解后端"的循环
- 一个项目足以承载阶段 1-11 的所有关键能力点演示

### 新方案：模块化单体 + 文档驱动演进

**核心思路：**

1. 搭一个模块化单体项目，实现核心业务链路（下单、扣库存、支付、通知）
2. 模块之间通过接口调用，不直接访问彼此的数据库表，为后续拆分留好边界
3. 文档驱动演进——每到一个阶段，用文档说清"如果此时压力增长，会遇到什么问题"
4. 代码验证方案行为——用测试或小 demo 展示关键方案的运行特征（如缓存一致性、线程池饱和、消息重复消费等），不需要真的打高并发流量
5. 到后期阶段，再将模块拆分为独立服务，关注"拆分引入了哪些新问题"

**项目结构：**

```
web-server-scaling/
├── pom.xml              # 父 POM
├── mall-common/         # 通用模块
├── mall-inventory/      # 库存模块（Stage 1 暂含商品信息）
├── mall-payment/        # 支付模块
├── mall-notification/   # 通知模块
├── mall-order/          # 订单模块
└── mall-app/            # 启动模块（主类、配置、Docker Compose）
```

**技术栈基线：**
- JDK 21, Spring Boot 3.5.11, Maven 3.9.x
- PostgreSQL 17, Spring Data JPA + Hibernate, Flyway
- Docker / Docker Compose 管理外部依赖

详细设计见 `PRD.md`。

### 各演进阶段在本项目中的体现

| 演进阶段 | 在本项目中验证什么 |
|---------|-----------------|
| 阶段 1：同步模型 | 基础业务链路，全同步执行 |
| 阶段 2：异步化 | 通知等附属流程改为异步 |
| 阶段 3：线程池 | 统一管理异步任务，观察队列和拒绝策略 |
| 阶段 4：性能优化 | 连接池配置、SQL 优化、指标观察 |
| 阶段 5：数据库优化 | 索引、慢查询、读写分离 |
| 阶段 6：缓存 | Redis 缓存热点数据，观察一致性问题 |
| 阶段 7：一致性 | 多副本状态同步策略 |
| 阶段 8：稳定性 | 超时、熔断、降级、限流 |
| 阶段 9：MQ 解耦 | 消息队列解耦同步链路 |
| 阶段 10：服务拆分 | 模块拆为独立服务，体验分布式协调成本 |
| 阶段 11：可观测性 | 日志、指标、链路追踪 |

### 关键原则

- **压力是推演出来的，不是压测出来的。** 文档负责说清因果，代码负责验证行为。
- **先把单体能力用深，再决定是否拆开。** 不一开始就上微服务架构。
- **先启动并迭代，不等待完美设计。**
