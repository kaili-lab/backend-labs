# stage-02：资源控制与池化

## 阶段目标
`stage-01` 证明了一个结论：
当请求内部存在多个彼此独立的耗时任务时，并发执行可以显著降低总响应时间。

但到了这一步，新的问题也自然出现了：
- 如果每次都临时创建执行资源，成本是否可控
- 如果并发任务越来越多，系统是否会失控
- 如果多个请求同时进入，线程数量应该如何管理

所以这一阶段要回答的问题是：
- 为什么“能并发”之后，下一步一定会进入“资源控制”
- 为什么线程池不仅是性能工具，更是资源治理工具
- 为什么系统需要用受控的方式提供并发能力

## 主场景
这一阶段仍然沿用商品详情聚合接口，避免更换业务语义带来额外理解成本。

也就是说：
- 业务场景还是商品详情聚合
- 核心变化不是业务本身
- 而是并发执行方式从“直接异步提交”进入“受控线程池”

## 当前系统形态
基于 `stage-01` 继续演进。

当前阶段仍然保持：
- 单体应用
- 单实例运行
- 不接 PostgreSQL
- 不接 Redis
- 不接消息队列

新增的是：
- 一个专门用于聚合任务的受控线程池
- 一个用于观察线程池复用与排队现象的实验接口

## 这一阶段要解决什么问题
`stage-01` 中，请求内并发已经成立，但还没有明确控制执行资源。

真实系统里，如果：
- 请求变多
- 每个请求都拆分多个并发任务
- 每类任务都任意占用线程

那么新的压力就会从“任务执行慢”转移为：
- 线程创建和调度成本上升
- 线程数量不可控
- 不同任务互相争抢资源
- 系统在高负载下行为不可预测

## 为什么线程池会成为下一步自然选择
因为线程池解决的不是“会不会并发”，而是：
- 如何复用线程
- 如何限制并发规模
- 如何让过多任务进入排队而不是无限扩张资源
- 如何让系统在压力上升时行为仍然可预期

换句话说：
- `stage-01` 重点是“并发为什么有价值”
- `stage-02` 重点是“并发为什么必须被控制”

## 当前实现
这一阶段提供两个入口：

1. 商品详情聚合接口
- `GET /products/{productId}/details/thread-pool`

这个接口继续做 3 个彼此独立的耗时子任务，
但这次不再使用默认异步执行器，而是显式提交到一个受控线程池。

2. 线程池批量任务观察接口
- `GET /experiments/thread-pool/batch?taskCount=6&delayMs=200`

这个接口的目标不是做真实业务，而是帮助观察：
- 线程是否被复用
- 当任务数大于线程数时，任务是否会排队分批执行
- 受控线程池为什么会让系统行为更可预测

## 当前线程池配置
当前阶段故意把线程池配置得比较小，便于观察现象：
- 核心线程数：2
- 最大线程数：2
- 队列容量：10
- 线程名前缀：`product-aggregation-`

这样做的原因是：
- 如果线程池太大，排队和复用现象不明显
- 如果线程池很小，更容易看清“资源受控”的含义

## 当前阶段的观察重点
这一阶段重点观察：
- 商品详情聚合接口已经不再依赖不受控的默认执行器
- 批量任务接口中，线程名是否被复用
- 当 `taskCount > poolSize` 时，总耗时是否按“分批次”增长
- 为什么线程池会让系统行为更稳定、更可预测

## 运行方式
在 `labs/web-server-scaling/stage-02/` 目录下执行：

```bash
./mvnw spring-boot:run
```

## 验证方式
服务启动后，可以执行：

```bash
curl http://localhost:8080/products/1/details/thread-pool
curl 'http://localhost:8080/experiments/thread-pool/batch?taskCount=6&delayMs=200'
```

通常会观察到：
- 商品详情聚合接口仍能保持较低总耗时
- 批量任务接口中，任务数超过线程池大小后，会分批完成
- 返回结果里的线程名会重复出现，说明线程被复用了

## 如何判断这个阶段达标
达到以下条件即可认为 `stage-02` 完成：
- 能展示使用受控线程池的聚合接口
- 能通过接口观察线程复用与任务排队
- 能清楚说明线程池解决的不只是“快”，还有“稳”和“可控”
- 能为下一阶段引入数据库后的资源竞争建立背景
